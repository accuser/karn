# Real-time Bynk — the `Stream[T]` primitive, streaming HTTP responses, and the WebSocket protocol with held connections

- **Status:** Draft (settling). Direction not yet merged; no slice authorised.
- **Realises:** `design/bynk-design-notes.md` §7 (*Services and Protocol Composition* — the **WebSocket** protocol: "the case where the boundary hands a long-lived resource to an agent") and §20 Example 2 (the chat-room with held connections); **sharpens** `design/bynk-type-system.md` §2.9 (*Held resources* — `Held[T]`/`Connection[F]`, the API-discipline linearity model, already *settled in shape*) and §2.9.6 (hibernation). Picks up the WebSocket/`Connection` portion of the **held-resources / delivery track** that [ADR 0122](../decisions/0122-queue-is-a-delivery-concern.md) anticipates ("not yet opened … already owns `Ref[A]` / `Held[T]` / `Connection[F]`").
- **Introduces (genuinely new):** `Stream[T]` — a value-over-time type. It is **not** in the design notes; §10's `Queue[T]` "durable async stream" is a delivery kind ([ADR 0122](../decisions/0122-queue-is-a-delivery-concern.md) removed it), not a temporal-sequence value type. The track's hard-to-reverse calls cluster here, which is why `Stream[T]` is the front-loaded foundation.
- **Posture:** Feature track per [ADR 0076](../decisions/0076-feature-track-posture.md). Qualifies on all three axes: **multi-increment** (a new core type + vocabulary + lowering, a streaming-response terminal, the held-resource linearity pass, and the protocol — five-plus slices); **surface not yet settled** (`Stream[T]`'s form is open; the held model is "settled in shape" but its fault-path and diagnostic surface, §2.9.7/§2.9.9, are not); and a **safety boundary** (a WebSocket upgrade authenticates at the service edge before a client holds a live channel into an agent — a wrong shape is a vulnerability, exactly the actor-track trigger). ADR 0076 names WebSocket as an obvious future track; this is it.
- **Proposed ADRs (numbered per-slice at authoring time):** the `Stream[T]` model (value-vs-effect, element semantics, relationship to `Query`/`Effect`); the streaming-HTTP response contract; held-resource linearity realised (API-discipline, the borrow/consume states); the WebSocket protocol surface (`from WebSocket(in:, out:)`); connection-transfer & edge-auth; the hibernation mapping. **No range is reserved.** Forward-reserving ADR numbers across settling tracks has already proved unreliable — 0126 (`httpresult-rfc9110-status-vocabulary`) and 0127 (`capability-requirement-provenance`) landed from unrelated work and overtook the `in-browser` track's provisional 0126–0130 claim. So each ADR here takes the next free number *when its slice is authored*, after the in-browser reservations settle — consistent with the steer that each ADR is written with its slice, not up front. This doc settles direction and frames the calls as `[DECISION]` blocks below.

## 1. Motivation

Bynk is "for … real-time interaction (e.g. WebSockets)" by its own charter (design notes §19), and the architecture already names the shape: §7's WebSocket protocol hands a long-lived connection from a stateless service to a stateful agent, after which inbound frames "arrive at the agent as ordinary typed messages, indistinguishable in shape from messages invoked by other agents or by tests." The type system has gone further and settled the carrier *in shape* — `Held[T]`/`Connection[F]`, an "API-discipline-driven linearity" model with owned/borrowed/stored/consumed states, storage rules, and a hibernation guarantee (`bynk-type-system.md` §2.9). What is missing is **realisation**: none of it is built, and one foundational piece it leans on does not exist as a language type at all.

That missing piece is a representation of **values produced over time**. `Effect[T]` resolves exactly once; it cannot express a token stream from a model, a server-sent progress feed, or any incremental response. Both the immediate, edge-shippable case (stream an HTTP response body) and the eventual capability case (a streaming `Ai.generateText`) need a first-class `Stream[T]`. The WebSocket protocol does *not* — §7 deliberately models inbound frames as discrete typed messages and outbound as imperative `connection.send(frame)` — so `Stream[T]` and `Connection[F]` are two distinct mechanisms for "real-time," and the track keeps them distinct on purpose (§3.4).

The work decomposes into two legs that share a settling phase but are independently valuable:

1. **The streaming leg** — the new `Stream[T]` type, and a streaming-HTTP response terminal that consumes it. Ships value with no sockets and no Durable Object; it is the cheap early payoff and the seam a future streaming `Ai` capability plugs into.
2. **The connection leg** — realise `Held[T]`/`Connection[F]` linearity, then the `from WebSocket` protocol that authenticates at the edge and transfers a held connection to an agent, mapping onto Cloudflare's hibernatable WebSockets.

## 2. Scope and non-goals

**In scope.** A `Stream[T]` value type with a minimal combinator vocabulary, classified non-serialisable/non-storable like its `Effect`/`Held` neighbours; a streaming-HTTP response terminal (SSE-shaped body) on the existing `from http` protocol; the realised `Held[T]`/`Connection[F]` linearity pass and its storage integration (`Cell[Option[Held[T]]]`, `Map[K, Held[T]]`); the `service … from WebSocket(in:, out:)` protocol — edge authentication via the `by` clause, the `on open` handler receiving a fresh `Connection`, transfer to an agent, inbound frames as typed agent messages, `connection.send`, close, and the Cloudflare hibernation mapping; a `TestConnection` capture-and-inspect for the bundle/test target (design notes §20: "A `TestConnection` records what was sent"); spec-in-place and a book chapter; the §20 chat-room compiling end-to-end as the closure proof.

**Non-goals (and why).**

- **The outbound enqueue / `Queue` capability and the broader delivery contract.** [ADR 0122](../decisions/0122-queue-is-a-delivery-concern.md) routes the outbound `Queue` capability and the at-least-once/`Idempotency` machinery to the same anticipated "held-resources / delivery track," but they are a separable concern with no socket or stream dependency. This track takes the WebSocket/`Connection`/`Stream` slice of that umbrella; `Queue`-out is a sibling proposal.
- **The Events and Alarm protocols.** ADR 0076 names Events and Alarm as their own future tracks sharing the actor surface. They are out here even though §7 lists them beside WebSocket; pub-sub fan-out and timer wakeups are distinct surfaces.
- **A streaming `Ai` capability.** The Workers AI `Ai` capability ships independently and returns whole completions first; `generateText`-streaming is a *named downstream consumer* of `Stream[T]`, built after this track lands the primitive — not part of it. (Keeps the track decoupled from the `bynk.cloudflare` `Ai` work and from any one platform's inference binding.)
- **Generalising `Held[T]` beyond `Connection[F]`.** §2.9.1 leaves the door open for future held instances (file handles, DB connections); the design-notes open item (§ "Storage type primitive set") flags this as answered only "as the platform capability surface grows." This track realises the kind with its **one** concrete instance and the closed-kind machinery — it does not add new held types.
- **A browser-side WebSocket client.** The `in-browser` track explicitly excludes the Workers topology (Durable Objects have no browser analogue); a `Connection` is server-held. Client code is out of Bynk's charter (§19).
- **General linear types.** §2.9 is deliberate: linearity is tracked through a *fixed vocabulary of runtime operations*, not a general affine-types system. The track holds that line.

## 3. Internal architecture

### 3.1 The two legs and their shared settling

The streaming leg (3.2–3.3) and the connection leg (3.5–3.7) share only the settling phase: `Stream[T]`'s defining calls (3.4) are made once, because the streaming-HTTP terminal consumes a `Stream` and any later streaming capability does too. Slices 0–1 (streaming) stand entirely alone — they need no Durable Object — so they can land and deliver before the connection leg is touched, mirroring the `in-browser` track's "first three slices stand on their own."

### 3.2 `Stream[T]` — the new primitive

`Stream[T]` is a lazy, pull-shaped sequence of values produced over time. It lowers to a host async iterable / `ReadableStream`; on the test target it is a deterministic in-memory sequence so a handler's streamed output is assertable. The vocabulary is deliberately minimal at v1 — construction from a source the runtime supplies, `map`, and a bounded `take` — with the lesson from the retired `query-algebra` track that a combinator surface wants its own ADR before it grows. One distinction to draw early, since users will conflate them: `Stream[T]` is **not** the effectful-iteration surface (`traverse`/`parTraverse`) that `query-algebra` deferred. That deferred surface walks an *already-materialised* collection applying an effect per element; `Stream[T]` is an *unbounded, externally-driven* source whose elements arrive over time and may never complete. They share a "do something per element" intuition and differ on everything that matters — source, boundedness, who drives — so the two should not be unified into one type.

Classification reuses existing machinery rather than inventing any. A live IO source is not a value you can persist or ship, so `Stream[T]` joins **function types and held resources** in the **non-serialisable** set the serialisability pass already enforces (`bynk-type-system.md`, the serialisability-check pass — it lists closures/function types and `Connection[F]`/`Held[T]`). That pass is the right home: it already rejects a non-serialisable value in a `store` field, across a context boundary, or captured by a serialised closure, and `Connection[F]` obeys exactly these rules — so adding `Stream[T]` extends an existing pass rather than introducing a concept. Two adjacent precedents reinforce the shape without being members of that pass: `Effect` is **non-storable** ([ADR 0031](../decisions/0031-effect-non-storable.md)) and function types are **non-boundary** ([ADR 0030](../decisions/0030-function-types-non-boundary.md)) — distinct properties from non-serialisability, cited as parallels for "a live thing is not a value," not as the same check.

### 3.3 The streaming-HTTP terminal (the early payoff)

A `from http` handler gains the ability to return a streamed body. The reconciling call (Q2) is *how status attaches to a stream*: [ADR 0126](../decisions/0126-httpresult-rfc9110-status-vocabulary.md) makes `HttpResult` a sum of status-bearing variants over a small set of payload **shapes** (`Value`/`Message`/`None`/`Location`), and a streamed `200` and a streamed error are both real — so a streamed body is most naturally a **fifth payload shape** (`Streamed`, carrying a `Stream[Chunk]`) under an existing status, exactly as 0126 D2 added `Location` as a header-shaped outcome, **not** a status-less `Streamed(...)` peer variant. The lowering is a `Response` whose body is the stream, content-type `text/event-stream` for the SSE framing. This delivers incremental responses — progress feeds, token relays once a streaming capability exists — on the existing protocol, existing actors, existing topology, with no connection lifecycle and no Durable Object. It is the slice that proves `Stream[T]` carries weight before the heavier leg begins.

### 3.4 Why `Stream[T]` and `Connection[F]` stay distinct

A tempting unification — express a `Connection`'s inbound frames as `Stream[F]` — is **rejected**, and the design notes already made the call: §7 models inbound frames as discrete typed messages precisely so a frame is "indistinguishable in shape from messages invoked by other agents or by tests," preserving handler uniformity and direct unit-testing of a single frame. `Stream[T]` is the *outbound/response* and *capability-output* primitive (server → client, pull-shaped, no ownership); `Connection[F]` is a *bidirectional owned resource* with imperative `send` and inbound-as-messages. Collapsing them would either impose ownership/linearity on a plain response stream or dissolve the message-uniformity §7 relies on. They share the "values over time" intuition and nothing else.

### 3.5 `Held[T]`/`Connection[F]` linearity, realised

`bynk-type-system.md` §2.9 is "settled in shape"; this leg builds it. The four ownership states (owned/borrowed/stored/consumed, §2.9.2), the closed origin (no public constructor — held values come only from runtime sources, §2.9.1), mandatory disposal at scope exit, and the storage rules (`Cell[Option[Held]]` and `Map[K, Held]` admit held values; `Set`/`Log`/`Cache` reject them, §2.9.3) become a **linearity-check pass** — step 11 of the §2.9.8 type-check pipeline. `connection.send(frame)` is a borrow-and-return non-consuming op (the binding stays owned); `close` and storage `put`/`take` are consuming. This leg ships the type machinery and storage integration with **no protocol yet**, so it is testable in isolation against hand-written held sources before the socket transport exists.

### 3.6 The `from WebSocket` protocol

The service header generalises cleanly — [ADR 0077](../decisions/0077-service-protocol-on-header.md) moved the protocol to `service <Name> from <protocol>` specifically so "the header shape generalises to the unbuilt Events/Alarm/WebSocket protocols." A WebSocket service binds its frame types on `from`:

```bynk
service ChatGateway from WebSocket(in: ClientFrame, out: ServerFrame) {
  on open by user: Participant (params: { roomId: RoomId })
      given Rooms: Ref[Room] {
    <- Rooms(params.roomId).join(user.identity, connection)
  }
}
```

The service performs the HTTP upgrade and **authenticates at the edge** via the `by` clause (an actor, exactly as HTTP routes do) *before* the connection is accepted; the `on open` handler receives a fresh `Connection[out]` as an owned binding and must dispose of it (here, transfer into the agent's storage). Thereafter inbound frames arrive at the agent as ordinary typed messages — `on join`, `on post`, `on leave` in the §20 example — and the agent sends with `connection.send(frame)`.

### 3.7 Platform mapping and hibernation

On **Workers**, the upgrade happens in the Worker (the stateless service), the server socket is accepted into the addressed Durable Object via the hibernatable-WebSocket API, and a `Connection[F]` stored in agent state "survives the agent's hibernation and is automatically available when the agent is rehydrated" — a platform-supplied guarantee the language relies on but does not implement (§2.9.6). Hibernation is transparent to the type system: a stored connection is just a stored value. On **bundle/test**, there is no DO; the design notes' `TestConnection` is the realisation — a capture-and-inspect channel so "the runtime-managed lifecycle in production becomes a capture-and-inspect mechanism in tests" (§20). The platform-lock machinery already in place ([ADR 0017](../decisions/0017-platform-lock-per-deployment-unit.md)) diagnoses a `from WebSocket` service on a target that cannot host it, the same way `bynk.cloudflare` units are policed.

## 4. Security and threat model

The WebSocket leg is the track's safety boundary; the streaming leg is a lesser, resource-shaped one.

- **Unauthenticated channel into an agent.** The upgrade authenticates at the *edge* via `by`, before acceptance, fail-closed — the same actor verification HTTP routes use. The threat is a protocol shape that lets a frame reach an agent before `on open`'s `by` clause has verified the actor, or that accepts the socket and authenticates late. The protocol must make edge-auth-before-accept structurally unavoidable; this is the slice that runs `/security-review`. The *carrier* the upgrade reads the credential from is itself an open call (Q8) — a browser cannot set an `Authorization` header on the handshake — so "exactly as HTTP routes do" holds for *when* auth happens but not necessarily *how* the credential arrives.
- **Resource exhaustion — connections.** A held connection is a real resource; a leaked one is a leak. §2.9's mandatory-disposal discipline is the defence in normal flow (the checker rejects a connection that escapes a handler unowned-and-unstored). The fault path (§2.9.7, still open — D5) must guarantee the runtime closes a handler-owned connection on abnormal exit, and that stored connections roll back transactionally with the rest of agent state.
- **Resource exhaustion — streams.** An unbounded `Stream[T]` (or an unbounded set of open connections) is a DoS surface. Backpressure and bounded `take` are the language-level tools; per-agent connection caps are a platform concern named here so a slice does not silently omit them.
- **Egress via streamed responses.** A streaming response holds an open channel; abusive long-lived responses are a resource vector. Bounded by the same backpressure story and by the platform's request limits.
- **Frame-type confusion.** The channel is typed (`in:`/`out:`); sending a wrong-shaped frame is a compile error at the `.send` site (§2.9.1), and inbound frames decode to `in:` or are rejected. The threat is a decode path that trusts client bytes; decoding is validated against the declared `in:` type, reusing the boundary-decode discipline.

## 5. Open questions (settle before slicing)

- **Q1 — `Stream[T]` form, and how much of it to commit in slice 0. [DECISION]** A value type vs an emit/yield *effect* vs a `Query`-sibling; and — the sharper call — whether slice 0 commits the *parametric element semantics* or only what the in-scope consumer exercises. *Recommendation:* a distinct value type in the non-serialisable family, **not** a variant of `Effect` (it resolves many times, breaking `Effect`'s once-semantics) and **not** `Query[T]` (a snapshot-consistent read over a materialised/storage source; a stream is an unbounded temporal source with no snapshot). Keep the **type parameter** from the start (retrofitting one is the disruptive move), but **defer the element semantics** that only a non-chunk consumer pins: whether a stream element may be an arbitrary value that *crosses a context boundary*, and whether such an element must be serialisable. Slices 0–1 exercise only wire-encodable chunks (`Stream[Chunk]`), so they do not force that call; the first consumer that needs `T ≠ Chunk` (a streaming `Ai` capability, or in-memory effectful iteration) settles it. This shrinks the genuinely irreversible surface to what is exercised while avoiding a later type-parameter retrofit. The slice-0 ADR records the type + parameter; the element-semantics ADR lands with that first consumer.
- **Q2 — Streaming-HTTP return shape, reconciled with ADR 0126. [DECISION]** 0126 widened `HttpResult` to the RFC 9110 status vocabulary, modelling each variant as a *status* over a payload *shape* (`Value`/`Message`/`None`/`Location`). A streamed body still carries a status (a streamed `200`, or a streamed error). *Recommendation:* add `Streamed` as a **fifth payload shape carrying `Stream[Chunk]`**, attached to status-bearing variants — following 0126 D2's `Location` precedent — rather than a status-less `Streamed(...)` peer variant, so a handler can stream under any admissible status. But scope the cost honestly: the **registry-driven surface** (resolver, completion, emitter dispatch) picks the new shape up as `Location` did, whereas the **exhaustive arms 0126 D3 names** — construction-check, pattern-bind, runtime status-map — are extended "once each," and for `Streamed` two of those are *not* trivial the way `Location` (a `String` "validated exactly as `Message`") was. The pattern-bind arm binds the **first non-serialisable payload** (`Stream[Chunk]` cannot cross a boundary, where every existing shape binds a serialisable value), and the runtime arm is **genuinely new behaviour** — a streaming `Response`, not a buffered one with a header tweak. So: the registry plumbing is free; the streaming lowering and the non-serialisable-payload interaction are the real work this slice does.
- **Q3 — `Stream` error/termination. [DECISION]** In-band (`Stream[Result[T, E]]`) vs a separate completion/error channel. *Recommendation:* in-band `Result` elements, matching the existing outcome-vs-fault split (a stream element that is an `Err` is an *outcome*, not a fault); a fault in the producer aborts the stream as faults abort handlers.
- **Q4 — Combinator vocabulary breadth. [DECISION]** Minimal (`map`, `take`) at v1 vs a fuller algebra (`filter`, `scan`, fan-in/merge). *Recommendation:* minimal at v1; a fuller vocabulary earns its own slice + ADR, as the query algebra did.
- **Q5 — Held-resource fault paths (§2.9.7). [DECISION, currently open in the notes]** The within-handler disposal rules are settled; what happens to a held value mid-transfer across a context boundary when a fault interrupts is open, as is the `Sagas.compensate` interaction (§2.9.7). *Recommendation:* settle conservatively — a connection mid-transfer is owned by exactly one side at the fault instant, and the owner's runtime closes it; a compensation may operate on a connection already in agent state but not on one the handler still held locally (the compiler rejects the latter statically, per §2.9.7). One caveat to state honestly: on Workers the service→DO "transfer" is **not** a same-address-space value move but a platform handoff (`acceptWebSocket` into the DO), so the single-owner invariant rests on the *handoff's* atomicity — the language's move semantics describe the source side, the platform binding must guarantee the destination side. The recommendation holds, but its backing is platform-handoff semantics, not language semantics alone. Lands with the connection-leg slice that emits disposal.
- **Q6 — Linearity diagnostics (§2.9.9). [DECISION]** Whether users ever see `&Held[T]` borrow notation, and the wording of "this branch leaves `conn` owned but the other consumes it." *Recommendation:* keep the borrow notation implicit; invest in the branch-divergence message, the most common real error. Cosmetic enough to fast-follow within the connection leg.
- **Q7 — Hibernation re-acquisition surface. [DECISION]** §2.9.6 says a stored `Connection` is transparently restored, but the Cloudflare hibernation API re-presents sockets via `getWebSockets()` on wake, not as a live JS reference. Confirm the stored-value model fully hides this, or whether the binding needs a re-association step keyed by a stored id. *Recommendation:* keep it transparent in the language (stored value in, stored value out) and absorb any re-association entirely in the platform binding (§18); flag for the platform-mapping slice as the one place "transparent" could leak.
- **Q8 — The upgrade's auth *carrier*. [DECISION, security-adjacent]** §3.6 authenticates the upgrade "exactly as HTTP routes do" via the `by` clause — but a browser's `WebSocket` constructor **cannot set request headers**, so a `Bearer` actor expecting `Authorization: Bearer …` has no carrier on the handshake. The token must arrive another way — `Sec-WebSocket-Protocol`, a cookie, or a query-parameter token — each with caveats (query strings land in logs; cookies bring CSRF surface; the subprotocol header is a hack). This directly shapes how `by` integrates with the upgrade and is the most common real-world WS-auth snag, so it is settled here, not discovered in slice 3. *Recommendation:* do **not** silently accept `Bearer`-over-WS; either define which carrier a WS upgrade reads (and have the actor scheme declare it) or admit only WS-appropriate schemes at the `from WebSocket` boundary, diagnosing a header-only scheme. Adjacent to the §4 unauthenticated-channel crux; resolved with the connection-leg security review.

## 6. Slice decomposition (ordered)

Each slice is an ordinary `vX.Y-<slug>.md` proposal citing this doc and authoring its own ADR(s); merging the proposal authorises the build. Slices 0–1 (streaming) stand on their own; 2–4 (connection) build in order.

- **Slice 0 — `Stream[T]` foundation.** The type, minimal vocabulary (`map`, `take`), the non-serialisable/non-storable classification extending the existing pass, and the lowering to host async-iterable / deterministic test sequence. Settles Q1, Q3, Q4. The load-bearing primitive; everything downstream cites its ADR.
- **Slice 1 — streaming-HTTP response.** A streamed body on `from http`, consuming a `Stream[T]`, SSE-framed (Q2). The early payoff: incremental responses with no socket, no DO. Stands alone.
- **Slice 2 — held-resource linearity.** Realise `bynk-type-system.md` §2.9 — the ownership states, the closed origin, mandatory disposal, the storage rules (`Cell[Option[Held]]`/`Map[K, Held]`), the §2.9.8 step-11 check — against hand-written held sources, **no protocol yet**.
- **Slice 3 — the `from WebSocket` protocol.** `service … from WebSocket(in:, out:)`, edge auth via `by`, `on open` receiving a fresh `Connection`, transfer to agent, inbound frames as typed messages, `connection.send`, close. The Workers hibernation mapping (Q7) and the bundle `TestConnection`. **Security-bearing** — carries its threat model (§4) and runs `/security-review` + `/code-review`. Settles Q5.
- **Slice 4 — closure.** The §20 chat-room compiles and runs end-to-end on both targets; linearity diagnostics (Q6); fault-path disposal proven; spec-in-place + book chapter; changelog; currency banner. The theme's completion proof.

## 7. Risks

- **`Stream[T]` is a permanent type-system commitment made largely for a deferred consumer.** The hard-to-reverse surface is the parametric type, yet the only in-scope consumer (slice 1) needs `Stream[Chunk]`; the generic `Stream[T]` is justified mainly by the streaming `Ai` capability, an explicit non-goal. Committing the full parametric element semantics now would front-load irreversibility for a consumer outside the track. *Mitigation (Q1):* split the commitment by reversibility — admit the **type parameter** in slice 0 (retrofitting one later is itself disruptive), but **defer the element semantics** (cross-boundary element transport, element serialisability) until the first non-chunk consumer pins them. Slices 0–1 only ever exercise wire-encodable chunks, so the irreversible part lands when, and shaped by, the consumer that actually needs it — and the value-vs-effect and non-serialisable calls that *are* made in slice 0 sit behind a conformance test, with the vocabulary kept minimal so even that surface is small. This is a deliberate timing choice, recorded here rather than left implicit.
- **The held model is "settled in shape," not in detail.** §2.9.7 (fault paths) and §2.9.9 (diagnostics, record-of-held fields, user-defined borrows) are genuinely open. *Mitigation:* Q5/Q6 settle the build-blocking subset; the rest (record-of-held, user-defined borrowing) is explicitly deferred and named in §2.9.9, not rediscovered mid-slice.
- **Security shape of the upgrade.** Edge-auth-before-accept is the one place a wrong protocol shape is a vulnerability. *Mitigation:* slice 3 is gated by `/security-review`; the threat model is stated in the track doc, per ADR 0076's security-bearing-slice rule.
- **Hibernation transparency could leak (Q7).** If the stored-value model cannot fully hide `getWebSockets()` re-association, the language's "stored connection is just a stored value" promise weakens. *Mitigation:* confine all re-association to the platform binding; the platform-mapping slice proves transparency with an end-to-end hibernate/wake test.
- **Two-leg scope creep.** The streaming and connection legs share only settling; pressure to fold a streaming `Ai` consumer or `Queue`-out into the track would balloon it. *Mitigation:* both are explicit non-goals (§2); they cite this track, they do not join it.

## 8. Relationship to the north star

This track realises a part of the architecture the design notes have committed since the start: §7's WebSocket protocol, §2.9's held-resource model, and the §20 chat-room worked example were written to be built. It sharpens them (the linearity pass, the fault-path and diagnostic calls) without contradiction, and it honours the architectural line §1 draws — connections and streams are runtime/delivery concerns that "appear in the language as typed messages or capabilities," never as agent-owned durable storage ([ADR 0122](../decisions/0122-queue-is-a-delivery-concern.md)). The one genuinely new idea, `Stream[T]`, fills a gap the notes left implicit: the language could express a single deferred value (`Effect`) and a snapshot read (`Query`) but had no word for a value produced incrementally over time. With it, "real-time interaction (e.g. WebSockets)" — in scope by charter since §19 — finally has its primitives.
