# v0.52 — Actors, slice 4 (multi-actor sum dispatch): an ordered sum of peer actors on the `by` clause, first-wins

- **Status:** Draft (proposal). **Merging is approval to build.** *(Feature-track slice — the fourth of
  the actors track, `design/tracks/actors.md`. The genuinely novel construct of the track: a handler
  whose `by` clause names an **ordered sum of peer actors** (`by who: User | Visitor`), resolved
  first-wins, with the body matching on the resolved party.)*
- **Sequences after v0.51 (Signature).** It composes the three landed schemes (`None`/Bearer/Signature)
  rather than adding a new one, and leans on the optional binder (v0.50) for the catch-all arm.
  Provisional version (`v0.52` assumes nothing jumps the queue — renumber if sequencing shifts).
- **Verification-bearing slice** → carries the `/security-review` + `/code-review` gate (ADR 0076) and
  the threat model below. It widens the verification seam from one actor to several tried in order, so
  the fail-closed guarantee is the thing to defend.

## Context — one route, several kinds of caller

The three landed slices each verify **one** party at a boundary: a public route (`None`), an
authenticated user (`Bearer`), or a webhook sender (`Signature`). Real routes often serve **more than
one kind of caller**: a notes endpoint that returns a richer view to a signed-in `User` and a public
view to an anonymous `Visitor`; an ingest endpoint that accepts either an internal `User` token or a
partner `Webhook` signature. Today that forces the app to either split the route in two or hand-roll
the "try auth A, else fall back to B" branching at the boundary — exactly the verification logic the
track exists to generate.

This slice lets a handler declare an **ordered sum of peer actors** on its `by` clause. The compiler
emits first-wins resolution — try each variant's scheme in declared order, bind the first that
verifies — and the body `match`es on the resolved actor, each arm carrying that actor's own identity.
No surveyed framework offers a closed, discriminated actor sum (Servant collapses multiple schemes to
one shared principal and loses *which* party verified); this is the track's novel contribution.

## Builds on (verified)

- **The three scheme seams** — `None` (no-op accept), `Bearer` (JWT/HS256, compose-wrapper seam, ADR
  0085), `Signature` (HMAC over the body, entry-dispatch seam, ADR 0089). A sum *composes* these; it
  adds no new scheme. `Scheme` is already a closed nominal set (ADR 0080).
- **The sealed-identity model (ADR 0081)** — each peer yields its own sealed identity type; the sum
  binder's type is the discriminated union of them, matched nominally by actor name.
- **Verify-then-body, fail-closed (ADR 0082) + the 401/403 split (Q6)** — a sum's total failure is
  unambiguously **401** (its members are peer base actors with no invariants; there is no 403 path).
- **The optional binder (ADR 0088)** — the catch-all `None` arm (`Visitor`) binds no principal, so the
  binder-less idiom already exists; a sum reuses the same binder machinery for its bound arms.
- **Sum types + `match` (the existing language core)** — the resolved-actor binder is an ordinary
  nominal sum the body matches; this slice forms that sum type at the boundary rather than inventing
  new match machinery.

## Settled (from the track / foundational ADRs)

- **▸ A multi-actor handler is an *ordered sum of peer base actors*** (Q4) — parties distinguished by
  **scheme**, each yielding its own identity type. **▸ Resolution is first-wins; the body matches on
  the resolved variant.** **▸ Overlap is allowed and ordered; a `None` catch-all (`Visitor`) must come
  last.** **▸ Unreachable arms are rejected by a decidable, scheme-level check** (a `None` followed by
  anything; a scheme an earlier peer already subsumes) — the compiler does **not** reason about
  predicate-level disjointness. **▸ Refinements are never sum variants** (`User | Admin` is rejected —
  every `Admin` is a `User`, Q3; narrowing is a guard *inside* the resolved arm). **▸ Sum failure →
  401, fail-closed** (Q6). **▸ Verification is side-effect-free and idempotent** — first-wins
  short-circuits, so the set and order of verifications attempted is observable; audit/logging belongs
  *after* resolution.

## Decisions (open — settle at review)

**[DECISION] Peer keying: by abstract scheme now; concrete-verification keying later.** The track flags
this as the one Q4 question left open *for this slice* (not closed by the prior-art pass). *Rec:* key a
peer by its **scheme** — two peers may not share a scheme (`by who: User | Visitor` ✓;
`by who: WebhookA | WebhookB`, both `Signature`, ✗ → `karn.actor.duplicate_sum_scheme`). This keeps the
reachability check decidable and resolution unambiguous. *Alternative:* key peers by concrete
*verification* (secret/audience), admitting same-scheme multi-provider routes (two `Signature` senders
with different secrets) — more expressive, with "peer = independent verifier, refinement = base
verifier + predicate" as the dividing line, but a softer reachability guarantee. Defer it; the
declaration shape (`A | B | …`) does not foreclose it, and a later slice can widen "distinct scheme" to
"distinct verifier."

**[DECISION] A sum requires a binder.** *Rec:* `by who: A | B (…)` — the binder is **mandatory** for a
sum, because the body must `match` to learn which party verified (and to reach its identity). A
binder-less sum (`by A | B`) is rejected (`karn.actor.sum_requires_binder`): it would verify-and-
discard *which* of several parties authenticated, which is never what a multi-actor route wants.
(Single-actor handlers keep the optional binder of ADR 0088.) *Alternative:* allow a binder-less sum
that just asserts "someone authenticated" — rejected: it throws away the discriminant the construct
exists to provide.

**[DECISION] The binder's type is a compiler-formed actor sum, matched nominally.** *Rec:* `who : User |
Visitor` is a synthetic, closed nominal sum whose variants are the peer actor *names*; the body matches
`match who { User(u) => …, Visitor => … }`, and each bound arm (`User(u)`) yields that actor's identity
type (`u.identity : UserId`), while a `None`-scheme arm (`Visitor`) binds no payload. Exhaustiveness and
arm-name resolution reuse the existing sum-type checker. *Alternative:* a positional tuple or an
open/structural union — rejected: loses the nominal, exhaustive match the rest of the language already
gives sums.

**[DECISION] Admissibility: HTTP-only, and every member must be admissible on the protocol.** *Rec:* a
sum is admissible on a protocol iff **every** member scheme is (today: HTTP admits `None`/`Bearer`/
`Signature`; the internal protocols admit only `Internal`). In practice this makes sums **HTTP-only**
for v0.52 — the only protocol with more than one admissible non-internal scheme. A member that is not
admissible on the handler's protocol is the existing `karn.actor.scheme_not_admissible`. *Alternative:*
allow internal-channel sums now — deferred: there is no second admissible internal scheme to sum.

**[DECISION] Seam composition under first-wins: header-discriminated, body read once.** This is the
emission crux. Bearer's seam reads the `Authorization` **header** (compose-wrapper, ADR 0085);
Signature's reads the **raw body** (entry-dispatch, ADR 0089). *Rec:* the entry dispatch resolves the
sum **before** the body handler: it reads the raw body once (if any member needs it), then tries each
member in order — a member verifies against the material already in hand (header for Bearer, the held
raw bytes for Signature) — and binds the first success; the resolved variant flows into the compose
surface, which dispatches the matched handler. No member re-reads the request. A `None` member always
succeeds (so it terminates the order as the catch-all). *Alternative:* keep each seam at its current
site and chain wrappers — rejected: two wrappers each wanting to own the body read reintroduces the
double-read footgun ADR 0089 closed. Settling the unified entry-dispatch resolution is the bulk of the
implementation risk and the focus of the security review.

**[DECISION] Failure response: 401, fail-closed (confirming Q6).** *Rec:* if no member verifies, the
boundary returns **401** and the body never runs — the body executes only with a verified, sealed
identity of one of the declared actor types. There is no 403 path (no refinement members). *Alternative:*
none — this is settled by Q6; restated here because it is the security guarantee the slice widens.

## Spec delta

- **`syntactic-grammar.md`.** The `by` clause's actor position generalises from a single actor to a
  **non-empty `|`-separated list** (`by_actor_sum := actor_ref ("|" actor_ref)+`), with the binder
  required when the list has more than one member. The single-actor form is the one-element case
  (unchanged).
- **`static-semantics.md`.** Multi-actor rules: a sum requires a binder; members must be peer base
  actors (no refinement member, `karn.actor.refinement_in_sum`); no two members share a scheme
  (`karn.actor.duplicate_sum_scheme`); a `None`-scheme member must be last and at most one
  (`karn.actor.unreachable_sum_arm`); every member admissible on the protocol (existing
  `scheme_not_admissible`); the binder's type is the closed nominal sum of the members, and the body
  `match` must be exhaustive over it (existing sum-exhaustiveness). The sum-failure response is 401.
- **`emission.md`.** The unified entry-dispatch resolution: read the raw body once when any member
  needs it; try members in declared order against material in hand; bind the first success into the
  resolved-actor variant; 401 if none; dispatch the matched handler. How each scheme's existing seam
  (Bearer header-verify, Signature HMAC, `None` accept) plugs into the ordered try.
- **`runtime-library.md`.** No new verifier — the slice composes `verifyBearerJwtHs256` and
  `verifySignatureHmacSha256`. A small resolution helper may be emitted inline (no new runtime export)
  or added as a thin combinator; settle during build.
- **`diagnostics.md`.** New `karn.actor.*` codes: `sum_requires_binder`, `duplicate_sum_scheme`,
  `refinement_in_sum`, `unreachable_sum_arm`. (`scheme_not_admissible` is reused for a member.)

## Tests

- **Positive golden:** a `from http` route `on GET("/notes/:id") by who: User | Visitor (id: NoteId)`
  whose body `match`es `User(u) => … | Visitor => …`; the entry dispatch emits the ordered resolution,
  binds the right identity, and `tsc --strict` is green. A second golden mixing a body-bearing member
  (`by who: User | Webhook (body: Event)`) exercises read-raw-once under first-wins.
- **Behavioral (the standing guard, extending `bearer_auth.rs`/`signature_auth.rs`):** a Node driver
  asserts first-wins resolution and the fail-closed total — a valid `User` token resolves the `User`
  arm; absent token + valid signature resolves the `Webhook` arm; neither → 401; the `Visitor` arm
  catches all when present; order is observed (an earlier member's success short-circuits a later one).
- **Negative:** `User | Admin` (refinement member); `WebhookA | WebhookB` (duplicate scheme);
  `Visitor | User` (`None` not last / unreachable arm); a binder-less sum; a sum member not admissible
  on the protocol; a non-exhaustive body match over the sum.

## Diagnostics / Risks

- **Seam-composition correctness** (the [DECISION] above) — the body-read-once invariant must survive
  composing a header member and a body member. *Mitigation:* the unified entry-dispatch read; golden +
  `tsc` + the behavioral mixed-member test.
- **Reachability check soundness** — the unreachable-arm rule must stay *decidable and scheme-level*;
  creeping toward predicate reasoning is out of scope. *Mitigation:* the check is purely over the
  member schemes and their order; refinements are excluded by construction.
- **Match/identity wiring** — each arm must see exactly its actor's identity type, and only bound
  (non-`None`) arms carry a payload. *Mitigation:* reuse the sum-type checker; negative test for a
  non-exhaustive match.

## Security / threat model

*Gates the slice (ADR 0076).*

- **Guarantees.** The body runs only with a verified, sealed identity of **one** of the declared peer
  actors, resolved first-wins; total failure is fail-closed → 401. Widening from one actor to an
  ordered sum does not weaken the guarantee: each member is verified by its existing (reviewed) seam,
  and the only new surface is the ordering and the single body read.
- **First-wins is order-observable but side-effect-free** — verification has no effects beyond its
  declared secret/key lookup, so trying members in order leaks nothing and re-running yields the same
  verdict. Audit/logging belongs after resolution.
- **No identity confusion** — the resolved variant is nominal; the body cannot mistake a `Webhook` for
  a `User`, and an unauthenticated request reaches a handler body only via an explicit `None` member.
- **Explicitly out:** same-scheme multi-provider routes (concrete-verification keying), refinement
  members and any 403 path (the invariants slice, Q3), and internal-channel sums.

## Definition of done

A `by who: A | B (…)` handler parses, resolves, and checks (binder required; peer base actors only; no
duplicate scheme; `None` last; every member admissible; exhaustive body match); the entry dispatch
emits first-wins resolution — raw body read once, members tried in declared order, the resolved
identity bound, 401 on total failure — with golden emission (incl. a mixed header/body-member route)
and `tsc --strict` green; a behavioral first-wins/fail-closed test (extending the auth-bypass guards)
passes; the `karn.actor.{sum_requires_binder,duplicate_sum_scheme,refinement_in_sum,unreachable_sum_arm}`
diagnostics land; the ADR lands and `design/tracks/actors.md` marks slice 4 done; LSP/grammar/spec/
changelog updated; `cargo build`/`test`/clippy + `cargo fmt --check` and CI green; `/security-review` +
`/code-review` gate the verification-bearing changes; the version is bumped. The implementing PR
**deletes this file**.
