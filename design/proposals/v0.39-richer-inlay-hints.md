# v0.39 — Richer inlay hints (parameter names + generic instantiations)

- **Status:** Draft (proposal). **Merging is approval to build.** *(Deferred-polish items 10 + 8. Two new
  inlay-hint kinds on top of the v0.27 inferred-type hints: **parameter-name** hints at call sites
  (`f(count: 5)`) and **generic-instantiation** hints showing inferred type arguments
  (`identity⟨[Int]⟩(5)`). Both are checker-internal + the inlay handler — no protocol or grammar change.
  The remaining polish items (semantic-tokens delta, `inlayHint/resolve`, InRange-swap) are separate.)*

## Context

The inlay-hint pipeline (v0.27, ADR 0056) harvests **inferred-type** hints — `let`/`let <-` bindings and
lambda params whose annotation is absent — into a `HintSink`, drained into `FileHints` and rendered by the
LSP `inlay_hints` handler. Two high-value hint kinds are missing: the **parameter name** before each
argument at a call, and the **inferred type arguments** at a generic call the user wrote without them.
Both data are already computed during checking; they just aren't recorded. The one shared enabler is a
**hint-kind discriminator** on the sink (the current `(span, label)` is implicitly type-kind).

## Builds on (verified — scout)

- **The sink is `(span, label)`, kind-implicit.** `FileHints = HashMap<PathBuf, Vec<(Span, String)>>`
  (`hints.rs:23`); `HintSink::record(span, label)` (`hints.rs:48`); the LSP handler hard-codes
  `kind: Some(InlayHintKind::TYPE)` and anchors at `span.end` with no padding (`inlay_hints.rs:20-35`).
  Labels are **pre-rendered** (no `Ty` leaks).
- **Parameter names + arg spans sit together in the checker.** Every call-checking loop has the callee's
  `Param { name, … }` (`ast.rs`) and each `arg.span` in hand — free fns/generics
  (`check_generic_call`/`check_call_against_fn`, `calls.rs:356/531`), methods (`check_method_call`,
  `:939`), capability ops, and record construction. `ctx.hints` is threaded to all of them.
- **Inferred type args are ground at the generic call.** `check_generic_call` builds `subst:
  HashMap<String, Ty>` and, by `calls.rs:~523`, every type param is a ground `Ty` — exactly the
  `[Int, String]` to render. Today the `subst` is local and dropped.
- **The capability is already declared.** `inlay_hint_provider` is on (`OneOf::Left(true)`); this
  increment adds no new capability — only more hints flow through the existing provider (and the existing
  `karn.inlayHints.enable` toggle still gates all of them).

## Decisions (open — settle at review)

**[DECISION] The hint-kind discriminator.** *Rec:* add `HintKind { Type, Parameter }` and widen the sink
entry to `(Span, String, HintKind)` (or a small `Hint` struct). The handler maps:
- **`Type`** → anchor `span.end`, `InlayHintKind::TYPE`, no padding (the existing binding-type hints
  *and* generic-instantiation hints — both render after a name with a pre-rendered label: `": Int"`,
  `"[Int]"`).
- **`Parameter`** → anchor `span.start`, `InlayHintKind::PARAMETER`, `padding_right` (renders
  `count: 5`).
`record(span, label)` keeps its meaning (Type); add `record_param(span, label)`. Generic-instantiation
needs no new method — it's `record(name.span, "[Int, …]")`.

**[DECISION] Parameter-name suppression.** *Rec:* skip the hint when it's noise — when the argument **is
an identifier equal to the parameter name** (`f(count)` where the param is `count`), matching
rust-analyzer. Settle at review whether to also skip single-argument calls (lean: keep them — a lone
`f(5)` still benefits). Literals and complex exprs always get the hint.

**[DECISION] Generic-instantiation: only when implicit.** *Rec:* record the type-arg hint **only when the
user wrote no explicit `[…]`** (else it's redundant). Render in declaration order over
`fn_decl.type_params`, each `Ty` through the shared type renderer, as `[A, B]`. Anchor at the function
name's end so it reads `identity[Int](5)`.

**[DECISION] Which call forms get parameter hints.** *Rec:* free-fn and generic calls first (the common
case), then method calls and record construction. Capability-op calls and constructor variants can follow
in the same slice if the loop is uniform; otherwise note the gap.

**[DECISION] Slicing.** *Rec:* **two slices.** **Slice 1:** the `HintKind` gate + **parameter-name
hints** across the call loops (the broadly-useful one; touches several checker sites + the handler).
**Slice 2:** **generic-instantiation hints** (one site, `check_generic_call`, reusing the gate). Each
ships its own tests/changelog/bump; the final slice deletes the proposal.

**[DECISION] New decision record.** *Rec:* **ADR 0072** — the `HintKind` discriminator; parameter-name
hints (anchor/kind/padding + the identifier-match suppression); generic-instantiation hints
(implicit-only, declaration order).

## Spec / docs delta (sketch)

- **`design/karn-lsp-spec.md`** — extend the inlay-hints section: three kinds now (inferred binding type,
  parameter name with the suppression rule, inferred generic type arguments); all behind the same toggle.
- No grammar/protocol change.

## Tests

- **`karnc` hints** (the existing inlay-hints test harness): parameter-name hints attach to the right arg
  spans with the right labels, and are **suppressed** when the arg is the identically-named identifier;
  generic-instantiation hints record the inferred args in declaration order and **only** when the call
  omitted explicit `[…]`.
- **`karn-lsp`**: the handler emits `InlayHintKind::PARAMETER` (anchored at the arg start, `padding_right`)
  vs `TYPE` (anchored at the name end) from the kind. Thin, as elsewhere.

## Diagnostics / Risks

- **Hint noise** — the suppression heuristic + the existing per-language toggle keep parameter hints from
  overwhelming; the density matches rust-analyzer's defaults.
- **`FileHints` tuple ripple** — widening the entry touches its few consumers (the sink drain, the LSP
  `Analysis.hints`, the handler, tests); mechanical, caught by the compiler.
- **Clean-file dependence** — generic-instantiation hints need the inferred `subst`, available only where
  the checker reached a successful instantiation; partial on a file with errors (same ceiling as today).

## Out of scope (→ later deferred-polish increments)

- **(11) Semantic-tokens `delta`** — a pure LSP-state optimisation (result-id cache + token diff); no
  hint overlap.
- **(12) `inlayHint/resolve`** — lazy tooltips; low ROI until a hint has heavy enrichable content.
- **(9) InRange-swap quick-fix** — gated on a *parser* change (per-bound spans on `InRange`); higher-risk,
  its own increment.

## Definition of done

Inlay hints gain **parameter-name** hints at call sites (suppressed when the argument is the
identically-named identifier) and **generic-instantiation** hints (inferred type args, only when the call
omitted them), via a `HintKind` discriminator on the sink and the inlay handler — no protocol or grammar
change, all behind the existing toggle. `karnc` hint tests + a `karn-lsp` handler test cover both kinds
and the suppression; `design/karn-lsp-spec.md` updated; ADR 0072 added; CI green; changelog; version →
0.39.0. Lands as two slices (parameter names; then generic instantiations); the final PR **deletes this
file**.
