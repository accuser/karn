# v0.29.8 — Refactor track, increment 8: eliminate the second TypeScript emitter

- **Status:** Draft (proposal). **Merging is approval to build.** *(Refactor track item 7. Gates on
  `29.2` (which surfaced `project/tests_emit.rs` as the seam) and the `29.1` pins are both **satisfied**
  (merged). Not purely mechanical — a **reconciliation** of divergent copies, not a naive collapse.
  Behaviour-preserving — emitted TS for every test/integration fixture is byte-identical.)*

## Context — a parallel TS generator outside `emitter/`

- The test/integration-emission helpers (now in `project/tests_emit.rs` after the `29.2` split) carry
  their **own** TypeScript formatters, parallel to the real emitter and at risk of escaping/formatting
  **drift** (verified, refs post-`29.2`/`29.7`):

  | concern | `emitter/` (canonical) | `project/tests_emit.rs` (duplicate) |
  |---|---|---|
  | string escaping | `escape_ts_string` (`emitter.rs:5646`) | `escape_ts_string` (`:2109`) |
  | type-ref rendering | `ts_type_ref` (`emitter.rs:5553`, `pub(crate)`) | `ts_type_ref_display` (`:4`), `ts_type_ref_emit` (`:2124`), `ts_type_ref_emit_qualified` (`:2170`) |
  | identifier sanitising | `sanitise_path_segment` (`emitter.rs:2293`) | `sanitise_suite` (`:899`), `sanitise_case_name` (`:2093`) |

- So there are **two `escape_ts_string`** and **three+ type-ref renderers** producing TypeScript from
  the same AST types. A change to escaping or type formatting in one is silently *not* reflected in the
  other — the exact drift hazard the consolidation removes.
- **The escaping divergence is now concretely confirmed** (the `29.1` pins, co-located in
  `tests_emit.rs`'s test module, pin it): `emitter::escape_ts_string` escapes `\r` → `\r`, while
  `tests_emit::escape_ts_string` leaves `\r` **raw**. They agree on `\\`/`"`/`\n`/`\t`. So the
  consolidation **cannot** naively adopt the emitter's copy for the test harness — it must reconcile
  the `\r` handling (parameterise, or accept a test-output change and re-bless). This is the increment's
  central reconciliation point.

## Builds on (verified)

- **`29.1` already pinned both sides (merged).** The seam-test increment characterised the helpers
  **and mirrored the escaping fixtures against `emitter.rs::escape_ts_string`** — precisely so the
  current agreement (and the `\r` *dis*agreement) between the copies is captured **before** they merge.
  Those pins were distributed into `tests_emit.rs`'s own test module by `29.2`, so they sit beside the
  helpers they guard. This increment consumes them as its correctness contract.
- **`29.2` gave the seam a home (merged).** The duplicates live in `project/tests_emit.rs`;
  consolidation is "move the call sites onto the emitter's functions and delete the local copies," a
  named, contained change in one file.
- **The canonical functions are already shaped for reuse** — `emitter::ts_type_ref` is `pub(crate)`
  (`emitter.rs:5553`), so `tests_emit` can call it directly; `escape_ts_string`/`sanitise_*` need only
  be made `pub(crate)` (or moved to a shared `emitter` sub-module) to be callable.

## Settled

- **▸ Reconcile, then delete — output byte-identical.** The acceptance gate is the `29.1` pins **plus**
  the test/integration golden fixtures, all green and unedited. The test harness's emitted TS must not
  move by a byte.
- **▸ One source of truth per concern.** After this increment, exactly one `escape_ts_string`, one
  type-ref renderer (parameterised if needed, see below), and the `sanitise_*` family live in the
  emitter module; `tests_emit` calls them.

## Decisions (open — settle at review)

**[DECISION] The type-ref renderers may genuinely differ — reconcile by parameter, not by fiat.** There
are three: `emitter::ts_type_ref`, `project::ts_type_ref_display`, `project::ts_type_ref_emit`(+
`_qualified`). *First action: diff their outputs* (the `29.1` pins make this concrete). *Rec:* if they
are output-identical, collapse to `emitter::ts_type_ref` outright. If they differ (e.g. `_qualified`
prefixes a module, `_display` formats for messages vs emission), unify into **one function with an
explicit mode/qualification parameter** that reproduces each call site's current output exactly — never
a "pick one and hope" merge. The divergence, if real, becomes an explicit argument rather than a
silent fork.

**[DECISION] `escape_ts_string`: confirm identity first.** The two copies are likely identical, but
*verify against the `29.1` mirrored fixtures* before deleting `project`'s — a hidden difference (an
extra escaped character, different unicode handling) would otherwise change test output. If identical,
delete `project`'s and call the emitter's; if not, the difference is a bug to surface separately, then
converge.

**[DECISION] Where the shared functions live + visibility.** *Rec:* keep them in the emitter module,
widening `escape_ts_string`/`sanitise_*` to `pub(crate)` (matching `ts_type_ref`); if the set grows,
an `emitter/ts_format.rs` sub-module is the natural home, but **don't** create it speculatively — move
there only if the call graph warrants. `sanitise_suite`/`sanitise_case_name` are test-emission-specific
naming; fold them onto `sanitise_path_segment` **only** if output-equivalent, else leave them in
`tests_emit` (they are not the drift hazard the *escaping/type* duplication is).

## Spec delta

**None.** No language surface, no `docs/src/spec/` chapter — a DRY consolidation of internal emitters.

## Tests

- **No new golden fixtures.** The `29.1` pins + the test/integration golden fixtures are the gate; all
  pass **unedited**. The pins specifically guard the escaping/type-ref behaviour being consolidated.
- **If a renderer is unified by parameter,** add a direct test that each mode reproduces the
  pre-merge output for a representative `TypeRef` (base, generic, qualified) — pinning the
  parameterised contract.

## Diagnostics / Risks

- **Silent divergence between the copies is the whole risk.** If the renderers differ and the merge
  picks one, test output changes — caught by the `29.1` pins **only if** they captured the difference,
  which is why the pins mirrored both sides. *Diff first, parameterise if needed, never assume
  identity.*
- **Visibility widening.** `pub(crate)` only; these are internal formatters, not API. Keep
  `tests_emit`'s own non-shared helpers private to it.
- **Scope discipline on `sanitise_*`.** The identifier sanitisers are lower-stakes than the
  escaping/type duplication; resist folding them in unless provably output-equivalent — an over-eager
  merge here risks churning test file names for no drift-safety gain.

## New decision records this increment adds

- **None.** Covered by the `29.1` track-level ADR. If a renderer is unified by parameter, note the
  parameter's semantics in the PR (not a new ADR).

## Out of scope (→ later / independent)

- **`CodeWriter`** (`29.6`) — these are pure `String`-returning formatters, not indented writers; the
  writer is incidental here.
- **The broader emitter/test-emission relationship** — this increment removes the *duplicate
  formatters* only; it does not merge the test-emission *pipeline* into `emitter/` (that is a larger
  question, not scheduled).

## Definition of done

`project/tests_emit.rs` calls the emitter module's single `escape_ts_string`, a single (possibly
mode-parameterised) type-ref renderer, and the shared `sanitise_*` where output-equivalent; the
duplicate copies in `tests_emit` are deleted; shared functions are `pub(crate)`; the `29.1` pins and
all test/integration golden fixtures pass **unedited** (plus any parameterised-renderer test); `cargo
build`/`test`/clippy and CI stay green; changelog notes the de-duplication. The implementing PR
**deletes this file**.
