# v0.49 — Security CI hardening: behavioral auth-bypass tests, SAST (CodeQL), and secret scanning

- **Status:** Draft (proposal, **queued**). **Merging is approval to build.** *(CI/tests only —
  no language surface, no emitted-output change; every existing gate stays green. A standalone
  hardening increment, not a feature track.)*
- **Build-gated on v0.48** (`design/proposals/v0.48-firstparty-testable-sources.md`). v0.48 relocates
  the emitted runtime — including the Bearer verifier `verifyBearerJwtHs256` — from the `RUNTIME_TS`
  string literal in `emitter.rs` into a real `firstparty/bindings/runtime.ts` behind `include_str!`.
  That relocation is what makes this increment *effective*: the auth tests get a real import target
  and CodeQL can finally see the verifier. Doing this before v0.48 would mean building against the
  string-literal runtime and reworking it after. The version is provisional (assumes v0.48 lands as
  v0.48 — renumber if sequencing shifts).
- **No new ADR-0076 verification surface** — this increment *strengthens* the safety net around the
  v0.47 Bearer verifier; it emits no new verification logic itself.

## Context — the auth verifier has no standing regression guard, and the riskiest code is SAST-blind

The v0.45–v0.47 actors work put a real security boundary in the emitted output: `verifyBearerJwtHs256`
and the fail-closed Bearer seam. Its correctness was established by a one-time `/security-review` at
v0.47 (clean — all bypass classes closed). But:

- **That review is a point-in-time gate, not a regression guard.** A future slice (Signature,
  multi-actor, a refactor of the seam) could reintroduce a bypass — `alg:none` slipping through, a
  fail-open path, the signature check inverted — and nothing in CI would catch it. `tsc --strict`
  checks *types*, not *that a tampered token is rejected*.
- **The verifier is invisible to SAST today** because it lives as a Rust string literal. Even after a
  tool like CodeQL is added, it sees nothing until v0.48 makes the runtime a real `.ts` file.

The existing supply-chain posture is already strong (`cargo audit`, `cargo-deny`, `dependency-review`,
OpenSSF Scorecard, SHA-pinned actions). The gaps are **(1)** a behavioral test of the emitted auth
boundary, **(2)** SAST, and **(3)** secret scanning. This increment closes all three, anchored on (1)
— the one that actually defends the feature we shipped.

## Builds on (verified)

- **v0.48's relocated `runtime.ts`** — a real file the auth tests import directly and CodeQL scans.
- **The `runtime_helpers` + `tsc_verify` harness** — already imports the emitted runtime and runs it
  under Node, with the discover-or-skip-loudly pattern (`KARN_REQUIRE_TSC=1` makes a missing toolchain
  a hard failure). The auth tests are an extension of this harness, not new machinery.
- **The v0.47 Bearer fixture** (`210_bearer_http_auth`) and `verifyBearerJwtHs256`'s shape
  (`Result<{ sub }, string>`, HS256, `exp`/`nbf`, `alg` pinning) — the unit under test.
- **The existing CI security jobs** (`ci.yml`: `audit`, `cargo-deny`; `scorecard.yml`) and the
  SHA-pinned-actions discipline — CodeQL/gitleaks workflows follow the same pinning.
- **OpenSSF Scorecard** already runs; its **SAST check** currently scores low — enabling CodeQL is a
  concrete, measured improvement in the framework the project already tracks.

## Decisions (open — settle at review)

**[DECISION] The headline: behavioral auth-bypass tests for the emitted Bearer verifier.** *Rec:* a
Node-driven test (extending `runtime_helpers`) that imports the relocated
`verifyBearerJwtHs256` and feeds **crafted JWTs**, asserting the verdict for each — the regression
guard the one-time review can't be:
  - **valid** (correctly signed, unexpired) → `Ok` with the expected `sub`;
  - **tampered signature** → `Err`; **`alg: none`** and **`alg: RS256`** (confusion) → `Err`;
  - **expired (`exp` past)**, **`nbf` in the future** → `Err`; **malformed `exp`/`nbf`** → `Err`;
  - **wrong secret** → `Err`; **missing/empty `sub`** → `Err`; **malformed token / base64** → `Err`.

  The test signs tokens with Node's `crypto` (HMAC-SHA256 + base64url), so a *valid* token is exercised
  too — proving the verifier accepts as well as rejects. *Optionally* a second, end-to-end test drives
  a compiled `210_bearer_http_auth` worker's compose wrapper with an `Authorization` header and asserts
  401 vs a minted identity. *Alternative:* rely on `tsc` + the one-time review — rejected: it leaves
  the auth boundary with no standing guard, which is the whole point of this increment.

**[DECISION] CodeQL: a committed, SHA-pinned workflow vs GitHub default setup.** *Rec:* a committed
`codeql.yml`, SHA-pinned like every other action (the project's posture), scanning
**`javascript-typescript`** (mature — now covers the relocated `runtime.ts` *and* the `vscode-karn`
extension) and **`rust`** (newer/maturing — modest yield over `clippy` + Rust safety, but free and it
bumps the Scorecard SAST check). *Set expectations:* CodeQL's real coverage here is the TS runtime +
extension; the Rust compiler is largely already covered by the type system and `clippy -D warnings`.
*Alternatives:* GitHub **default setup** (zero-maintenance, but not SHA-pinnable — off-posture); or
**JS/TS only** (drop the low-yield Rust leg). Recommend the committed workflow with both languages,
Rust included for the Scorecard credit and as cheap defense-in-depth.

**[DECISION] Secret scanning: GitHub native push protection vs a gitleaks CI job.** *Rec:* **both** —
enable GitHub-native **secret scanning + push protection** (free on this public repo; *prevents* a
secret reaching the remote at push time) and add a **SHA-pinned gitleaks** CI job (catches secrets in
PR diffs and scans history; reproducible and committed, in keeping with the other gates). A
`.gitleaks.toml` allowlist covers the test fixtures' dummy JWT secrets (e.g. `AUTH_JWT_SECRET`
env-*names*, not values). *Alternative:* native only — cheaper, but no in-CI/history coverage and not
a committed gate.

**[DECISION] JS dependency audit parity.** *Rec:* an `npm audit --audit-level=high` job over
`vscode-karn` and `tree-sitter-karn`, matching what `cargo audit` does for the Rust crates (closing the
one un-audited dependency surface). Cheap; honour a threshold so low-severity advisories don't gate.

## Spec delta

**None.** No language surface, grammar, checker, or emission change. CI workflows + Rust/Node test
files only. Emitted output is byte-identical; every emitter golden, `tsc_verify`, and `runtime_helpers`
test stays green and unedited.

## Tests

- **The auth-bypass tests are the deliverable** (the test-scope decision) — a new `tests/` module
  (Rust harness invoking Node over the relocated runtime, or a small Node test the harness shells),
  discover-or-skip-loudly, honouring `KARN_REQUIRE_TSC`/a Node guard.
- **No new emitter fixtures**; the existing suite is the byte-identical-output acceptance gate.
- CodeQL / gitleaks / `npm audit` are CI jobs, not unit tests; each must pass on a clean tree (with
  documented allowlists/thresholds so they gate on *real* findings only).

## Diagnostics / Risks

- **CodeQL noise / Rust immaturity.** The Rust leg may surface low-value or false-positive alerts.
  *Mitigation:* triage to the Security tab (not a hard PR gate initially — `fail-on` tuned), revisit
  once signal is known; the JS/TS leg is the load-bearing one.
- **CI flakiness for the Node-driven auth tests.** Node/tsc availability varies across the matrix.
  *Mitigation:* reuse the established discover-or-skip-loudly pattern; `KARN_REQUIRE_TSC=1` on the
  Linux leg makes the skip a hard failure there, so the guard can't silently vanish.
- **Secret-scanning false positives.** Fixture JWT-secret *names* and example tokens could trip
  gitleaks. *Mitigation:* a committed `.gitleaks.toml` allowlist scoped to `tests/fixtures`.
- **`npm audit` advisory churn.** Transitive dev-dep advisories with no fix. *Mitigation:* an
  `--audit-level` threshold; this is a signal, not a blocker, for low severities.

## New decision records this increment adds

- **One ADR** recording the security-CI posture: *the emitted security boundary (the auth verifier)
  carries a standing behavioral test, not only a one-time `/security-review`; SAST (CodeQL) and secret
  scanning are committed, SHA-pinned gates.* The durable home for "why behavioral auth tests are
  required for emitted verification code," so future auth slices (Signature, etc.) extend the tests by
  default. (Sits alongside ADR 0076's verification-slice gate.)

## Out of scope (→ later / independent)

- **Fuzzing the compiler** (parser/checker robustness) — a separate, larger effort.
- **DAST / a deployed pen-test** of an emitted Worker — needs a deploy target; out of a CI increment.
- **The broader emitted-project security model** (CSP headers, CORS, rate limiting) — application
  concerns, not the verification boundary this increment guards.
- **Signing/SBOM beyond what Scorecard + release provenance already provide.**

## Definition of done

The emitted Bearer verifier has a **standing behavioral test suite** (valid + every bypass class →
asserted verdict) running in CI over v0.48's relocated `runtime.ts`, discover-or-skip-loudly with a
hard-fail leg; **CodeQL** runs as a committed, SHA-pinned workflow (`javascript-typescript` + `rust`)
and the **Scorecard SAST check** improves; **secret scanning** (native push protection + a SHA-pinned
gitleaks job with a fixture allowlist) and an **`npm audit`** parity job are in CI; every existing
gate stays green and emitted output is byte-identical; the security-CI-posture ADR lands;
`cargo build`/`test`/clippy + `cargo fmt --check` and CI stay green; the changelog notes the hardening
(internal — no language change). The implementing PR **deletes this file**.
