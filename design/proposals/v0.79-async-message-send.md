# v0.79 — Asynchronous message send: a caller-chosen fire-and-forget form

- **Scope:** a **single, small language increment**. It adds one call-site form —
  a fire-and-forget send, spelled with a leading `~>` ([DECISION A], now settled) —
  with its marker token, the static rule that gates it, and **one** emission path
  (`ctx.waitUntil`). It does **not** add the at-commit/buffered tier, and it does
  **not** migrate any existing code. Touches lexer, parser, checker, emitter, and
  the spec (lexical grammar, `§4.8` statements, `§5` static semantics, `§7`
  emission) plus a concept page in the book. With [A] settled, this proposal is
  the build authorisation.
- **Realises:** the one present-day wart — `let _ <- Logger.info(msg)` verbosity
  on a `-> Effect[()]` op — and **pre-establishes** the call-site form for the
  async channels that are *designed but unshipped* (`Events`, `Push`, `Queue`),
  so they inherit a consistent spelling instead of each inventing one.

## What this is, and is not, fixing today

Be honest about the present state, because it is the strongest argument for the
change. **Today `<-` is not overloaded.** The parser requires a binder
(`bynk-syntax/src/parser/statements.rs` — an effect statement is `let <name> <-
…`), so every effectful statement uniformly *awaits*. The binder-less `<-` that
*would* be ambiguous exists only in the design notes (§1071 deliberately reuses
`<-` as a leading fire-and-forget marker); it is not in the shipped grammar. And
none of the "fire-and-forget channels" are shipped: `bynk-check/src/firstparty/`
exposes only `Clock`, `Random`, `Logger`, `Fetch`, `Secrets` — no `Events`, no
`Push`, no `Queue`. The only present-day `-> Effect[()]` capability operations are
`Logger.info` / `Logger.error`.

So the sole beneficiary today is removing the `let _` from `let _ <-
Logger.info(...)`. Everything else here is **pre-emptive**: we are choosing the
call-site distinction *before* `<-`-as-release calcifies. That is the right time
to do it — retrofitting a marker once code and conventions depend on
`<-`-meaning-release is far more expensive than reserving it now against a
sketch.

## The model

A call is a message to a recipient. `Recipient.op(args)` *sends* `op(args)` to
`Recipient`; synchronous delivery awaits the reply, asynchronous delivery does
not. In the author's mental notation:

- synchronous — `let r <- (op(args) -> Recipient)`: send to the recipient, await
  the reply.
- asynchronous — `(op(args) ~> Recipient)`: send to the recipient, no reply.

This is UML's synchronous/asynchronous message distinction (filled arrowhead with
a dashed reply, versus an open arrowhead and no reply), and it sits naturally on
the actor framing the language already uses (agents receive keyed messages;
`on call` handlers). The notation above is the *semantics*; the surface marker is
[DECISION A].

**[DECISION B — taken] The caller chooses the form; the operation only publishes a
type contract.** An operation's signature stays a plain `-> Effect[...]`
describing the *reply* — **no `call`/`cast` handler keywords, no `Oneway` return
type**. Sync-versus-async is a call-site choice, *bounded* by what the contract
permits. The recipient cannot know whether *this* caller needs the completion, so
the choice belongs at the call site.

## Surface — the call-site grid

The form follows from two independent questions: does the reply carry a value, and
does the caller wait?

|              | **await**                          | **don't await**                |
|--------------|------------------------------------|--------------------------------|
| **value**    | `let r <- Recipient.op(a)`         | forbidden — [DECISION D]       |
| **no value** | `let _ <- Recipient.op(a)` (durable ack) | `~> Recipient.op(a)` — `Effect[()]` only |

Mapped to UML: `let r <-` is a synchronous message with a valued reply; `let _ <-`
is a synchronous message whose reply is an empty **ack** you still wait on (the
durable-write case — storage ops are `Effect[()]` yet must be awaited for
read-your-writes and atomic commit, design notes §388); the new `~>` form is the
asynchronous message — open arrowhead, no reply.

**Erroring fire-and-forget does not become a send.** The design notes' binder-less
`<- Sagas.compensate(...)?` and `<- Idempotency.dedup(...)?` (design notes §722,
§630) carry an error via `?`; the error gate ([DECISION C]) forbids those from
collapsing into `~>`, because the error would vanish. Their honest spelling is
`let _ <- e?` — await, discard the unit, still propagate the error. So the new
form absorbs exactly the *empty, non-erroring* sends and nothing else; the grid
above is total.

## Static rules

- **[DECISION C — adopt] The error gate.** A send is legal only when the reply is
  `Effect[()]`. `~> Fetch.send(req)` is rejected (proposed
  `bynk.send.drops_error`) because its `FetchError` would vanish. The contract
  bounds the caller's menu; it does not dictate the choice. This is what keeps
  caller-decides from being a silent-drop footgun.
- **[DECISION D — adopt] No forfeiting of values.** If an operation returns a real
  value, the spelling for "I don't want it" is `let _ <-` (await, discard), not a
  send that silently drops data. So `~>` *requires* `Effect[()]`, not merely
  "no error".
- **Context.** The `~>` statement is valid only in an effectful position (handler
  or provider body), exactly as `<-` is, and it is a statement, never an
  expression.

## Delivery tier — **[DECISION E]**

A send says only "I am not waiting." Whether it fires **immediately, best-effort**
(logging) or is **buffered and released at commit** (events) is a property of the
providing **capability**, not the call site or the signature.

**Adopt the principle, but build only the immediate tier now.** The at-commit half
has nothing to attach to today: there is no output-gate / commit-buffer machinery
in `bynk-emit/`, and the only at-commit channel (`Events`) is unshipped. Building
it now yields dead, untestable code. So this increment wires and tests exactly one
path — `~> Logger.info(...) → ctx.waitUntil` — and **defers the at-commit tier to
the `Events` increment**, where it can be co-designed against real semantics.

## Emission — and why Cloudflare makes the immediate tier real

On the Workers target the immediate tier has a precise home: `ctx.waitUntil(promise)`.
This is more than "don't `await`" — **`ctx.waitUntil` lets the call settle away from
the call site.** The runtime keeps the worker alive until the promise resolves, so
the send completes *after* the handler has returned its response, decoupled from the
synchronous request/response flow. A bare un-awaited promise would be killed when the
handler returns; `waitUntil` is the platform primitive that makes a fire-and-forget
send actually *deliver*. That is exactly the semantics `~>` should carry — dispatched,
settles away from the call site, no reply — and it is why the immediate tier is
buildable and testable on its own.
Synchronous forms keep emitting `await`. (Direction, not normative prose; the spec
text is written during implementation.)

## Scope of this build (explicit)

In: the `~>` marker token; the `~>` statement (parse + check); the error gate; the
`ctx.waitUntil` emission for `Logger.info`/`error`; the spec and one concept page.
Out, by decision: the at-commit tier ([E], → `Events`); first-party and example
migration ([F]); any `call`/`cast`/`Oneway` machinery ([B]).

## Decisions

- **[A — settled] The marker is a leading `~>` glyph.** It must **not** be `<-`
  (reusing the await arrow rebuilds the very overload this prevents). The choice was
  between a *contextual* `send` keyword and a glyph, and the glyph wins on cost:

  - A `send` keyword is **not** trivially reservable (correcting this proposal's
    earlier framing). A hard `#[token("send")]` would break `Fetch.send` — the most
    prominent capability method in the shipped language (`bynk.bynk:63`, used across
    the examples) — and the designed `Queue.send`, because member names after `.` go
    through `expect_ident`, which rejects keyword tokens. The viable form is a
    *contextual* keyword (lexed as `Ident`, recognised as the marker only at
    statement start), which is feasible and precedented — `integration` does exactly
    this (`bynk-syntax/src/parser/declarations.rs` ~56) — but it costs a
    statement-start disambiguation rule (an `Ident` at the head of a statement,
    followed by a recipient call, versus an ordinary expression statement). That is a
    real parser design point, not a free reservation.
  - A leading `~>` glyph has **zero identifier collision** and needs **no
    disambiguation** (no expression statement begins with `~>`), giving the cleanest
    LL(1) at statement start. It satisfies the §1022–1024 ligature/LL(k) constraints
    directly, and it *coincides with the author's own mental notation* (`op(args) ~>
    Recipient`), so surface and semantics read the same. Its only cost — "earns less
    than a word" — is the lightest of the trade-offs, and it is symmetric with `<-`
    already being an arrow glyph for effect/result flow.

  Either was defensible; `~>` is chosen for the collision-free, disambiguation-free
  path. **Regression guard:** the increment adds a one-line positive fixture asserting
  `Fetch.send(req)` still parses as an ordinary member call, so the marker cannot
  silently disturb member access.
- **[B — taken] Caller-decides; operation publishes a contract.** No new handler
  kinds, no `Oneway` type.
- **[C / D — adopt] Error gate + no value-forfeiting.** `~>` requires `Effect[()]`.
- **[E — adopt principle, defer half] Tier is the capability's property.** Ship the
  `waitUntil` tier; defer at-commit to `Events`.
- **[F — follow-on] Migration.** Default to a later pass. Migrating first-party
  `bynk.bynk`, the examples, and the guides onto a *visible, hard-to-revert* surface
  must wait until (a) [A] is signed off and ideally (b) a second async channel exists
  to justify it.
- **[G — settled by scope] Single small increment.** Scoped to marker + `waitUntil` +
  `Logger`, this is plainly one increment; the sliced-track option is dropped as
  over-engineering.

## Risks & mitigations

- **Silent-drop footgun** — bounded by the error gate ([C]/[D]); `~>` is `Effect[()]`
  only.
- **Two unit-shaped meanings** (`Effect[()]` awaited vs sent) — the *call-site form*
  distinguishes them, so the type stays single.
- **Designing a marker against a sketch** — accepted: cheaper than retrofitting once
  `<-`-as-release calcifies (the central argument). Mitigated by shipping the real,
  testable `waitUntil` path so the marker is exercised, not hypothetical.
- **Glyph regret** — settle [A] before [F]; nothing visible migrates until then.

## Citations (corrected from the first draft)

- "`Effect[()]` must be awaited" is supported by **design notes §388** (all storage
  operations are Effect-typed, requiring `<-`), *not* ADR 0051 (which is only about
  KV method naming).
- The motivating overload is **design notes §1071's own convention** of reusing `<-`
  for fire-and-forget — this proposal argues that convention is the mistake. The
  chat-room example (§1832) is **not** inconsistent: its broadcast is
  `<- conns.parTraverse(c => c.send(MessageBroadcast(msg)))` — the send is a
  `parTraverse` lambda body awaited by `<-`, not a bare statement.
- The "distinct from `<-`/`->`/`=>`" requirement is **this proposal's**, not a spec
  claim. §1022–1024 establishes only ligature-friendliness and LL(k)-tractability.

## Spec delta (sketch — not normative content)

- **Lexical grammar:** one new `~>` marker token.
- **§4.8 statements:** a new `effect_send_stmt` (`~>` + a recipient call, no binder),
  sibling to `effect_let_stmt` (§4.8.4).
- **§5 static semantics:** the error gate (`~>` requires `Effect[()]`); effectful
  context only.
- **§7 emission:** the immediate tier (`ctx.waitUntil`); synchronous forms unchanged;
  the at-commit tier explicitly out of scope.
- **Regression fixture:** a one-line positive fixture that `Fetch.send(req)` still
  parses as a member call (guards member access against the new token).
