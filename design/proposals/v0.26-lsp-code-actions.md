# v0.26 — LSP code actions (quick-fixes from structured suggestions)

- **Status:** Draft (proposal). **Merging is approval to build.** *(The A-1 headline, on the
  complete A-0 surface. Touches `karnc` (diagnostic surface + a little checker plumbing) +
  `karn-lsp`.)*
- **Builds on (verified):**
  - `CompileError` (`error.rs:14`) carries `category`, `span`, `message`, `labels`, `notes` —
    **no machine-applicable fix data**; the suggestion layer is net-new.
  - **The prescriptive `given`-clause fixes are now reachable** (v0.24's `diagnose_project` checks
    contexts): `karn.given.unused_capability` (`checker.rs:282`, anchored at **`return_ty_span`**
    `:283`) and add-to-`given` — `karn.given.undeclared_capability` at the **usage site**
    (`type_name.span` `:4509`; `receiver.span` `:5848`). **Crucially, the edit lands in the
    `given` clause, away from the diagnostic's span** — this reshapes the code-action rule (below).
  - **The `given`-cap names reach the check as `Vec<String>`** (`given_declared`, `checker.rs:211`)
    — *names only*; the spans live on `CapRef` (`ast.rs:348`) but aren't plumbed in. So the seed
    fixes need a little checker-entry-point plumbing (sized below), not just a `.with_suggestion`.
  - `diagnose_project` → `FileDiagnostics { source_path, text, diagnostics }` (`lib.rs:270`). The
    retained `Analysis` (`main.rs:50+`) holds the **index + snapshots + versions + diagnostic
    *categories*** (the rename validator's baseline) — but the **full `CompileError`s, where
    suggestions ride, are published and dropped**. So serve-from-cache needs `Analysis` **extended
    to retain the round's per-file diagnostics** — a small change, the same shape as v0.25's index
    retention. *(Not an "already sufficient" cache.)*
  - The LSP does **not** advertise `codeActionProvider`. The v0.25 test harness
    (`karn-lsp/tests/rename_validation.rs`) is **pure functions, no transport** — it includes the
    handler modules via `#[path]` and never speaks LSP (this bounds the test plan, below).
- **Spec sections touched:** `design/karn-lsp-spec.md` (Code actions + the capability; §4.3);
  `karnc` `error.rs` (the `Suggestion` type) + the seed checker sites.
- **Scope:** the **suggestion mechanism** + the **two `given`-clause quick-fixes** + the LSP
  `codeAction` provider. A **secondary section** folds in two near-free index riders
  (`workspace/symbol`, `documentHighlight`).

## Settled (recorded as ADRs ≥0054)

- **▸ Quick-fixes are structured data authored at the diagnosis site — not re-derived in the
  LSP.** The checker knows the fix; prose `notes` are not machine-applicable; co-location keeps it
  correct, unit-testable in `karnc`, and CLI-consumable later. LSP-side re-derivation is rejected.
- **▸ Code actions key on the *diagnostic's* span, not the edit's.** Clients request actions at the
  cursor/squiggle (the diagnostic span); for the `given` fixes the edit lands elsewhere (the
  clause). So the rule must surface a diagnostic's suggestions when the **diagnostic span**
  intersects the request — see the computation decision.

## Decisions (open — settle at review)

**[DECISION] Suggestion representation.** *Rec:* add to `CompileError`
```
suggestions: Vec<Suggestion>
struct Suggestion { message: String, edits: Vec<(Span, String)>, applicability: Applicability }
enum Applicability { MachineApplicable, HasPlaceholders }
```
with `.with_suggestion(...)` (mirroring `.with_note`); `edits` are span→replacement, empty =
deletion. The LSP maps each to a `CodeAction { kind: QuickFix, edit: WorkspaceEdit }`. *Both*
seed fixes are `MachineApplicable`; `HasPlaceholders` is **unused this increment** — kept only to
mirror rustc and gate a future `--fix` (no behaviour hangs on it yet).

**[DECISION] Seed catalogue — the two `given`-clause fixes (InRange deferred).** *Rec:*
- **Remove unused capability** — `karn.given.unused_capability` (`checker.rs:282`): delete the
  capability from the `given` clause.
- **Add capability to `given`** — `karn.given.undeclared_capability` (`:4509` `Cap`, `:5848`
  `B.Cap`): insert the named capability into the handler's `given` clause.
*Deferred:* the **InRange bound-swap** — `PredKind::InRange(i64,i64)` (`ast.rs:767`) carries **no
bound spans or lexemes** (only `pred.span`), so "swap the two argument spans" is **not computable**
without extending the predicate AST to carry bound spans (a parser change) or re-rendering the
whole predicate (lossy for `Int` bounds — the formatting loss `InRangeF`'s lexemes deliberately
avoid). Not worth dragging into a code-actions increment; it returns as a fast-follow once the
predicate carries bound spans.
*Fix in passing:* the notes at `checker.rs:4513`/`:5854` are **plain string literals** — `{name}`
/ `{consumed}.{cap}` are never interpolated (users see literal `{name}`); add the missing `format!`
while these sites gain `.with_suggestion`.

**[DECISION] Checker plumbing the seed fixes need (sized honestly).** Authoring the fix at the
diagnosis site requires:
- **Remove** — plumb the unused cap's `CapRef.span` (`ast.rs:348`) into the unused-capability check
  (today it receives `given_declared: Vec<String>`, `checker.rs:211` — names only).
- **Add** — plumb the `given` clause's **insertion point** (the last entry's span, or where to
  synthesise an absent clause) to the usage-site diagnostics.
None is hard, but each is a **checker entry-point signature change**, not a one-line note — the
slice is sized accordingly.

**[DECISION] List-aware edits (the seed fixes *are* `given`-list edits).** *Rec:* the suggestion
author computes list-aware edits so the result never double-/leading-commas or leaves `given ,`:
**remove** spans the capability + one adjacent comma + surrounding space; **remove the *only*
capability** deletes the `given` keyword too (no dangling `given`); **add** inserts `, Cap` after
the last entry, or `given Cap` when the clause is absent. A fixture matrix pins exact text for
first / middle / last / **only** positions.

**[DECISION] `codeAction` computation — key on the diagnostic span, serve from cache.** *Rec:* on
`textDocument/codeAction`, read the **analysis cache** for the document (`Analysis`, **extended this
increment to retain the round's per-file diagnostics** alongside the snapshot it already keeps) —
**not** a fresh `diagnose_project` (slow, and can disagree with the squiggles the client is
showing). Offer a QuickFix for every suggestion whose **owning diagnostic's span** intersects the
requested range (optionally unioned with the edit spans). Positions convert against the **cached
snapshot** (the v0.24 rule); edits are **versioned** `TextDocumentEdit`s. A request arriving
**before the first analysis round** (or for a file **outside the project**) finds no cached round →
the handler returns an **empty list**. *(Once `Analysis` retains the round's full per-file
diagnostics, the existing `diags`-categories field — the rename validator's baseline — becomes
derivable from them; collapse the two rather than carrying parallel representations.)*

**[DECISION] Riders — `workspace/symbol` + `documentHighlight` (folded in).** Both are
`ProjectIndex` reads, natural neighbours of `index_queries`: workspace-symbols enumerates index
definitions; document-highlights returns a symbol's in-file occurrences (the `references` query,
file-scoped). *Note:* `documentHighlight` has a `kind` (Read/Write/Text); the v0.25 index does not
distinguish read/write, so **`kind` is omitted (or `Text`)**. They add the
`workspaceSymbolProvider` / `documentHighlightProvider` capabilities; **no `karnc` change**.

## Tests (the LSP analogue of fixtures)

- **`karnc`** *(`cargo test`, no LSP):* each seed diagnostic carries the expected `Suggestion`
  (`MachineApplicable`, right edits); the **list-aware** fixtures assert exact `given` text for
  first / middle / last / **only**-capability positions (incl. the only-case dropping `given`).
- **`karn-lsp` pure-function level** *(matching the v0.25 harness — no transport):* a function that
  computes code actions from a `FileDiagnostics` + a range (keyed on the diagnostic span) returns
  the expected `CodeAction` + versioned `WorkspaceEdit`; unit-tested directly. The
  `codeActionProvider` advertisement is a **trivial unit check** (or left to manual verification) —
  **a transport round-trip is not claimed**, since the harness speaks no LSP. *(Riders:*
  `workspace/symbol` and `documentHighlight` as index-query unit tests.)

## Diagnostics

None new. This increment *attaches fixes to existing* diagnostics; no `karn.*` codes added.

## Risks
- **List-item edit correctness** — the dominant risk; mitigated by the list-aware decision + the
  position matrix (incl. only→drop-`given`).
- **Checker signature churn** — plumbing spans/insertion-points touches the unused-cap and
  usage-site check entry points; keep it localised.
- **Applicability discipline** — only `MachineApplicable` auto-applies; all seeds are.
- **Suggestion drift** — co-location + `karnc` unit tests keep fixes correct as diagnostics evolve.
