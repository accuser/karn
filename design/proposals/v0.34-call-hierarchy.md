# v0.34 — Call hierarchy

- **Status:** Draft (proposal). **Merging is approval to build.** *(Tooling-queue item 5, the first
  half of A‑3 navigation. `prepareCallHierarchy` + incoming/outgoing calls — "who calls this fn / what
  does this fn call", with peekable call sites. The caller→callee attribution it needs is **already
  collected** by the index; it's currently dropped at assembly.)*

## Context

LSP call hierarchy is a three-call protocol: `textDocument/prepareCallHierarchy` resolves the symbol
under the cursor to a `CallHierarchyItem`; `callHierarchy/incomingCalls` lists its callers (each with the
call-site ranges); `callHierarchy/outgoingCalls` lists what it calls. The binding index records *that* a
reference occurs and *which symbol* it references, but the assembled `SiteRef` does not say *which
declaration encloses* the reference — so today there is no caller→callee edge to traverse.

That enclosing-declaration attribution, however, **already exists one layer up**: every `RefEdge` carries
`owner` — "the display name of the enclosing top-level declaration" (`index.rs:73`) — set around each fn,
service, agent, provider, and capability body (`checker.rs:201`, `resolver.rs:364/368`,
`project/validate.rs:585/613/772/835`). The index uses `owner` at assembly only for *file
re‑attribution* (`index.rs:385`) and then discards it (`SiteRef { path, span }`, `index.rs:415`).
Preserving it — resolved to the caller's `SymbolKey` — is the whole of this increment.

## Builds on (verified — scout)

- **The caller attribution is collected.** `RefEdge.owner: Option<String>` (`index.rs:78`) is set at
  every body via `RefSink::set_owner` (the seven sites above) and is already in hand at assembly
  (`index.rs:385`, the re‑attribution lookup `owner_files.get((namespace, owner))`).
- **The caller's key is resolvable.** `IndexBuilder::add_def` (`index.rs:311`) already records every
  def's `SymbolKey` *and* an `owner_files` entry keyed by `(unit, name)` — the same `(namespace, owner)`
  shape the re‑attribution lookup uses. A parallel `owner_keys: HashMap<(unit, name), SymbolKey>`
  populated in the same call resolves an `owner` string to the **caller's `SymbolKey`** by construction.
- **The call site is the ref site.** The span pushed at `index.rs:415` (callee-name span, re‑attributed
  to the owner's file) is exactly the call-site range both directions of the protocol want
  (`IncomingCall.fromRanges` and `OutgoingCall.fromRanges` are both the call-site spans).
- **prepare reuses goto-def.** `index_queries::definition_at` (`:40`) + `symbol_at` (`index.rs:220`)
  already resolve a position to `(SymbolKey, def SiteRef)`; `site_to_location` / `span_to_range`
  (`main.rs:472`) build the LSP range. A `CallHierarchyItem` is that plus a kind mapping.
- **No call-hierarchy provider is declared yet** — this adds `call_hierarchy_provider` next to
  `definition_provider` (`main.rs:1351`).

## Decisions (open — settle at review)

**[DECISION] How the call graph is stored.** *Rec:* a `calls: Vec<CallEdge>` on `ProjectIndex`, where
`CallEdge { caller: SymbolKey, callee: SymbolKey, site: SiteRef }`. Built in `IndexBuilder::build`
alongside the existing ref push (`index.rs:424`): when an edge's `owner` resolves via `owner_keys` to a
caller key **and** the qualified callee is a `Fn`, also push a `CallEdge`. Pure, in‑memory, no
serialization (the index is LSP‑only). Queries filter the vec — fine at project scale; an adjacency map
is a later optimisation if it bites.

**[DECISION] What counts as a callee.** *Rec:* **`SymbolKind::Fn` only** (free fns; method callees are
not index symbols — deferred index kinds). Capability‑op calls and agent/service dispatch are not
distinct index symbols and so are out of this graph. This is the honest, non‑lossy MVP: every edge it
shows is a real fn call, and the gaps line up exactly with the deferred‑index‑kinds item.

**[DECISION] What can be a caller.** *Rec:* **any indexed owner** that has an `owner_keys` entry —
free fns, services, agents, providers, capabilities. So "a service handler that calls `validate`" shows
the *service* as an incoming caller (useful). Methods‑as‑callers (`"T.m"` owners, `add_owner`,
`index.rs:351`) have no key and are excluded — same deferred‑index‑kinds boundary, flagged in the spec.

**[DECISION] Carrying identity across the three calls.** *Rec:* encode the resolved `SymbolKey`
(`unit`, `kind`, `name`) into `CallHierarchyItem.data` (a `serde_json::Value`) in `prepare`, and read it
back in incoming/outgoing — so the follow‑up calls resolve directly off the key, never re‑inferring from
a position. Items whose `data` doesn't round‑trip a known key return no calls (defensive).

**[DECISION] Item kind mapping.** *Rec:* map `SymbolKind` → `SymbolKind` (LSP) once — `Fn→Function`,
`Capability/Service/Agent/Provider→Interface/Class/...` — reusing whatever the document‑symbol provider
already maps so the icons match. Settle the exact LSP-kind table against the existing mapping at review.

**[DECISION] New decision record.** *Rec:* **ADR 0067** — call hierarchy from a `CallEdge` side table
built by preserving `RefEdge.owner` (resolved to a caller key); Fn callees only; both directions;
methods/op-calls deferred with the index kinds.

## Tests

- **`karnc` index** (the `IndexBuilder`/`build` harness): a fixture where `fn a` calls `fn b` twice and
  `fn c` calls `fn b` once asserts the `calls` graph has the three edges with the right caller/callee keys
  and the two distinct `b` call‑site spans under `a`; a service handler calling a free fn yields a
  CallEdge with the service as caller; a method caller (`"T.m"`) yields **no** CallEdge.
- **`karn-lsp` pure-function** (the `index_queries` harness): `incoming_calls(index, key)` groups call
  sites by caller; `outgoing_calls(index, key)` groups by callee; an unknown key returns empty.
- LSP‑layer position/`CallHierarchyItem` conversion stays thin (covered as elsewhere).

## Diagnostics / Risks

- **Owner/key resolution misses.** An `owner` with no `owner_keys` entry (a method, or a synthetic
  owner) simply produces no CallEdge — never a wrong edge. The `(namespace, owner)` lookup mirrors the
  proven re‑attribution path (`index.rs:389`), so it can't be *more* wrong than that.
- **Recursion / cycles.** `a` calls `a`, or `a↔b`, are ordinary edges; the client owns tree expansion, so
  no special handling — the queries are flat (one level), as the protocol intends.
- **Clean‑file dependence.** Like every index‑served feature, the graph reflects the last clean analysis
  round; mid‑error edits fall back to the prior round (§3.2).

## Out of scope (→ later)

- **Method / capability‑op / agent‑dispatch edges** — gated on the deferred index kinds (item 3); they
  join the graph for free once those identifiers are indexed.
- **Type‑definition / implementation navigation** (`given Cap` → provider) — item 6, the *other* half of
  A‑3; a separate increment (it needs provider metadata exported into `Analysis`).

## Definition of done

`textDocument/prepareCallHierarchy`, `callHierarchy/incomingCalls`, and `callHierarchy/outgoingCalls` are
declared and answer for free fns (and indexed‑owner callers): prepare resolves the symbol under the
cursor to a `CallHierarchyItem` carrying its key; incoming lists callers with call‑site ranges; outgoing
lists called fns. Built on a `CallEdge` graph assembled by preserving `RefEdge.owner`. `karnc` index
tests + `karn-lsp` query tests cover both directions and the method‑caller exclusion;
`design/karn-lsp-spec.md` gains §3.18; ADR 0067 added; CI green; changelog; version → 0.34.0. The
implementing PR **deletes this file**.
