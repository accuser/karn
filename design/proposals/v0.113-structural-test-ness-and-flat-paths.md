# v0.113 — structural test-ness, suites stripped from the build, and flat `include`/`exclude` (testing track, slice 1b)

- **Scope:** a single-increment **compiler + tooling + build-model** change — the
  second slice of the testing track and the file/build half of it. It makes
  *test-ness a property of the declaration* (a `suite` block), not of a file's name
  or its directory: `suite` becomes legal in any `.bynk` file, the emitter **strips**
  every `suite` from the production build, discovery scans the **whole** source tree,
  and the role-named `[paths] src`/`tests` manifest config collapses to a flat
  `include`/`exclude` pair. No new *language surface* (no keyword, no grammar rule) —
  the change is where test declarations may live and how the toolchain finds and
  builds them. Version → **v0.113**.
- **Realises:** the testing track's file/build model (`design/tracks/testing.md`,
  DECISION S) and, downstream, the in-browser track's single-file/shareable case — a
  playground snippet holding `commons` + `context` + `suite` in one file that ships
  its production halves and drops its tests. It follows directly from ADR 0144: the
  language already gates `expect` at the *block*, not the path, so making test-ness
  structural is the honest extension of what exists.
- **Relates:** ADR 0144 (one predicate surface — test-ness is already a declaration
  property at the predicate level; this extends it to the file/build level); ADR 0136
  (strip-only emission — the emitter already has a "what is emitted" discipline this
  slice narrows further, dropping test decls before lowering); ADR 0098
  (`--format json` — the runner's `path:line:col` locations, now project-relative
  without a synthetic `tests/` prefix). Absorbs the v0.9.1 split-paths model and the
  `#47` `.test.bynk` suffix, both retired here.
- **ADRs at implementation:** one, numbered at authoring (next free after 0146; do not
  pre-allocate): **S** — file & build model (structural test-ness; strip suites from
  the build; tree-wide discovery; flat `include`/`exclude`). It cites ADR 0144 and is
  the **load-bearing build-model ADR** the track flags.

## The model, before and after

Today a `.bynk` file is **one top-level unit** whose kind the parser reads from its
single declaration (`bynk-emit/src/project/discovery.rs:180-186`,
`parser::parse_unit` returns one `SourceUnit`). Test-ness is inferred from that
kind *plus* a separate `tests/` root: sources live under `[paths] src`, tests under
`[paths] tests`, a test file's path must mirror its target's name under `tests/`
(`bynk.project.inconsistent_test_path`), and a `.test.bynk` suffix is accepted as a
self-identifying convenience (`strip_test_infix`, `bynk-emit/src/project/paths.rs:167`).

After this slice:

```text
-- before (v0.9.1 split model)          -- after (v0.113 structural model)
my-project/                             my-project/
├── bynk.toml  [paths] src/tests        ├── bynk.toml  ([paths] optional)
├── src/counters.bynk                   ├── counters.bynk      -- commons + suite, or
└── tests/counters.test.bynk            └── counters_test.bynk -- a pure-suite file, anywhere
```

- **One extension, no filename marker.** Everything is `.bynk`; `.test.bynk` is gone.
  The suffix only ever restated what the `suite` keyword already says.
- **`suite` is legal in any file, next to source declarations.** A single *atomic*
  file may hold `commons` / `context` **and** a `suite` together — the shareable,
  single-file, playground case. Conventionally they are separate files; that is
  convention, not a rule.
- **The build strips test-only declarations.** `bynkc compile`/`build` skips every
  `suite` (and `suite integration`) wherever it sits — never type-checked for the
  build, never emitted. An atomic file ships its `commons`/`context` and drops its
  `suite`; a pure-suite file emits nothing. `bynkc test` compiles and runs them.
- **Discovery scans the whole tree** for `suite` declarations, not a designated
  folder.
- **`[paths]` names the tree, not roles.** `src`/`tests` collapse to a flat
  `include`/`exclude`:

      [paths]
      exclude = ["vendor"]     # include defaults to the project root

  `[paths]` is optional — `include` defaults to the project root minus the build-output
  cache — so a conventional `src/` + `tests/` layout needs no config at all; `tests/`
  survives as a pure human convention the build never mentions.

A consequence to accept, called out in the track: with roles gone from config,
placement is inert in both directions — a `context` misplaced under `tests/` now emits
like any other unit. A team wanting the old separation enforces it with a lint, not
the build.

## What is genuinely new — and where the risk is

The keyword and grammar are untouched. The work is in three places, one of which is
the crux:

1. **Multi-unit files (the crux).** Today `ParsedFile` holds exactly **one** `unit`
   and one `kind` (`discovery.rs:188-200`); `parser::parse_unit` returns one
   `SourceUnit`. The atomic file — `commons X { … }` **and** `suite X { … }` in one
   `.bynk` — is *not representable* in this model. Delivering "`suite` legal in any
   file" in its full form requires the parser to return **multiple top-level units per
   file** and the project model (`ParsedFile`, grouping at `project.rs:847-866`,
   consistency checks) to treat *declarations*, not files, as the unit of test-ness.
   **[DECISION — atomic file scope]** *Recommend* including multi-unit parsing in 1b:
   without it, "`suite` legal in any file" degrades to "a suite file may sit anywhere"
   (still one unit per file), and the in-browser single-file payoff — the decision's
   stated motivation — is not delivered. The fork the reviewer owns: full atomic
   support now (larger parser + project-model change) vs. ship structural discovery +
   strip + flat config with one-unit-per-file preserved and atomic files as a named
   follow-on. This is the single largest risk in the slice.

2. **Strip suites from the build.** Production emission already separates test groups
   from production groups (`project.rs:847-866`) and only emits the latter, so a
   *whole-file* suite is already absent from a `bynkc compile`. What is new is
   dropping a suite that shares a file with a source unit **before type-checking and
   lowering** — the `#[cfg(test)]`-with-the-keyword-as-the-cfg behaviour — so an
   atomic file's `commons` compiles for the build while its `suite` is skipped
   entirely (never checked, never emitted). Tied to the multi-unit change above.

3. **Flat discovery + config.** `discover_bynk_files` (`discovery.rs:203`) already
   walks the whole tree collecting `.bynk`; today the split-paths mode discovers `src`
   and `tests` roots separately (`project.rs:602-650`). Tree-wide discovery **unifies**
   these into one walk from `include` (default: project root) minus `exclude` and the
   build-output cache — the split disappears. `ProjectPaths { src, tests }`
   (`paths.rs:59-78`) and its parser (`read_project_paths`, `paths.rs:84-120`) become
   `{ include, exclude }`; `Roots::Split`/`Single` (`project.rs:195-234`) collapse to
   a single rooted-tree model.

## What changes, against the current code

The rename-free change is broad but concentrated in the project/build layer:

1. **Manifest** (`bynk-emit/src/project/paths.rs`) — `ProjectPaths { src, tests }` →
   `{ include: Vec<PathBuf>, exclude: Vec<PathBuf> }` (defaults: `include = [root]`,
   `exclude = [<out-cache>]`); `read_project_paths` (`:84-120`) parses `[paths]
   include`/`exclude` instead of `src`/`tests`. `ProjectPaths::conventional()`
   (`:72-77`) becomes the root-scanning default. Retire `strip_test_infix` (`:167-174`)
   with the `.test.bynk` suffix.
2. **Roots** (`bynk-emit/src/project.rs:195-234`) — collapse `Roots::Single`/`Split`
   and `resolve`/`tests_prefix` into one rooted-tree model; `source_path` becomes
   project-root-relative uniformly, so the v0.59 `tests_prefix` (`:223-233`) for
   `assert` location resolution is no longer needed (locations are already
   project-relative).
3. **Discovery** (`bynk-emit/src/project.rs:602-650`, `discovery.rs:203-235`) — one
   tree walk from `include`, skipping `exclude` and the output cache; drop the
   separate tests-root discovery and the `split_mode = src_root != tests_root` flag
   (`project.rs:2576`).
4. **Parser + project model** (per the crux decision) — `parse_unit` →
   multiple top-level `SourceUnit`s per file; `ParsedFile` carries a `Vec` of units
   (or discovery flat-maps one file into several `ParsedFile`-equivalents); grouping
   (`project.rs:847-866`) partitions **declarations** into production/test/integration
   sets regardless of file.
5. **Consistency** (`bynk-emit/src/project/consistency.rs`) — **retire**
   `check_test_path_alignment` (`:119-153`) and the `bynk.project.inconsistent_test_path`
   code: a suite no longer has a path-identity requirement (it targets by name, from
   anywhere). `check_path_name_alignment` (`:80-113`) and
   `check_directory_name_consistency` (`:11-65`) still govern **source** units — but
   an atomic file's source unit still anchors the path, and a pure-suite file's path
   is unconstrained.
6. **CLI** (`bynkc/src/main.rs`) — `project_options` (`:17-23`) no longer branches on
   `src/` presence for split-vs-single; the `rel.starts_with("tests/")` assumptions in
   `run_test` (`:120,123,132,154`) that route test emission and the integration-test
   overlay must key off **declaration kind**, not the `tests/` path prefix; the
   synthetic test-runner source path (`project.rs:2930`, `"tests/main.test.bynk"`) and
   the `out/tests/main.ts` layout (`main.rs:179,207`) are emitter-internal and can stay
   or be renamed, but must not depend on user files living under `tests/`.
7. **LSP manifest parser** (`bynk-lsp` — the manifest.md-declared "authoritative
   parser") — mirror the `[paths] include`/`exclude` change so the editor and compiler
   agree on the project's file set.

## Migration (clean-slate; codemod-assisted)

The split model is removed, not deprecated (the track's posture). A codemod:
- **renames `*.test.bynk` → `*.bynk`** and, where that would collide with a same-named
  source file in a single tree, leaves the file in place (test-ness is now structural,
  so co-location is legal);
- **rewrites `bynk.toml`**: drop `[paths] src`/`tests`; add `[paths] exclude` only if
  the old `tests`/`src` were non-conventional. A conventional `src/`+`tests/` project
  needs **no** config after migration (both are just directories under the default
  root scan).
- The ~35+ testing fixtures under `bynkc/tests/fixtures` move from `tests/…test.bynk`
  to plain `.bynk` and their expected `--format json` locations lose the `tests/`
  prefix (`test_json.rs:45,79-82,105`; `reference_index.rs:139,204,229`). Negative
  fixtures for `inconsistent_test_path` (`negative/103_inconsistent_test_path`) are
  removed; the code is retired.

## Risks

- **The multi-unit-per-file change is the slice.** Moving `ParsedFile` from one unit
  to many touches parsing, discovery, grouping, and every `matches!(pf.kind, …)` site
  (`project.rs:858,860,2368,2377`). Scoping it out (the DECISION fork) is the pressure
  valve if it over-runs — but then the headline "`suite` in any file" is only
  partially delivered.
- **`tests/`-prefix coupling in the driver.** `bynkc/src/main.rs` uses
  `rel.starts_with("tests/")` to route test emission and the integration-test Worker
  overlay (`:120-154`). These must switch to keying off declaration kind, or a
  structurally-placed suite under `src/` silently fails to run. High-value fixture:
  a suite in a source-named file that must still be discovered and run.
- **Build-strip must precede type-checking.** A dropped suite must not be
  type-checked, or an atomic file with a deliberately-broken-in-tests-only construct
  would fail the build. The strip has to happen at the grouping/checking boundary, not
  just at emission.
- **LSP/compiler manifest divergence.** Two `[paths]` parsers (compiler + LSP) must
  land together or the editor's project file-set drifts from the build's.
- **Drift guards.** `reference_index.rs`/`test_json.rs` hardcode `tests/…test.bynk`
  paths; the manifest/layout doc pages are drift-guarded. Regenerate and re-bless.
