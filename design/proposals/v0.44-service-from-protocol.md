# v0.44 — Service protocol on the header: `from <protocol>`, method-builders, and `QueueResult`

- **Status:** Draft (proposal). **Merging is approval to build.** *(Language-surface
  increment. Realises the design notes' *Services and Protocol Composition* protocol/transport
  *concept*, and deliberately **supersedes their provisional handler surface** — the notes' own
  examples use capitalised `from HTTP` and retain `on POST "/orders"` (verb + bare route on the
  handler), whereas this increment chooses lowercase sealed keywords and the method-builder, a
  revision the notes explicitly invite ("the surface … will be revisited"). Behaviour-changing: existing `on http`/`on cron`/`on queue`
  fixtures migrate in the same increment. No emitted-target change — the generated Worker
  topology is identical; only the source surface and the queue verdict type move.)*

## Context — two service paradigms, and a dishonest queue verdict

The design notes (*Services and Protocol Composition*) frame a service as conforming to a **protocol** — "`from HTTP`
is a conformance claim" in the Swift/Clojure sense — with the **transport** (which
broker/runtime delivers the bytes) a separate, platform-level concern. The implemented
language drifted away from that: a `service` is a bare handler container, and the
protocol lives **per-handler** in the keywords `on http` / `on cron` / `on queue`
(`syntactic-grammar.md §4.4.4/4.4.6/4.4.7`). Three consequences:

- **Two paradigms coexist.** The `on <kind>` family (protocol per handler) sits beside
  the design's `from <protocol>` family (protocol on the service). The yet-to-be-built
  Events surface (`from Events(...)`, `via schema(...)`) only fits the latter — its
  config is a parameterised subscription pattern that has nowhere to live on an
  `on`-handler. So the drift is not cosmetic; it blocks Events from sharing a shape.
- **HTTP carries two bare config tokens.** `on http GET "/notes/:id"` is the only
  handler with two adjacent literals after the kind (verb *and* route); every other
  handler has one. The rhythm "one config literal after the kind" is broken exactly once.
- **The queue verdict is overloaded and dishonest.** Queue handlers return
  `Effect[Result[(), E]]`, the *same type* as cron, yet `Err` means **retry** for queue
  and **log-and-drop** for cron. The two protocols' verdicts are distinguished only by the
  handler *keyword*, not by the type a reader sees — against *outcomes-are-typed-parts-of-
  contracts* (*Separations Enforced*) and the `HttpResult` precedent, where the verdict is a named sum.

A separate question fell out of the same thread: if the protocol moves onto the header as
`from <protocol>`, **where is `HTTP` defined**, and may a developer write `from Kafka`?
The answer (settled below) is that protocols are a closed, compiler-known set and Kafka/MQTT
are transports, not protocols — which means **no `protocol` declaration kind** this increment.

## Builds on (verified)

- **`HttpResult` is the verdict-as-sum precedent** (`reference/http.md`): a named sum whose
  variants carry the wire outcome (`Ok`/`Created`/`BadRequest(msg)`/…). `QueueResult` is the
  same move for queue. The `HttpResult.Ok` / `Result.Ok` ambiguity already has a resolution
  (`karn.types.ambiguous_constructor`, qualify the constructor) — `QueueResult` inherits it.
- **`http_method` already exists as a production** (`§4.4.5`) — the verb set
  `GET|POST|PUT|PATCH|DELETE`. The method-builder reuses it in config position
  (`GET("/route")`) rather than introducing new tokens.
- **The contract-mediated surface is `on call`** (`§4.4.3`), realised today as the emitter's
  **internal Service Binding** dispatch (`/_karn/call/`, `emitter/workers_entry.rs`/`workers.rs`) —
  *not* a wire protocol. (The design notes' `auth = Internal` is the doctrine for this surface; it
  has no grammar/checker presence yet, so nothing here leans on it.) `on call` is the natural home
  for the no-`from` default (see decisions). The restructure touches only http/cron/queue.
- **Adapters are the transport seam** (`reference/adapters.md`): the *outbound* host boundary,
  where `karn.cloudflare` already supplies `Queue`/`Kv`. This is where a Kafka/MQTT *transport*
  binds — confirming transports already have an openable home, so the *protocol* set need not open.
- **The closed-set-openable-later posture is established** — auth schemes ship closed "that can
  be opened later" (*Actor Declarations as Contracts*); platform capabilities ship sealed in the `karn` surface with
  uniform resolution. Protocols follow the same posture.

## Settled

- **▸ Protocol moves to the service header: `service <Name> from <protocol> { … }`.** One
  protocol per service; a service is a single protocol adapter again (*Services and Protocol
  Composition* / *Separations Enforced*). The protocol keyword is **lowercase** (`from http`,
  `from cron`, `from queue`) to signal it is a sealed keyword, not a resolvable name — consistent
  with today's `on http`. **The trade:** the protocol is no longer local to the handler line —
  after this change `on GET(…)` / `on schedule(…)` / `on message(…)` / `on call(…)` are
  heterogeneous, and a reader scanning a long body must look up to the `from` header to know the
  protocol. That is the deliberate DRY win, paid for with one indirection.
- **▸ Config splits by arity along one axis — *one binding* vs *many endpoints*.** This is not
  "uniformity" — it retires HTTP's two-literal anomaly but introduces a *cleaner irregularity*: a
  new conditional readers must learn, namely that **where config lives depends on protocol
  cardinality**. The predicate that fixes a protocol's cardinality: **a protocol that binds a single
  named transport resource** (a queue name, an events type) **is single-binding** — the binding goes
  on `from` and the service carries one handler; **a protocol that addresses many endpoints** (HTTP
  routes, cron schedules) **is multi-endpoint** — `from` carries the bare protocol and each handler
  its own endpoint. This per-protocol cardinality is a fixed, compiler-known fact, so a future
  protocol inherits a rule rather than guessing from precedent. Multi-endpoint protocols put the
  protocol bare on `from` and the endpoint on each handler; single-binding protocols put the binding
  on `from` and carry one handler:

  ```karn,ignore
  service api    from http                    { on GET("/notes/:id") (id: String) -> Effect[HttpResult[Note]] { … } }
  service sweeper from cron                    { on schedule("*/5 * * * *") (at: Int) -> Effect[Result[(), E]] { … } }
  service outbox  from queue("outbound-email") { on message(m: EmailJob) -> Effect[QueueResult] { … } }
  ```

  (Events, when built, is single-binding: `from events(PaymentConfirmed { … })`, one `on event`.)
- **▸ HTTP method-as-config-builder.** `GET("/route")` collapses verb+route into one config
  expression, restoring the one-config-per-handler rhythm and leaving a slot for per-route options
  later (`POST("/x", idempotent: true)`) without a grammar change. Method constructors on the way in
  mirror `HttpResult` status constructors on the way out. (Whether a method-builder *denotes a value*
  — a constructor of a route-config type — or is pure handler-config grammar is settled in the
  decisions below.)
- **▸ `QueueResult` replaces `Result[(), E]` for queue handlers.** A named verdict sum; the runtime
  routes on it instead of on the keyword. Minimum viable set `Ack | Retry` (see open decision for the
  exact shape). Cron and Events keep `Effect[Result[(), E]]`.
- **▸ Cron stays `Result[(), E]` — and the rule is recut around *agency*.** *A protocol earns a
  verdict type when the handler makes a **dispatch decision** the type should name.* A queue handler
  decides ack-vs-retry, and that decision is **independent of success/failure** — you can `Ack` a
  logical failure to drop a poison message, or `Retry` despite a partial success — so the verdict
  must be named. HTTP's status is the analogous decision (which response to send). Cron makes **no
  dispatch decision**: success/failure is the entire story, and the platform's logging of `Err` is
  transport behaviour, not a handler choice — so `Result[(), E]` is exactly right, and a `CronResult`
  would name a decision that doesn't exist. *(The earlier "protocol-specific outcomes" framing was
  weaker: "failed-and-logged" is itself a protocol-specific reading of `Err`, so a sceptic could
  argue cron also earns a verdict — agency forecloses that.)* The return column reads
  `HttpResult[T]` / `QueueResult` / `Result[(), E]` *by principle*, and the rule generalises to
  Events/Alarm later (a verdict iff the handler dispatches).
- **▸ Protocols stay a closed set — no `protocol` declaration kind in v0.44.** Adapters are
  *outbound* (the app calls a set of functions; openness earns its keep because libraries are
  unbounded). A protocol is *inbound* — a driver that owns dispatch, lifecycle, boundary
  validation, actor verification, and atomicity-at-the-edge — far too sharp a surface to hand to a
  TS binding, and the five service-tier shapes are a near-complete cover. The thing that genuinely
  varies (transport/codec) is already openable via adapters. **Reserve the `protocol` keyword** so
  the door stays open; if ever opened, `protocol` should pair a baked-in lifecycle shape with an
  adapter-supplied transport, not author a driver from scratch.
- **▸ `from kafka` / `from mqtt` are rejected with a fix-it.** They are transports, not protocols
  (`from queue` or `from events`, with the broker bound at the transport/platform layer).

## Decisions (open — settle at review)

**[DECISION] `QueueResult` variant set.** *Rec:* ship `Ack | Retry`, with `Retry` carrying an
**optional reason** for observability (`Retry` / `Retry(reason: String)` via an `Option`, or two
variants). Defer `RetryAfter(Duration)` (needs a duration type) and an explicit `DeadLetter` (DLQ is
configured outside Karn today; redelivery already lands there on persistent failure). *Alternative:*
bare `Ack | Retry` only — smallest surface, add reason later — but a silent retry with no logged cause
is the first thing operators ask for, so the reason channel is cheap insurance.

**[DECISION] No-`from` services, the `call` surface, and the wire/`call` exclusion.** A `service` with
only `on call` handlers is the contract-mediated internal-RPC surface, which is not a wire protocol.
*Rec:* **bare `service Foo { on call … }` stays the default** (no header) — demanding `from calls`/`from
contract` for the architectural default is ceremony. This makes the exclusion **explicit and
intentional, not an emergent side-effect of the mixing rule**: a `from`-less service admits *only*
`on call`; a `from <wire>` service admits *only* that protocol's handler form — so a single service
**may not** expose both an internal-RPC surface (`on call`) and a wire surface (`on http`) together
(`karn.service.mixed_protocols`). Rationale: a service is one protocol adapter; an internal escape
hatch on an HTTP service is two adapters wearing one name, and the split keeps the per-context diagram
("services in, agents below") honest. This matches current practice — no in-repo fixture mixes them;
fixture 135 deliberately splits `service create { on call }` from `service api { on http }`, its own
comment noting "a service with only `on http` handlers, no `on call`". Stated as a well-formedness rule
so the first person wanting an HTTP-plus-RPC service reads a deliberate "no", not an oversight.
**Fix-it for the header-less cliff:** because the default is the *absence* of a header, a forgotten
`from http` silently makes a service `call`-only, and its `on GET(…)` then fails as "call-only admits
only `on call`" rather than "you forgot `from http`". So a `from`-less service that contains
non-`call` handlers is diagnosed with a fix-it suggesting the matching `from <protocol>`
(`karn.service.missing_from`), never the bare call-only rule.
*Alternative:* require `from internal` for symmetry (every service has a header) — cleaner table, more
boilerplate on the commonest case. Recommend bare.

**[DECISION] Cron handler config form.** With `from cron` on the header, the schedule is all the handler
needs. *Rec:* `on schedule("*/5 * * * *")` — a builder, parallel to `GET("/route")`, keeping the
`on <builder>(…)` rhythm consistent across multi-endpoint protocols. *Alternative:* bare `on "*/5 * * * *"`
(the kind is already on `from`, so the literal is unambiguous) — terser, but breaks the builder symmetry
and reads oddly next to `on GET(...)`. Recommend the builder.

**[DECISION] Migration posture for the old surface.** *Rec:* **hard cutover** — pre-1.0, spec-first;
the increment rewrites all in-repo fixtures/examples and the old `on http`/`on cron`/`on queue` forms are
removed, not deprecated. A one-shot `karn-fmt` migration (or a `karnc fix` codemod) rewrites the three
forms mechanically. **Caveat — queue is not purely mechanical:** the codemod rewrites *headers*
(`on queue "n"` → `from queue("n") { on message }`), but queue handler *bodies* change type
(`Result[(), E]` → `QueueResult`), so `Ok(())`/`Err(e)` must become `Ack`/`Retry` — a *semantic*
mapping (only a human knows whether a given `Err` meant retry or drop-this-poison-message). Budget
per-handler review of queue verdict bodies; a spot-check is insufficient. *Alternative:* a deprecation window accepting both surfaces — softer for any external
code, but doubles the grammar and the diagnostic surface for a pre-1.0 language with no stability promise.
Recommend hard cutover + codemod.

**[DECISION] Internal protocol-descriptor refactor — now or later.** *Rec:* **now**, since the increment
already rewrites all three handler productions: replace the near-duplicate http/cron/queue parser+checker
arms with a single protocol-descriptor representation (shape constraints + actor defaults + result type +
emitter), sealed but uniform — so a future opening is new *surface* only, not a re-architecture (mirrors the
`karn`-surface treatment of capabilities). **Caution:** the descriptor must encode not just config-shape,
return-type, and actor defaults but also the **per-protocol param obligations** the three
`validate_*_handler` functions enforce today — queue's *message mandatory* (`karn.queue.bad_params`),
cron's *`at` optional* (`karn.cron.bad_params`), HTTP's *body optional, never on GET/DELETE*
(`karn.http.body_on_get_or_delete`) and *path-param stringy* (`karn.http.path_param_not_stringy`). These
are easy to drop on the floor during consolidation; pin each with the existing negative fixtures so the
descriptor reproduces every check. *Alternative:* keep three hand-written arms and defer the refactor —
less churn this increment, but leaves the triplication the `from` restructure is meant to retire.

**[DECISION] Are method-builders denotable values or pure handler-config grammar?** `GET("/route")`
(and the deferred `POST("/x", idempotent: true)`) must be one or the other, and the choice propagates
to the descriptor and the LSP. *Rec:* **pure handler-config grammar** — a method-builder is admitted
only in the handler-config slot, denotes no value, and lowers straight into the protocol descriptor.
This keeps the parse-ambiguity fix simple (a bare `GET(...)` in expression position stays undefined),
needs no `Route`/`Endpoint` type in the surface, and per-route options are just more config grammar;
the LSP tokenises `GET`/`schedule`/`message` as a builder, not a constructor. *Alternative:* make them
real constructors of a route-config record (so `POST("/x", idempotent: true)` is record construction
with typed fields) — more uniform with `HttpResult` constructors and more extensible, at the cost of a
denotable boundary type and a value-position story. Recommend pure grammar; revisit if per-route
options grow into a structured config worth typing.

## Spec delta

- **`syntactic-grammar.md §4.4`.** Rewrite `service_decl` to carry an optional `from`-clause
  (`from <protocol>` or `from <protocol>(<binding-config>)`). Replace `http_handler`/`cron_handler`/
  `queue_handler` with the header-relative handler forms (`on <Method>(route)`, `on schedule(expr)`,
  `on message(param)`); `http_method` (§4.4.5) is retained, now used in builder position. `call_handler`
  (§4.4.3) unchanged; document it as the no-`from` default surface.
- **`static-semantics.md`.** New well-formedness rules: one-protocol-per-service; `from`-less ⇒ `call`-only;
  unknown protocol after `from` (with the Kafka/MQTT transport fix-it); the per-protocol handler-shape and
  return-type checks re-expressed against the protocol descriptor; `QueueResult` as the required queue return.
- **`runtime-library.md`.** Add the `QueueResult` sum to the prelude (alongside `HttpResult`). State
  explicitly that **`QueueResult` is non-generic** (no type parameter), unlike `HttpResult[T]`, so
  nobody mirrors the HTTP generic by reflex — but **non-generic is not payload-free**: the `Retry`
  variant may still carry a reason string (per the variant-set decision); it simply takes no type
  argument.
- **`lexical-grammar.md`.** Reserve `from` in service-header position and reserve the `protocol` keyword
  (no production yet).
- **`emission.md`.** No target-output change — `fetch`/`scheduled`/`queue` entry points and the
  `wrangler.toml` stanzas are byte-identical; only the source→descriptor→dispatch wiring is re-described.
  `QueueResult.Ack`→ack / `Retry`→retry replaces the `Ok`/`Err` routing prose.
- **`diagnostics.md` + guides.** Rename the `karn.http.*` / `karn.cron.*` / `karn.queue.*` families to the
  header-relative forms; add `karn.service.{mixed_protocols,unknown_protocol,missing_from}` and
  `karn.queue.return_not_queue_result`. Rewrite `guides/entry-points/{http,cron,queue}.md` and
  `reference/{http,cron,queue}.md` to the new surface.
- **LSP / tooling.** Introducing `from`, demoting the protocol keywords, and promoting `GET(…)` into a
  builder-call position move these through the highlighter and the LSP providers. Confirm whether the
  **semantic-token legend** changes (`from` likely needs theming as a keyword; `GET`/`schedule`/`message`
  shift token kind from `http_method`/keyword to a builder/function position), and re-check the
  **binding/member indices**, **inlay hints**, and **semantic-token** providers that currently key off the
  `http`/`cron`/`queue` keyword tokens. The existing legend-drift guard (ADR 0058) should catch a missed
  legend update, but scope it now rather than discover it in CI.

## Tests

  does the bulk; spot-check the HTTP method-builder and the `from queue("name")` binding by hand).

- **`QueueResult` routing fixtures:** `Ack` acks, `Retry`(/`Retry(reason)`) retries; reason surfaces in the

- **Golden emission unchanged** for the migrated HTTP/cron cases (same Worker output proves a pure surface move);

- **`karn-fmt` round-trip** over the new forms; tree-sitter + TextMate grammars updated and pinned.
- **LSP providers:** semantic-token snapshot over a `from`-header service (legend + token kinds for

## Diagnostics / Risks

- **Fixture-migration blast radius.** The largest risk is mechanical churn across many fixtures. Mitigation:
  the codemod is exercised on the repo itself and golden emission must stay byte-identical for HTTP/cron — any
  diff there is a migration defect, not an update.
- **`QueueResult.Ack`/`Retry` constructor ambiguity.** Like `HttpResult.Ok`, bare `Ack`/`Retry` could collide
  in scopes that also see other sums. The existing `karn.types.ambiguous_constructor` path covers it; pick variant
  names (`Ack`, `Retry`) that don't clash with `Result`/`HttpResult`.
- **Method-builder parse ambiguity.** `GET("/x")` must parse as a handler-config builder, not a value-level call,
  in `on`-position only. Confine the method-builder to the handler-config grammar slot; a bare `GET(...)` in
  expression position remains undefined. Pin with a parser fixture.
- **`from` keyword introduction.** `from` is currently only prose/operator-adjacent; introducing it in
  service-header position must not collide with identifiers elsewhere. Reserve it narrowly (header position) and
  add a lexer fixture.
- **Events scope creep.** Events shares the `from` shape but is **not** built here. Keep it out of the grammar
  this increment (reserve nothing Events-specific beyond what `from(...)` already allows); the proposal only
  guarantees Events *fits* the rule.

## New decision records this increment adds

- **ADR 0077 — Service protocol on the header.** `from <protocol>`, one protocol per service, lowercase
  keyword, HTTP method-builders, `from`-less ⇒ `call` default. Supersedes the per-handler `on <kind>` surface.
- **ADR 0078 — `QueueResult` typed verdict.** Named, non-generic ack/retry sum; the **agency rule** — a
  protocol earns a verdict type iff the handler makes a dispatch decision the type should name — and why
  cron (no dispatch decision) keeps `Result[(), E]`.
- **ADR 0079 — Protocols are a closed set.** No `protocol` declaration kind; reserve the keyword; protocol/transport
  split (Kafka/MQTT are transports); the internal protocol-descriptor refactor as the openable-later seam.

## Out of scope (→ later / independent)

- **Events / Alarm / WebSocket entry points.** Events fits `from events(...)` and is the obvious next increment;
  Alarm and WebSocket (*Services and Protocol Composition*) remain unbuilt. This increment only ensures the
  header shape generalises to them.
- **A `protocol` declaration kind.** Deferred behind a reserved keyword; the descriptor refactor is its seam.
- **Transport selection surface.** Which broker a `from queue` binds to (Cloudflare Queues, SQS, Kafka, MQTT) is a
  platform/adapter concern, untouched here.
- **`RetryAfter` / explicit `DeadLetter`.** Deferred with the duration type and any in-language DLQ policy.

## Definition of done

`service <Name> from <protocol> { … }` replaces the per-handler `on http`/`on cron`/`on queue` surface; HTTP
handlers read `on <Method>("route")`, cron `on schedule("expr")`, queue `from queue("name") { on message(…) }`;
`from`-less services hold `on call` only; mixing protocols, unknown protocols (with the transport fix-it), and a
non-`QueueResult` queue return are diagnosed; `QueueResult` lands in the prelude and the runtime routes ack/retry on
it; the three handler productions collapse to one protocol descriptor; the `protocol` keyword is reserved; all
fixtures/examples migrate (codemod + spot-checks) and HTTP/cron golden emission is byte-identical; `karn-fmt`,
tree-sitter, and TextMate grammars round-trip the new forms; ADRs 0077–0079 land; the spec chapters, diagnostics,
guides, and changelog are updated; `cargo build`/`test`/clippy and CI stay green; `scripts/bump-version.sh` sets
v0.44. The implementing PR(s) **delete this file**.
