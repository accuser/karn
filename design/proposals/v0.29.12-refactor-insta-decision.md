# v0.29.12 — Refactor track, increment 12: decide `insta` — adopt or drop

- **Status:** Draft (proposal). **Merging is approval to build.** *(Refactor track item 10 — a quick
  **standalone decision**, not a structural change. Independent of every other increment. The outcome
  is either a one-line dependency removal or a scoped snapshot-test adoption.)*

## Context — a declared dev-dependency that nothing uses

- `insta` is a declared **workspace dev-dependency** (`insta.workspace = true`, `karnc/Cargo.toml:28`)
  but is **entirely unused**: no `assert_snapshot!`/`assert_debug_snapshot!`, no `.snap` files anywhere
  (verified). A hand-rolled golden-fixture harness does the equivalent today.
- An unused dependency is dead weight: it carries a compile/build cost, a supply-chain surface, and an
  implied-but-false "we snapshot-test" signal. Either use it or remove it.

## Builds on (verified)

- **The bespoke golden harness already works — now demonstrably.** Every landed refactor increment
  (29.1–29.11, plus the `tsc_verify` strict-compile gate) used "golden fixtures pass unedited" as its
  behaviour-preservation gate and it held throughout — the mechanism isn't just load-bearing, the whole
  track is proof it's sufficient. `insta` was never reached for once across that work.
- **`insta`'s genuine edge is emitter TS output** — reviewing a snapshot diff (`cargo insta review`) is
  cheaper than maintaining a bespoke differ for large generated-text outputs, which is exactly what the
  emitter produces.

## Settled

- **▸ Decide, don't drift.** This increment's deliverable is a *resolved* dependency status — adopted
  with at least one real use, or removed — not a deferral.

## Decisions (open — settle at review)

**[DECISION] Drop now, or adopt for emitter snapshots?** *Rec: **drop now**.*
- **Drop (recommended):** remove `insta` from the workspace dev-dependencies. Rationale: the bespoke
  harness is proven and **the refactor track depends on fixture-mechanism stability** ("pass
  unedited" is the per-step gate) — changing the verification substrate *mid-track* adds risk for no
  track benefit. Removal is a one-line change with zero behaviour impact. Adoption can be revisited as
  its own post-track increment if snapshot ergonomics are wanted.
- **Adopt (alternative):** introduce `insta` for **new** emitter TS tests only, leaving existing
  golden fixtures untouched, so the per-step "unedited" gate still holds against the old harness while
  new coverage uses snapshots. Choose this only if the team wants snapshot review enough to run two
  mechanisms during the track.

*Honest framing:* this is a genuine judgement call, not a forced hand — the recommendation is "drop now
to keep the track's substrate stable," **not** "snapshots are bad." If adopted, scope it tightly
(emitter output, new tests) so it never churns an existing fixture.

**[DECISION] If dropped, is anything lost?** *Rec:* nothing in use — confirm with a final grep for
`insta`/`assert_snapshot`/`.snap` at implementation; if still zero, removal is safe and total.

## Diagnostics / Risks

- **Adopting mid-track is the only real risk** — two verification mechanisms, and a temptation to
  migrate existing fixtures (which would break the "pass unedited" gate the other increments rely on).
  The drop recommendation avoids this entirely; if adopting, the **"new tests only"** constraint is
  mandatory, not advisory.
- **Trivial to reverse.** Whichever way it goes, the change is small and easily revisited — this is the
  lowest-stakes increment in the track.
