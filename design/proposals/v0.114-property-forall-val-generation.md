# v0.114 — `property`/`for all`, `Val[T]`, and generation over refinement domains (testing track, slice 2)

- **Scope:** a single-increment **language + runtime** change — the third testing-track
  slice and its first *generative* rung. It adds `property "…" { for all … }` (a
  generative test whose subjects the runner produces), a generator that draws **valid
  inhabitants** of a type from its refinements, a repro **seed** and **shrinking** on
  failure, and `Val[T]` / `Val[T](pin)` for value fabrication — **retiring `Mock[T]`**.
  Version → **v0.114**.
- **Realises:** the testing track's Pillar II — *subjects are supplied or generated*
  (`design/tracks/testing.md`) — at the value/domain level. A `case` supplies subjects
  (pinned `Val[T](v)`, `expect`); a `property` generates them (`for all x: T`), so a
  reader sees at a glance whether generation, with its seed and cost, is in play. It is
  the same one predicate surface (ADR 0144): `for all … { expect … }` uses the invariant
  predicate verbatim, now over generated inputs.
- **Relates:** ADR 0144 (one predicate surface — a property's body is the invariant
  predicate); ADR 0146 (`suite`/`case` — `property` is the third vocabulary word, the
  generative sibling of `case`); the retired `Mock[T]` surface (v0.9.4) whose value
  fabricator this replaces; ADR 0098 (`--format json` — the runner document, extended
  with a property's case-count/seed on failure). It does **not** touch the `mocks`
  collaborator block (that retires with `provides` in slice 6), so the word "mock" is
  *not yet* fully gone — only the `Mock[T]` value fabricator leaves here.
- **ADRs at implementation:** two, numbered at authoring (next free after 0147; do not
  pre-allocate): **C** — value fabrication is `Val[T]`, `Mock[T]` retired; **P** — a
  generated subject is a *valid inhabitant* of its type (the generator **is** the
  type's inhabitant space), and agents are the exception (generate handler *sequences*,
  not fabricated states — deferred to slice 7). Both cite ADR 0144; **P** is the
  load-bearing ADR that *defines generation* for the whole track.

## The surface

A `property` is a generative sibling of `case`, legal in the same `suite`. Its body is
`for all <bindings> [where <pred>] { … expect … }` — the invariant predicate over
inputs the runner produces:

```bynk
suite money {
  case "a tenth off a tenner" {
    expect discount(1000, 10) == 900          -- supplied: a pinned witness
  }

  property "more discount, never a higher price" {
    for all p: Price, a: Percent, b: Percent where a <= b {
      expect discount(p, b) <= discount(p, a) -- relational: no single `ensures` says this
    }
  }
}
```

- **`for all x: T`** binds `x` to a *generated* inhabitant of `T`. Multiple bindings
  are a comma list; an optional **`where <pred>`** (the same pure predicate grammar)
  filters generated tuples before the body runs.
- The runner generates each `T` from its **refinement domain** (below), draws a bounded
  number of cases per property (a configurable default), and **includes the boundary
  values** so edge cases are not left to chance.
- The body is one or more `expect`s — the one predicate surface, unchanged.

**`Val[T]` replaces `Mock[T]`** for the *supplied* form: `Val[T]` fabricates a valid
inhabitant (the boundary-inclusive seed the runner also draws from), `Val[T](v)` pins a
specific one, checked against the refinement exactly as `Mock[T]`/`Mock[T](v)` were:

```bynk
let q = Val[Quantity]        -- a valid inhabitant (was Mock[Quantity])
let n = Val[Quantity](50)    -- pinned, refinement-checked
```

## Generation: a type is its own inhabitant space (DECISION P)

The generator for `T` produces **only values that satisfy `T`'s refinements** — it *is*
the type's inhabitant space, so a generated subject is valid by construction:

| `T` | Generated inhabitants (boundary-inclusive) |
|---|---|
| `Int where Positive` | `1`, small positives, `Int` max boundary |
| `Int where NonNegative` | `0`, small non-negatives |
| `Int where InRange(a, b)` | `a`, `b`, and interior values |
| `String where MinLength(k)` / `Length(k)` | strings at and above the length bound |
| `String where Matches(re)` | **out of scope** — no refinement-driven generator for an arbitrary regex; a bare `for all`/`Val` of a `Matches` type is rejected (`bynk.val.needs_pin`, the renamed `bynk.mock.needs_pin`), pin instead |
| sum | each variant (payloads recursively generated) |
| record | every field generated |
| opaque | over the base type's domain via `.unsafe` |

**Restating a refinement is redundant.** A property whose predicate merely re-checks a
refinement — `for all q: Quantity { expect q > 0 }` when `Quantity` is `Int where
Positive` — asserts a guarantee the type already gives. The runner flags it
(`bynk.property.restates_refinement`, a **conservative** check — see Risks). Domain
properties earn their keep by asserting **behaviour over valid inputs**.

**Agents are the exception (DECISION P).** Fabricating an agent *state* that satisfies
every invariant does not make it reachable, so `for all`/`Val` over an **agent** is
rejected (`bynk.val.agent_not_generable`); behavioural agent generation is
handler-*sequence* generation — the history rung, **slice 7**. Snapshot/step invariants
run at the real commit boundary, never over a fabricated state.

## Seeds, shrinking, and reproduction

On failure a property reports the **case count**, the **seed**, and a **shrunk**
counterexample, with a copy-paste reproduce line:

```text
commerce.money › more discount, never a higher price
  property failed after 41 cases (seed 0x5f3a)
  shrunk counterexample:  p = 100, a = 10, b = 20
  expected:  discount(p, b) <= discount(p, a)
  actual:    discount(100, 20) == 95  >  discount(100, 10) == 90
  reproduce: bynkc test commerce/money.bynk --seed 0x5f3a
```

- **Seed.** Each run draws one root seed (printed only on failure); every property's
  generation is derived deterministically from it, so `--seed <hex>` reproduces a run
  exactly. A new `bynkc test --seed` flag threads the seed to the runner.
- **Shrinking.** On a counterexample the runner minimises each generated input toward
  its boundary (integers toward the refinement floor, strings toward minimum length,
  sums toward the first variant) while the predicate still fails, and reports the
  shrunk tuple — reusing the slice-1a expected-vs-actual renderer for the `expect`.
- **Determinism** is the generator's, not the world's: a `property` body stays pure
  (the one predicate surface), so a fixed seed fully determines the case.

## What changes, against the current code

The `Val` rename is mechanical; `property`/`for all` and the generator/runner are the
new surface.

1. **Lexer / keywords** — `property`, `for`, `all` become keywords (`for all` is the
   binder). `Val` is recognised in expression position exactly as `Mock` is today (an
   identifier + `[` lookahead, not a keyword), so `Mock`→`Val` is a lookahead rename.
2. **AST** — add `ExprKind`/decl nodes for a `PropertyDecl` (name + `for all` bindings +
   optional `where` + body) as a `suite` body item beside `Case`; rename
   `ExprKind::Mock`/`MockExpr` → `Val`/`ValExpr` (`bynk-syntax/src/ast.rs`).
3. **Parser** — `parse_mock_expr` (`parser/expressions.rs:357`) → `parse_val_expr`
   (trigger `Val` + `[`); a new `parse_property` in the suite-body parser
   (`parser/declarations.rs`), reusing the predicate/`where` parsers.
4. **Checker** — `Val[T]` fabrication/pin checks reuse the `Mock[T]` logic
   (`bynk-check` refinements/expressions) renamed; a `property`'s `for all` bindings
   type each `x: T`, `where`/`expect` type as pure `Bool`; the generability check
   (refinement-driven, agent rejection, `Matches` needs-pin) and the
   restates-refinement flag are new.
5. **Diagnostics** — `bynk.mock.*` (`diagnostics.rs:578-628`) → `bynk.val.*`
   (relocated for sort order); add `bynk.property.*` (`restates_refinement`,
   `empty_forall`, …) and `bynk.val.agent_not_generable`.
6. **Emitter / runner** (`bynk-emit/src/project/tests_emit.rs`) — the real new work: a
   per-type **generator** derived from its refinement, a seeded PRNG, the case loop,
   **shrinking**, and the property failure record (count + seed + shrunk tuple) threaded
   into the NDJSON `message`/`--format json`. `Mock[T]` lowering → `Val[T]`.
7. **CLI** — `bynkc test --seed <hex>` (`bynkc/src/cli.rs` + `main.rs`), forwarded to
   the runner; the reproduce line prints the resolved seed.
8. **tree-sitter** — `mock_expr`/`mock_arg` → `val_expr`/`val_arg`; add `property` /
   `for_all` rules; keyword list + `highlights.scm`; re-bless `grammar.json` and the
   corpus.

## Migration (clean-slate; codemod-assisted)

`Mock[T]` is removed, not deprecated. A codemod rewrites `Mock[` → `Val[` (both bare
and pinned). The ~6 `Mock[T]` fixtures and their expected outputs move to `Val`;
`bynk.mock.*` negative fixtures re-point to `bynk.val.*`. New fixtures cover: a passing
`property` over a refined domain, a failing property with a shrunk counterexample and
fixed `--seed`, a `restates_refinement` rejection, and an `agent_not_generable`
rejection. The `mocks` collaborator block is **untouched** (it retires in slice 6).

## Risks

- **The generator + shrinking is the slice.** A per-type generator over the refinement
  domains, a seeded PRNG, boundary inclusion, and shrinking are real runtime code in
  `tests_emit`. Bounding it to the refinement kinds `Val[T]` already fabricates keeps
  it tractable; a custom-distribution `Gen[T]`/`Arb[T]` is a reserved future, unneeded
  while a type is its own inhabitant space (DECISION C).
- **`restates_refinement` must be conservative.** Flagging a property that merely
  re-checks a refinement risks false positives on subtle predicates; the check must
  only fire when the predicate is *syntactically* the refinement over the bound var,
  never guess. Under-flagging is acceptable; over-flagging is not.
- **Determinism of reproduction.** `--seed` must reproduce byte-for-byte, so all
  generation derives from the one root seed via a documented, stable derivation; any
  nondeterminism (map iteration order in generation) would break repro.
- **`Matches` generation.** No refinement-driven generator for an arbitrary regex —
  keep the `needs_pin` rejection `Mock[T]` already has; a regex-aware generator is out
  of scope.
- **Agent generation false positives.** Rejecting `for all`/`Val` over an agent up
  front (DECISION P) avoids valid-but-unreachable states eroding trust; the real
  behavioural path (handler sequences) is slice 7, not this slice.
