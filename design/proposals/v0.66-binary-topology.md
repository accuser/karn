# v0.66 — Binary topology: the `bynk` driver links the pipeline; a thin `bynkc` survives (crate-decomposition slice 7 — the finale)

- **Track:** [crate-decomposition](../tracks/crate-decomposition.md), slice 7 —
  **the last slice; it retires the track.**
- **Realises ADRs:** [0101](../decisions/0101-front-end-links-pipeline-binary-topology.md)
  (the front-end links the pipeline; a thin `bynkc` survives), [0100](../decisions/0100-structured-data-rendering-separation.md)
  (the driver finally adopts `bynk-render`).
- **Amends ADR:** [0084](../decisions/0084-doctor-output-exit-contract.md) — the
  `doctor` compile/skew contract (the one decision worth sign-off; below).
- **Supersedes the [[0083]] note** that "the driver does not link the pipeline"
  (already partly untrue; ADR 0101 began this).
- **Posture:** **behaviour-preserving for the developer**, but the *mechanism*
  under `bynk dev` changes (shell → in-process). No language surface.

## Why now / why this shape

After slices 1–6 the compiler is clean leaf libraries, and `bynkc` is already the
thin CLI over them. The remaining question is binary topology: **does `bynk` shell
`bynkc` or link it?** The decision (ADR 0101, confirmed by the project owner's
framing — *most developers use `bynk`; `bynkc` is a background tool*): **`bynk`
links the pipeline and becomes the self-contained developer front-end; `bynkc`
survives as the minimal background/CI binary.**

The win is concrete: today a fresh `cargo install bynk` does **not** give a working
`bynk dev` — it shells `bynkc`, which must also be installed at a matching version
(the entire reason `doctor` carries a `bynkc`-resolution + version-skew check).
Linking the compiler removes that failure mode: `cargo install bynk` just works, no
separate binary, no `PATH` resolution, no skew on the primary path.

## What this slice does

1. **`bynk dev` compiles in-process.** Replace the `Command::new(bynkc).arg("compile")…`
   shell-out (`dev.rs`) with `bynk_emit::compile_project(&options)` (target
   `Workers`, output the managed `.bynk/dev/` dir), then **write the output** via a
   new shared `bynk_emit::write_output(&ProjectOutput, &dir)` helper (factored from
   the ~6-line loop `bynkc`'s `main.rs` already duplicates 3×).
2. **The driver adopts `bynk-render`.** On a `ProjectFailure`, `bynk dev` renders
   in-process via `bynk-render` — `bynk` gets the front-end's flatten-then-delegate
   (the slice-6 pattern: `ProjectFailure` flattening stays in the front-end, the
   per-error rendering delegates to `bynk-render`). Slice 6 built `bynk-render` for
   exactly this second consumer.
3. **`bynk` drops the `bynkc` crate dependency.** Its leaf-helper imports re-point:
   `bynkc::lexer::tokenize` → `bynk_syntax::lexer`, `bynkc::read_project_paths` →
   `bynk_emit::`, `bynkc::NODE_MAJOR_FLOOR` → relocated to `bynk_emit` (see
   DECISION 2). `bynk`'s deps become `bynk-emit` + `bynk-syntax` + `bynk-render`
   (parallel to how `bynk-lsp` dropped `bynkc` in slice 5).
4. **`bynkc` survives unchanged in role** — the thin `compile`/`check`/`fmt`/`test`
   binary for CI, `cargo install bynkc`, and build determinism. (It is *already*
   this after slices 1–6; this slice doesn't slim it further, it just stops `bynk`
   depending on it.)
5. **The v0.59 `bynk test` deferral re-mechanises** from "shell `bynkc test
   --format json`" to "link `bynk-emit`'s test emission + run node" — a **note
   change only** (`bynk test` is still unbuilt).
6. **Retire the track** on merge.

## Decisions

**[DECISION 1 — sign-off] Amend ADR 0084: the `doctor` compile/skew contract.**
Today `doctor`'s `Compile` capability *is* "the external `bynkc` resolves + isn't
majorly skewed." Once `bynk` compiles in-process that is no longer how the driver
compiles, so:
- The **`Compile` capability becomes "in-process — always available"** (the
  compiler is built into `bynk`; it cannot be missing or skewed against itself). It
  is no longer a `Fail` for `bynk dev`.
- The **external-`bynkc` resolution + skew check narrows to the `BYNK_BYNKC`
  override path** — a power user pointing `bynk` at an *external* compiler is the
  only way a second compiler (and thus skew) enters the picture; only then does
  `doctor` resolve + skew-check it. With no override, there is no second compiler to
  check.
- `doctor` still gates the genuinely-external tools it shells: Node, `tsc`/`tsx`
  (`bynk test`), `wrangler` (dev/deploy). Those are unchanged.

Net: `doctor` gets **simpler and more honest** — it stops reporting a skew failure
mode that no longer exists for normal use. *Recommendation: accept the amendment.*
*Alternative:* keep `doctor` reporting `bynkc`-presence informationally even
without an override — rejected as noise (the driver doesn't use it; `cargo install
bynkc` users check `bynkc` by running it).

**[DECISION 2] `NODE_MAJOR_FLOOR` relocates to `bynk-emit` — recommend yes.** It is
the single source of truth for the Node runtime floor, used by `bynkc` (the
`node`-platform binding), the emitted code's target, and `bynk doctor`. With `bynk`
dropping the `bynkc` crate dep it needs a shared home below both binaries;
`bynk-emit` (which emits the TS that runs on Node) is the natural owner. `bynkc`
re-exports it. Same shape as `line_col` (slice 4) / `Severity` (slice 5).

**[DECISION 3] Version — recommend a minor `v0.66`. No new crate → no
bootstrap-seed.** Unlike slices 1, 3–6, this adds **no** published crate (it
re-points deps and relocates a const). Per-increment minor cadence (v0.65 → v0.66).

## Risks & mitigations

- **`bynk dev`'s behaviour must be preserved** (same build output, same exit codes,
  same error display). Mitigated by the `bynk` dev integration tests / e2e: the
  emitted `.bynk/dev/` tree and the failure rendering must match the shelled path.
  The compile/write/render logic is the *same code* `bynkc`'s `main.rs` runs —
  `compile_project` + `write_output` + `bynk-render` — so output is identical by
  construction.
- **`cargo install bynk` now links the compiler libs.** This is the *intended*
  shape (the front-door tool contains the compiler, like `cargo` contains the
  builder), not a regret — called out so it is a chosen consequence.
- **The `doctor` amendment is user-visible** (DECISION 1) — hence the sign-off.
  Mitigated by `doctor`'s own output/exit tests, updated to the narrowed contract.

  dev + doctor suites pass with the new mechanism.
