# v0.99 — Agents own their capabilities: encapsulation at the agent boundary + a generic requirement-provenance surface

- **Scope:** a language-semantics decision (an agent method **owns and provisions its own
  capabilities**; a call across the agent boundary forwards nothing) **plus** the tooling
  that makes a capability requirement *discoverable* (a generic provenance model, surfaced
  via LSP hover, CodeLens, and a materializable `given` inlay hint). Touches the **checker**,
  the **bundle emitter**, and **bynk-lsp** + **vscode-bynk**.
- **Realises / supersedes:** resolves the soundness gap in the v0.98 line where a
  capability-carrying agent method called from a capability-free handler emitted TypeScript
  that `tsc` rejects. The interim diagnostic `bynk.given.unsatisfied_capability` is
  **retired** here (see [DECISION F]). The cross-boundary half of the open *"Inference of
  `given`"* question (`bynk-type-system.md` §2.6.4) is **answered by choosing encapsulation,
  not propagation**; §2.6.4's intra-context helper-inference half is out of scope.
- **Citations corrected (review):** ADR **0008** (*Cross-context capabilities wire by local
  instantiation*) and ADR **0092** both exist and are cited accurately below; the open
  inference question is `bynk-type-system.md` §2.6.4 (not the design notes). The
  Annotation-policy and Constrained-refinement principles are `bynk-design-notes.md`.

## The problem, stated once

A storage `Cache`/`Log` op reads the clock for TTL eviction / timestamping, so the *agent
handler* that runs it must declare `given Clock` — but nothing in the source names `Clock`.
The requirement is real, derivable, and **invisible at the declaration site**. Worse, in the
v0.98 bundle emitter that invisible requirement *leaks across the agent boundary*: a service
handler calling such an agent method has the caller's `deps` forwarded into the call, so it
too must declare `given Clock` or emit TS that `tsc` rejects — and if it declares it, the
clause is then flagged `unused_capability` because nothing in the body names it. One missing
idea underlies both: **the language has no notion of *where a capability requirement stops*,
and no model of *why* it arises.**

## Two strands

1. **The boundary decision (load-bearing, language-level).** An agent method owns its
   capabilities; calls into it forward nothing. This removes the unsound/noisy class.
2. **The provenance + surfacing (the discoverability layer).** Make a requirement visible at
   the right place, generically for *any* capability.

## Strand 1 — Encapsulate capabilities at the agent boundary

**[DECISION B — load-bearing, gated on the spike below] An agent method owns and provisions
its own capabilities; a call `Agent(k).m()` forwards nothing.** Calling an agent method is a
*boundary crossing*, exactly like a cross-context service call — `B.svc(...)` already
forwards no `deps` (`bynkc/tests/cross_context_caller.rs`; the in-process, locally-wired
model is ADR **0008**, the caller-value boundary is ADR **0092**). The callee's `given` is
satisfied at the agent's composition site, not by the caller. The **bundle** emitter must
therefore stop threading the calling handler's `deps` into the agent method and instead give
the agent its own capability deps — which is what the **workers** target already does (a DO
provisions its own providers). *Recommend: accept, contingent on the spike.*

**Why encapsulation, not Option-1 propagation.** Propagating the requirement up to the caller
(the v0.98 stopgap's direction) makes a pure orchestration handler restate the *union* of its
callees' capabilities, leaking each agent's internals (that `login` happens to use a `Cache`)
into every caller's contract — adding a cache to an agent would break unrelated signatures.
Encapsulation keeps the awaited `<-` as the visible cost and the specific capability as the
agent's own business. **B alone fixes the original bug *and* the `unused_capability` noise:**
the service handler no longer declares — or needs — `given Clock`, so there is nothing to be
unused and nothing to mistype.

### [DECISION A — REVISED after measuring the blast radius] Do **not** confine capability use
### to agent handlers. Capabilities remain available to service / cron / queue handlers for
### their **own direct use**.

The draft proposed confining all capability use to agent `on call` handlers, framed as
"store access is the agent's role." The blast-radius audit the review asked for **overturns
that framing**:

- **Store access is *already* agent-only** — enforced by the grammar. The audit found **zero**
  `store` ops outside agents (a `store` field can only be declared in an `agent`). So the
  stated architectural goal needs no new rule.
- **What confinement would *additionally* forbid is legitimate stateless boundary work.** The
  audit found **~97 capability-use sites outside agents**, dominated not by store access but by
  honest, direct platform-capability use at the edge: `Logger.info(...)` (hello-world),
  `Kv.get/put/list` (feature-flags), `Random.uuid()` + `Kv.putTtl` (link-shortener),
  `Fetch.send` + `Secrets.get` + `Logger.info` (webhook-relay), a cron `on schedule(...)
  given Fetch, Kv, Logger` (uptime-monitor). Every one declares `given X` and uses `X.op()`
  in the body — **none has the invisibility problem, none is unsound, none is noisy.**

Confining these would force a stateless HTTP handler that merely logs or reads KV to
instantiate a Durable Object — heavier, and a category error (`Kv` is a *stateless* platform
capability; agents are for durable single-writer state). That is not "correcting an
architectural error"; it imposes statefulness on stateless glue, and would require rewriting
13+ committed example files and 3 tutorials against no soundness benefit.

*Recommend: drop confinement. Keep B (the boundary is where capabilities stop **propagating**,
not where they may be **used**).* This still honours "the agent boundary is the capability
boundary": the agent owns what it needs; the edge owns what *it* needs; neither borrows the
other's. **This reverses the draft's accepted [DECISION A]; flagged explicitly for the
reviewer, since it changes an accepted call on the strength of the measurement they
requested.** If the intent really is an austere "edge is pure routing, all effects in agents"
architecture, that is a separate, far larger proposal and should be argued on its own.

## Strand 2 — A generic capability-requirement provenance

The review's crux: a hand-written reason like *"Cache TTL eviction reads the clock"* must be
**generatable for any capability**, and "reads the clock" presumes we know *how* the
capability is used — which we won't in general. So the reason is **derived from the source of
the requirement, never from the capability**.

**[DECISION C — accepted] Model a requirement as `{ capability, site, source }`; render the
reason as a total function of `source` (a small closed enum), with no per-capability text.**

```
Requirement { capability: CapRef, site: Span, source: RequirementSource }

RequirementSource =
  | DirectCall { op }                 // body calls `Cap.op(...)` directly
  | StoreOp    { kind, op, field }    // a storage op that consumes a capability
  | Builtin    { feature }            // a language builtin that draws on one (e.g. Uuid → Random)
```

- `DirectCall { op }` → *"calls `{cap}.{op}`"*. Mechanical, correct for **any** capability,
  including user-defined ones (`Payments.authorise` → *"calls `Payments.authorise`"* for
  free). The call site *is* the explanation; we never invent a "why".
- `StoreOp { kind, op, .. }` → a fragment owned by the **storage feature**, keyed by
  `(kind, op)` — the only code that knows *why* a store needs a capability. This promotes the
  string already inline in `check_store_cache_op` / `check_store_log_op` into a small data
  table `(StoreKind, Op) -> (CapName, reason-fragment)`.
- `Builtin { feature }` → a fragment owned by the builtin (`firstparty` surface).

Decisive property: **adding a new capability requires zero new reason text.** Fragments are
authored only when a new capability-*consuming feature* is added (a new store kind, a new
builtin) — a closed, compiler-internal set.

**[DECISION D — accepted] Record requirements always, not only on absence.** `require_clock`
today only speaks when the capability is missing. Generalise to `require_capability(req)` that
**records every requirement** into a per-handler ledger (the capability analogue of the
inferred-type `HintSink`) *and* errors when the enclosing agent handler does not declare it.
The ledger — `site → Requirement` plus the reverse `capability → [Requirement]` — is the
single producer the three editor surfaces consume.

## Strand 2 (cont.) — The editor surfaces

All three already exist as LSP provider methods (`bynk-lsp/src/main.rs`: `hover` :755,
`code_lens` :872, `inlay_hint` :1405) and reuse the existing `given_insertion_edit` quick-fix.

**[DECISION E — accepted] The "annotation" is a *materializable inlay hint*, not source
syntax.** On a handler whose body has requirements its `given` does not cover, render a ghost
clause after the return type:

```
on call login(token: Token, user: UserId) -> Effect[()]  «given Clock»
```

Accepting the hint writes real `given Clock` via `given_insertion_edit`; when already
declared, no hint. **Reject** a source-level `@requires(Clock)`: the requirement is derivable
from `kind + @ttl`, so authoring it restates an implied internal (the reason use-site
refinement was rejected, `bynk-design-notes.md` §Constrained refinement).

**Hover (the "why").** Reads the ledger: on a store field / kind / `@ttl`, on an op or direct
call (the rendered reason for that site), and on a handler's existing `given Clock` the
**reverse** map (*"satisfies: `live` — Cache TTL"*) so a reader sees what a capability is *for*.

**CodeLens (the ledger at the boundary).** Above each handler, a glanceable summary —
`requires Clock` (click → add) when unsatisfied, quiet when covered. A new lens family
alongside the reference-count lenses (precedent: the client-side test CodeLens,
`vscode-bynk/src/codelens.ts` / `testCodeLens.ts`).

## Decisions still open / to confirm

**[DECISION F — accepted] Retire the interim `bynk.given.unsatisfied_capability` — it is a
*contradiction*, not just dead code.** Its registered description encodes the **old transitive
model** (*"the caller forwards its `deps` into the call"*) that [DECISION B] overturns: under
B the caller forwards nothing. Remove the diagnostic, its registry entry, its two tests, and
re-bless the generated diagnostics docs.

**[DECISION G — open, now de-risked by the spike] One sliced v0.99 increment.** Slices:
1. **Emitter + checker** — [DECISION B]/[F]/[H]: per-agent deps in the bundle target; reject
   `by` on agent `on call` handlers (the deps-split taxonomy's category-3 invariant); retire
   the stopgap. The correctness landing.
2. **Provenance** — [DECISION C]/[D]: the `Requirement` model + `require_capability` ledger.
3. **Inlay hint** — [DECISION E]: the materializable ghost `given`.
4. **Hover + CodeLens** — the read-only surfaces.
**The shippable "correctness fix + the affordance that prevents the error" is slices 1 + 2 +
3** — slice 3 reads the ledger slice 2 produces, so slice 2 is *folded in, not fast-followed*
(corrected from the draft). Slice 4 may fast-follow. The spike (below) shows B is a bounded
emitter change, **not** structural, so a **single v0.99 increment** is appropriate; promote to
a track only if implementation uncovers that the deps-split (below) pulls in the composition
root. *Recommend: single increment, slices 1–3 together.*

## Spike result — bundle per-agent deps (gates B and G)

Traced; **feasible, bounded, not structural:**

- The caller-forwarding is one localized site: `lower.rs:1083` pushes `"deps"` onto the agent
  method call. Stop pushing it.
- Agent method bodies already resolve capability access through `cx.cap_deps_expr`, an
  existing indirection that is `"deps"` in a handler body and **`"this.deps"` in a composed
  provider body** (`emitter.rs:1806`). Pointing an agent method's body at its own `this.deps`
  reuses proven machinery; the workers target already provisions per-agent providers as a
  reference implementation.
- **The one real wrinkle:** today's `deps` object *multiplexes* capabilities **and** per-call
  runtime context — `env` (DO namespaces, read by `__makeAgent`), `__exec.waitUntil`
  (`lower.rs:296`). So B is "**split** agent-owned capabilities from threaded runtime context,"
  not "stop passing deps." The agent factory/constructor takes the agent's composed
  capabilities (compose → `__makeAgent` → `new Agent(state, caps)` → `this.deps`); the method
  signature drops the capability `deps` param; request/runtime context that genuinely must
  cross the boundary stays a passed argument. This is the part to get right in slice 1; it is
  bounded to the agent factory, constructor, method-signature/body lowering, the single
  call-site, and compose wiring.

## Deps-split taxonomy (slice-1 precondition)

The review's one addition: pin down, before slice 1, what is **agent-owned** vs what **must
cross the boundary**, naming `__exec` explicitly rather than resolving it ad hoc. The `deps`
object is already assembled from gated facets (e.g. `__exec` only when the body uses `~>` —
`block_uses_send`, `emit.rs:874`; `identity`/`who` only under actor auth), so the split is a
re-partition of an existing facet set, not a new abstraction. Three categories:

**1. Agent-owned** — provisioned where the agent is composed/constructed; **never forwarded**:
- **Capabilities** (`Clock`, `Random`, `Fetch`, `Logger`, `Secrets`, `Kv`, …) — declared by the
  agent handler's `given`; read via `this.deps` (the provider-body lowering already in place).
- **`env`** (DO-namespace bindings used to construct sub-agents) — a platform binding identical
  regardless of caller. Today `agent_construct` threads `deps.env` (`emitter.rs:1896`); under B
  it is sourced at the agent's own construction (the workers DO has its own `env`; bundle has
  the one global namespace map), not from the caller's per-call deps.

**2. Execution context** — owned by the **physical execution unit**, supplied at the run site;
neither a capability nor request data. **This is `__exec` (`waitUntil`), and it is resolved
here, not deferred.** Principle: execution context is never owned by the *logical agent* — it
belongs to whatever unit is actually running the code.
- On **workers**, the agent is a Durable Object with its own `ctx` → `__exec` is agent-local;
  **nothing crosses**.
- On **bundle/test**, there is a single shared request context → `__exec` is the **one
  must-cross item**, threaded into the agent invocation, gated on the agent body using `~>`
  exactly as handler deps already gate it (`block_uses_send`). A non-sending agent method
  threads nothing.
This target-dependent asymmetry is precisely why the split is "bounded but not zero"; it is the
spot the implementation must encode explicitly (a small `RuntimeContext`/`exec` argument
distinct from both `this.deps` and the method's payload params), and the only place where
"bounded" could quietly become "structural" if left implicit.

**3. Service-boundary-only** — never reaches an agent method:
- **`identity` / `who`** — actor auth, established by `by Actor` at the service edge. An agent
  `on call` handler must carry no `by`, so these never cross — **but today this is an
  unenforced assumption, and slice 1 must turn it into a rule.** The parser accepts a `by`
  clause on an agent handler (`parser/declarations.rs:~2755`, ungated on `is_agent`) and the
  agent emit path never reads `handler.by_clause`, so a `by Actor` on an agent silently parses
  and is dropped. **[DECISION H — accepted] Slice 1 adds a checker rule rejecting `by` on an
  agent `on call` handler** (diagnostic e.g. `bynk.actor.by_on_agent` — *"`by` is a
  service-edge clause; an agent handler has no actor"*), making this category an *enforced*
  invariant rather than a presumed one. The [DECISION B] ADR records it as a **rule, not a
  description** — the deps-split's "actor auth never crosses the agent boundary" guarantee
  holds by construction only once `by`-on-agent is rejected. **Blast radius: zero** — a
  repo-wide sweep (every `.bynk`, fixture, doc, and embedded test source) found **no** agent
  `on call` handler carrying a `by` actor clause (the only `by` tokens inside `agent` blocks
  are `@indexed(by: …)` annotation args and prose in comments). The rule is purely additive;
  it breaks no existing example or green test.
- **`surface`** — the service-to-service dispatch surface; agents invoke agents via the factory
  (`__makeAgent`), not via `surface`.

**Request payload** crosses as the method's explicit parameters — already the case, unchanged.

So an agent method under B reads capabilities + `env` from `this.deps`, takes `__exec` from the
execution unit (its own DO on workers; a threaded `exec` arg on bundle, gated on `~>`), and its
payload as params; `identity`/`who`/`surface` do not appear. The ADR for [DECISION B] records
this taxonomy as the normative split.

## Risks & mitigations

- **The deps-split is the load-bearing detail.** Mitigation: the spike above scopes it; the
  workers target is the reference; land it behind a `tsc_verify`-style end-to-end case proving
  a capability-free service handler can call a `given Clock` agent method and the **bundle**
  output type-checks.
- **Blast radius is near-zero under the revised [DECISION A].** Keeping capabilities available
  at the edge means the ~97 audited sites keep compiling unchanged; the only churn is removing
  the v0.98 stopgap and any in-flight code that declared `given` purely to forward (e.g. the
  uncommitted `examples/sessions`). If confinement is *not* dropped, the radius is ~97 sites +
  13 example files + 3 tutorials — measured here so the call is made with eyes open.
- **Reason quality for `DirectCall`.** *"calls `Clock.now()`"* is honest but thin; accept it —
  inventing richer per-capability prose is the trap the review flagged. Feature-mediated
  sources carry the richer fragment.
- **Agent addressing from each ingress (relaxed under revised A, still worth stating).** With
  capabilities kept at the edge, no ingress is *forced* to delegate to an agent. Where one
  *chooses* to (stateful work), the agent key must be expressible: **http** has the path/body;
  **queue** (`on message`) derives a key from the payload; **cron** (`on schedule`) has no
  inbound entity, so it addresses a **fixed singleton** (`Agent("default")`) or a known fan-out
  set. All expressible today (agent construction + `<- Agent(k).m()` lowers identically in
  http/cron/queue/test bodies, `lower.rs:1069`); the cron singleton convention is the only one
  worth a doc line.

## Docs & tests

- **Spec/book:** the capability chapter gains the boundary rule (an agent owns its
  capabilities; a call across the agent boundary forwards nothing) and contrasts it with the
  cross-context boundary it mirrors (ADR 0008/0092). A guide note: "an agent's store needs a
  capability — declare it on the agent handler (the editor offers it)." Changelog row; advance
  the currency banner; an ADR for [DECISION B] and the provenance model.
- **Tests:** checker — `require_capability` records-and-errors; `by` on an agent `on call`
  handler is rejected (`bynk.actor.by_on_agent`, [DECISION H]); an agent method called from a
  capability-free service handler compiles **and** emits tsc-clean (end-to-end, bundle target).
  LSP — the inlay hint appears with its materialization edit; hover renders the reason from
  each `RequirementSource` variant; the CodeLens reports the ledger. Provenance — a table-level
  totality test over `(StoreKind, Op)` and a check that a user-defined capability's
  `DirectCall` reason generates with no bespoke entry.

## Done when

- An agent method owns its capabilities; a capability-free service/cron/queue handler can call
  it and the **bundle** output type-checks; no `unused_capability` noise; edge handlers keep
  their own direct capability use.
- `by` on an agent `on call` handler is a clean compile error ([DECISION H]); the deps-split
  taxonomy's "actor auth never crosses the agent boundary" guarantee holds by construction,
  not by assumption.
- A single `Requirement { capability, site, source }` ledger drives all three surfaces; the
  reason renders from `source` with **no per-capability text**, and a user-defined capability
  needs no new fragment.
- The ghost `given` inlay hint materializes the clause; hover explains the why; the CodeLens
  shows the per-handler ledger.
- `bynk.given.unsatisfied_capability` and its tests are removed; the registry + generated
  diagnostics docs re-blessed.
- Spec + book + changelog current; ADR(s) written; version bumped.
- On merge: **delete this proposal.**
