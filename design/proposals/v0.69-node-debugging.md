# v0.69 — Node / test-runner debugging (debugging track, slice 2)

- **Track:** [debugging](../tracks/debugging.md), slice 2 — the smallest *consumer* of
  slice 1's maps, and the end-to-end proof that a `.bynk` breakpoint binds and pauses
  without `workerd` in the loop.
- **Realises ADR:** [0104](../decisions/0104-debug-launch-model.md) D3 (Node target);
  **consumes** [0103](../decisions/0103-source-map-contract.md) (the maps slice 1 emits).
- **Posture:** tooling. Adds a debug-launch mode to `bynkc test`; no language surface.

## Why now / why this shape

Slice 1 made the emitter carry `.bynk → .ts` maps. Slice 2 is where that first
*pays off*: compile a project, launch the emitted test entry under `node
--inspect-brk`, and confirm a breakpoint set in `.bynk` source binds and pauses on
the running process. The track deliberately does the Node path before workerd —
it proves the **map round-trip** cheaply and isolates any later `wrangler`-inspector
trouble (slice 3) to the *attach*, not the maps.

But the path from "maps on the `.ts`" to "a debugger stopping on `.bynk`" is not a
straight line, and a spike (below) found four concrete obstacles. The shape of this
slice is therefore: **settle the run-and-map mechanism against an observed end-to-end
step, then ship the thinnest launch surface (`bynkc test --inspect`) plus an automated
proof.** The VS Code extension UX (the `DebugConfigurationProvider`, the Debug
CodeLens) stays in slice 4 — slice 2 proves the mechanism and the maps.

## Current state — the reality check (all spike-confirmed)

A throwaway project (one `commons` with a free `fn`, one `test`) compiled with the
real `bynkc`, then probed under Node. Four facts set the work:

1. **`bynkc test` writes no maps.** `bynkc compile` routes through `write_output`
   (slice 1) and emits `.ts.map` siblings + trailers. But `bynkc test`
   (`bynkc/src/main.rs` ~91–93/137–139) writes each file with a **bespoke
   `std::fs::write(&target, &file.typescript)` loop that ignores `source_map`** — so
   the test `out/` has no maps on disk at all. The debugger has nothing to read.
2. **The run is `tsc → node out-js/*.js`, and tsc does not chain input maps.** The test
   runner compiles the emitted `.ts` with `tsc -p tsconfig.json` and runs the resulting
   `.js`. A `.bynk` breakpoint would need the running `.js` mapped to `.bynk`, but slice
   1's map is `.ts → .bynk` and **tsc cannot consume an existing input source map** — so
   the chain `.js → .ts → .bynk` is broken at the transpile. (`sourceMap: true` would
   give `.js → .ts`, but the second hop to `.bynk` is not followed by `vscode-js-debug`
   by default — a two-hop chain.)
3. **Running the `.ts` directly fails on `.js` import specifiers.** The emitted modules
   import siblings as `import … from "./x.js"` (NodeNext convention). `node
   out/tests/main.ts` under type-stripping throws `ERR_MODULE_NOT_FOUND` for `./x.js` —
   Node does not resolve a `.js` specifier to the `.ts` on disk.
4. **Even with `.ts` specifiers, strip-only mode rejects the emitted classes.** The test
   runtime emits TS **parameter properties** (`constructor(public location: string, …)`),
   which Node's *strip-only* type-stripping refuses
   (`ERR_UNSUPPORTED_TYPESCRIPT_SYNTAX`). `node --experimental-transform-types` *does*
   run it (`✓ doubles`) — but transform mode **rewrites syntax and shifts line numbers**
   (parameter properties become `this.x = x;` assignments), which would invalidate slice
   1's *line-level* map for code after such a class.

The slice-0 spike already proved the *good* end state: running a `.ts` whose imports
are `.ts` and whose syntax is strip-clean, under line-preserving type-stripping, makes
slice 1's `.ts.map` apply directly and a `.bynk` breakpoint resolve. Slice 2's job is
to make the *real* emitted output reach that state.

## The load-bearing decision — how the map reaches the running process

This is the ADR-grade call, and it is **gated on a spike** (first task below), because
the obstacles interact and the right answer is empirical, not armchair. Three viable
mechanisms:

- **(i) Direct `.ts`, line-preserving strip.** Emit a *debug build* with `.ts` import
  specifiers and strip-clean syntax (no parameter properties in the test runtime — emit
  explicit `this.x = x;`), run `node --inspect-brk out/tests/main.ts` under *strip-only*
  type-stripping. Slice 1's `.ts.map` applies **directly and unchanged** — no transpile,
  no chaining. Cleanest map story; cost is a debug-emit variant + a small emitter tweak.
- **(ii) Compose through esbuild.** Transpile the emitted `.ts` with esbuild (which
  *does* fold an input `//# sourceMappingURL` into its output map), producing one
  `.js.map` that resolves all the way to `.bynk`; run that `.js`. Handles `.js`
  specifiers and TS-only syntax for free; cost is an esbuild dependency on the debug
  path and trust in its map composition.
- **(iii) tsc `sourceMap` + confirm the chain.** Keep `tsc → .js`, enable `sourceMap`,
  and verify whether `vscode-js-debug` follows `.js.map → .ts → .ts.map → .bynk`. Lowest
  code cost *if* the two-hop chain resolves; the spike decides — finding 2 suggests it
  will not without help.

**Recommendation: lean (i)** — it keeps slice 1's line-level map authoritative and
sidesteps every transpile-composition risk, and the emitter tweak is small and arguably
worth having anyway (strip-clean output runs on more Node versions). **(ii) is the
fallback fork** if the debug-emit variant proves more invasive than expected. The spike
ratifies the choice from an observed end-to-end step, exactly as slice 0 did for
granularity.

## What this slice does

1. **Map the test path to disk.** Route `bynkc test`'s file-writing through the same
   map-aware writer as `write_output` (or share its logic), so the test `out/` carries
   `.ts.map` siblings + trailers (closes finding 1).
2. **Map the test modules** (extends slice 1). Slice 1 left `emit_test_main` /
   `emit_test_module` with `source_map: None`; thread the `SourceMapBuilder` through the
   test-body lowering (the `LowerCtx` sites in `lower.rs` slice 1 skipped) so a
   breakpoint in a *test body* (`assert`, `let`) binds too — not only in production code
   the test calls.
3. **The debug-emit variant + launch** (per DECISION A). `bynkc test --inspect`:
   compile in debug-emit mode, write maps, launch the emitted test entry under `node
   --inspect-brk=<port>`, and **surface the inspector `ws://` URL** (parsed from Node's
   stderr) on stdout for a debugger to attach. The existing node-spawn seam
   (`main.rs` ~231) is where `--inspect-brk` slots.
4. **An automated, headless proof** that a `.bynk` breakpoint binds and pauses —
   reusing the slice-0 CDP harness shape (Node's global `WebSocket`, no deps): set a
   breakpoint by `.bynk` line (resolved to the generated location through the map, as
   `pwa-node` does), attach over CDP, assert it **binds and pauses at the expected
   generated line**, mapped back to the right `.bynk` line.

## Decisions

**[DECISION A — sign-off] The run-and-map mechanism (spike-gated).** Forks (i)/(ii)/(iii)
above. **Recommend (i) direct `.ts` / line-preserving strip**, with (ii) esbuild-compose
as the fallback, ratified by the spike. This is the one genuinely uncertain call and the
reason the slice spikes first.

**[DECISION B — sign-off] Deliverable = `bynkc test --inspect` + headless proof; extension
UX deferred to slice 4.** The launch mechanism lives in the CLI (compile → launch under
inspector → print the `ws://` URL); slice 4's `DebugConfigurationProvider` will *shell
that* and tell VS Code to attach `pwa-node`. So slice 2 owns compile-and-launch + the
automated map proof; slice 4 owns the VS Code surface (the `debuggers`/`breakpoints`
contribution, the Debug CodeLens beside the existing Test Explorer "Run"). This honours
the track's 2-vs-4 split and gives slice 4 a clean thing to wrap. *Recommend as stated.*

**[DECISION C] Extend slice-1 mapping to test modules.** Map `emit_test_main` /
`emit_test_module` output so test-body breakpoints bind, and write those maps to disk.
*Recommend yes* — debugging a test you cannot break inside the test body is half a
feature, and the test-body lowering already flows through the same `emit_block_as_function_body`
path, so it is the `LowerCtx`-threading slice 1 deferred, not new machinery.

**[DECISION D] The debug path may require a newer Node than `NODE_MAJOR_FLOOR`.**
Line-preserving type-stripping of `.ts` needs Node ≥ 22.18 / 23.6 (or `--experimental-strip-types`
on 22.x); strip mode also needs the strip-clean emitter tweak (finding 4). The normal
`tsc → node` test path keeps `NODE_MAJOR_FLOOR = 18`; only `--inspect` raises the floor,
with a clear `doctor`/error message when unmet. *Recommend: the debug run states its
Node requirement explicitly rather than silently failing.* (Moot if the spike picks (ii).)

**[DECISION E] Version — recommend minor v0.69.** A new tooling capability
(`bynkc test --inspect`), no language change → minor, over the 0.68.0 baseline.

## The spike (first task, before any production code)

Like slice 0: reuse the CDP harness to drive the *real* emitted output end-to-end under
each candidate mechanism, and **observe** a `.bynk` breakpoint binding and pausing at the
correct line. Confirm: the chosen run mode (a) resolves imports, (b) runs the emitted
classes, (c) preserves the lines slice 1's map assumes (or composes a correct map), and
(d) a breakpoint set on a `.bynk` line stops on the matching generated statement. The
spike ratifies DECISION A from an observed step, not a guess. No production code until it
does.

## Risks & mitigations

- **Map composition is the whole risk.** Findings 2–4 are all "the map doesn't survive
  the run." Mitigated by spiking the mechanism first and preferring (i), where there *is*
  no transpile to survive.
- **Debug-emit divergence.** A `.ts`-specifier debug build differs from the normal `.js`
  build, risking "works in debug, not in `tsc` run" skew. Mitigated by keeping the debug
  variant a thin specifier/syntax toggle over the same emitter, and by the automated
  proof running the real output.
- **`pwa-node` fidelity vs. the headless proof.** The slice-2 test emulates what
  `vscode-js-debug` does (map `.bynk` → generated, set breakpoint, assert bind). The
  *actual* `pwa-node` session is exercised in slice 4's extension integration test. Stated
  so slice 2 is not read as proving the VS Code UX.
- **Node floor friction.** Raising the debug-path Node requirement could surprise. Mitigated
  by DECISION D's explicit message and by `bynk doctor` surfacing it.
