# v0.110 — `Bytes`: a binary primitive, erased to `Uint8Array`, base64 on the wire

- **Scope:** a new base type `Bytes` and its **usable-primitive** surface — construction, String interop, length, content equality, and a JSON/base64 codec — following the `Float`/`Duration`/`Instant` distinct-base-type playbook (ADRs 0040/0112/0114), **with one genuine departure from it**: `Bytes` is the first base type whose `==` is not host `===`, so equality is real emitter work, not a kernel-table row (D4), and that departure ripples into keyability (D7) and boundary-crossing (D8). The binary *consumers* that motivate it (`Stream[Bytes]`, binary HTTP bodies, an R2 capability, AI image/audio) are **named follow-ons, out of this increment** — each cites the landed primitive.
- **Realises:** `bynk-type-system.md` §1.1, which already lists `Bytes` in the primitive set (`Int | Decimal | String | Bool | Bytes | Timestamp | Duration | Unit`) but ships it as **unbuilt** — the spec/impl primitive divergence flagged in `design/bynk-status-and-roadmap.md` (§ "Spec/impl primitive divergence"). Building `Bytes` closes part of that gap (`Decimal` remains).
- **Motivation (the forcing function):** every honest binary capability hits the same wall — `String` cannot hold arbitrary octets without corruption. R2 objects are binary; a binary HTTP body has no home (`bynk`'s `Response.body` is `String`); a streamed download wants `Stream[Bytes]` (the just-landed `Stream` shipped `String`-only, ADR 0129). The shared prerequisite under all of them is a value type for octets. This proposal builds *only* that prerequisite.

## The problem, stated once

Bynk has no representation for arbitrary binary data. The only "bag of bytes" available is `String`, which is UTF-8 text — feeding it raw octets (an image, a gzip blob, an HMAC digest, an R2 object) corrupts them at the first encode boundary. So the entire class of binary platform surfaces is unreachable today, and the type-system spec already concedes the gap by listing a `Bytes` primitive it never built. The fix is the same one `Float`/`Duration`/`Instant` used: a **distinct base type the checker refuses to confuse with its neighbours**, erased to the host's natural representation — here `Uint8Array`, not `number`.

## Decisions

**[D1 — accepted] `Bytes` is a seventh base type, erased to `Uint8Array`.** It joins
`Int`/`String`/`Bool`/`Float`/`Duration`/`Instant` as a `BaseType`, but unlike all of
them it does **not** lower to `number` — a `Bytes` lowers to a TS `Uint8Array`, an
immutable finite octet sequence at the Bynk level. Usable anywhere a type is written:
`Cell[Bytes]`, `Map[ObjectKey, Bytes]`, `{ payload: Bytes }`, parameters, returns. The
`Uint8Array` erase target is a plain runtime value, so it is **strip-only-clean** (the
in-browser track's emission invariant is untouched — no type-directed construct).

**[D2 — accepted] No source literal; construction is explicit, mirroring `Instant`.**
Arbitrary octets written in source are an unportable foot-gun (`Instant`'s D2 reasoning).
`Bytes` is **constructed**, never written literally:

- `Bytes.fromUtf8(s: String) -> Bytes` — total; the UTF-8 encoding of a string.
- `Bytes.fromBase64(s: String) -> Option[Bytes]` — partial; `None` on invalid base64 (see Q1).
- `Bytes.empty` — the zero value (an empty sequence).

`fromHex`/`toHex` are **not** in v1 (see Q2).

A `b"…"` byte-string literal is **rejected for v1** (deferred): rare in source, and the
constructors cover the constant case. No parser, lexer, or tree-sitter change — *that* much is
exactly `Instant`'s shape. But the resemblance stops there: unlike every prior base type, `Bytes`
is **not** a kernel-table-only addition, because its equality needs new emitter codegen (D4).

**[D3 — accepted] String interop is the load-bearing surface: UTF-8 and base64, both
directions.** The whole point is a clean bridge to and from text:

- `b.decodeUtf8() -> Option[String]` — partial; `None` on an invalid UTF-8 sequence (Q1).
- `b.toBase64() -> String` — total; the base64 encoding.

With `fromUtf8`/`fromBase64` (D2) this is the full round-trip. Encoding (text → bytes) is total;
decoding (bytes → text) is partial, because not every octet sequence is valid UTF-8 and not
every string is valid base64 — the partiality is real and surfaced, not hidden.

**[D4 — accepted] Length and content equality; no arithmetic, not orderable (v1). Equality is
the proposal's one piece of genuinely new emitter work.**

- `b.length() -> Int` — the octet count (the `Uint8Array.length`, **not** any string length).
- `==` / `!=` compare **by content**, byte for byte — and this is where `Bytes` stops being
  "just another branded primitive." Equality lowers today to a bare `===`
  (`bynk-emit/src/emitter.rs`, `ts_binop`: `BinOp::Eq => "==="`), which is content-correct for
  *every existing base type* because they all erase to a JS primitive (`number`/`string`/`bool`).
  A `Bytes` erases to `Uint8Array`, and `===` on two `Uint8Array`s is **reference** equality —
  so `Bytes.fromUtf8("a") == Bytes.fromUtf8("a")` would be `false` under the current lowering.
  Content equality therefore needs **dedicated codegen**, and the propagating case matters most:
  a record or sum carrying a `Bytes` field gets correct structural/derived equality only if that
  codegen threads the byte-compare through every `Bytes` field. This is the first crack in the
  base-type layer's standing "branded primitives keep JS value equality" assumption (ADR 0038's
  words), and it is load-bearing for D7.
- `Bytes` is **not** in the orderable base set (no `<`/`sortBy` key) and has **no** arithmetic
  or concatenation in v1. Lexicographic byte ordering, `concat`, `slice`, and `fromHex`/`toHex`
  are **named follow-ons** (none needed by the motivating consumers; deferred exactly as
  `Duration` deferred division).

**[D5 — accepted] Codec: a `Bytes` is a base64 JSON string on the wire; zero is `empty`.**
JSON has no binary type, so a `Bytes` **serialises as a base64-encoded JSON string** and
**deserialises requiring a valid base64 string** (an invalid or non-string wire value is
rejected, as `Instant`/`Duration` reject a non-integer). So a `Bytes` in a record or `store`
field round-trips. Its implicit zero is `Bytes.empty` (the empty sequence, `""` in base64).
Note a `Bytes` and a `String` field both appear as JSON strings but are distinct Bynk types
decoded by different rules (base64-validated vs. raw) — the same harmless surface-coincidence
as `Instant`/`Int` both being JSON numbers.

**[D6 — accepted, the key contrast] `Bytes` is a fully ordinary, serialisable value — the
opposite of `Stream`/`Connection`.** A `Bytes` is a finite immutable blob: it **is**
serialisable (D5), **is** storable in any `store` kind, and **may** cross a context boundary —
it sits with `Int`/`String` in the *serialisable* set, **not** with `Stream`/`Connection[F]`
in the non-serialisable one. This is worth stating because the motivation is binary
*streaming*: the live source `Stream[Bytes]` is non-serialisable (it is a `Stream`), but its
*element* `Bytes` is an everyday value. The primitive carries no linearity, no lifetime, no
held-resource discipline. Only the streaming of bytes is special; a byte blob is not.

**[D7 — accepted] `Bytes` is equatable but **not** `Map`-keyable; the keyable set stays
`String`/`Int`.** ADR 0038 confines `Map` keys to value-keyable types — `String`, `Int`, and
refined/opaque types over them — on the explicit premise that "branded primitives keep JS value
equality." `Bytes` is the first base type to break that premise: it is equatable (D4) yet
neither JS-value-equal nor orderable, a combination no existing base type has. So `Map[Bytes, V]`
is **rejected** with `bynk.types.unkeyable_map_key`, and `Bytes` is *not* added to the keyable
set. A keyed-by-bytes need is served by keying on `b.toBase64()` (a `String`) explicitly. This
is the same crack as D4, surfaced at the `Map`-key chokepoint rather than the `==` site.

**[D8 — accepted] Boundary-crossing `Bytes` is guaranteed for the *typed* paths in v1; the
erased workers edge is scoped out, not assumed.** D5/D6's "the decoder keys on the field's static
type" holds for the `bundle` path and for `store`/record codecs, which are fully typed. But
`workers`-mode cross-context boundary emission leans on `any` + runtime serialisation helpers
(`bynk-status-and-roadmap.md`, "Workers-edge type safety"), and that is exactly where `Bytes`
diverges from `Instant`: a raw `number` round-trips through JSON whether or not the static type
survives, so `Instant` is fine; a `Uint8Array` must be **base64-encoded**, so at an erased
boundary it silently mis-round-trips (re-emerging as a `String`, or dropping). Therefore v1
**guarantees** `Bytes` round-trip in `bundle` cross-context calls and in `store`/record fields,
and a `Bytes` crossing a `workers` cross-context boundary is **diagnosed as not-yet-supported**
rather than mis-encoded — it waits on the roadmap's named workers-edge `any` fix. This is the one
place "just like `Instant`" actively misleads, so it is pinned by a fixture, not assumed.

## Open questions

- **Q1 — partial-decode shape. [DECISION]** `Option[Bytes]` vs `Result[Bytes, EncodingError]`
  for `fromBase64`/`decodeUtf8`. *Recommendation:* `Option`, on the **`Int.parse`/`Float.parse
  -> Option[T]`** precedent (ADR 0048) — the right analogy, because like a parse these are "a
  value *was* supplied and is malformed," not "a value is absent." The mis-analogy to avoid is
  `Secrets.get -> Option`, whose `None` means *absent*: collapsing absent and malformed into one
  `None` is the ambiguity the browser-`Secrets` binding was careful about (the in-browser track's
  "unavailable, not a silent empty"). Here there is only the malformed case, so a single `None`
  is unambiguous and an `EncodingError` enum is heavier than the one failure mode justifies.
  Revisit only if a consumer must branch on *why* a decode failed.
- **Q2 — v1 method breadth. [DECISION]** Ship `concat`/`slice`/ordering/hex now, or defer?
  *Rec:* defer all four (D4) — none is needed by R2/binary-body/`Stream[Bytes]`. Hex is the one
  worth calling out: it serves only the HMAC-digest aside, so it is deferred for *consistency*
  with the consumer-driven rule (not kept for round-trip symmetry — base64 already gives a
  complete text bridge). Each is additive later with no design debt, as `Duration / Int` was.
- **Q3 — consumer reach. [DECISION, scope-confirming]** Confirm `Stream[Bytes]`, binary HTTP
  bodies, and the R2 capability are **out** of this increment. *Rec:* out. `Stream[Bytes]`
  depends on `Stream`'s deferred generic-element work (ADR 0128 D5); a binary body re-types the
  first-party `bynk` `Response`/`Request` surface (its own increment); R2 is Cloudflare-adapter
  work. Each is a clean follow-on citing this primitive — keeping this proposal a pure
  type-system addition with no protocol or adapter churn.

## Risks

- **A second "JSON string" representation.** `Bytes` and `String` both wire as JSON strings,
  decoded differently. *Mitigation:* this is precisely the `Instant`/`Int`-as-number situation
  the codec already handles per-type; the decoder keys on the field's static type, so there is
  no ambiguity — the risk is only reader surprise, addressed in the book.
- **`Uint8Array` mutability vs. Bynk value semantics.** The host type is mutable; the Bynk type
  is an immutable value. *Mitigation:* the kernel surface exposes no in-place mutator (no
  `set`/`fill`); all operations are pure and return new values, so the mutable host type is
  never observably mutated through Bynk. Flag for the emitter to defensively copy at the
  capability boundary if a future binding hands back a view.
- **Scope creep toward the consumers.** The motivation is binary I/O, and it will be tempting to
  pull `Stream[Bytes]` or R2 in. *Mitigation:* Q3 fixes them as out; this increment is the
  prerequisite alone, and its done-when forbids consumer surface.
- **Partiality ergonomics.** Three partial decoders returning `Option` add `match`/`?` noise at
  call sites. *Mitigation:* total encoders cover the common direction; the partial direction is
  genuinely partial and the `Option` is honest. Accept.

## Spec & docs delta

- **Spec (normative, in place):** `bynk-type-system.md` §1.1 — promote `Bytes` from
  listed-aspirational to **built**, with its erase target, codec, and kernel surface; remove the
  "`Bytes` … not built" line from the spec/impl-divergence note in
  `design/bynk-status-and-roadmap.md`. The base-type operator/equality rules gain the `Bytes`
  arms (equality only; no order/arithmetic). Two rules reach beyond the kernel table: the
  **equality emitter** gains a content-compare path for `Bytes`, with structural/derived equality
  propagating it through `Bytes` fields (D4), and the **`Map`-key keyability** rule (ADR 0038)
  explicitly excludes `Bytes` (D7).
- **Book (reader-facing):** a `reference/` primitive entry for `Bytes` (constructors,
  conversions, `length`, equality, the base64 codec) **and** a short guide note on
  encoding/decoding (the UTF-8/base64 bridge, and when each partial decoder returns `None`).
  A new *concept* on-ramp is not needed — `Bytes` is an ordinary value type, not a new idea like
  agents or capabilities.
- **Currency & changelog:** advance the "written against vX.Y" banner and
  `spec/appendix-version-history.md`; add the `reference/changelog.md` row; keep the roadmap
  honest — move `Bytes` out of "planned/diverging" into shipped, leaving `Decimal` as the one
  remaining unbuilt spec primitive.
- **Examples:** none required for the primitive alone; the first consumer increment (binary body
  or R2) carries the worked example.

## Done when

- `Bytes` is a base type erased to `Uint8Array`, strip-only-clean; `fromUtf8`/`fromBase64`/
  `empty` construct it, `decodeUtf8`/`toBase64`/`length` operate on it (hex deferred, Q2), and
  `==` compares by content.
- `==`/`!=` on `Bytes` compares **by content** via dedicated equality codegen (not the bare
  `===`), and derived structural/sum equality threads that byte-compare through `Bytes` fields —
  proven by a fixture where two independently-built equal `Bytes` (and records over them) compare
  equal.
- `Map[Bytes, V]` is rejected with `bynk.types.unkeyable_map_key` (D7); the keyable set is
  unchanged.
- A `Bytes` in a record or `store` field round-trips through the base64 codec; it is
  **serialisable** and crosses a `bundle` context boundary (the explicit contrast with `Stream`/
  `Connection`), **and** the `workers` cross-context path is exercised to prove the D8
  not-yet-supported diagnostic fires rather than mis-encoding — both proven by fixtures.
- Emission is `tsc`-clean; no parser/grammar/tree-sitter change; LSP hover/completion cover the
  kernel.
- No consumer surface ships (`Stream[Bytes]`, binary bodies, R2 are absent and named as
  follow-ons).
- Spec + book + changelog current; the spec/impl-divergence note updated; an ADR for the base
  type authored on merge (the `Float`/`Duration`/`Instant` lineage); version bumped.
- **On merge: delete this proposal.**
