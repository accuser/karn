# v0.30 — Comprehensive completion, slice 1: positional (type / keyword / snippet)

- **Status:** Draft (proposal). **Merging is approval to build.** *(The biggest outstanding
  interactive gap — completion is still the v0.17 surface. Sliced; this is **slice 1**, mostly
  `karn-lsp` with a small `karnc` reuse.)*
- **Builds on (verified):**
  - Completion today is **narrow**: `complete(line_prefix, doc_text, src_root)` (`completion.rs`)
    recognises only three **lexical** contexts — `consumes U { … }`, `consumes <prefix>`,
    `given …` — and emits only `Unit`/`Capability` items (`main.rs:607-617`, mapping
    `Unit→MODULE`, `Capability→INTERFACE`). The handler builds `line_prefix` from the text up to the
    cursor (`main.rs:597`).
  - **Lexical context detection is the right model** — completion fires *mid-edit*, when the buffer
    rarely parses; `completion.rs` already enumerates candidates by parsing the *other* (valid)
    project files with recovery, so it works while the current file is broken. This increment
    extends the *contexts* and *candidate sources*, not the approach.
  - **Reusable candidate sources:** the **`keywords.rs::KEYWORDS`** registry carries names **with
    docs** (`Bool`/`Int`/`Float`/`Option`/`Result`/`Effect`/…) — ready-made `CompletionItem`
    label+detail; the **cached `ProjectIndex`** has project `SymbolKind::Type`/`Fn`/`Capability`
    defs; the **karn surface** `exports transparent { Uuid, Method, FetchError, Request, Response }`
    (`firstparty.rs:207`) and `List`/`Map` are built-ins.
  - **No scope-at-offset query exists** (the index's `in_scope` is span-range filtering, not lexical
    scope). So **locals/params** and **`.`-member** completion need machinery this slice doesn't pull
    in — deferred to slice 2 (below).
  - First-party symbols are **excluded from the index** (the v0.28 finding), so built-in/surface type
    names must come from the **static registries** (`keywords.rs` + `firstparty`), not the index.
- **Spec sections touched:** `design/karn-lsp-spec.md` (Completion — the contexts, sources, item
  kinds); `karn-lsp` `completion.rs`/`main.rs`; possibly expose `karnc`'s keyword/built-in list if
  not already public.
- **Scope (slice 1):** **positional** completion needing neither receiver typing nor scope tracking —
  **type-position** type names, **keyword** completion, **snippets**, and the existing
  `consumes`/`given`. **Not** here: `.`-member completion (value methods/fields, `Type.`/`Cap.`
  statics/ops) and **locals/params in scope** → **slice 2**.

## Settled (recorded as ADRs ≥0058)

- **▸ Context detection is lexical (line-prefix), candidates are semantic.** Completion runs on an
  unparseable buffer, so *where am I?* is decided from the line prefix (as today), while *what can go
  here?* is drawn from the cached index + static registries. No reliance on the current file parsing.
- **▸ Built-ins/surface come from static registries, not the index.** Because first-party symbols
  aren't indexed (v0.28), `List`/`Map`/`Option`/`Result`/`Effect`/base types/`Uuid`/… are sourced
  from `keywords.rs` + `firstparty` — the index supplies only *project* symbols.

## Decisions (open — settle at review)

**[DECISION] Slice the increment; ship positional completion first.** Comprehensive completion has
three machinery tiers: (1) **positional** (type-position, keywords, snippets) — needs only the index
+ static lists; (2) **`.`-member** (`x.method`/`x.field`, `Type.of`, `Cap.op`) — needs resolving
*what's before the dot* and, for value receivers, the **typed model** mid-edit; (3) **locals/params
in scope** — needs a **scope-at-offset** query that doesn't exist (overlaps the deferred index
kinds). *Rec:* **slice 1 = tier (1)** now (tractable, high constant value, reuses the existing
lexical+project approach); **slice 2 = tiers (2)+(3)**. *(Honest note: `.`-member is the daily-driver
that users will most expect, so slice 2 should follow promptly — but it's the part that needs the
receiver-typing + scope machinery, so it's a separate increment, not a bolt-on.)*

**[DECISION] Contexts + candidates for slice 1.** *Rec:*
- **Type position** (after `:`, in `-> T`, inside `[ … ]` type args, record field types): project
  **types** (cached index `SymbolKind::Type`) + **built-ins** (`List`/`Map`/`Option`/`Result`/
  `Effect` + `Int`/`Bool`/`Float`/`String`) + **karn-surface transparent types** (`Uuid`/`Method`/…)
  — built-ins/surface from the static registries, with `keywords.rs` docs as `detail`.
- **Keyword position** (declaration/statement starts): the relevant `KEYWORDS` (`context`,
  `adapter`, `capability`, `service`, `fn`, `let`, `given`, `consumes`, `match`, `on`, …) with their
  registry docs.
- **Snippets** (`CompletionItemKind.SNIPPET`): `context`, `adapter`, `capability`, `service`,
  `on call`, `test` scaffolds.
- Plus the **existing** `consumes`/`given` contexts (unchanged).

**[DECISION] Item kinds + sourcing.** *Rec:* `Type → STRUCT`, keyword `→ KEYWORD`, snippet `→
SNIPPET`, alongside the existing `Unit → MODULE`, `Capability → INTERFACE`. Project symbols from the
**cached index when a round exists, else the existing project-parse** (robust pre-first-round /
mid-edit); built-ins/keywords/surface always from the static registries.

## Spec delta (sketch — written in-place during implementation)

- **`design/karn-lsp-spec.md`** — a Completion section: the lexical contexts (type / keyword /
  snippet / `consumes` / `given`), the candidate sources (index + static registries), item-kind
  mapping, and that `.`-member + locals are slice 2.
- **`karn-lsp`** — extend `completion.rs` contexts + candidate builders; map the new kinds in
  `main.rs`. **`karnc`** — expose the keyword/built-in registry to the LSP if not already public.

## Tests

- **`karn-lsp` pure-function level** *(the v0.25 harness):* `complete()` over crafted line prefixes
  returns the expected items — a type position yields project types **and** built-ins/surface (with
  docs), a declaration position yields keywords, a snippet trigger yields scaffolds, and the existing
  `consumes`/`given` cases stay green. Out-of-context prefixes yield `[]`.
- **`karnc`** — if the keyword/built-in registry is newly exposed, a trivial test it's reachable.
- **Not auto-tested:** end-to-end completion UX (no transport harness) — the producer is pure-function
  tested.

## Diagnostics / Risks
- **Context-detection precision** — lexical heuristics can over-/under-fire; keep them conservative
  (better to offer nothing than wrong items) and cover each context with a prefix fixture.
- **Built-in source drift** — the static built-in/surface list must track the language; reuse
  `keywords.rs`/`firstparty` rather than a fresh hardcoded copy (one source of truth).
- **Noise** — don't flood expression position with every project symbol in slice 1; positional
  completion is deliberately scoped to type/keyword/snippet, where the candidate set is bounded.

## New decision records this increment adds

- **0061** *(light)* — completion is **sliced**: positional (type/keyword/snippet, this increment)
  vs `.`-member + locals (slice 2, needs receiver typing + scope-at-offset); context detection is
  **lexical**, candidates from the **index + static registries** (built-ins not in the index).
  *(Numbered after v0.29's 0058–0060.)*

## Out of scope (→ slice 2 / later)

- **`.`-member completion** — value `x.method`/`x.field` (needs the typed model mid-edit) and
  `Type.of`/`Cap.op` statics (needs before-the-dot resolution) → **slice 2**.
- **Locals / params in scope** — needs a scope-at-offset query (overlaps the deferred index kinds) →
  **slice 2**.
- **Signature help** (the queue's next item after completion); **auto-import / add-`consumes`**
  completion resolve; postfix/snippet expansion beyond the scaffolds above.

## Definition of done

`completion.rs` recognises the **type / keyword / snippet** contexts (plus the existing
`consumes`/`given`) and sources candidates from the **cached index (else project-parse) + the
`keywords.rs`/`firstparty` static registries** (built-ins/surface never from the index); `main.rs`
maps the new item kinds; a **pure-function** test suite covers each context (incl. docs on built-ins
and `[]` out of context); `design/karn-lsp-spec.md` gains the Completion section noting `.`-member +
locals as slice 2; decision record added; the `extension`/`test` CI jobs stay green; changelog
updated. The implementing PR **deletes this file**.
