# v0.77 — Quiet the lowered-temp noise: stepping shows your bindings, not the lowering's (semantic-debugging track, slice 4)

- **Track:** [semantic-debugging](../tracks/semantic-debugging.md), slice 4 — the last
  of Phase 2's planned reshapes. Slices 1–3 made values, the frame's variables, and the
  call stack read in Bynk; this removes the *compiler temporaries* that clutter the
  Variables pane when you step through lowered `?`/`match`.
- **Realises:** ADR [0105](../decisions/0105-semantic-debug-interposition.md) — the same
  editor-side `DebugAdapterTracker` rewrite of the Local-scope `variables` response
  (slice 2's machinery), now *suppressing* recognised temporaries.
- **Posture:** tooling only. **Inference-only — no emitter change** (see below); the
  grounding overturns the "partly an emitter slice" expectation the track carried.

## The gap

Stepping through a handler, the Variables pane carries the lowering's spill bindings
next to yours. From a real emitted handler (`?`/`match`/effect-let):

```ts
const __r0 = await deps.Logger.info("hit");   // effect-let result
const r = Some(1);
switch (r.tag) { ... const __d = r; ... }       // match discriminant
```

so the pane shows `__r0`, `__d`, … interleaved with `r` and the user's bindings. Slice 4
suppresses the compiler temporaries so what's left is the Bynk program's state.

## Why this is inference-only (the grounding)

The track expected this to need an emitter-formalised temp-naming convention. It does
not: **Bynk's lexer already reserves the `__` prefix.** A user `let __x = …` is a *parse
error* — `_` is the discard token, so `__x` cannot be written. Verified:

```
$ bynkc check   # on `let __mine = n + 1`
[bynk.parse.expected_token] expected `=` or `<-` after the let-binding's name, found `_`
```

So a Local-scope variable whose name begins with `__` is **exclusively a compiler
temporary** — recognition is safe inference, with no false positives and **no change to
the emitter or the emitted `.ts`**. (Module-level `__` names like `__CounterRegistry`
aren't handler locals, so they never appear in a handler's Local scope.)

## The one genuine uncertainty — hide or deemphasise (spike-first)

DAP has no "hidden variable" flag, so there are two ways to get a temp out of the way,
and which reads best in VS Code is the spike:

1. **Deemphasise — `presentationHint.visibility: "internal"`** (non-destructive). The
   variable stays in the response but is tagged internal; VS Code may grey it or tuck it
   behind a "show internal" affordance — out of the way but still inspectable. *Preferred
   if it reads well.*
2. **Remove** — drop the temp from the `variables` array entirely (guaranteed gone, but
   you can't inspect it even when you want to). The fallback if `visibility` does nothing
   visible on the pinned VS Code.

The spike sets `visibility: "internal"` on the `__`-temps and observes the pane; if VS
Code doesn't act on it, the slice removes them instead. Either way the
`bynk.debug.semanticValues` toggle restores the raw view.

## What this slice does

1. **Extend the Local-scope rewrite** (`src/semanticValues.ts` + `src/debug.ts`): in the
   `variables`-response rewrite (where slice 2 relabels `deps`/`currentState`), mark each
   variable whose name starts with `__` as internal (or drop it, per the spike). Total,
   and only on Bynk sessions + behind the toggle.
2. **Both runtimes** — the same editor-side path; no runtime or compiler dependency.
3. **Tests:** extend the `relabelBynkLocals` unit coverage (temps suppressed, user
   bindings + the `Capabilities`/`State` groups intact, references preserved) and the
   Node frame-structure integration test (assert a `__`-temp is gone/deemphasised).
4. **Docs:** drop the "compiler temporaries are still listed for now" caveat from the
   guide; changelog; the bump.

## Decisions

**[DECISION A] Inference-only; no emitter change.** The `__` prefix is lexer-reserved, so
the interposer recognises temps by name with zero false positives — no temp-naming
formalisation, no emitted-`.ts` change, no golden churn. (This corrects the track's
slice-4 framing.)

**[DECISION B — recommend, spike-gated] Deemphasise via `visibility: "internal"`; remove
as the fallback.** Non-destructive is better — a temp stays inspectable when you need it.
Adopt removal only if the spike shows VS Code ignores `visibility` on the pinned build.

**[DECISION C] Suppress, don't group.** Grouping temps under a synthetic "(internals)"
node needs a virtual `variablesReference` — the synchronous-rewrite problem slice 2 hit
with scopes. Suppression (deemphasise/remove) needs none.

**[DECISION D] Version — minor v0.77.** A new extension capability; no language change.

## Risks & mitigations

- **A non-temp `__` local slips through (or a real temp is missed).** The lexer guarantee
  makes a user `__` binding impossible, and handler temps are uniformly `__`-prefixed; the
  rewrite is name-only and behind the toggle, so a surprise is recoverable. Unit-tested
  against the real temp names (`__r0`, `__d`).
- **`visibility: "internal"` is a no-op on some VS Code builds.** Spiked first; the
  removal fallback (DECISION B) is unconditional.
- **Over-suppression.** Only `__`-prefixed names are touched; `Capabilities`/`State` (slice
  2) and user bindings are byte-identical — the slice-1 totality discipline.
