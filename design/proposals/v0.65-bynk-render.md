# v0.65 — Introduce `bynk-render`: the shared diagnostic-rendering layer (crate-decomposition slice 6)

- **Track:** [crate-decomposition](../tracks/crate-decomposition.md), slice 6 —
  the structured-data/rendering split (D1) made real.
- **Realises ADRs:** [0100](../decisions/0100-structured-data-rendering-separation.md)
  (the single most important commitment: libraries emit structured data, a shared
  `bynk-render` renders; depends on `bynk-syntax::CompileError` **only**; the
  `AttributedError → CompileError` flattening stays **above** render — no
  `render → emit` edge), [0099](../decisions/0099-crate-layering-dependency-direction.md),
  [0102](../decisions/0102-foundation-types-boundary.md) (`Severity`/`line_col`
  already pre-positioned in the leaf for exactly this).
- **Depends on:** slices 1–5. **Pre-positioned by** slices 4–5: `line_col` and
  `Severity` already live in `bynk-syntax`, so a render crate can depend on the
  leaf alone.
- **Posture:** a **pure internal refactor** — no language/CLI surface, no output
  change. Every diagnostic transcript / golden-error fixture must render
  byte-identical.

## What this slice does

Create **`bynk-render`** over **`bynk-syntax` only** (+ `ariadne`), and move the
**pure-`CompileError` renderers** down into it. `bynkc` re-exports them so its
binary, the diagnostic transcripts, and the tests are unchanged.

**Moves into `bynk-render`** (each already takes `&[CompileError]` + `source` +
`filename` — the shape that makes the split trivial):
`render_errors`, `render_errors_plain`, `print_errors` (ariadne human),
`render_errors_short`, `print_errors_short`, `render_project_errors`,
`print_project_errors`, and the helpers `short_line` + `severity_word` (promoted
to `pub`). Deps: `bynk-syntax` (the `CompileError::report` methods + `Severity` +
`span::line_col`) and `ariadne`. **Nothing else.**

**Stays in `bynkc` — the flattening layer** (ADR 0100's invariant): the three
`ProjectFailure`/`AttributedError`-aware functions —
`print_project_failure`, `print_project_failure_short`,
`project_failure_short_lines`. `ProjectFailure`/`AttributedError` live in
`bynk-emit::project`; these functions flatten an `AttributedError` to
`(CompileError, text, label)` and then **delegate** to `bynk_render::{print_errors,
short_line, severity_word}`. The flattening (the `render → emit` edge if it
crossed) stays **up** here, never inside `bynk-render`.

```
bynk-syntax (CompileError · Severity · line_col)
   ▲
bynk-render   ariadne human + short/json over CompileError ONLY   [+ ariadne]
   ▲
bynkc         ProjectFailure flatteners delegate to bynk-render; re-exports it
```

## Decisions

**[DECISION 1] The flatteners stay in `bynkc`, not `bynk-render` — recommend yes
(it is ADR 0100, restated as code).** A reviewer's instinct may be "let
`bynk-render` take a `ProjectFailure` so the CLI calls one function." That creates
the `render → emit` cycle the layering forbids (`ProjectFailure` is in
`bynk-emit`, above render). The flatten-then-delegate split is the whole point of
D1. *No alternative* — this is the invariant the ADR exists to protect.

**[DECISION 2] The `bynk` driver does not adopt `bynk-render` this slice —
recommend defer to slice 7.** ADR 0100 says "both front-ends adopt it", but the
`bynk` driver today **shells** `bynkc` and renders the *text* it already produced
(`report.rs` over human/`short`/`json` strings) — it holds no `CompileError`s to
render. It gains `CompileError`s only when it **links** the pipeline (D2, slice 7);
it adopts `bynk-render` then. Slice 6 = `bynkc` adopts; the driver follows in 7.
*Alternative:* force it now — rejected, there is nothing for it to render until it
links.

**[DECISION 3] Version — recommend a minor `v0.65`.** New published crate
(`bynk-render`) → a new release surface needing the **bootstrap-seed**, added to
both publish loops (after `bynk-syntax`, before `bynkc` — it depends only on the
leaf). Per-increment minor cadence (v0.64 → v0.65).

## Risks & mitigations

- **A rendering regression.** Mitigated by the committed diagnostic transcripts
  (`docs/diagnostics/`, byte-stable plain output), the golden-error fixtures, and
  the `bynkc check --format short` problem-matcher pin — all must stay identical.
- **Accidentally pulling `ProjectFailure` into render.** Guarded by DECISION 1 and
  the crate graph itself: `bynk-render` does **not** depend on `bynk-emit`, so a
  `ProjectFailure` parameter would not compile there.
- **Boundary surfacings** (`short_line`/`severity_word` → `pub`) — mechanical.
