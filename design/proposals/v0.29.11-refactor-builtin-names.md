# v0.29.11 — Refactor track, increment 11: centralise the stringly-typed built-in names

- **Status:** Draft (proposal). **Merging is approval to build.** *(Refactor track item 8. Independent;
  slots into any calm increment. Behaviour-preserving — same string values, same comparisons, named.)*

## Context — bare string literals as the built-in vocabulary

- Built-in type/method names are compared as **bare string literals** scattered across the checker
  (verified counts in `checker.rs`):

  | literal | sites | | literal | sites |
  |---|---|---|---|---|
  | `"Json"` | 5 | | `"HttpResult"` | 3 |
  | `"List"` | 2 | | `"foldEff"` | 2 |
  | `"Int"` | 2 | | `"Map"` | 1 |
  | `"Float"` | 1 | | `"of"` | 1 |
  | `"unsafe"` | 1 | | `"raw"` | 1 |

  ~19 sites in `checker.rs` alone (and ~13 more across `emitter.rs` and the `project/` submodules,
  verified post-`29.2` split). A typo (`"Jsonn"`) compiles
  to a silently-never-matching branch; the full built-in vocabulary is impossible to enumerate without
  grepping; a rename touches an unknowable set of files.

## Builds on (verified)

- **These are stable, language-defined names** — base types (`Int`/`Float`/`Json`), kernels
  (`List`/`Map`), the HTTP result type (`HttpResult`), and privileged method/constructor names
  (`of`/`unsafe`/`raw`/`foldEff`). They change only when the language does, so a central registry is
  the right shape (one edit point per name).
- **The comparison shape is uniform** — `== "Json"`, `"Json" =>`, `Some("of")` — so the sweep is
  mechanical: replace each literal with a named constant, no logic change.

## Settled

- **▸ Byte-identical behaviour.** The constants hold the **same string values**; every comparison
  resolves identically. Pure renaming of literals to names; all fixtures pass unedited.
- **▸ One source of truth.** After this increment the built-in vocabulary is enumerable in one place
  and renameable in one edit.

## Decisions (open — settle at review)

**[DECISION] `const` module vs enum.** *Rec:* a `mod builtin_names { pub const JSON: &str = "Json"; … }`
(plus method names `OF`/`UNSAFE`/`RAW`/`FOLD_EFF`). It is the **minimal behaviour-preserving** change —
the values stay `&str`, comparisons stay `== JSON`, no match-logic rewrite. *Alternative:* an `enum
Builtin` with `as_str`/`from_str` — stronger typing and exhaustive matching, but it forces converting
comparison sites to enum dispatch (more invasive, more risk) and the parse boundary (`&str` →
`Builtin`) adds its own failure surface. *Rec: ship the `const` module now*; an enum is a possible
*later* increment if exhaustiveness proves valuable, built on the constants.

**[DECISION] Where it lives + scope of the sweep.** *Rec:* a small `builtin_names` module in `karnc`
(near the AST or a new `builtins.rs`), used by `checker.rs`, `emitter.rs`, and the `project/`
submodules. The sweep covers **all three** areas, not just the checker — the drift hazard is cross-file
(a checker rename not mirrored in the emitter). Drive the sweep by grepping each literal to exhaustion; **`log` any literal
deliberately left bare** (e.g. a string that is coincidentally equal but semantically unrelated) so the
exclusion is visible, not silent.

**[DECISION] Disambiguate type names from method names.** *Rec:* group as `builtin_names::types::*`
(`JSON`/`LIST`/`MAP`/`INT`/`FLOAT`/`HTTP_RESULT`) and `builtin_names::methods::*` (`OF`/`UNSAFE`/`RAW`/
`FOLD_EFF`) so a constant's role is legible at the use site and the two namespaces can't be crossed by
accident.

## Spec delta

**None.** No language surface, no `docs/src/spec/` chapter — an internal literal→constant refactor.

## Tests

- **No new golden fixtures.** Every fixture passes **unedited** — the values are unchanged.
- **Optional:** a trivial test asserting the constants hold their expected values (guards an accidental
  edit to the registry from silently changing matching behaviour).

## Diagnostics / Risks

- **Incomplete sweep undercuts the point.** The value is *enumeration*; a missed literal leaves a
  silent fork. Mitigation: grep each name to zero remaining bare occurrences across `checker.rs`,
  `emitter.rs`, and the `project/` submodules, and `log`/comment any intentional exclusion.
- **False matches.** A literal `"of"` or `"raw"` may appear in an unrelated string (a message, a key);
  replace **only** the built-in-vocabulary comparison sites, not every textual occurrence. Read each
  site before swapping.
- **Lowest-risk item in the track.** No control flow, no data model, no API — pure literal naming.

## New decision records this increment adds

- **None.** Covered by the `29.1` track-level ADR.

## Out of scope (→ later / independent)

- **An `enum Builtin` with exhaustive dispatch** — a possible follow-up built on these constants, not
  this increment.
- **Non-built-in stringly-typed comparisons** (diagnostic codes, capability names, etc.) — different
  concern; this increment is the language-defined built-in vocabulary only.

## Definition of done

A `builtin_names` module (`types::*` + `methods::*` constants holding the current string values) is the
single source for the built-in vocabulary; the bare-literal comparison sites in `checker.rs`,
`emitter.rs`, and the `project/` submodules use the constants, with any intentional exclusion logged;
all fixtures
pass **unedited** (plus an optional registry-value test); `cargo build`/`test`/clippy and CI stay
green; changelog notes the centralisation. The implementing PR **deletes this file**.
