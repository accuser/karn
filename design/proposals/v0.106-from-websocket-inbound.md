# Proposal v0.106 — `from WebSocket` slice 3b-iii: inbound `on message` + `on close` (the receive path)

- **Status:** Draft, for approval.
- **Scope:** a single increment — the **inbound/close half** of the `from WebSocket` protocol. Slices 3a/3b-i/3b-ii built the outbound (server→client) path: an edge-authenticated `on open` transfers a `Connection` to an agent that `send`s to it, and the connection survives hibernation. The `in: ClientFrame` type has been declared since slice 3 but **no handler consumes it** — a client cannot yet talk *back*. 3b-iii adds the two remaining lifecycle handlers the §20 design commits to: **`on message(frame)`** (an inbound frame) and **`on close`** (the socket closed), completing the bidirectional contract.
- **Provenance:** the inbound-dispatch piece ADR 0133 deferred ("`webSocketMessage(ws, msg)` decoding against `in:` and routing to an agent handler — a new protocol surface"). Realises design notes §7 / §20 (the chat-room's `on message`/`on close`) and the **DECISION E** framing of ADR 0132. Consumes the hibernation attachment of [ADR 0133](../decisions/0133-from-websocket-hibernation.md) and the DO-hosted-handler + self-agent lowering of [ADR 0132](../decisions/0132-from-websocket-protocol-workers.md). **Security-bearing** (a client now drives agent logic over the channel): runs `/security-review` + `/code-review`. ADR provisionally **0134**, version **0.106.0**.

## 1. The model — `on message`/`on close` are service handlers that run in the hosting DO

The §20 design notes are explicit: the inbound handlers are **service** handlers
(like `on open`), authenticated by the same `by` actor, and they `match` the
decoded frame and dispatch to the agent's ordinary `on call` methods:

```
service ChatGateway from WebSocket(in: ClientFrame, out: ServerFrame) {
  on open    by user: Participant (roomId: RoomId) given Rooms { … Room(roomId).join(user.identity, connection) }
  on message by user: Participant (roomId: RoomId) given Rooms (frame: ClientFrame) {
    match frame { Send(text) => <- Room(roomId).post(user.identity, text); Typing(_) => () }
  }
  on close   by user: Participant (roomId: RoomId) given Rooms { … Room(roomId).leave(user.identity) }
}
```

So **variant routing is just a `match`** — no new routing machinery. The work is
the *plumbing*: on an inbound frame, the hosting Durable Object's
`webSocketMessage(ws, raw)` must (1) **decode** `raw` against `in:` (reject an
undecodable frame, never dispatch — DECISION E), (2) recover the **sender identity
and route params** the handler's `by user`/`(roomId)` need, and (3) re-present the
firing **`connection`**, then run the `on message` body in the DO (where the agent
is local, exactly as `on open` does via the self-agent lowering).

The identity and params are **not re-derived from the frame** (a client must not
re-authenticate or re-route per message): they were fixed at `on open`, when the
edge authenticated the actor. 3b-iii carries them in the socket's
**`serializeAttachment`** (extending 3b-ii's `{ connId }` to `{ connId, identity,
params }`) — server-side, not client-forgeable — so a waking `webSocketMessage`
recovers a *trusted* sender without re-verification.

## 2. Scope and non-goals

**In scope.**

- **`on message(frame: In)` and `on close` service handlers** on a `from WebSocket`
  service: parse (`HandlerKind::Message` is the queue kind — the WebSocket ones are
  distinct, disambiguated by the `from WebSocket` protocol; provisionally
  `HandlerKind::WsMessage` / `HandlerKind::WsClose`), check (the `frame` param's type
  must equal the service's `in`; `by`/params/`given` validated as for `on open`), and
  emit DO-hosted bodies (`__wsMessage_<Service>` / `__wsClose_<Service>`) lowered with
  the same self-agent rewrite `on open` uses.
- **The runtime/DO `webSocketMessage` + `webSocketClose` handlers:** decode the raw
  message against `in:` via the boundary-decode discipline (reject-and-close on
  failure), `deserializeAttachment` → `{ connId, identity, params }`, re-wrap the
  socket as the `connection`, and dispatch to the emitted body.
- **The attachment carries `{ connId, identity, params }`**, written at `on open`.
- **Bundle/test:** `on message`/`on close` lower to callable surface methods a test
  drives directly with a `TestConnection` (symmetric with how `on open` is tested);
  no `webSocketMessage` runtime off-Workers.
- A **decode-reject negative path** and an inbound vertical (an `on message` that
  echoes a reply to the sender's connection) emitting + `tsc --strict`-clean.

**Non-goals (and why).**

- **Broadcast-to-all-connections** (`connections.values.parTraverse(c => c.send(…))`)
  — the held-aware iteration borrow surface, deferred since slice 2; the **slice-4
  closure**. 3b-iii's inbound vertical replies to the *sender's* connection (a single
  `connection.send`), not a fan-out, so it does not need the borrow surface.
- **Presence/`Idempotency`/`Events`** richness of the full §20 `post` — those are
  ordinary capability calls the agent already supports; the slice proves the inbound
  *plumbing*, not the chat's full feature set.
- **Per-frame re-authentication** — explicitly rejected (the socket is authenticated
  once, at `on open`); the attachment is the trust carrier.

## 3. Decisions

**[D1 — `on message`/`on close` are DO-hosted service handlers, identity/params from
the attachment.** They run in the connection-hosting DO (the same one `on open`
resolved to), lowered with the self-agent rewrite. The `by user` identity and the
route params come from `deserializeAttachment`, set at `on open` — **not** re-verified
per frame and **not** taken from the frame. *Recommended;* it reuses the 3b-i/3b-ii
machinery and keeps the channel authenticated-once.

**[D2 — the inbound frame is decoded against `in:`, fail-closed.** `webSocketMessage`
runs the raw message through the boundary deserialiser for the service's `in` type; a
structurally-invalid or refinement-violating frame is **rejected and the socket
closed** (`1003`/`1008`), never dispatched (DECISION E). The decode is the trust
boundary on client bytes. *Recommended;* it is the §7 "frames are typed messages"
model with the decode as the only place client bytes are trusted.

**[D3 — the attachment carries `{ connId, identity, params }`, written at `on open`.**
3b-ii wrote `{ connId }`; 3b-iii adds the verified `identity` (the actor's `sub`,
re-constructed to the identity type) and the route `params`. Bounded well under the
2 KB `serializeAttachment` limit; server-side, so not client-forgeable. *Recommended;*
it is the hook ADR 0133 named.

**[D4 — `connection` in `on message`/`on close` is the firing socket.** Per the design
notes, "the `connection` identifier refers to the connection that produced the frame":
`webSocketMessage(ws, …)`/`webSocketClose(ws)` re-wrap `ws` (with its attachment
connId) as a `WorkersConnection`, bound as `connection` in the body — so an `on
message` can reply to its sender and `on close` can transfer the closing connection
out. *Recommended.*

**[D5 — `on close` is optional cleanup, not the only disposal.** A `from WebSocket`
service may declare `on close` to run on socket close (e.g. `Room(roomId).leave(…)`,
which `remove`s the stored connection). It is **not required**: a stored connection
whose socket has closed already resolves to `None` (3b-ii fail-soft), so the absence of
`on close` is a stale map entry, not a live leak. *Recommended;* keeps `on close` a
domain hook, not mandatory boilerplate. (Whether the runtime should also auto-prune the
held map on close — independent of a user `on close` — is a small open question for the
ADR; recommend **no** auto-prune at v1, to keep one disposal path.)

**[D6 — bundle: callable handlers, test-driven.** Off Workers there is no
`webSocketMessage`; `on message`/`on close` lower to surface methods a test invokes
with a `TestConnection` and a constructed frame — symmetric with `on open`. *Recommended.*

## 4. Internal architecture

- **`bynk-syntax`:** parse `on message (frame: T)` and `on close` inside a `from
  WebSocket` service; new `HandlerKind` variants (the queue `Message` kind is a
  service-`from queue` concern — kept distinct). Reuse the `by`/params/`given` parsing
  `on open` uses.
- **`bynk-check`:** validate the WS message/close handlers — at most one of each per
  service; the `on message` `frame` param's type **must equal** the service's `in`
  type; `by` mandatory (same edge-auth-before-anything rule); the body may transfer to
  agents (no single-transfer shape constraint — that governs `on open`'s connection
  hand-off only).
- **`bynk-emit`:** emit `__wsMessage_<Service>` / `__wsClose_<Service>` DO methods
  (lowered with `ws_self_agent`); add `webSocketMessage`/`webSocketClose` to the DO
  class that decode + `deserializeAttachment` + dispatch; write `{ identity, params }`
  into `serializeAttachment` at the `on open` accept. The edge wrapper is unchanged
  (auth still happens once, at upgrade).
- **Runtime (`connection.ts`):** a decode helper for the inbound boundary; `WorkersWebSocket`
  already has `deserializeAttachment`. `resolveConnection`/`connIdOf` unchanged.

## 5. Risks

- **Trusting the attachment.** The whole inbound path trusts `{ identity, params }`
  from `deserializeAttachment`. This is sound — the attachment is set server-side at an
  edge-authenticated `on open` and the platform binds it to the socket — but it is the
  one claim `/security-review` must confirm: there is no path where an attacker sets or
  alters the attachment, and a frame on an un-attached socket (no `on open` ran) is
  rejected.
- **Decode is the client-bytes trust boundary.** A malformed frame must never reach a
  handler; the reject-and-close path is the defence and is pinned by a negative/behaviour
  test.
- **`HandlerKind` proliferation / parser ambiguity.** `on message` already exists for
  queues; the WS variants must not bleed into `from queue` (or vice-versa). Disambiguated
  strictly by the service protocol; a `from queue` `on message` is unchanged.
- **No real-`webSocketMessage` test in the unit harness** (needs Miniflare/workerd) —
  coverage is the bundle behaviour test (drive `on message` with a `TestConnection`,
  assert the reply captured) + a shape-snapshot Workers fixture + `tsc --strict` + the
  strip-types guard, plus a manual `wrangler dev` note. Never claim a runtime proof that
  did not run.

## 6. Testing

- Bundle behaviour test: an `on message` echo vertical — drive the handler with a
  decoded frame + `TestConnection`, assert the reply frame was captured; a malformed
  frame is rejected.
- Workers shape-snapshot fixture: the `webSocketMessage`/`webSocketClose` DO handlers +
  the `__wsMessage`/`__wsClose` bodies + the extended attachment, `tsc --strict`-clean.
- Negatives: an `on message` whose `frame` type ≠ the service `in`; an `on message`
  without `by`.
- Full gate `BYNK_REQUIRE_TSC=1 cargo test --workspace --locked` + `fmt` + `clippy`.

## 7. Rollout

ADR 0134; version 0.106.0; changelog + spec §7/§20 status update; delete this proposal
on merge. `/security-review` + `/code-review` before the gate. With 3b-iii the
bidirectional channel is complete; the **slice-4 closure** (broadcast / the held-aware
iteration borrow surface, plus the full §20 chat-room running end-to-end) is the track's
final increment.
