# v0.54 — Actors, slice 6 (cross-context `Internal` actors): the live `CallerId` value

- **Status:** Draft (proposal). **Merging is approval to build.** *(Feature-track slice — the sixth of
  the actors track, `design/tracks/actors.md`. Cashes the deferred half of Q7: the **runtime value**
  behind the already-typed `Caller` prelude actor, so a cross-context `on call` handler's `by c: Caller`
  yields the *real* calling context, not the placeholder `undefined`.)*
- **Sequences after v0.53 (authorisation invariants).** It is the smallest remaining actors slice and a
  different subsystem — the v0.6 cross-context Service-Binding pipeline, not the HTTP auth seams. The
  *typed* half (`Caller`/`CallerId`) landed in Foundations (v0.45); this is the value half only.
  Provisional version (`v0.54` assumes nothing jumps the queue — renumber if sequencing shifts).
- **Verification-bearing? Lightly.** It carries the `/security-review` + `/code-review` gate (ADR 0076)
  because it touches an identity-bearing boundary, but it introduces **no crypto** — verification is
  static / channel-trust (the `Internal` scheme: "the channel itself is the assertion"). The thing to
  defend is the trust model, below.

## Context — the callee knows *that* it was called, not *who* called it

Cross-context calls already work (v0.6): context A calls context B's `on call` service over a
Cloudflare Service Binding, dispatched at `/_karn/call/<service>` with a JSON args body
(`callService` in the runtime). The `Internal` scheme and the `Caller` prelude actor are in place, and
`CallerId` is typed as `String` — but `c.identity` on a `by c: Caller` handler currently lowers to the
placeholder `undefined` (`lower.rs`), and the wire format carries **only the arguments**: nothing
identifies the calling context. The v0.45 notes flagged this explicitly — "the calling-context value
arrives with a later slice."

This slice transmits the calling context's identity end-to-end, so a callee can finally read *who*
called it: `on call … by c: Caller (…)` → `c.identity : CallerId` is the caller's context name, minted
at the boundary, sealed, before the body runs.

## Builds on (verified)

- **The typed `Caller`/`CallerId` half (Foundations)** — `Caller` is a prelude `Internal` actor
  yielding `Identity::CallerId`, resolved to `Ty::Base(String)`; `on call` defaults to `Caller`. All
  static machinery is in place; only the value is missing.
- **The v0.6 cross-context pipeline** — `callService(binding, "<service>", argsJson, deser)` POSTs to
  `/_karn/call/<service>`; the callee's worker entry switch deserialises the args and invokes the
  surface. The calling context's qualified name is a **compile-time constant** at the call site (the
  emitter knows `commons.name`), so stamping it needs no new runtime input.
- **The Bearer deps-threading pattern (ADR 0085)** — a minted identity threads through the handler's
  `deps`, and `<binder>.identity` lowers to `deps.identity`. The `Caller` value reuses this exact
  shape: read at the boundary, threaded into `deps`, never re-derived in the body.

## Settled (from the track / foundational ADRs)

- **▸ Identity is a context-sealed value** (ADR 0081) — `CallerId` is read-only `.identity`-access,
  minted at the boundary, never constructed or passed by the body. **▸ Verify-then-body, fail-closed**
  (ADR 0082) — the value is established before the body runs. **▸ The `Internal` scheme trusts the
  channel** (ADR 0080) — a Service Binding is platform-dispatched and not externally routable, so the
  channel *is* the assertion; cross-context verification is **static-only** (Q7), no crypto.

## Decisions (open — settle at review)

**[DECISION] The `CallerId` value is the calling context's qualified name.** *Rec:* `c.identity` is the
caller context's name string (e.g. `"shop.orders"`), stamped as a compile-time constant at the call
site. It answers exactly "which context called me" — the natural sealed identity for an `Internal`
channel. *Alternatives:* (a) a structured `CallerId` record (context + binding + timestamp) — over-rich
for the channel-trust model and would change the `CallerId` type; (b) an opaque token — pointless when
the value is a known first-party name. Keep `CallerId = String` (the name), already the resolved type.

**[DECISION] Transmit via a request header, not an args envelope.** *Rec:* the call side stamps the
caller name into a reserved header (`X-Karn-Caller`); the args body stays exactly the arguments. This
keeps the wire format and its (typed) deserialisation **untouched** — the identity rides beside the
payload, not inside it. *Alternative:* wrap the body as `{ caller, args }` — rejected: it churns the
args serialise/deserialise path and the `/_karn/call/` contract for every service, for a value that is
metadata, not a parameter. The header is additive and backward-compatible.

**[DECISION] Trust model: static-only, first-party — the channel is the assertion.** *Rec:* the
compiler stamps the *real* context name as a constant; there is no app-controlled path to forge it, and
the `/_karn/call/` endpoint is reachable only over the internal Service Binding (not externally
routable). A malicious *first-party* context forging the header is **out of the threat model** — all
contexts in a deployment are one trust domain (this slice mints *identity*, not *authorisation*; there
is no inter-context permission check, and none is implied). *Alternative:* sign the caller header
(HMAC) for inter-context integrity — deferred; it presumes a multi-trust-domain model the platform does
not have, and adds the crypto this slice deliberately avoids.

**[DECISION] Absent header on a `by c: Caller` handler: fail-closed.** *Rec:* a Karn caller always
stamps the header, so its absence means a malformed / non-Karn call on the internal channel — reject it
(the `Internal`-channel analogue of 401), before the body, returning the boundary error the
`/_karn/call/` dispatch already uses. A **binder-less** `on call` (or one inheriting the `Caller`
default without binding) does not read the value, so it is unaffected — the requirement applies only
where the identity is actually captured. *Alternative:* default an absent caller to an empty/unknown id
— rejected: it is the fail-open footgun the sealed-identity model exists to prevent.

**[DECISION] Threading mirrors Bearer.** *Rec:* the callee's `/_karn/call/` dispatch reads the header,
threads the caller name into the handler's `deps` (a `callerId` field, present only when the handler
binds a `Caller` identity), and `c.identity` lowers to it — reusing the Bearer deps pattern verbatim
rather than passing the request into the handler. *Alternative:* hand the raw request to the handler —
rejected: breaks the sealed boundary (the body would see transport, not a minted value).

## Spec delta

- **`syntactic-grammar.md` / `static-semantics.md`.** No grammar change. Document that a `by c: Caller`
  `on call` handler now binds a live `CallerId` (the calling context's name); an absent caller on the
  internal channel is rejected fail-closed.
- **`emission.md`.** The call side stamps the `X-Karn-Caller` header with the compile-time context
  name; the callee's `/_karn/call/` dispatch reads it (fail-closed if absent and the handler binds a
  `Caller` identity), threads it into `deps`, and `<binder>.identity` lowers to the threaded value
  (replacing the `undefined` placeholder for `CallerId`).
- **`runtime-library.md`.** `callService` gains a caller-name parameter and sets the reserved header; a
  small helper reads/validates it on the callee side. No crypto.

## Tests

- **Positive golden:** two contexts where A calls B's `on call … by c: Caller (…)` handler; the call
  side stamps `X-Karn-Caller`, the callee threads it, and `c.identity` reads A's name; `tsc --strict`
  green. (A cross-context worktree golden, like the existing v0.6/v0.18 cross-context fixtures.)
- **Behavioral (extending the cross-context tests):** a driver invokes the callee's `/_karn/call/`
  dispatch with and without the header — with → the body sees the caller id; without (a `by c: Caller`
  handler) → fail-closed rejection.
- **Negative:** an absent caller header on a `by c: Caller` handler (fail-closed); confirm a binder-less
  `on call` is unaffected (no requirement, output unchanged).

## Diagnostics / Risks

- **Fail-open on an absent caller** — the gravest risk: a `by c: Caller` body running with an
  empty/placeholder identity. *Mitigation:* the absent-header rejection (the [DECISION] above);
  behavioral test for the missing-header path.
- **Wire-format compatibility** — the header is additive, but a mixed-version deployment (new callee,
  old caller) would see no header. *Mitigation:* the fail-closed rule treats a missing header as a
  rejected call; a binder-less handler is unaffected. Documented as a deploy-ordering note.
- **Scope discipline** — this mints *identity*, not *authorisation*. No inter-context permission check
  is added or implied; cross-context authorisation (if ever) is a separate, later question.

## Security / threat model

*Gates the slice (ADR 0076).*

- **Guarantees.** A `by c: Caller` body runs only with a caller identity established at the boundary
  (fail-closed if absent), sealed and read-only. The value is the calling context's name, stamped by
  the compiler as a constant — not app-controlled.
- **Trust boundary.** The `/_karn/call/` channel is the assertion: a Service Binding is
  platform-dispatched and not externally routable, so an external party cannot reach it or stamp the
  header. All contexts are first-party / one trust domain.
- **Explicitly out:** signed/authenticated caller headers (multi-trust-domain integrity), inter-context
  **authorisation** (permissions, not just identity), structured `CallerId`, and non-`Internal`
  cross-context schemes.

## New decision records this increment adds

- **ADR — the cross-context `CallerId` value is the calling context's qualified name, stamped by the
  compiler into a reserved `X-Karn-Caller` header beside the args body, read at the callee's
  `/_karn/call/` boundary and threaded into `deps`; an absent caller on a `by c: Caller` handler is
  fail-closed; trust is static / channel-based (no crypto), first-party.** Closes Q7's value half.

## Definition of done

A cross-context `on call … by c: Caller (…)` handler reads a live `CallerId` (the calling context's
name): the call side stamps `X-Karn-Caller`, the callee's `/_karn/call/` dispatch reads it (fail-closed
if absent), threads it into `deps`, and `<binder>.identity` lowers to it (no more `undefined`
placeholder); a binder-less `on call` is byte-unchanged; with golden emission (a cross-context fixture)
and `tsc --strict` green; a behavioral with/without-header test passes; any `karn.actor.*` diagnostic
lands; the ADR lands and `design/tracks/actors.md` marks slice 6 done (and the track's Q1–Q7 scope
complete); LSP/spec/changelog updated; `cargo build`/`test`/clippy + `cargo fmt --check` and CI green;
`/security-review` + `/code-review` gate the changes; the version is bumped. The implementing PR
**deletes this file**.
