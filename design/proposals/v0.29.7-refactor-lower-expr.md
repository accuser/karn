# v0.29.7 — Refactor track, increment 7: decompose `lower_expr`

- **Status:** Draft (proposal). **Merging is approval to build.** *(Refactor track item 3.
  **Independent** — its only prerequisite is the `29.1` safety net + the emitter golden fixtures.
  `lower_expr` is an *expression* lowerer (`-> String` + a `stmts` accumulator), **not** an
  indented-buffer writer — which is exactly why it never depended on the queue's "land `CodeWriter`
  first" note. (`29.6`/`CodeWriter` is now **shelved** — its premise didn't hold — so that ordering is
  moot; `lower_expr`'s decomposition stands on its own.) Behaviour-preserving — emitted TS is
  byte-identical.)*

## Context — a 600-line match with one 284-line arm

- `lower_expr(e: &Expr, stmts: &mut Vec<String>, cx: &mut LowerCtx) -> String`
  (`emitter.rs:3758`–`4361`, ~600 lines) is a single `match e.kind` over `ExprKind`. It returns the
  lowered **expression** string and pushes any required **statements** onto `stmts` — it does not write
  into an indented `out` buffer.
- The arms are wildly uneven (verified offsets within `lower_expr`): leaf arms (`IntLit`, `BoolLit`,
  `StrLit`, `None`, `UnitLit`) are one-liners, while a few are large:
  - **`MethodCall` ≈ 284 lines** (`~3983`–`4267`) — **the real monster**, not `Call`/`BinOp`. This is
    where the **string-kernel dispatch** lives: a cascade of `if let ExprKind::Ident(id) =
    &receiver.kind && …` branches selecting per-method lowerings.
  - `Call` (~50 lines), `BinOp` (~46), `Lambda`, `ConstructorCall`, `RecordConstruction`,
    `FieldAccess`, `If`.
- **The extraction template already exists in the same function.** `Match`, `Is`, and `Block` arms
  already delegate to `lower_match_as_iife`, `lower_is`, and `lower_block_as_expr` — per-`ExprKind`
  helpers with exactly the signature shape the larger arms should adopt.

## Builds on (verified)

- **Delegation precedent:** `ExprKind::Match { … } => lower_match_as_iife(discriminant, arms, cx)`,
  `ExprKind::Is { … } => lower_is(value, pattern, stmts, cx)`, `ExprKind::Block(b) =>
  lower_block_as_expr(b, cx)` (all in `lower_expr`). The new helpers (`lower_call`, `lower_bin_op`,
  `lower_method_call`, …) follow this shape: take the arm's destructured pieces + `stmts` + `cx`,
  return `String`.
- **`MethodCall` is internally sub-dispatchable.** Its body is a flat sequence of receiver-typed
  branches (`if let … = &receiver.kind` over kernel method names) — each branch is a candidate
  sub-helper (`lower_string_method`, `lower_list_method`, `lower_map_method`, …), turning the 284-line
  arm into a short dispatcher.
- **Golden fixtures cover the output** — every kernel-method and operator lowering has fixture
  coverage; that is the acceptance gate.

## Settled

- **▸ Behaviour-preserving per-arm extraction.** Bodies move into helpers verbatim; the `match` becomes
  a dispatcher of one-line delegations. No lowering logic changes; emitted TS is byte-identical.
- **▸ Helpers mirror the existing `lower_*` shape.** Same `(pieces, &mut stmts, &mut cx) -> String`
  contract as `lower_match_as_iife`/`lower_is`/`lower_block_as_expr` — no new threading.

## Decisions (open — settle at review)

**[DECISION] Which arms extract, and how far to sub-split `MethodCall`.** *Rec:* extract `Call`,
`BinOp`, `UnaryOp`, `ConstructorCall`, `RecordConstruction`, `FieldAccess`, `Lambda`, and
`MethodCall`; leave the one-line leaf arms inline. For `MethodCall`, **two-level**: a `lower_method_call`
dispatcher that routes by receiver kernel to `lower_string_method` / `lower_list_method` /
`lower_map_method` / … sub-helpers, so no single function carries the whole string-kernel cascade.
*Honest note:* the receiver-typed branches share local context (the lowered receiver, `cx` flags); the
sub-helpers must take that explicitly rather than re-deriving it — verify each branch's inputs before
lifting it.

**[DECISION] Where the helpers live.** *Rec:* keep them in `emitter.rs` adjacent to `lower_expr` (or a
`#[path]` `emitter/lower/` submodule if the file is split later — out of scope here). This increment is
**extraction, not relocation**; a file split is a separate mechanical step if wanted.

**[DECISION] One PR or staged?** *Rec:* **two PRs** — the straightforward arms (`Call`/`BinOp`/
`ConstructorCall`/`FieldAccess`/`Lambda`) first, then the `MethodCall` two-level carve (where the
shared-context risk concentrates). Both golden-verified.

## Diagnostics / Risks

- **Shared local context in `MethodCall`.** The branches lean on the lowered-receiver string and `cx`
  state; an extraction that drops or reorders a `cx` mutation changes output. Mitigation: lift one
  branch at a time, run the kernel-method fixtures after each, and pass context explicitly (a missing
  input is a compile error, not a silent diff).
- **`stmts` ordering.** Arms push side-statements onto `stmts` in a specific order; helpers must push
  in the same order at the same point. Preserve the push sites exactly.
- **Borrow-checker friction over output drift** — as elsewhere in the track, the likely failure is
  non-compilation (overlapping `&mut cx`/`&mut stmts`), the *safe* failure for a refactor; resolve by
  tightening borrows, never by cloning in a way that changes semantics.
