# v0.116 — step invariants `transition`: the invariant subject widens to the step (old→new), checked at the commit boundary (testing track, slice 4)

- **Scope:** a single-increment **language + runtime** change — the fifth testing-track
  slice and the rung that widens the invariant subject from a *snapshot* to a *step*. It
  adds a `transition <name>: <pred over old/new>` declaration on an `agent`, beside
  `invariant`, where `old` and `new` are the pre- and post-commit state records. It is
  checked at **one** point — the **commit boundary**, exactly where snapshot invariants
  already run — so it fires at every tier for free and adds no test-runner surface.
  Version → **v0.116**.
- **Realises:** the testing track's subject ladder — `value → domain → call → snapshot →
  step → history` (`design/tracks/testing.md`). Snapshot invariants (`invariant`, live
  since v0.80) constrain a *committed state*; `transition` constrains the *move between
  two committed states* (`old → new`). Both are declared on the agent and carried by the
  code, not written in a test. It is the same one predicate surface (ADR 0144): a
  transition clause *is* the invariant predicate, now over two states.
- **Relates:** ADR 0144 (one predicate surface — a transition clause is the invariant
  predicate verbatim, `implies`/`is`/pure `Bool`); the v0.80 invariant machinery that
  evaluates snapshot predicates in `commitState` before `storage.put` (transitions extend
  that gate, they do not add a new one); the slice-3 contract clauses (`requires`/`ensures`
  — the *call* rung, with `result` as its contextual binding; `transition` is the *step*
  rung, with `old`/`new` as its contextual bindings, parsed the same contextual way);
  DECISION P (a fabricated agent state that satisfies every invariant is **valid but not
  reachable**, which is why `transition` gets **no** runner attack in this slice — see
  *One enforcement point*).
- **ADR at implementation:** one, numbered at authoring (next free after 0150; do not
  pre-allocate): **E** — *the invariant subject widens to the step (`transition`) — but
  not to emission.* The ADR also settles **DECISION I** (the step-rung keyword surface) as
  one of its decisions: a **dedicated `transition` keyword** over `old`/`new`, not a
  folded `invariant … step old -> new { … }`. It records the **DECISION O** disposition:
  this slice ships with `is`/`implies` only; enum positional `Ord` (needed for `>=`
  transitions like `new.status >= old.status`) is a **separate prerequisite**, out of
  scope here. Cites ADR 0144.

## The surface

A `transition` sits on the agent, in the same declaration group as `invariant` — a named
predicate, but over the *pair* of committed states rather than one:

```bynk
-- commerce/order.bynk
type OrderStatus = enum { Pending, Placed, Paid }

agent Order {
  key id: OrderId
  store status:     Cell[OrderStatus] = Pending
  store paymentRef: Cell[Option[AuthId]]

  -- snapshot: a committed state is internally consistent (exists today)
  invariant paid_has_payment_ref:
    status == Paid implies paymentRef.isSome()

  -- step: a paid order can never become unpaid (uses `is`/`implies`; needs no enum Ord)
  transition paid_is_terminal:
    old.status is Paid implies new.status is Paid

  on call pay(ref: AuthId) -> Effect[()] {
    paymentRef := Some(ref)
    status     := Paid
    ()
  }
}
```

- **`transition <name>: <pred>`** — a predicate over **two** contextual bindings, `old`
  and `new`, each the agent's state record (every `store` field, at its declared type).
  `old` is the last committed state; `new` is the proposed state this commit would
  persist. Both are read-only records; the predicate is pure `Bool` in the invariant
  surface: `implies`, `is` with narrowed bindings, operators, pure methods — no effects,
  capabilities, `expect`, or `Val` (`bynk.transition.impure_predicate`,
  `bynk.transition.not_bool`).
- **`old` / `new` are contextual identifiers**, special *only* inside a `transition`
  predicate (as `result` is only inside an `ensures`, and `for`/`all` are only in a
  `for all` head). Outside a transition they remain ordinary identifiers, so existing code
  naming a value `old` or `new` still parses — verified by a fixture. They never become
  reserved words.
- Clauses are **named** like invariants (the name rides the failure report), and a
  `transition` declared with a duplicate name is `bynk.transition.duplicate_name`.
- A `transition` whose predicate references **neither** `old` nor `new` is not a step
  claim at all — it is a snapshot invariant misfiled. Flag it conservatively
  (`bynk.transition.no_step_reference`, a recommendation) and point the author at
  `invariant`.

Transitions attach to **agents** only (the sole subject with a commit boundary and a
prior state). A `transition` on a value `type` is `bynk.transition.not_on_agent`.

## One enforcement point — the commit boundary (DECISION E)

A snapshot `invariant` is evaluated in the generated `commitState(s)` against the proposed
state `s`, before `storage.put`, throwing `InvariantViolation` so the offending commit
never persists. A `transition` rides **the same gate**, with one addition: the gate needs
the *old* state as well as the new.

- **Old state is threaded to the gate.** A handler loads the current state at entry
  (`loadState()`), mutates a copy, and calls `commitState` at handler end. The entry
  snapshot — the state *before* this handler's writes — is exactly `old`; the proposed
  record is `new`. The commit gate gains that `old` binding and evaluates each transition
  after the snapshot invariants, throwing the same `InvariantViolation`-family fault
  (agent type + transition name, never the key — ADR 0107) before the write.
- **The genesis commit has no `old`.** An agent's first commit has no prior committed
  state, so there is nothing to transition *from*. Transition checks are skipped on that
  commit (detected by the same state-presence check the agent already uses for the
  load-time invariant twin); the genesis state is constrained by snapshot invariants
  alone. From the second commit onward, every commit checks every transition.
- **It fires at every tier for free.** Because the check lives at the commit boundary, a
  transition is enforced under `unit`, `integration`, and `system` alike (Pillar IV) — so
  promoting a `case` can surface a collaborator's transition a stub was hiding, with no
  new test code. This is the invariant↔test duality the track builds toward: you author
  the step claim once, on the agent, and it is checked everywhere the agent runs.

**Why there is no runner attack (DECISION P).** Slice 3's contracts got a *second*
enforcement point — the runner generates arguments and attacks `ensures`. A `transition`
does **not**, and deliberately. Attacking a transition generatively would mean fabricating
an `old` state and a `new` state — but a fabricated state that satisfies every invariant
is **valid, not reachable**: no handler sequence need ever produce it, so a counterexample
over fabricated states is a false positive that erodes trust in the runner (DECISION P,
Risks). The sound way to attack a transition is to **drive the real handlers from the
initial state and observe the old→new pairs they actually reach** — handler-sequence
generation, which is the *history* rung (slice 7). Until then, a transition is checked
against the *real* old→new of every actual commit, at every tier — which is already strong
coverage and never lies.

## `is`/`implies` only; ordered transitions are a separate prerequisite (DECISION O)

The track's transition examples are written with `is`/`implies`
(`old.status is Paid implies new.status is Paid`) — a *terminality* / *monotonicity of
membership* shape that needs no ordering. An **ordered** transition
(`new.status >= old.status`, "status only ever advances") needs enums to be orderable, and
today they are not: the changelog enumerates the orderable types (`Instant` v0.90,
`Duration` v0.86) and enums never appear. This slice ships `transition` with
`is`/`implies`/`==`/`!=` and the ordering already available on numeric and temporal
fields; granting enums a **declaration-positional `Ord`** so `>=` becomes available on a
status enum is a **separate prerequisite** (DECISION O), decided and shipped on its own
before ordered-status transitions are authored. Out of scope here, named so the boundary
is deliberate.

## What changes, against the current code

`transition`/`old`/`new` are new; the commit-gate predicate lowering and the
`InvariantViolation` fault are reuse.

1. **Lexer / keywords** — `transition` becomes a keyword (heads an agent transition
   declaration, beside `invariant` at `bynk-syntax/src/lexer.rs`); `old` and `new` stay
   **contextual** identifiers (special only inside a `transition` predicate), so existing
   code naming a value `old`/`new` still parses.
2. **AST / parser** — add a `Transition { name, predicate, span }` and a
   `transitions: Vec<Transition>` field on the agent declaration (beside `invariants`),
   parsed in the same declaration loop as `invariant`. Reuse the predicate /
   `implies` / `is` parsers; bind `old`/`new` contextually within the predicate.
3. **Checker** — type each transition predicate as pure `Bool` in the invariant-predicate
   context, with scope = `old: <StateRecord>` + `new: <StateRecord>` (the agent's state
   record type). Reject non-`Bool` (`bynk.transition.not_bool`), impurity/effects
   (`bynk.transition.impure_predicate`), duplicate names
   (`bynk.transition.duplicate_name`), a transition on a non-agent
   (`bynk.transition.not_on_agent`), and — conservatively — a predicate referencing
   neither `old` nor `new` (`bynk.transition.no_step_reference`). Reuse the existing
   invariant purity gate (`check_invariants`).
4. **Emitter — commit gate** (`bynk-emit/src/emitter/emit.rs`) — thread the entry-snapshot
   `old` state into `commitState` (currently `commitState(s)` → `commitState(old, s)`, or
   capture the pre-mutation load and bind it), and, after the snapshot-invariant checks,
   emit a transition check per clause: lower the predicate with `old`/`new` bound to the
   two state records, `throw invariantViolation(agent, name)` on violation. Skip the block
   on the genesis commit (no prior state), reusing the load-time presence check. No new
   runtime-library export.
5. **Diagnostics** — add the `bynk.transition.*` family (`not_bool`, `impure_predicate`,
   `duplicate_name`, `not_on_agent`, `no_step_reference`) and a "Transitions" category
   (or fold into the existing invariant/predicate category — decide at authoring for the
   `bynk.<position>.not_bool` family consistency ADR 0144 wants).
6. **tree-sitter** — a `transition_decl` rule on the agent body; `old`/`new` as contextual
   highlights inside a transition predicate; corpus + `highlights.scm` (`transition` to
   `keyword.declaration`); re-bless `grammar.json`, the corpus, and the embeddable-rule
   count. **vscode-bynk** — add `transition` to the `keyword.declaration` match in
   `bynk.tmLanguage.json`.

## Migration (clean-slate; additive)

Transitions are **new surface** — nothing is removed or renamed, so no codemod is needed.
New fixtures cover: a passing transition (`old.status is Paid implies new.status is Paid`,
held across a legal commit); a transition **violation** faulting at the commit boundary
(an illegal `Paid → Pending` move throws `InvariantViolation` and the commit does not
persist); the **genesis-skip** (the first commit is not transition-checked); an
`impure_predicate` / `not_bool` rejection; a `no_step_reference` flag (a `transition` using
only `new`-free snapshot terms); a `not_on_agent` rejection; and an ordinary-code fixture
binding a value named `old`/`new` to prove they stay contextual. The v0.115 contract
surface is untouched.

## Risks

- **Old-state threading correctness.** The commit gate must see the *true* pre-commit
  state as `old`, not a partially-mutated copy. Capture the entry snapshot before the
  handler's first write and thread it unchanged to `commitState`; a fixture asserts a
  handler that writes then reads still evaluates the transition against the *entry* `old`.
- **Genesis commit.** A transition has no meaning on the first commit (no prior state).
  Skipping it must use the same presence check the agent already relies on, and the skip
  must be a genuine "no prior state", not "prior state equals the zero value" — a fixture
  covers a first commit whose new state would *violate* a transition and confirms it is
  allowed (and snapshot invariants still apply).
- **No false-positive attack (DECISION P).** This slice must **not** add a generative
  attack on transitions; fabricated states are valid-but-unreachable and would produce
  counterexamples production can never hit. The proposal's coverage is the real commit
  boundary at every tier; handler-sequence generation is slice 7. Stated so the absence is
  a decision, not an omission.
- **Ordering temptation (DECISION O).** Authors will want `new.status >= old.status`;
  enums are not ordered today. The slice ships `is`/`implies` and documents the ordered
  form as a named prerequisite, so a `>=` on an enum field fails with a clear "enums are
  not ordered" message, not a confusing type error.
- **Two-state predicate typing.** `old`/`new` must both be the state record with `is`
  narrowing and `impure_predicate` biting exactly as in a snapshot invariant. Reuse the
  invariant typing context with a two-binding scope; a fixture exercises `is`-narrowing on
  both `old` and `new`.

## Out of scope (named, deferred to their slices)

- **Generative / behavioural attack on transitions (handler-sequence generation).** The
  sound way to attack a step claim — drive the real handlers and observe reached old→new
  pairs — is the *history* rung, **slice 7** (DECISION H). Transitions here are
  commit-boundary only.
- **Enum positional `Ord` (ordered `>=` transitions).** A **separate prerequisite**
  (DECISION O), shipped on its own before ordered-status transitions are authored.
- **Emission is not an invariant rung (DECISION E).** What a handler emits is derivable
  from its body; there is no `transition`-style `emits` clause. A scenario-specific
  emission is a test observation — slice 5 (DECISION F); a *universal* emission guarantee
  is a cross-cutting policy / use-case minimal guarantee — deferred (DECISION U).
- **Observation (`called`/`never called`/`trace`)** — slice 5 (DECISION F).
- **Tier dial + `provides`** (retires `mocks`) — slice 6 (DECISIONS G/M/R).
