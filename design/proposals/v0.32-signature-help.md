# v0.32 — Signature help

- **Status:** Draft (proposal). **Merging is approval to build.** *(Tooling-queue item 2 — the natural
  partner to the now-complete comprehensive completion. `textDocument/signatureHelp`: while typing a
  call's arguments, show the callee's signature with the **active parameter** highlighted. `karn-lsp`
  with a small shared-rendering tidy in `karnc`; most of the machinery already exists.)*

## Context — completion's other half

Completion shipped over v0.30–v0.31 (positional → name-member → value-member → locals). Its partner is
**signature help**: completion tells you *what to call*; signature help tells you *what the call
expects* once you've opened the `(`. The queue pairs them ("pairs with completion; medium"), and after
the completion work the pieces are largely in place — the rendering, the registries, and the
receiver-typing are all reusable.

## Builds on (verified — scout citations)

- **The signature *rendering* already exists.** `karn-lsp/src/symbols.rs` has `type_ref_str(&TypeRef)`
  (`:280+`) rendering Karn-syntax types (`Result[T, E]`, `A -> B`, `List[T]`, …), and renders free-fn /
  method signatures (`:107-112`, `"{name}: {type}"` params + return) and **capability-op** signatures
  (`:132-138`) for hover. Signature help renders the same way — reuse it (make `type_ref_str` and the
  per-kind helpers shareable).
- **The registries are display-ready.** `karnc::kernel_methods`'s `KernelMethod.signature` is already a
  signature string (`fold(init: U, step: (U, T) -> U) -> U`), and `completion.rs`'s `BUILTIN_STATICS`
  carries `(name, signature)` for `Int.parse`/`Json.decode`/… — both built for completion, both exactly
  what signature help shows.
- **Value-receiver typing is built.** `expr_types` + `type_at_offset` + `value_receiver_rewrite`
  (v0.30.2) already type a value receiver mid-edit — so `list.fold(`/`str.split(` resolve to a kernel
  method's signature, with the same clean-file ceiling.
- **The lexical-context template is the completion contexts.** `FnDecl` exposes `params`/`return_type`
  (`ast.rs:830`); `CapabilityDecl.ops` (used in completion slice 2); the call-context detection is the
  same line-prefix scanning slice 1–3 use.
- **The capability isn't declared yet** — `completion_provider` is set (`main.rs:1245`) but no
  `signature_help_provider`; this adds it (trigger chars `(` and `,`, plus retrigger).

## Decisions (open — settle at review)

**[DECISION] Call-context detection (lexical).** *Rec:* from the cursor, scan the line(s) back to the
**innermost unclosed `(`** (bracket-balanced, so a nested `f(g(x|))` targets `g`); the **callee** is the
identifier (or `Recv.member`) immediately before that `(`; the **active parameter** is the count of
top-level commas between the `(` and the cursor. Mirror the completion detectors' conservatism
(string/comment-aware enough to not misfire); a call we can't resolve yields no help.

**[DECISION] Callee resolution — name vs value (the completion split).** *Rec:*
- **Name callees** (lexical, recovery-parse — no typed model): a free fn `bar(`, a method/static on a
  *type* `Email.of(` / `Color` variant payloads, a capability op `Clock.now(`, and built-in statics
  `Int.parse(`/`Json.decode(`. Resolve the `FnDecl`/`CapabilityOp`/registry entry and render.
- **Value-receiver methods** (`list.fold(`, `str.split(`): type the receiver via the v0.30.2 machinery
  → the kernel method's `signature`. Carries the clean-file ceiling (no help when the file doesn't
  check) — graceful, as in value-member completion.

**[DECISION] Share the renderer.** *Rec:* lift `symbols.rs::type_ref_str` + the fn/op signature helpers
into a small shared `signatures` module (or `pub(crate)` them) so hover and signature help share one
Karn-syntax renderer — no second copy. (A genuinely public `karnc` Karn-syntax `TypeRef` renderer is an
option, but keeping it in `karn-lsp` next to hover is lower-surface and sufficient.)

**[DECISION] The response shape.** *Rec:* one `SignatureInformation` (label = the rendered signature;
`parameters` = per-parameter ranges into the label) with `activeParameter` set; `activeSignature` 0.
Overloads don't exist in Karn (one signature per callee), so no multi-signature handling.

**[DECISION] Slicing.** *Rec:* **slice 1 — name callees** (free fns, statics, capability ops, `of`/
`unsafe`, variants): pure recovery-parse + the registries, no typed model — the bulk of calls, lowest
risk. **slice 2 — value-receiver methods**: the kernel-method signatures on a typed receiver, reusing
the v0.30.2 receiver-typing. *(Or fold into one increment — the machinery for both exists; settle at
review.)*

**[DECISION] New decision record.** *Rec:* **ADR 0065** — signature help mirrors completion's
name-vs-value callee split and shares hover's renderer; lexical call-context detection, semantic
signatures.

## Spec delta (sketch)

- **`design/karn-lsp-spec.md`** — a new "Signature help" capability section (§3.x): the lexical call
  context (innermost unclosed `(`, callee, active-parameter comma count), the signature sources (FnDecl
  / CapabilityOp / kernel registry / built-in statics / `of`-`unsafe`), value-receiver typing + its
  ceiling, and that Karn has no overloads. Note the shared renderer with hover (§3.3).

## Tests

- **`karn-lsp` pure-function:** call-context detection over crafted prefixes — innermost-paren targeting
  in nested calls, active-parameter index across commas (and not counting commas inside nested
  `()`/`[]`), and `[]` outside any call. Signature resolution per callee kind (free fn, `Int.parse`,
  `Clock.now`, `Email.of`, `xs.fold`) yields the expected label + parameter count.
- **Analysis-level:** a value-receiver method in a clean file yields its kernel signature; in a file
  with errors, none (the ceiling).

## Diagnostics / Risks

- **Active-call disambiguation in nested/multiline calls** — the chief detection risk; the innermost
  unclosed-paren rule + bracket-balanced comma counting handle the common cases; fixture them.
- **Clean-file ceiling** (value receivers only) — inherited from v0.30.2; name callees are unaffected.
- **Renderer drift** — sharing one renderer with hover prevents two divergent signature formats.

## Out of scope (→ later)

- Parameter-name *inlay* hints at call sites (queue item 10 — a different surface). Documentation/markdown
  in the signature label beyond the type signature (doc blocks could enrich it later, as hover does).
  Generic type-argument display in the signature (the checker doesn't store instantiations queryably —
  queue item 8's gate).

## Definition of done

`textDocument/signatureHelp` is declared (trigger `(`/`,`) and serves, for the call under the cursor,
one `SignatureInformation` with the callee's Karn-syntax signature and the active parameter highlighted
— for free fns, methods, capability ops, built-in statics, and `of`/`unsafe` (slice 1) and typed
value-receiver kernel methods (slice 2); the Karn-syntax renderer is shared with hover; pure-function +
analysis-level tests cover detection and each callee kind; `design/karn-lsp-spec.md` gains the section;
ADR 0065 added; CI green; changelog; version → 0.32.0. The implementing PR(s) **delete this file**.
