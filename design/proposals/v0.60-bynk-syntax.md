# v0.60 — Extract `bynk-syntax`: the foundation leaf (crate-decomposition slice 1)

- **Track:** [crate-decomposition](../tracks/crate-decomposition.md), slice 1 —
  the first extraction slice, the largest mechanical move and the **validation
  gate** for the whole track.
- **Realises ADRs:** [0099](../decisions/0099-crate-layering-dependency-direction.md)
  (downward layering, published lockstep), [0102](../decisions/0102-foundation-types-boundary.md)
  (the foundation-types boundary — `span`/`CompileError`/diagnostics registry in
  the lowest leaf).
- **Posture:** a **pure internal refactor** — behaviour-preserving, no language
  surface, no grammar/checker/emitter change. Every existing test passes
  unchanged; the only "new" behaviour is the crate boundary.

## What this slice does

Create a new published leaf crate **`bynk-syntax`** and move the foundation
modules **down** into it. `bynkc` then depends on `bynk-syntax` and re-exports
nothing it doesn't today (its public API is preserved via `pub use`).

**Modules that move** (verified to have **zero upward `crate::` edges** — they
reference only each other and external crates, so the extraction is mechanical):

| Module | Lines | Notes |
|---|---|---|
| `lexer` | ~1060 | `logos`-backed |
| `parser` | ~1050 (+ `parser/`) | |
| `ast` | ~1480 | |
| `span` | 35 | the position type every node/diagnostic carries |
| `keywords` | 180 | |
| `error` (`CompileError`) | 136 | the structured diagnostic `bynk-render` will consume |
| `diagnostics` (registry) | ~1490 | the `bynk.*` code registry — see DECISION 2 |

Total ≈ 5.4k lines, no logic change.

**What stays in `bynkc`:** everything else (resolver, checker, emitter, project,
fmt, the IDE surface, cli, the registries `kernel_methods`/`builtin_names`,
`firstparty`, `actors`). They become **downward consumers** of `bynk-syntax`
(they already import these modules via `crate::`; the imports re-point to
`bynk_syntax::`).

## Why this slice first

It is the lowest-risk, highest-leverage move: the leaf has no upward
dependencies, so nothing else changes behaviour, and it is the foundation every
later slice depends on. Per the track, **slices 1–2 are independently valuable
even if the track later stalls** — once `bynk-syntax` exists, `bynk-fmt` (slice
2) can re-point onto it and stop linking the compiler.

## Decisions

**[DECISION 1] Version bump — recommend a minor `v0.60`.** Adding a published
crate (`bynk-syntax`, `version.workspace = true` per ADR 0099) is a new release
surface, so it is more than a patch even though behaviour is identical. Follows
the established per-increment minor cadence (v0.57/v0.58/v0.59). *Alternative:* a
patch under a refactor-track-style posture (ADR 0059) — rejected because this
adds a crate to the release set, which 0059's behaviour-only patches did not.

**[DECISION 2] Include `diagnostics.rs` in this slice — recommend yes.** The
track's slice-1 enumeration lists six modules and omits `diagnostics`, but ADR
0102 places the diagnostics **registry** in `bynk-syntax` (codes must cross every
crate from the leaf). `diagnostics.rs` is confirmed clean (no upward edges), so
moving it now matches the crate's final shape and avoids a second migration.
*Handling its pins:* the `diagnostics_registry` (codes-vs-usage) and
`doc_diagnostics` (generates `docs/src/reference/diagnostics.md`) tests **stay in
`bynkc`** for now — `bynkc` depends on `bynk-syntax`, so a test there still sees
both the moved registry and the in-`bynkc` emission sites. They migrate to a
workspace integration test only when `bynk-check`/`bynk-emit` later split the
emission sites out (ADR 0102), not in this slice. *Alternative:* defer
`diagnostics.rs` to a later slice to match the literal slice-1 list — rejected as
needless churn given it is clean and foundational.

**[DECISION 3] The "source-cache type" (ADR 0102) is deferred — recommend
noting, not building.** ADR 0102 lists a source-cache type for `bynk-syntax`, but
there is no bespoke cache type today: rendering builds `ariadne::Source` from the
raw source string at the boundary (`lib.rs`). Since `bynk-render` does not exist
until slice 6, no cache type is introduced here. If slice 6 wants a shared cache
type, it lands then, in `bynk-syntax`. This slice moves only what exists.

## Risks & mitigations

- **The mechanical churn touches every downward consumer's imports.** Mitigated
  by the clean-leaf verification (no upward edges) and by keeping `bynkc`'s public
  API stable with `pub use bynk_syntax::…` so external consumers (`bynk`,
  `bynk-lsp`, `bynk-fmt`, tests) see no change in slice 1 — their re-pointing is
  their own later slices.
- **The drift-test pins (`diagnostics_registry`, `doc_diagnostics`).** Addressed
  in DECISION 2: they stay in `bynkc` and keep passing; the generated
  `diagnostics.md` is byte-identical.
- **Validation gate.** The whole workspace test suite (incl. golden emission
  fixtures) must pass unchanged. If anything but imports/paths had to change, the
  leaf was not actually a leaf — that is the signal to stop and re-scope.
