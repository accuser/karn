# v0.108 — Strip-only emission invariant (in-browser track, slice 0)

- **Scope:** the first slice of the [in-browser track](../tracks/in-browser.md) —
  an **emitter + tooling** increment. No language change, no grammar change, no
  server change. It closes the gap between "the emitter is 99% strip-only-clean"
  and a guaranteed, tested invariant.
- **Realises:** the track's §3.1 (Layer 1, the strip-only half) and its slice-0
  line in §6. Lands the **strip-only emission invariant ADR**
  ([ADR 0136](../decisions/0136-strip-only-emission-invariant.md)).
- **Why first:** smallest, lowest-risk slice; its value is independent of
  everything above it. A strip-only emitter is *required* for an in-browser eval
  that ships no transpiler, and it immediately repairs the `bynkc test --inspect`
  debug path (which runs emitted `.ts` under Node `--experimental-strip-types`).

## The invariant

Pure type-stripping erases type syntax by replacing it with whitespace. It cannot
erase **type-directed** constructs — constructor **parameter properties**
(`constructor(private x: T) {}`), `enum`, and `namespace` with values — which
imply runtime code. `tsc` accepts all of these, so the existing
`emitted_typescript_passes_tsc_strict` gate does not catch them. The invariant:
**every `.ts` bynkc emits is erasable by pure type-stripping**, across the whole
emitted surface, with no per-target branching.

## The audit (and a correction to the track)

The track's §3.1 stated an audit found **exactly one** type-directed construct —
the provider `given` constructor at `bynk-emit/src/emitter/emit.rs`. Re-running the
audit during implementation found **three more**, all parameter properties, in the
**shipped first-party bindings**: `SecretsProvider` in `bynk-{cloudflare,node}.ts`
and `WorkersKv` in `cloudflare.binding.ts`, each
`constructor(private env?: unknown) {}` using `this.env`. These are emitted output
too, and run under strip-only Node on `--inspect`; the parameter property
**silently broke every `--inspect` session that exercised `Secrets`** (or a
`given`-clause provider) — the module fails to parse before any breakpoint binds.
The emitted `AssertionError` test-scaffold class was already de-sugared (it carries
a comment citing ADR 0104); the four sites above were not.

## What this adds (against the current code)

1. **De-sugar the provider `given` constructor** (`emit.rs`), unconditionally, to a
   declared typed field + assigning constructor:
   `private deps: {…}; constructor(deps: {…}) { this.deps = deps; }`. Under the
   emitted ES2022 tsconfig (`useDefineForClassFields` on) the end state is identical
   to the parameter property, stays typed for the `tsc`/strict path, and strips
   clean. This *removes* the one type-directed site rather than adding a branch.
2. **The same de-sugaring in the three shipped bindings.**
3. **The fixture bindings** (`182`–`185`, hand-authored `*.binding.ts` that mirror
   the old emitter shape) adopt the de-sugared form, so the whole emitted bundle of
   every fixture is strip-clean — the property the playground needs.
4. **A load-bearing regression test** (`bynkc/tests/tsc_verify.rs ::
   all_emitted_typescript_strips_under_node`): every `.ts` emitted for the
   project-form positive fixtures is checked with `node:module`'s
   `stripTypeScriptTypes(code, { mode: 'strip' })` via a committed harness
   (`tests/support/strip_check.mjs`), in one Node process over the staged tree.
   `node --experimental-strip-types --check` is deliberately **not** used: a file
   leading with a `type`/`declare` statement trips its module detection and
   false-fails even though it strips cleanly. The check skips loudly when Node is
   absent or `stripTypeScriptTypes` is unavailable (< 22.13), matching the existing
   `tsc`/runtime strip gates.

## Decisions

- **[DECISION A] De-sugar unconditionally, not per target.** The provider `given`
  constructor was the only type-directed site in the emitter proper; rewriting it
  removes a special case instead of adding one. The form keeps the field typed.
- **[DECISION B] The invariant covers the shipped bindings.** They are emitted
  output and run under strip-only Node; the audit gap (above) is closed here.
- **[DECISION C] `stripTypeScriptTypes`, not `node --check`, is the oracle.** It is
  the exact strip-only transform, operates on a source string (no execution, no
  import resolution), throws precisely on the non-erasable constructs, and avoids
  the `--check` leading-statement false-fail.
- **[DECISION D] Version — minor v0.108; emitter + tooling.** A language/tooling
  increment per the proposals lifecycle.

## Docs delta

## Risks & mitigations

- **Strip-only as a standing constraint** forbids future type-directed emitter
  constructs forever. *Mitigation:* the regression test turns any violation into a
  failing build, with a message naming the de-sugar fix.
- **Third-binding parity.** The bindings stay in lockstep; the strip test now covers
  all three plus any user binding copied into a fixture.
- **The existing `embedded_runtime_strips_types_under_node` test** still uses
  `node --check` (it passes — runtime.ts does not lead with a `type`/`declare`).
  Migrating it to the harness is a tidy-up follow-on; the new comprehensive test
  already covers runtime.ts (it is emitted into every fixture).
