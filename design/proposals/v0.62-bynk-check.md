# v0.62 — Extract `bynk-check`: the semantic-analysis layer (crate-decomposition slice 3)

- **Track:** [crate-decomposition](../tracks/crate-decomposition.md), slice 3 —
  the largest slice, and the one the track flagged as the first with real
  conceptual risk (the check↔IDE seam). An upfront dependency investigation
  (below) shows the seam is **much thinner than feared**.
- **Realises ADRs:** [0099](../decisions/0099-crate-layering-dependency-direction.md)
  (downward layering, published lockstep), [0102](../decisions/0102-foundation-types-boundary.md)
  (the captured-table / drift-test homes).
- **Depends on:** slices 1–2 (`bynk-syntax`, `bynk-fmt` — shipped).
- **Posture:** a **pure internal refactor** — behaviour-preserving, no language
  surface, no checker/resolver behaviour change. The whole workspace suite
  (including golden emission + all the index/hints/capture drift tests) must pass
  unchanged.

## What this slice does

Create a new published crate **`bynk-check`** over `bynk-syntax`, and move the
semantic-analysis layer **down** into it. `bynkc` then depends on `bynk-check`
and re-exports its modules so the public API and the rest of the pipeline are
unchanged (the slice-1/2 pattern).

**Modules that move:** `resolver`, `checker` (+ `checker/`), `expr_types`,
`locals`, `kernel_methods`, `builtin_names`, `firstparty`, `actors`, **`index`,
and `hints`** (see DECISION 1 for why the last two come too). ~6k+ lines — the
largest slice.

**What stays in `bynkc`:** `emitter`, `project` (+ subdirs), `test_json`, `cli`,
and the binary/driver glue (`diagnose`, `diagnose_project`, the rendering
helpers). These become **downward consumers** of `bynk-check` — they already
import these modules via `crate::`; the imports re-point to `bynk_check::`.

## The seam, grounded (investigation result)

The track warned about a three-module check↔IDE seam (`expr_types` / `locals` /
`kernel_methods`) and "where the drift tests live." A file-level dependency scan
(real `crate::` code edges, excluding rustdoc links) shows the seam is almost
entirely cosmetic:

| Module | Real upward edge |
|---|---|
| `checker`, `kernel_methods`, `builtin_names`, `firstparty`, `actors` | **none** |
| `expr_types` → `hints`, `locals` → `hints` | **doc-comment links only** (no code) |
| `resolver` → `index`, `checker` → `index` | `use crate::index::{RefSink, SymbolKind}` — the binding-index **sink** |

So the *only* real coupling is resolver/checker writing into the index sink. With
`index` + `hints` placed in `bynk-check` (DECISION 1), the full set is
**self-contained over `bynk-syntax` with zero upward edges** (verified).

## Decisions

**[DECISION 1] `index` and `hints` live in `bynk-check`, not `bynk-ide` —
recommend yes.** The track's schematic graph drew `index`/`hints` in `bynk-ide`,
but the code shows they are **captured tables**, written *during* resolution and
checking — exactly what the track's prose assigns to `bynk-check` ("the captured
types/tables live in `bynk-check`; the queries live in `bynk-ide`"). `index.rs`
splits cleanly into a sink side (`RefSink`/`SymbolKind`/`SymbolKey`/`RefEdge`/
`SiteRef`/`SymbolModifiers`, written by resolver/checker) and a query side
(`ProjectIndex` + `symbol_at`/`sites`/`calls_into`/…, `IndexBuilder`, read by the
LSP). Both stay together in `bynk-check` as a captured table; **slice 5 builds
`bynk-ide` as the query/diagnose layer on top** (relocating `diagnose_project`,
the scope-at-offset / completion *entry points*, and `unit_sources` — not the
table types). *Alternative:* split `index.rs` now, sink-half down and query-half
up — **rejected**: it fragments a cohesive 623-line file mid-track for no slice-3
benefit, and the query side has no home until `bynk-ide` exists.

> **Track-graph correction (folded into this slice):** the graph's `bynk-ide`
> row loses `index`/`hints` as *type homes*; `bynk-ide` owns the *queries* over
> them. The track doc is updated to match on merge.

**[DECISION 2] Drift-test homes — recommend keep-in-`bynkc`, re-point scans.**
- `kernel_registry` (registry vs. checker dispatch): both halves move into
  `bynk-check`, so ADR 0102's "straddles check↔IDE" worry **dissolves** — the
  pin only needs `bynk-check`. It keeps working as a `bynkc` test (which depends
  on `bynk-check` and sees both via re-export); no new cross-crate dev-dep needed
  this slice.
- `diagnostics_registry` (codes vs. usage; already scans `bynkc/src` +
  `bynk-syntax/src` since slice 1): add **`bynk-check/src`** to its scan, since
  the resolver/checker emit sites move there.
- The capture/index drift tests (`expr_types_capture`, `locals_capture`,
  `inlay_hints`, `reference_index`, `call_hierarchy_index`, `method_index`,
  `field_op_index`, `implementation_index`, `semantic_tokens_index`,
  `suggestions`, `diagnose_project`) consume via the `bynkc::` API and keep
  working through the re-export — no change beyond what the compiler forces.

**[DECISION 3] Version — recommend a minor `v0.62`.** A new published crate
(`bynk-check`), so — like slice 1 — it is a new release surface and needs the
**bootstrap-seed before trusted publishing** (`release-bootstrap.yml`; the
crate is added to both publish loops, after `bynk-syntax` and before `bynkc`).
Per-increment minor cadence (v0.61 → v0.62).

## Risks & mitigations

- **Largest mechanical surface (~6k+ lines, many downward consumers).** Mitigated
  by the zero-upward-edge verification: the move is purely re-pointing imports,
  and `bynkc`'s re-export keeps consumers stable. If anything but imports/paths
  must change, the boundary was wrong — stop and re-scope.
- **The index/hints placement (DECISION 1) changes what slice 5 inherits.**
  Mitigated by recording it in the track doc now, so slice 5 is cut against the
  corrected graph (queries up, tables down).
- **Validation gate:** the full suite — golden emission, all capture/index drift
  tests, `kernel_registry`, `diagnostics_registry` — must pass byte-identical.
