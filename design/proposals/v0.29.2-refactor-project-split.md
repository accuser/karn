# v0.29.2 — Refactor track, increment 2: split `project.rs` into submodules

- **Status:** Draft (proposal). **Merging is approval to build.** *(Refactor track item 1 — the
  single highest-value structural refactor in `design/karn-refactor-proposal-queue.md`. Gate on
  `v0.29.1` is **satisfied** (merged): the characterisation pins for the helpers moved here are in
  place. Pure relocation: **no behaviour change, no API change** — `project.rs`'s public items keep
  their paths via re-export.)*

## Context — why this one, why now

- `project.rs` is **8,264 lines** (7,912 of production code plus the landed `v0.29.1` `seam_tests`
  module) — the largest file in the crate — and mixes **at least six unrelated
  jobs** in one flat namespace: path/convention resolution, file discovery, structural-consistency
  checks, dependency-cycle detection, symbol-table construction, error attribution, semantic
  validation, and test/integration emission. Nothing *depends* on them sharing a file; the coupling is
  incidental.
- This is the **lowest-risk high-value** item: the work is **moving definitions between files** behind
  a re-exporting facade — no logic edited. The crate has **done exactly this cleanly before**: the
  `emitter/` submodule split (`emitter/{workers,workers_entry,wrangler,serialisation}.rs`) is the
  working template, including the `mod`-with-re-export shape that keeps callers unchanged.
- Splitting first **lowers cognitive load before anything else in the track** — the pipeline
  decomposition (item 2), the duplicate-emitter elimination (item 7), and the `compile_project*` API
  collapse (item 5) are all far easier to reason about once their target code lives in a named,
  single-purpose module rather than at some offset in an 8k-line file.

## Settled

- **▸ Move, do not edit.** Function bodies are relocated verbatim. The only permitted edits are the
  **mechanical** consequences of the move: `use`/path qualification, `pub(crate)` widening **only**
  where a now-cross-module call requires it, and `mod`/`pub use` wiring. No logic, no signatures, no
  renames in this increment.
- **▸ Output is byte-identical.** The acceptance gate is the existing golden fixtures plus the
  `v0.29.1` helper tests, all green and unchanged. Any fixture diff means the move was not clean and
  is a defect, not an update.
- **▸ Facade preserves caller paths.** `crate::project::compile_project_full` (and siblings) resolve
  exactly as before. Callers (`main.rs`, `karn-lsp`, `lib.rs`) are not in this diff except, at most,
  unchanged re-export lines.

## Decisions (open — settle at review)

**[DECISION] Visibility widening discipline.** Cross-module calls will force some today-private `fn`s
to widen. *Rec:* widen to **`pub(crate)`, never `pub`** (these are internals, not new API), and keep
each submodule's *own* helpers private to it — only items with a genuine cross-submodule caller widen.
The privacy diff is itself a useful coupling map; resist blanket `pub(crate)`.

**[DECISION] Where the boundary-straddlers land.** A few helpers are used by two jobs (e.g. the
`*_span_of` lookup helpers `uses_span_of:3918`/`consumes_span_of:4160`/`parsed_alias_span:4171`, used by
both symbol construction and validation). *Rec:* place by **primary caller**; if genuinely shared,
a small `project/util.rs` rather than forcing an arbitrary home or a circular `mod` dependency.

**[DECISION] Where the landed test modules go.** `project.rs` now carries two `#[cfg(test)]` modules
that must move with their subjects (verified): `platform_lock_tests` (`:7858`, exercises
`lock_violation`) → `project/validate.rs`; and the `v0.29.1` `seam_tests` (`:7915`), which reaches
**many** helpers via `use super::*` — `normalize_rel`/`unit_path_matches`/the path maps (→ `paths`),
`canonicalise_cycle`/`detect_consumes_cycles` (→ `graph`), `escape_ts_string`/`ts_type_ref_emit`/
`sanitise_*` (→ `tests_emit`). A single `super::*` module cannot span the new submodules. *Rec:*
**distribute `seam_tests` across the submodules where each helper lands** — each submodule gets a small
`#[cfg(test)] mod` over its own helpers (so `super::*` stays in-module and **no helper widens visibility
merely to be test-reachable**). The shared test fixtures (`strs`/`graph`/`named`/`base`) move to
whichever submodule uses them, duplicated only if genuinely needed by two. *Alternative (rejected):*
keep one `seam_tests` in `mod.rs` and `pub(crate)`-widen every pinned helper just for the test — that
inflates the visibility surface the move is trying to minimise.

**[DECISION] One PR or staged?** *Rec:* **one PR**, because a half-moved file is harder to review than
a fully-moved one and the move is mechanical. The submodules land together with the facade. *Honest
caveat:* if the diff proves unwieldy at review, fall back to **two PRs split by the natural fault line**
— the *emission* half (`tests_emit` + `paths`) vs the *analysis* half (`consistency`/`graph`/`symbols`/
`validate`/`diagnostics`) — since those clusters barely reference each other.

## Tests

- **No new test *assertions*.** This increment's safety comes from `v0.29.1`'s helper pins plus the
  existing golden fixtures — every assertion stays green **unchanged**. Re-running the same `v0.29.1`
  pins against the relocated helpers is the before/after identity proof the track is built on.
- **The two `#[cfg(test)]` modules relocate with their subjects** (see the decision above):
  `platform_lock_tests` → `project/validate.rs`; `seam_tests` **distributed** across `paths` / `graph`

  unchanged" still holds; the only edits are `mod`/`use` mechanics, exactly like the production move.

## Diagnostics / Risks

- **Hidden coupling surfacing as a `pub(crate)` cascade.** If widening one helper forces widening its
  callees transitively, that is a real coupling signal — record it (it informs item 2's pipeline
  decomposition), but do not let it tempt a logic change *in this PR*. Move now, untangle later.
- **`#[path]`/`include!` or macro surprises.** The crate has at least one `#[path]`-include test
  pattern (per the v0.28 work); confirm no module here is pulled in by a non-obvious path before
  moving it. (Verified: `project.rs` is a plain `mod project;` — but check at implementation.)
- **Merge exposure is near-zero by design.** This is exactly why the track runs under a feature
  freeze: with no concurrent edits to `project.rs`, this 8k-line relocation cannot collide. Landing it
  **fast** (one focused PR) keeps that window short.
- **Reviewer fatigue.** An 8k-line move is a large diff even when mechanical. Mitigate by reviewing
  per-submodule (`git diff` is dominated by pure relocations) and leaning on the unchanged-fixtures +
  unchanged-`v0.29.1`-tests gate rather than line-by-line reading.
