# v0.43 — String interpolation

- **Status:** Draft (proposal). **Merging is approval to build.** *(Issue #45, the chosen surface. Replace
  `"Hello, ".concat(subject).concat("!")` with `"Hello, \(subject)!"`. A real lexer/parser/emitter
  surface + the surface tooling; a multi-slice language increment. The two cheaper options — operand-typed
  `+`, and the already-shipped `karn.string.join` — were weighed and set aside in favour of this.)*

## Context

`+` is `Int`/`Float`-only (ADR 0046 deliberately kept `+` numeric, `concat` a method), so building a
string means a `.concat()` chain — the least elegant line in `examples/hello-world`, and string-building
is exactly what first programs do. **Interpolation** is the chosen fix (over admitting `+` for `String`,
or leaning on the existing `karn.string.join`).

## Builds on (verified — scout)

- **`\(` is free.** The string lexer regex (`lexer.rs:181`) admits only `\n \t \" \\`; `\(` is currently
  an **invalid escape**, so no valid existing literal contains it — repurposing `\(…)` is
  backward-compatible. (`${…}` is *not* free — `$`/`{` are ordinary string chars today, so `${x}` is a
  literal; choosing it would silently re-mean existing strings.)
- **`StrLit` is one token → one AST node.** `ExprKind::StrLit(String)` (`ast.rs`), parsed trivially
  (`parser/expressions.rs:~393`), lowered to `"…"` via `escape_ts_string` at two sites
  (`emitter/lower.rs:292,322`). Interpolation adds a sibling node and a template-literal lowering.
- **The hole is just an expression.** Each hole parses as a full Karn `Expr`, so the checker, emitter,
  and (later) the LSP reuse the existing expression machinery — a hole is not a new expression kind, only
  a new *container*.
- **Operand-typed precedent + #44.** `Div` is operand-typed (ADR 0042) and `Int/Float.toString()` now
  exists (ADR 0074) — relevant to the hole-typing decision below.

## Decisions (open — settle at review)

**[DECISION] Syntax: `\(expr)`.** *Rec:* Swift-style `"… \(expr) …"`. Backward-compatible (it's an
invalid escape today); reads cleanly; `\\(` is the escape for a literal `\(`. Rejected `${…}` (breaks
existing literals) and `{…}` (ambiguous with record syntax / too easy to hit by accident).

**[DECISION] What a hole may contain — the central call.** *Rec:* **auto-convert the base scalar types**
— a hole of `String` interpolates as-is; `Int`, `Float`, `Bool` are rendered (Int/Float via the ADR 0074
`toString` contract, `Bool` as `"true"`/`"false"`); **every other type is rejected** (`record`, `sum`,
`Option`, `Result`, … → `"type T has no string form here — map it to a String first"`). Rationale:
interpolation's whole point is *display*, so requiring `.toString()` on every number would gut the
ergonomic win; bounding the implicit to base scalars avoids JS's `[object Object]` footgun; and a display
conversion *in a display context* is a far narrower, more defensible implicit than arithmetic coercion.
  - **Alternative (more philosophy-consistent): holes must already be `String`** — no implicit
    conversion at all; `"count: \(n.toString())"`. Consistent with Karn's no-coercion stance
    (operand-typed everything), at an ergonomic cost. *This is the one call worth your explicit sign-off —
    flag it at review if you'd prefer strict over the scalar auto-convert.*

**[DECISION] The AST node.** *Rec:* `ExprKind::InterpStr(Vec<InterpPart>)` where `InterpPart` is
`Chunk(String)` or `Hole(Box<Expr>)` — chunks and holes alternate; a plain `"…"` with no holes stays
`StrLit` (so existing code is untouched and the formatter/emitter fast-path is preserved).

**[DECISION] Lexing the holes.** *Rec:* a custom scanner (logos can't balance parens) that, on `\(`,
consumes a **balanced-parenthesis** run to the matching `)` as the hole's source, then resumes string
scanning — so `"\(f(x))"` takes `f(x)` as the hole. The parser re-parses each hole's source as an `Expr`
(reusing the entry parser). Nesting (a hole containing another interpolated string) is the main risk;
the balanced scan handles the common cases, and the slice-1 tests pin the corners.

**[DECISION] Emission.** *Rec:* `InterpStr` → a **TS template literal** — chunks as literal text (escaped
for backtick/`${`/backslash), each hole `${ <lowered expr> }`, with the scalar auto-convert wrapped in
`String(...)` for `Int/Float`/`Bool` (a `String` hole needs no wrap). Byte-stable.

**[DECISION] Slicing — three increments.**
- **Slice 1 (core, compiles):** lexer scanner + `InterpStr` AST + parser + checker (the hole-typing rule)
  + emitter (template literal). `karnc` compiles and runs interpolated strings end-to-end.
- **Slice 2 (surface tooling):** `karn-fmt` round-trip (re-emit `\(…)`), the **tree-sitter** grammar, and
  the **TextMate** grammar (`vscode-karn`) — so formatting and highlighting handle holes.
- **Slice 3 (LSP polish):** confirm/extend hover, completion, and semantic tokens *inside* holes — likely
  light, since holes are ordinary expressions with real spans once slice 1 lands; this slice is mostly
  verification + any semantic-token coloring of the string-vs-hole boundary. Deletes the proposal.

**[DECISION] New decision record.** *Rec:* **ADR 0075** — `\(…)` (backward-compat rationale), the
hole-typing rule (scalar auto-convert, or strict if you redirect), the `InterpStr` node, and the
template-literal emission. Notes it does **not** override ADR 0046 (`+` stays numeric; `concat` stays).

## Spec / docs delta (sketch)

- **`docs/src/spec/lexical-grammar.md`** — the string-literal grammar gains `\(expr)` holes (and `\\(` to
  escape); **`static-semantics.md`** — the hole-typing rule; **`emission.md`** — `InterpStr` → template
  literal. **A guide** (string-building) shows interpolation alongside `concat`/`join`.

## Tests

- **Slice 1:** lexer (holes, balanced parens, `\\(` escape, an unterminated hole error); parser
  (`InterpStr` parts); checker (a `String`/`Int`/`Float`/`Bool` hole types; a `record`/`Option` hole is
  the new error); an **e2e fixture** emitting a template literal (blessed), run under tsc.
- **Slice 2:** a `karn-fmt` round-trip fixture (idempotent re-emit of `\(…)`); tree-sitter corpus +
  TextMate scope checks.
- **Slice 3:** LSP hover/completion resolve inside a hole.

## Diagnostics / Risks

- **Lexer/parser coordination on nested holes** — the one real risk; the balanced scan + the corner-case
  tests are the guard, and a malformed hole degrades to a clear lex/parse error (not a panic).
- **`[object Object]`** — foreclosed by the base-scalar bound (non-scalars are a compile error, never
  emitted).
- **Formatter idempotence** — the `InterpStr` round-trip fixture guards it (slice 2).

## Out of scope (→ later)

- Format specifiers / padding / alignment inside holes (`\(n, width: 4)`) — a separate, larger feature.
- Operand-typed `+` for `String` and any change to ADR 0046 — explicitly *not* touched.

## Definition of done

`"… \(expr) …"` interpolates: `\(…)` holes parse as expressions, type-check by the hole rule (base scalars
auto-convert; non-scalars error), and emit a TS template literal — backward-compatible (`\(` was invalid),
ADR 0046 untouched. Lands as three slices (core → tooling → LSP) with tests at each; spec
(lexical-grammar + static-semantics + emission) updated; ADR 0075 added; CI green; changelog; version →
0.43.0. The final slice **deletes this file**.
