# Proposal: `bynk new` — scaffold a new project

**Increment:** v0.58 (tooling — the `bynk` driver)
**Status:** draft for sign-off
**Depends on:** ADR 0083 (driver is a thin orchestrator over `bynkc` + Node).
Stands beside `bynk doctor` (v0.46) and `bynk dev` (v0.57). The conceptual
**second** step of the driver arc **`doctor` → `new` → `dev`**
(`bynk-tooling-roadmap.md` §5.1) — shipping third, because `dev` was tractable
without it.

---

## 1. Why

Starting a Bynk project today is undocumented archaeology. There is no "create a
project" command, so a newcomer must already know three things the tooling could
tell them:

- that a project is *marked* by a `bynk.toml` at its root (and what keys it
  carries);
- that sources live under `src/` and a source file's path must **match** its
  qualified name (`context links` ⇒ `src/links.bynk`, or `bynk.project.*` errors);
- what a minimal, *compiling* unit even looks like.

The closest the book gets is tutorial 01's "create `demo.bynk` with this
content" — a bare file, not a project, and not something `bynk dev` can serve.
The driver already owns project rooting and the surrounding toolchain; this
increment gives it the zero-to-one step:

```sh
bynk new hello
cd hello
bynk dev          # already serving on http://localhost:8787
```

### The unlock: `new` closes the `doctor → new → dev` loop

`new` is the missing first link. With it the arc becomes a three-command
on-ramp — *is my machine ready* (`doctor`), *make me something* (`new`), *run
it* (`dev`) — and the proof is that the scaffold `new` writes is **immediately
serveable by `dev`** with no edits. That end-to-end loop is the whole point of
the increment: not "write a config file for me", but "hand me a running service
to start editing".

And unlike `dev`, **`new` needs no toolchain at all** — it writes files. It works
before `bynkc`, Node, or `wrangler` are installed (it is, after all, what you run
*before* you have set any of that up), which is exactly why it can be the true
first step.

## 2. What ships

A new driver subcommand. Surface (final flag spelling is an implementation
detail; the shape is what's being signed off):

```
bynk new <PATH> [--name NAME]
```

Behaviour, in order:

1. **Resolve the target** — `PATH` is the directory to create (e.g. `bynk new
   ./hello` or `bynk new hello`). The project name defaults to `PATH`'s final
   component; `--name` overrides it.
2. **Derive & validate the identifier** — the name becomes both the `[project]
   name` and the starter's **context** name, so it must be a legal Bynk
   identifier (a single dotless segment — the exact lexical rule is the
   implementation detail). A path like `my-app`, whose component carries a dash,
   fails with a fix-it pointing at `--name` (D3).
3. **Refuse to clobber** — if the target exists and is non-empty, fail before
   writing anything. `new` never overwrites a file it didn't create (D5).
4. **Write the scaffold** — create the directory and write a **complete,
   runnable** single-context HTTP service (D2):

   ```text
   hello/
   ├── bynk.toml            # [project] name/version + [paths] src/tests
   ├── .gitignore           # /.bynk, /out
   └── src/
       └── hello.bynk       # context hello — a GET "/" HTTP service
   ```

5. **Print next steps** — a short, copy-pasteable `cd hello && bynk dev`, plus a
   pointer to `bynk doctor` to check the toolchain. (`new` is stateless and
   offline — it can't know whether you've run `doctor` — so it always suggests
   it rather than guessing.)

`new` shells nothing, compiles nothing, and reads no network. It is pure,
offline file-writing.

### The starter

`src/<name>.bynk` is a minimal but real service — the single-file analogue of
`examples/hello-world`, chosen so `bynk dev` serves it unmodified:

```bynk
context hello

consumes bynk { Logger }

service api from http {
  on GET("/") by v: Visitor () -> Effect[HttpResult[String]] given Logger {
    let _ <- Logger.info("hello from hello")
    Ok("Hello from hello!")
  }
}
```

The template lives as an **embedded, fmt-clean, compile-tested** source (the
`include_str!` firstparty precedent, ADR 0086), with the name substituted at
write time — so a standing test compiles and `bynk-fmt`-checks it, and the
scaffold can never rot into something that doesn't build.

### Implementation surface

Small, and mostly reuse:

- `cli.rs` — a `Command::New { … }` variant.
- `main.rs` — a `run_new` arm.
- `new.rs` (new) — derive/validate the name, the clobber check, render the
  template, write the tree.
- An embedded `firstparty/`-style template source + a test that **renders it
  with a non-default name** and compiles + `bynk-fmt`-checks the result.
- Reused as-is: nothing from `dev`'s serve path (no `bynkc`/probe needed); the
  driver's existing CLI/exit scaffolding.
- New dependencies: **none** (`std::fs` only).

## 3. Decisions

### [DECISION D1] `new <path>` vs an `init` (new dir vs in-place)

`cargo` splits `new <path>` (make a directory) from `init [path]` (scaffold into
an existing/current one). Options: **(a)** ship only `bynk new <path>`; **(b)**
ship both `new` and `init` now.

**Recommendation: (a).** `new <path>` is the dominant zero-to-one motion and
keeps the conflict policy simple (the target is ours to create). `init` — which
must reason about scaffolding *into* a directory that already has files — is a
distinct safety problem (which files may coexist? is a lone `README.md` ok?)
that deserves its own slice. Land `new`; name `init` as a fast-follow. *The
scaffolding logic is shared, so `init` is later mostly a target-dir + empty-check
policy, not a rewrite.*

### [DECISION D2] What gets scaffolded — recommend a runnable service

Options: **(a)** an empty project (`bynk.toml` + empty `src/`); **(b)** one
opinionated **runnable HTTP service** starter; **(c)** multiple templates via
`--template lib|service|agent`.

**Recommendation: (b).** A runnable starter is what makes `new` worth more than
`mkdir` + a heredoc: it's what closes the loop with `dev` (§1). An empty project
(a) leaves the newcomer facing the same blank-file problem `new` exists to
solve; a template menu (c) front-loads a taxonomy decision (what *are* the
canonical project shapes?) better made once the library of examples has settled.
Ship one good default — a single-context `from http` service — and add
`--template` as a named follow-up when there's a real second template to offer.

### [DECISION D3] Project name & identifier validation

The name feeds two sinks with **different** rules, and it's worth being precise
about which:

- `[project] name` is a **liberal** string. `examples/feature-flags/bynk.toml`
  carries `name = "feature-flags"` — dash and all — and compiles fine.
- The starter's **context** identifier is not liberal: a context name must be a
  legal Bynk identifier (and, transitively, a legal worker-dir name under
  `--target workers`), and it must match its file's path or the compiler raises
  `bynk.project.inconsistent_commons_name`. That diagnostic
  (`bynkc/src/project/consistency.rs`) is about **context-name ↔ path**
  alignment — *not* about `[project] name`.

So a `feature-flags` *project* legitimately holds a `flags` *context* in
`src/flags.bynk` — a liberal project name beside a separate, legal context
identifier. That hand-built split is the real fork:

- **(a) liberal name, derived ident** — `[project] name` = the directory
  (`my-app`), with a separately-derived legal context identifier (sanitised, or
  supplied via `--name`). Matches what `feature-flags` does by hand; the on-disk
  name stays whatever the user typed.
- **(b) one identifier everywhere** — require the name to be a legal identifier
  and use it for both `[project] name` and the context.

**Recommendation: (b) — on simplicity, not necessity.** (a) is perfectly legal,
so this is a preference, not a constraint: (b) is chosen because a scaffold that
quietly contains *two* names (a project string and a separately-mangled context
ident) is a reconciliation the newcomer shouldn't have to do, whereas one
validated identifier makes the mental model just "the name is the name". When the
directory component isn't a legal identifier, **fail with a fix-it** ("`my-app`
isn't a valid Bynk name — pass `--name <ident>`") rather than silently mangling,
so the user *chooses* the identifier. `--name` is also the escape hatch for
anyone who deliberately wants the liberal split of (a).

### [DECISION D4] Does `new` touch the toolchain — recommend no

Options: **(a)** pure file-writing; **(b)** pre-flight the compile floor (resolve
`bynkc`, like `dev` does) and/or compile the starter to prove it builds on the
user's machine.

**Recommendation: (a).** `new` is the step you run *before* you have a
toolchain; coupling it to `bynkc` resolution would make the first command fail
for exactly the person it's meant to help. The starter's correctness is
guaranteed at *our* build time (the compile-tested embedded template, §2), not
at the user's `new` time. Keep `new` offline, instant, and dependency-free; let
`doctor` be the (separate, suggested) environment check.

### [DECISION D5] Clobber policy — recommend refuse-non-empty

**Recommendation:** `new <path>` creates the directory; if it already exists and
is **non-empty**, fail before writing anything and touch nothing. An existing
*empty* directory is fine (mirrors `cargo`, and matches the common "I just
`mkdir`ed it" case). Crucially, "empty" **ignores VCS/OS cruft** — `.git`,
`.gitignore`, `.DS_Store` and the like — so a freshly `git init`ed dir, or one
Finder has touched on this macOS-heavy contributor base, still counts as empty
(again as `cargo` does); otherwise the policy papercuts the exact case it's
meant to accommodate. `new` never overwrites a file — the look-before-you-leap
default for a command whose whole job is to write files.

### [DECISION D6] `.gitignore` and `git init` — recommend ignore-file only

The scaffold's `dev`/compile outputs (`/.bynk`, `/out`) shouldn't land in
version control. Options: **(a)** write a `.gitignore` only; **(b)** also run
`git init` (cargo's default).

**Recommendation: (a).** Write a `.gitignore` covering two paths: **`/.bynk`** —
the build dir `bynk dev` actually writes, so a `dev` run is clean from the first
commit (consistent with `dev`'s own `.bynk/.gitignore`, hoisted to the project
root) — and **`/out`**, the conventional `bynkc compile --output out` directory
the examples use and `[paths] out` documents. The honest distinction: `/.bynk` is
produced by the scaffold's own workflow; `/out` is *pre-empting* the documented
compile convention, not something `new` or `dev` writes. (If `[paths] out` is the
place to anchor this, the scaffold's ignore entry should track it.) Don't run
`git init`: it's a second tool dependency and an opinion about VCS the thin
driver needn't hold. A repo-less scaffold drops cleanly into an existing repo;
`git init` is one obvious command away. Revisit with `init`/`--vcs` if asked.

## 4. Risks

- **Starter rot.** A hand-maintained template silently stops compiling as the
  language evolves. *Mitigation:* the template is an embedded source compiled and
  `bynk-fmt`-checked by a standing test (ADR 0086 precedent), so a surface change
  that breaks it fails CI — the same guard the first-party sources already have.
  The test **renders with a non-default name** before compiling, so the
  name-substitution step is covered too, not just drift in the canonical
  template.
- **Name/identifier mismatch surprises.** A user expects `bynk new my-app` to
  "just work". *Mitigation:* D3 fails fast with a fix-it naming `--name`, rather
  than mangling silently and surfacing a confusing path-alignment error later.
- **Template opinion lock-in.** Shipping one starter blesses one project shape.
  *Mitigation:* D2 scopes to a single, deliberately minimal `from http` service
  and defers `--template`; the shape matches the canonical `hello-world` example,
  so it's the shape already documented as the on-ramp.
- **Scope creep toward a generator.** "Also scaffold a context / an adapter / a
  test" is a tempting adjacency (B-2's snippet scaffolds). *Mitigation:* this
  slice is **project** scaffolding only; in-project generators (`bynk new
  context …`) are explicitly out of scope and noted as later intent.

## 5. Spec / contract delta

No language-surface change — driver tooling, so no normative *language* spec
delta. The contract that changes is the **driver CLI**: `bynk new` joins `bynk
doctor` and `bynk dev` with its own argument/exit contract (success = scaffold
written; failure = a clobber refusal or an invalid name, each non-zero and
touching nothing). It is documented in the driver's CLI reference and pinned by
output goldens in the style of `doctor`/`dev` (the deterministic surface: the
"next steps" success message and the two failure messages). The **emitted
scaffold** — `bynk.toml`, the starter `.bynk`, `.gitignore` — is itself pinned by
a golden tree, and the starter additionally compile-tested.

ADR to record on implementation: a **single `bynk new` decision record**
(consistent with 0096 covering `bynk dev` as one record), whose defining calls
are the **runnable, offline scaffold** — D2 + D4, *what* `new` writes (an
embedded, compile-tested, immediately-`dev`-serveable starter) and *why* it
touches no toolchain — and the **name-validation rule** (D3). D1/D5/D6 are
captured as posture. `init` earns its own record if and when it ships.
