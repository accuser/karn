# v0.40 — InRange-swap quick-fix

- **Status:** Draft (proposal). **Merging is approval to build.** *(Deferred-polish item 9, the last
  user-facing one. A machine-applicable code action on the `karn.types.inverted_range` diagnostic —
  `Int where InRange(120, 0)` → one click swaps the bounds to `InRange(0, 120)`. The gate is a small
  parser/AST change: record a **span per bound** so the fix can target each numeral. Int and float
  together.)*

## Context

`InRange(lo, hi)` with `lo > hi` already raises `karn.types.inverted_range` with a *textual* note
("swap the arguments, e.g. `InRange(min, max)`") — but no machine-applicable fix, because the AST keeps
only the bound **values**, not their source spans, so the checker can't say *which* text to replace. This
increment records a span per bound (the gate), then turns the existing note into a one-click swap via the
v0.26 `Suggestion` mechanism the LSP already renders as a `CodeAction`.

## Builds on (verified — scout)

- **The diagnostic is already there** for both variants (`checker/refinements.rs:272-300`) — `InRange`
  (ints) and `InRangeF` (floats) each push `karn.types.inverted_range` with a `.with_note(...)`. Adding a
  `.with_suggestion(...)` beside the note is the whole fix.
- **The suggestion → code-action path exists.** `CompileError::with_suggestion(msg, Vec<(Span, String)>,
  Applicability)` (`error.rs:74`); `karn-lsp/src/code_actions.rs` turns every `MachineApplicable`
  suggestion into a `CodeAction` with its `TextEdit`s, keyed on the diagnostic span — and **multi-edit,
  edits-away-from-the-squiggle suggestions already work** (the `given`-clause fixes; the code_actions
  test). A two-edit "swap" is directly expressible.
- **The bounds lack spans.** `PredKind::InRange(i64, i64)` and `InRangeF(FloatBound, FloatBound)` carry
  only values/lexemes (`ast.rs:765-779`); `FloatBound { value, lexeme }` has no span. `RefinementPred`
  has a whole-predicate span only.
- **The numeral spans exist at parse time, then are dropped.** `parse_signed_num_literal`
  (`parser/types.rs:385-428`) reads each `IntLit`/`FloatLit` token (`t.span`) and keeps the value/lexeme
  but not the span. Capturing it is a few lines.
- **Codegen and formatting need only the values** — the emitter (`emit.rs:162-180`, `lower.rs:2069-2072`)
  and formatter (`fmt.rs:753/1287`) read the bound values (and float lexemes for byte-stability); none
  read a span. So adding spans is behaviourally inert — the ~20 consumer sites become `.value` accesses.

## Decisions (open — settle at review)

**[DECISION] The gate shape.** *Rec:* a new **`IntBound { value: i64, span: Span }`** and a **`span` field
on `FloatBound`**; `PredKind::InRange(IntBound, IntBound)` / `InRangeF(FloatBound, FloatBound)`. **Keep
`IntBound` value-only (no lexeme)** — the formatter prints `{value}` for ints (stable; integers have one
canonical form), so there's *no* idempotence risk; floats keep their `lexeme` (so `1e10` never normalises
to `10000000000`) and gain the span alongside it.

**[DECISION] The blast radius is mechanical.** *Rec:* the ~20 read sites (checker predicate
eval/compat/zero-value, AST `name()`, emitter codegen, formatter, the `checker.rs` unit-test
constructors) change from a bare `i64`/`FloatBound` to `.value` / the struct — the compiler enumerates
every one (non-exhaustive match / field error), so none is missed. No behaviour changes; the e2e and
formatter-idempotence fixtures are the guard.

**[DECISION] The bound span.** *Rec:* each bound's span covers the whole literal **including a leading
`-`** (the parser eats `Minus` then the numeral) — so the swap replaces `-10` as a unit. The replacement
text is the *other* bound's canonical form: `value.to_string()` for ints, `lexeme` for floats.

**[DECISION] The fix.** *Rec:* at each `lo > hi` branch, add
`.with_suggestion("swap the bounds", vec![(lo.span, hi-text), (hi.span, lo-text)],
MachineApplicable)` beside the existing note. Two edits, swapping the two numerals' text in place; the
note stays for non-LSP consumers.

**[DECISION] Scope: int + float together.** *Rec:* both in one increment — they're symmetric (the float
variant already carries a lexeme; it just gains a span). No slicing; the gate and the fix are
inseparable and the whole change is bounded.

**[DECISION] New decision record.** *Rec:* **ADR 0073** — the `IntBound`/`FloatBound`-span gate
(value-only ints for idempotence), and the two-edit swap suggestion on `karn.types.inverted_range`.

## Tests

- **`karnc` parser/AST**: `InRange(10, 5)` parses with each bound's span covering its numeral (incl. a
  signed `-3`).
- **`karnc` suggestions** (the v0.26 suggestions harness): the `inverted_range` error on `InRange(10, 5)`
  carries a `MachineApplicable` two-edit suggestion whose edits, applied, yield `InRange(5, 10)`; same for
  a float `InRange(5.2, 1.1)` → `InRange(1.1, 5.2)` (lexemes preserved).
- **Regression**: the existing negative fixtures (`02_inverted_range`, `173_mixed_inrange_bounds`) still
  diagnose; the positive `InRange` fixtures still emit byte-identical TypeScript; `karn-fmt` stays
  idempotent on `InRange` (a fmt round-trip fixture).

## Diagnostics / Risks

- **Formatter idempotence** — the one real risk, neutralised by keeping `IntBound` value-only; covered by
  a fmt round-trip assertion.
- **A missed consumer site** — impossible to ship silently: changing the variant shape makes every reader
  a compile error until updated.
- **Swap fidelity** — `value.to_string()`/`lexeme` reconstructs each bound; a cosmetic `+5` would
  normalise to `5` on swap (ints), which is acceptable and rare.
