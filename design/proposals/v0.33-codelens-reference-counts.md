# v0.33 — CodeLens: reference counts

- **Status:** Draft (proposal). **Merging is approval to build.** *(Tooling-queue item 4, the cheap
  half. A `textDocument/codeLens` above each top-level definition showing its reference count, clickable
  to peek the references. Small, self-contained — it falls straight out of the binding index.)*

## Context

The binding index (ADR 0053) already records, per top-level symbol, its definition site and **every
reference** (`SymbolEntry { def, refs }`, `index.rs:183`). A reference-count CodeLens is just that count
surfaced above the definition — no new analysis. The test-run-lens half of item 4 (a "▶ Run test" lens)
needs test discovery + a run command and is **out of scope** here.

## Builds on (verified — scout)

- **The count is already there.** `SymbolEntry.refs: Vec<SiteRef>` (`index.rs:188`) → `refs.len()`. The
  def site is `SymbolEntry.def`. `index_queries::workspace_symbols` (`:64`) already iterates
  `index.symbols` filtered to those with a def — the exact template for "one lens per definition".
- **It's served from the cached round**, like references/tokens — positions convert against the analysed
  snapshot (§3.2's rule). No re-analysis.
- **`code_lens` is not declared yet** — this adds `code_lens_provider` to the server capabilities.

## Decisions (open — settle at review)

**[DECISION] Which definitions get a lens.** *Rec:* every **index symbol whose def is in the file** —
types, free fns, capabilities, services, agents, providers (the v0.25 index set). Locals/methods/fields
are not in the index and get none (a lens per local would be noise anyway).

**[DECISION] The lens action.** *Rec:* title `"{n} reference(s)"` with the standard VS Code command
`editor.action.showReferences` (args: the def URI, the def position, and the reference `Location`s) —
clicking peeks the references, no extension change needed. Non-VS-Code clients still render the title
(informative without the click).

**[DECISION] Zero-reference defs.** *Rec:* **show `"0 references"`** rather than hiding it — an unused
top-level symbol is worth flagging (dead-code signal), and matches rust-analyzer's behaviour.

**[DECISION] Resolution.** *Rec:* compute eagerly (the count is `O(1)` off the index) — no
`codeLens/resolve` lazy pass.

**[DECISION] New decision record.** *Rec:* a **light ADR 0066** — reference-count lenses from the index
`sites()`, served from the cached round, `showReferences` action; test-run lenses deferred.

## Spec delta (sketch)

- **`design/karn-lsp-spec.md`** — a short "CodeLens" capability section (§3.x): one reference-count lens
  per top-level index definition, served from the cached round, the `showReferences` action, zero shown,
  test-run lenses deferred.

## Tests

- **`karn-lsp` pure-function** (the `index_queries` harness): `code_lenses(index, path)` returns one
  `(def site, count)` per definition in `path` with the right counts (incl. a 0-ref symbol), and nothing
  for a file with no defs. Positions/locations are an LSP-layer conversion (thin, as elsewhere).

## Diagnostics / Risks

- **Lens noise** — one per top-level def is the established density (rust-analyzer etc.); acceptable.
  Visibility is the client's (`editor.codeLens` setting).
- **First-party `karn.*` symbols** — excluded from the index (the v0.28 finding), so they get no lens —
  correct (they aren't user-editable).

## Out of scope (→ later)

- **Test-run lenses** ("▶ Run") — needs test discovery + a run task/command; the larger half of item 4.
- CodeLens for non-index kinds (methods/fields/locals); implementation/override lenses.
