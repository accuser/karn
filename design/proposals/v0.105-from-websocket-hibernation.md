# Proposal v0.105 — `from WebSocket` on Workers, slice 3b-ii: hibernation re-association

- **Status:** Draft, for approval.
- **Scope:** a single increment — make a held `Connection` **survive Durable Object eviction** on Cloudflare Workers, realising design notes §2.9.6 ("a `Connection[F]` stored in agent state survives the agent's hibernation and is automatically available when the agent is rehydrated"). Slice 3b-i (ADR 0132) shipped a working edge-authenticated upgrade using the **non-hibernatable** `server.accept()` model — the socket lives in the DO's memory and is lost on eviction. 3b-ii swaps that for the **hibernatable** WebSocket API so a stored connection is durable across hibernation cycles, entirely behind the `Connection<F>` interface (the language still sees "a stored value").
- **Provenance:** the **Q7** piece ADR 0132 explicitly deferred ("hibernation re-association lives in the runtime binding, keyed by a connection id via `serializeAttachment`"). Consumes the protocol surface + edge-auth + DO-hosted on-open of ADR 0132 and the held-resource model of ADR 0130. **Security-bearing** (it is on the same connection-into-an-agent boundary): runs `/security-review` + `/code-review`. ADR provisionally **0133**, version **0.105.0**.

## 1. The problem — `server.accept()` is in-memory; eviction drops the socket

3b-i accepts the socket inside the DO with `server.accept()` and keeps the live
`WorkersConnection` in an **in-memory side-table** (`heldStore`, keyed by the
durable state object). That table is plain isolate memory: when the DO is evicted
(idle, or the platform reclaims it), the isolate is torn down and **every live
connection is lost** — a later `agent.send(frame)` finds nothing. §2.9.6 promises
the opposite: a stored connection is *automatically available* after hibernation.

Cloudflare's **hibernatable WebSockets** deliver that promise, but only if the
socket is accepted through `state.acceptWebSocket(ws, tags)` (not `ws.accept()`):
the platform then preserves the socket across hibernation and re-presents it via
`state.getWebSockets(tag)` when the DO wakes. The language relies on this platform
guarantee; it does not implement it (§2.9.6) — 3b-ii is the *binding* that wires
the held `Connection` onto it.

```
on open  ── state.acceptWebSocket(server, [connId]); server.serializeAttachment({connId})
            store conns: Map[K, Connection]  ──persists──▶  Record<string(K), connId>   (durable, serialisable)
                                              ⋮ (DO hibernates; isolate torn down) ⋮
later send ── conns.get(k) → connId → state.getWebSockets(connId)[0] → WorkersConnection(ws).send(frame)
```

The held value the language stores is **the connId, not the socket** — a stable,
serialisable string. Every `Connection` access re-resolves connId → live socket
through the platform, so no live socket is ever held across requests and
hibernation is transparent.

## 2. Scope and non-goals

**In scope.**

- **Hibernatable accept** — the DO `fetch` upgrade branch accepts via
  `state.acceptWebSocket(server, [connId])` (a fresh `crypto.randomUUID()` connId
  tag) and `server.serializeAttachment({ connId })`, instead of `server.accept()`.
- **The held `store Map[K, Connection]` becomes durable on Workers** — it persists
  as `Record<string(K), connId>` in the agent state record (reversing 3b-i's
  in-memory `heldStore` split, now that the value is a serialisable id). `put`
  stores the connection's connId; `get`/iteration resolve connId → socket via
  `state.getWebSockets(connId)`; `remove` resolves-and-**closes** then deletes.
- **`WorkersConnection` carries its connId**, and two runtime helpers:
  `acceptHibernatableConnection(state, server) → WorkersConnection` (assigns the
  connId, accepts, attaches) and `resolveConnection(state, connId) → Option<WorkersConnection>`
  (getWebSockets-by-tag, re-wrap).
- **`remove`-closes** — the §2.9 "removes-and-closes" contract is now emitted (it
  resolves the connId to the socket and closes it), closing the 3b-i follow-up.
- The §20 chat-room **re-emits + `tsc --strict`-clean** on Workers with the
  hibernatable handlers; the held-map snapshot fixture updates.

**Non-goals (and why).**

- **Inbound frame dispatch (DECISION E) — split to a named slice 3b-iii.**
  `webSocketMessage(ws, msg)` decoding against `in:` and routing to an agent
  handler needs a **new protocol surface** (inbound message handlers + frame→handler
  routing), independent of and larger than the hibernation binding. 3b-ii keeps the
  **send path** (server→client) durable; the receive path is its own increment.
  Server-push alone (a stored connection an agent sends to from a cron/http/event)
  is a coherent, useful capability without it.
- **Broadcast-to-all-connections** — the held-aware iteration borrow surface; the
  slice-4 closure.
- **Non-Cloudflare platforms** — `Connection` is a Cloudflare-DO-hosted resource.

## 3. Decisions

**[D1 — hibernatable accept] Accept through `state.acceptWebSocket(server, [connId])`,
not `server.accept()`.** A fresh `connId = crypto.randomUUID()` tags the socket so
`getWebSockets(connId)` can find it after hibernation; `server.serializeAttachment({ connId })`
persists the id on the socket (forward-compat with the 3b-iii inbound path, where a
waking `webSocketMessage` needs to recover the connId without loading state).
*Recommended;* it is the one platform call that makes a socket survive eviction.

**[D2 — the held value is the connId] A `store Map[K, Connection]` persists
`Record<string(K), connId>` and resolves connId → socket per access.** This
reverses 3b-i's in-memory split: the stored value is now a serialisable string, so
it belongs back in the durable state record. `Connection` operations never hold a
live socket across requests — `get` does `getWebSockets(connId)[0]` then re-wraps,
returning `None` if the platform has no such socket (a closed/gone connection).
*Recommended;* it is the only representation that is both durable and
hibernation-safe, and it makes "stored value in, stored value out" literally true.

**[D3 — resolution is total and fail-soft] A connId that resolves to no socket is
`None`, not a fault.** A connection may have closed (client gone) while its connId
lingered in the map until the next `remove`. `get` returning `None` is the honest
shape; `send` on a `None` is a no-op the caller already handles via the `Option`.
*Recommended;* a missing socket is normal lifecycle, not corruption (contrast the
rehydration gate, which faults on malformed *persisted* data — a connId string is
always structurally valid).

**[D4 — `remove` closes] `conns.remove(k)` resolves the connId, closes the socket,
then deletes the entry.** This finally emits the §2.9 "removes-and-closes" contract
(the 3b-i lowering only deleted). A close on an already-closed socket is a no-op.
*Recommended;* it honours the linearity discipline's disposal semantics and
prevents a leaked hibernatable socket.

**[D5 — bundle unchanged] The bundle/test target keeps the in-memory
`TestConnection` model.** There is no `acceptWebSocket`/`getWebSockets` off-Workers;
a held map on bundle stays the in-memory test-state record of `TestConnection`s
(3b-i behaviour). The connId representation is **Workers-only**, selected at emit by
target — exactly as 3b-i's `heldStore` split was.

## 4. Internal architecture

- **`emit_agent` (held maps):** on Workers, a held `Map[K, Connection]` rejoins the
  persisted state record typed `Record<string, string>` (K-string → connId). The
  `heldStore` in-memory side-table and its `agent_held_maps` lowering are replaced
  by connId-record ops: `put` → `state.conns[String(k)] = conn.connId`; `get` →
  `resolveConnection(this.state, state.conns[String(k)])`; `remove` →
  resolve-close-delete; iteration → map connIds through `resolveConnection`. The
  rehydration gate validates the K key (textual) as before; the connId value is an
  opaque platform string (no refinement).
- **DO `fetch` upgrade branch:** `acceptHibernatableConnection(this.state, server)`
  replaces `newWebSocketPair().server.accept()` + `new WorkersConnection(server)`.
- **Runtime (`connection.ts`):** `WorkersConnection` gains a `connId`;
  `acceptHibernatableConnection(state, server)` and `resolveConnection(state, connId)`
  added; structural `DurableObjectState` view extended with `acceptWebSocket`,
  `getWebSockets`, and the socket's `serializeAttachment`/`deserializeAttachment`.
- **`wrangler.toml`:** no change — the DO binding + migration already exist; the
  hibernation API needs no manifest stanza.

## 5. Risks

- **Re-wrap allocates per access.** `get` builds a fresh `WorkersConnection` each
  time. Sockets are cheap to wrap (a thin object); no caching needed and caching
  would re-introduce a cross-request live reference. Stated, not mitigated away.
- **`serializeAttachment` is forward-looking.** 3b-ii does not yet *read* it back
  (no inbound path); it is written so 3b-iii's `webSocketMessage` can recover the
  connId. If 3b-iii's design changes, the attachment shape may change — it is
  internal, versionable, and unobserved by the language.
- **No real-hibernation test in the harness.** Eviction/wake needs Miniflare/workerd
  state the unit harness can't drive; coverage is a **shape-snapshot fixture +
  `tsc --strict`** (the resolution path is exercised structurally), plus a manual
  `wrangler dev` note. Never claim a runtime hibernation proof that did not run.
- **Security surface unchanged but re-reviewed.** Auth still happens at the edge
  before accept; 3b-ii changes only how the *accepted* socket is stored/resolved.
  `/security-review` confirms no path resurrects a socket without its edge-authed
  provenance (the connId only ever enters the map via the on-open transfer).
