# v0.64 — Extract `bynk-ide`: the IDE/LSP analysis surface; `bynk-lsp` drops `bynkc` (crate-decomposition slice 5)

- **Track:** [crate-decomposition](../tracks/crate-decomposition.md), slice 5.
  The slice that pays off the *original* concern most visibly: the language
  server stops linking the compiler binary's crate.
- **Realises ADRs:** [0099](../decisions/0099-crate-layering-dependency-direction.md)
  (downward layering), [0102](../decisions/0102-foundation-types-boundary.md)
  (`Severity` is a `CompileError` classification → the leaf).
- **Depends on:** slices 1–4. **Interacts with:** the [LSP track](../tracks/lsp.md)
  — these are its queries *relocated*, not rewritten. The validation gate is the
  full `bynk-lsp` suite + the 12 index/diagnose drift tests passing unchanged.
- **Posture:** a **pure internal refactor** — no language surface, no LSP
  behaviour change.

## What this slice does

Create **`bynk-ide`** over `bynk-syntax` + `bynk-check` + `bynk-emit`, holding the
IDE analysis surface that today lives in `bynkc`'s lib glue, and **re-point
`bynk-lsp` off `bynkc` entirely**.

**Moves into `bynk-ide`** (from `bynkc/src/lib.rs`): `diagnose` (single-file
diagnostics), `diagnose_project` (the non-bailing, overlay-aware project
analysis — it calls `bynk_emit`'s `analyse_project`), and the result types
`Diagnostic`, `FileDiagnostics`, `ProjectDiagnostics`.

**`bynk-lsp` is re-pointed:** its dependency changes from `bynkc` + `bynk-fmt` to
**`bynk-ide` + `bynk-check` + `bynk-syntax` + `bynk-fmt`** — `bynkc` is dropped.
The investigation shows almost all of `bynk-lsp`'s ~80 `bynkc::…` imports are
*pass-through* re-exports: `bynkc::span`/`ast`/`lexer`/`parser`/`error` →
`bynk_syntax::`; `bynkc::index`/`checker`/`expr_types`/`locals`/`kernel_methods`/
`firstparty`/`builtin_names`/`hints` → `bynk_check::`; the IDE entry points
(`diagnose`, `diagnose_project`, `Diagnostic`, `Severity`, `ProjectDiagnostics`)
→ `bynk_ide::`/`bynk_syntax::`. ~7k lines of mechanical import rewrites, no logic
change.

## Why this is the headline slice

`bynk-lsp` today links **all of `bynkc`** — cli/clap, `test_json`, the ariadne
render path — none of which a language server needs. After this it links only the
analysis libraries it uses. This is the standing "the LSP shouldn't pull the whole
compiler" concern finally closed (the original motivation, with `bynk-fmt` in
slice 2 as the warm-up).

## Decisions

**[DECISION 1] Move `Severity` down to `bynk-syntax` — recommend yes.** `Severity`
(`Error`/`Warning`, via `for_error(&CompileError)`) is the one item **shared**
between the IDE `diagnose` path (→ `bynk-ide`) and `bynkc`'s `short`/`json`
rendering (`severity_word`). If it went to `bynk-ide`, `bynkc`'s render path would
gain an **upward** edge. It classifies a `CompileError` by category, so it belongs
beside `CompileError` in `bynk-syntax` (ADR 0102). This also pre-positions slice 6
— `bynk-render` (which depends on `bynk-syntax` only) needs `Severity` for the
`short` form. *Alternative:* keep it in `bynkc` and have `bynk-ide` define its own
— rejected (two `Severity` types, and the render path still needs one). Same
shape as slice 4's `line_col`.

**[DECISION 2] `bynkc` re-exports `bynk-ide` — recommend yes.** `bynkc`'s *binary*
does not use the IDE surface (verified), so it need not depend on `bynk-ide` for
its own sake. But 12 `bynkc` integration tests (`diagnose_project`,
`reference_index`, `inlay_hints`, the `*_index` suites, …) use
`bynkc::diagnose_project` / `bynkc::Diagnostic`. Re-exporting
(`pub use bynk_ide::{diagnose, diagnose_project, Diagnostic, ProjectDiagnostics,
FileDiagnostics};`) keeps those tests and the public API stable — the slice-1..4
pattern — at the cost of one downward `bynkc → bynk-ide` edge (harmless; `bynkc`
is the kitchen-sink binary). *Alternative:* move the 12 tests into `bynk-ide`'s
suite — more churn, deferred; they can migrate later.

**[DECISION 3] What stays in `bynkc`.** The **rendering** glue (`print_errors*`,
`render_errors*`, `print_project_failure*`, `render_project_errors`,
`short_line`/`severity_word`) stays — it is the CLI's ariadne/`short`/`json`
output, used by `main.rs`, and is slice 6's `bynk-render` target, not slice 5's.
`compile`/`compile_project` glue stays. So `bynk-lsp` dropping `bynkc` does **not**
strand anything the LSP needs — it never used the render path.

**[DECISION 4] Version — recommend a minor `v0.64`.** New published crate
(`bynk-ide`) → a new release surface needing the **bootstrap-seed**, added to both
publish loops after `bynk-emit`, before `bynkc`. (`bynk-lsp` keeps its place at the
end.) Per-increment minor cadence (v0.63 → v0.64).

## Risks & mitigations

- **The `bynk-lsp` re-point is the largest import rewrite of the track (~7k
  lines).** Mitigated by it being purely mechanical (same functions, new module
  paths) and by the full **`bynk-lsp` test suite** as the gate — if a query
  changed behaviour, those tests fail. This is the LSP track's surface; the
  contract is "relocated, not rewritten."
- **Boundary surfacings** (private items now crossing into `bynk-ide`/`bynk-lsp`)
  are expected — mechanical `pub(crate)`→`pub` promotions, compiler-surfaced.
- **Validation gate:** the 12 index/diagnose drift tests (via the `bynkc`
  re-export) **and** the entire `bynk-lsp` suite must pass byte-identical.
