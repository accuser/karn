# v0.80 — Agent invariants: runtime-checked contracts at the commit boundary

- **Scope:** a single-increment **language + compiler** proposal (no track — see the
  posture note below). Touches lexer/keywords, AST, parser, checker, and emitter;
  `bynkServerVersion` and the workspace version → 0.80.0.
- **Realises:** `bynk-design-notes.md` §14 ("Invariants on agents") — the *second*
  validation commitment, alongside test contexts (§14, already shipped). Agents may
  declare universally-quantified predicates that must hold of every committed state;
  a commit that would violate one **faults and reverts**.

## Why a proposal and not a track

ADR 0076's bar for a feature track is two-or-more of {spans several increments,
surface unsettled, security/safety boundary}. Invariants meet at most one, weakly.
They are a *correctness* feature, not a trust boundary. §14 already pins the
semantics (commit-boundary checking, faults-not-outcomes, per-agent scope, the
`implies`/`is`/deref predicate language). And the implementation is narrow: one
predicate phase in `parse_agent_decl`, one well-formedness pass in `bynk-check`, one
gate in the emitted `commitState`. The one genuinely separable ambition — static
provable-violation analysis — is carved out below as a **named follow-on** (its own
later proposal), which is how follow-ons are handled elsewhere (actor Q8, the LSP nav
issues). That does not by itself justify a track.

## The reframe the current implementation buys us

§14 was written against the `Cell`/`store` model with an **implicit** commit at handler
end, so it speaks of the runtime reconstructing "the proposed committed state" at an
implicit boundary and of *implicit `Cell` dereference* in predicates. The compiler has
since diverged, and in our favour:

- An agent is `key id: T` then a `state { …record fields… }` block then handlers
  (`AgentDecl`, `bynk-syntax/src/ast.rs:508`).
- Commit is an **explicit statement** — `commit expr` lowering to
  `await this.commitState(expr)` (`bynk-emit/src/emitter/lower.rs:262`), and
  `commitState(s)` is emitted today as a one-line `storage.put` (`emit.rs:~1560`).

So "the proposed committed state" is already a first-class value (the argument to
`commit`), and there is already **one** generated chokepoint. The runtime check is a
gate at the top of `commitState`, evaluating each invariant against `s` before the
`put`. The abort-and-revert semantics are automatic: throw before the write and the
state never persists. The implicit-`Cell`-deref clause of §14 mostly dissolves —
field access on the state record (`s.status == Paid`) is the natural, already-parseable
surface. **Invariants are predicates over the agent's state record, evaluated against
the value passed to `commit`.**

## Decisions for sign-off

**[DECISION A] Predicate scope — what an invariant may reference.** *Recommend:* the
agent's own state fields, base operators, `implies`, `is`, and a **closed** set of
pure value methods already needed by the §14 examples (`Option.isSome`/`isNone`, sum
`is` checks). No capabilities, no `<-`/`Effect`, no storage *mutation*, no reference to
another agent. This is load-bearing and hard to reverse — it defines what an invariant
*is* — so it lands as **ADR 0107 (agent-invariant predicate surface)** up front.
*Fork:* state-fields-only-with-builtins (recommended, keeps predicates pure and
agent-local and the checker bounded) vs. arbitrary pure-helper calls (more expressive,
but pulls call-graph purity analysis into invariant checking and widens the surface a
reader must trust). Cross-agent predicates are rejected outright — §14 closes that door
(line 884); the right tools are sagas/scenarios.

**[DECISION B] Static provable-violation analysis — now or later.** *Recommend:*
**later**, as a named follow-on. The MVP is runtime-checking, which realises the core
of §14 on its own. The static pass (flag a handler all of whose paths provably commit a
violating state) needs state-field value tracking to each `commit` site and is cleanly
separable. §14 itself frames it as a layer *on top of* runtime checking, and explicitly
defers static *satisfaction* proving beyond even that.

**[DECISION C] `implies` — surface spelling only.** The desugaring is already settled:
the notes' IR-lowering step lowers `is` and `implies` to boolean expressions, so
`P implies Q` → `!P || Q` is effectively pinned, and `is`-as-expression already exists
(ADR 0007). The *only* genuinely open part is the spelling — a reserved keyword `implies`
(costs one reserved word; reads directionally, matching §14's prose-readability intent)
vs. a non-keyword form. *Recommend:* the keyword; the cost is trivial and the readability
is the whole point. This is a light call, not a heavy one.

**[DECISION D] Body placement.** *Recommend:* invariants form a **phase between the
`state { }` block and the handlers**, matching every §14 worked example. Pinned order
keeps `parse_agent_decl` a straight-line three-phase parse and keeps the agent readable
top-to-bottom (identity → state → contracts → behaviour).

**[DECISION E] The fault class and what the caller observes.** This is the load-bearing
question §14 and the current code do *not* answer between them, so it is settled here
rather than deferred to build. *Recommend:* a **dedicated `InvariantViolation` runtime
fault — not a `BoundaryError`.** `BoundaryError` (`runtime/src/boundary.ts:14`) is the
cross-Worker *call/refinement* layer (`MalformedJson | StructuralMismatch |
RefinementViolation | Transport`, mapping to 400); a commit-time invariant breach is an
*internal agent fault*, a different surface, and conflating them would mis-signal a 4xx
for what is a server-side refusal to enter an inconsistent state. Instead, `InvariantViolation`
rides the **existing uncaught-fault channel**: thrown in `commitState` before the `put`, it
propagates out of the Durable Object, `callDurableObjectMethod` rethrows it
(`agent.ts:78`), and the service entry's `catch` maps it to a fault response (today: a bare
`500 Internal Server Error`, `workers_entry.rs:220`). *What the caller observes:* a
**fault, not an outcome** — the handler's `Result` is never produced, consistent with
Bynk's failure model (faults abort and are not typed parts of contracts; only outcomes
are). A 500-class status is the honest surface for "the agent refused an inconsistent
commit." *Sub-fork — caller-visible structured faults, deferred and reframed:* enriching
the bare 500 with a recognizable body (`{ kind: "InvariantViolation", invariant, agent }`,
with the key subject to the same redaction question as the log line — see Slicing) is
**not an invariant-specific change**. The type is erased at two hops for *every*
uncaught agent fault — `agent.ts:78` rewraps as `new Error(text)`, and the entry `catch` is
bindingless — so preserving fault identity to the caller means a three-file change to the
DO→service wire protocol (serialise a recognisable shape in the DO response → detect-and-
rethrow-typed at `agent.ts:78` → bind-and-map at the entry `catch`). Building that for
`InvariantViolation` alone would create a bespoke second fault path; the right home is a
**general typed-agent-fault channel** that serves invariant breaches, non-exhaustive match,
and every other fault uniformly. So this is a named follow-on increment, not part of v0.80.
Whenever it lands, the surface must stay a fault envelope, never a `Result` variant the
caller pattern-matches. The v0.80 MVP therefore ships the honest bare 500 plus the
`commitState` log line (see Slicing) — refusal visible in logs, not yet to the caller.

## Implementation delta (against current code)

1. **Lexer + keywords** (`bynk-syntax/src/lexer.rs`, `keywords.rs`): add `invariant` and
   `implies` as `#[token]`s with `KeywordInfo` rows. The drift test lives in the *driver*
   crate (`bynkc/tests/keywords_reference.rs`) and reads across the crate boundary, so the
   two edits are mandatory and self-checking even though the test is not co-located.
2. **AST** (`ast.rs:508`): add `invariants: Vec<Invariant>` to `AgentDecl`, with
   `Invariant { name: Ident, predicate: Expr, documentation: Option<String>, span }`.
   `ActorRefinement.predicate: Expr` (`ast.rs:599`) is the existing
   parsed-predicate-on-a-declaration precedent to mirror.
3. **Parser** (`parser/declarations.rs:2356`, `parse_agent_decl`): after the `state { }`
   block and before the handler loop, parse zero-or-more `invariant <name>: <expr>`
   using the existing expression parser. The doc conventions already anticipate
   `invariant name: expr` as an inline declaration (notes §17, line 1157), so
   `collect_item_lead`/`finalize_doc` apply unchanged. The handler loop's current
   "expected `on`" rejection (`bynk.parse.expected_handler`) tightens to allow the
   invariant phase to have closed first.
4. **Checker** (`bynk-check`): a well-formedness pass — predicate types to `Bool`;
   references resolve to state fields or the DECISION-A builtins; no capabilities, no
   `Effect`, no mutation, no foreign-agent reference. New diagnostics:
   `bynk.invariant.not_bool`, `bynk.invariant.impure_predicate`,
   `bynk.invariant.cross_agent_reference`, `bynk.invariant.duplicate_name`. §9 already
   lists "invariant predicate well-formedness" as an intended architectural-validation
   rule (notes line 1303), so this has a designated home. `implies` desugars to `!P || Q`
   here; reuse the refinement-predicate purity check shape.
5. **Emitter** (`bynk-emit`): emit each invariant as a pure TS predicate over the state
   record; gate `commitState(s)` by evaluating them **before** `storage.put`, throwing the
   dedicated `InvariantViolation` fault on failure (DECISION E). Gating inside `commitState`
   rather than at each `commit` site means one central check regardless of how many `commit`
   statements a handler has — though note this centralises the *check*, not handler
   atomicity (see Risks). No new commit machinery beyond the gate and the fault type.

## Slicing

The MVP is **one increment**: items 1–5 above minus DECISION B. Result: invariants parse,
type-check, and abort offending commits at runtime — the core of §14.

**MVP observability limitation (stated, not glossed).** The `InvariantViolation` thrown in
`commitState` is structured, but the *response* the caller sees is not: `agent.ts:78`
rewraps the throw as `new Error(text)` (erasing the type) and `workers_entry.rs:220`'s
bindingless `catch` discards it, so the programmatic caller observes an **unadorned 500
with no body — observationally identical to a panic, OOM, or any other bug.** The MVP does
*not* make the refusal caller-distinguishable. It does add one cheap honesty measure: a
`console.error` of the violation emitted inside `commitState` *before* the throw — no
wire-format change — so the refusal is distinguishable from a crash **in the logs**, just
not yet to the caller. This matches the runtime's existing failure-logging convention (the
cron/queue paths already `console.error("<ctx> <what>", payload)`, `workers_entry.rs:279`),
so it's a house-style line, not a new channel. **What it logs is a deliberate call:**
invariant name + agent *type* unconditionally, but **not the agent key value** by default.
The key is `key id: T` where `T` is domain-chosen and frequently a sensitive identifier (an
email, a user id, an account number); the runtime has no redaction policy today to defer to,
so dumping it on every violation would route PII into logs. The key is omitted (or behind an
opt-in) until a general logging/redaction convention exists to carry it safely. Carried as named follow-ons (each
its own later `vX.Y` proposal): (i) the static provable-violation pass (DECISION B; static
*satisfaction* proving remains deferred per §14); (ii) caller-visible structured faults —
see DECISION E, scoped as a *general* typed-fault channel, not an invariant-specific body.

## Risks

- **Predicate-language creep.** Every builtin admitted to DECISION A is a permanent part
  of the invariant surface. Keep the initial set to exactly what the §14 examples need;
  widen only by later ADR. Mitigated by making A an ADR rather than an inline call.
- **"Revert" means non-persistence, not effect rollback — and §14's wording overstates
  it.** Throwing in `commitState` before the `put` guarantees only that *the faulting
  committed value is never written*. It does **not** undo effects the handler already
  performed: a handler that does `<- sendEmail(...)` then `commit badState` has already
  sent the email when the invariant throws; cross-agent calls and event emissions made
  before the commit likewise stand. The handler is **not** transactional. §14's "the agent
  reverts to its pre-handler state" (line 880) was written for the implicit single-commit
  model and is inaccurate under the current eager-commit implementation — the spec must
  restate it as *non-persistence of the offending commit*, not whole-handler revert.
- **Centralising the check does not make the handler atomic.** Because `commit` is an
  eager statement that can appear more than once, `commit good; …; commit bad` persists
  `good`, *then* faults at `bad`. The gate is per-commit. Restoring true handler atomicity
  (buffer all commits, apply once at handler end) is a larger semantic change out of scope
  here; this proposal documents the per-commit reality and flags the reconciliation with
  §14's atomic-handler framing as a spec note, not a silent assumption.
- **Reads inside predicates vs. handler-cached state.** Predicates run against the value
  passed to `commit`, not against re-read storage — this is the intended semantics and
  must be stated, lest a reader expect the persisted (pre-commit) state.

## Spec delta (sketch — normative prose written in-place during build, not here)

- §14 (validation): promote invariants from design-notes commitment to normative spec —
  declaration grammar, the predicate surface (per ADR 0107), commit-boundary checking
  semantics, per-agent scope. **Restate "revert" precisely** as non-persistence of the
  offending commit (not whole-handler rollback), and reconcile with the atomic-handler
  framing given the eager per-commit reality (see Risks).
- Grammar appendix: the `invariant` production inside the agent body; `implies` operator
  precedence (below `||`).
- Diagnostics index: the four `bynk.invariant.*` codes.
- Failure-model section: `InvariantViolation` as an internal fault distinct from
  `BoundaryError`, and what the caller observes (DECISION E).
- ADR 0107: agent-invariant predicate surface (DECISION A) **and fault semantics**
  (DECISION E — the dedicated fault class and the non-persistence guarantee), with B/C/D
  recorded as consequences. E is load-bearing enough to be ratcheted by the ADR, not just
  the proposal.
