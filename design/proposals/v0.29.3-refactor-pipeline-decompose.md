## Context вАФ what this is and why it's harder than 29.2

- `compile_project_pipeline` (`project.rs:238`, **~1,810 lines**) is the orchestrator the `v0.29.2`
  split deliberately left whole. This increment carves it into per-phase functions.
- Unlike 29.2 (move definitions, edit nothing), this **reshapes control flow**: each phase becomes a
  function taking explicit inputs and returning its products. That is why it is sequenced *after* the
  split вАФ reasoning about phase boundaries is far easier once their helpers live in named submodules
  (`project/{discovery,consistency,symbols,graph,validate}.rs`) rather than inline.
- The payoff: the back half is currently **unmarked and deeply nested** (~700 lines of per-unit loop
  body); after extraction every phase is a named, separately-testable unit with a signature that
  *documents* what it consumes and produces вАФ the precondition for the `UnitInfo` aggregate (item 6)
  and the `CompileOptions` API collapse (item 5) that follow.

## Settled

- **вЦЄ Behaviour-preserving extraction, golden-verified.** Output stays byte-identical; the existing
  end-to-end golden fixtures are the acceptance gate (they cover the pipeline's observable output
  comprehensively вАФ unlike 29.1's pure helpers, the *right* net here is integration-level, already in
  place). Any fixture diff is a defect.
- **вЦЄ Phases become functions with explicit signatures.** Each takes what it reads and returns what it
  produces вАФ no phase reaches back into a later phase's locals. Where a phase mutates shared
  accumulators (the parallel maps, the sinks), they pass as explicit `&mut` params, not captured
  closure state.

## Decisions (open вАФ settle at review)

**[DECISION] Extract first, aggregate later вАФ do *not* fold in `UnitInfo` (queue item 6) here.** The
queue suggests folding the `UnitInfo` aggregate into the pipeline work. *Rec: keep them separate.* This
increment threads the **existing** parallel maps (`kinds`, `unit_tables`, `unit_uses`, `unit_consumes`,
`exports_visibility`) as explicit params вАФ preserving the data model keeps the extraction a pure,
golden-verifiable refactor. Collapsing those maps into one `HashMap<String, UnitInfo>` is a **data-model
change** and earns its own increment (`v0.29.4`), which is then *trivial*: the phase signatures already
exist, so it's a mechanical param-type swap (`&HashMap<String, UnitInfo>` for five `&HashMap<String,
_>`). Mixing the two means moving code **and** retyping it at once вАФ harder to review, harder to bisect.
*(This is a deliberate deviation from the queue's "fold in" note, on the move-then-retype principle that
served 29.2.)*

**[DECISION] Granularity of the back-half carve.** The phase-8 loop body is the genuinely nested part.
*Rec:* extract the per-unit body into named steps вАФ `compose_unit_symbols` (uses mixin),
`merge_consumed_exports` (consumes + visibility), `collect_unit_methods`, `check_unit_files`,
`emit_unit` вАФ each taking the composed symbol space and returning its product, with the `for` loop
reduced to a readable sequence of calls. Keep the loop itself in `pipeline`; extract its *body*.

**[DECISION] One PR or staged by half?** The front half (1вАУ7) extracts cleanly phase-by-phase; the back
half (8) is the deeper surgery. *Rec:* **two PRs** вАФ front-half phase extraction first (low-risk,
marker-guided), back-half loop carve second (where the nesting and shared state concentrate). Both
behaviour-preserving, both gated on the same golden fixtures. *(Front/back is a real fault line here, vs
29.2 where one PR was preferred вАФ because this increment edits control flow, smaller diffs materially
help review.)*

## Spec delta

**None.** No language surface, no `docs/src/spec/` chapter вАФ internal control-flow reshaping only.

## Tests

- **No new golden fixtures.** The pipeline's observable output is already comprehensively golden-tested;
  extraction must leave every fixture **unchanged and green**. That invariance is the proof.
- **Optional pins for any pure sub-helper that falls out.** If carving a phase surfaces a genuinely pure
  sub-function (e.g. a symbol-merge or conflict-detection helper with no I/O), add a direct
  characterisation test for it in the same PR вАФ same rule as 29.1, applied opportunistically to
  whatever the carve exposes. Phases that are inherently I/O-or-sink-driven stay covered by the golden
  fixtures.

## Diagnostics / Risks

- **Shared mutable state is the real difficulty.** The five parallel maps plus the sinks are read and
  written across phases; an extraction that gets a `&mut` aliasing or ordering detail wrong changes
  output. Mitigation: extract **one phase at a time**, run the full golden suite after each, and lean on
  the explicit-signature discipline (a phase that needs a map it wasn't given is a borrow error at
  compile time вАФ the type system enforces the data-flow honesty).
- **Phase ordering is load-bearing.** Several phases depend on earlier accumulators (e.g. consumes-cycle
  detection needs the fully-built `unit_consumes`). Extraction must preserve the *sequence*; returning
  products and feeding them forward (rather than mutating shared state in place) makes the ordering
  explicit and harder to get wrong вАФ prefer that shape where a phase has a clean product.
- **Borrow-checker friction over output drift.** The likely failure mode is the code not compiling
  (overlapping `&mut`), not silently wrong output вАФ which is the *safe* failure for a refactor. Resolve
  by tightening what each phase borrows, never by cloning-to-dodge in a way that changes semantics.
- **Diff size on the back half.** ~700 nested lines becoming ~5 functions is a large diff even when
  faithful; the two-PR split and per-phase golden runs keep it reviewable.

## Definition of done

exposes; `cargo build`/`test`/clippy and CI stay green; changelog notes the decomposition. The
implementing PR(s) **delete this file**.
