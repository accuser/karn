- **Spec sections touched:** §4 (syntactic grammar — only if [A]=marker), §5 (static

## Goal

Ship the first **platform adapter** — `cloudflare` providing a minimal `Kv`
(get/put/delete) — and turn **platform-lock enforcement** from latent (0017) into
live. The increment is **collection-free** (0023): `Kv` needs only `String` / `Option`
/ `Effect`, which already exist. It proves the platform-adapter *and* locking path
end-to-end on the simplest possible capability. Extending `Kv` (`list`, JSON values)
and adding `Queue` are explicitly **v0.22**, after the functional core (v0.20) and the
wider stdlib (v0.21).

## Why the delta is small

Most of the machinery shipped already:

- the `adapter` kind, external providers, the `binding` clause (0010, 0013);
- adapter `consumes` + external-provider `given` wiring, and the **transitive
  given-closure walk** built for compose imports (0019) — 0017 names this *the
  lock-propagation primitive*;
- env-taking first-party providers via a **metadata hook** (0021), flagged there as
  "what platform-adapter resource derivation extends";
- two `--platform` values, `cloudflare` and `node` — so a platform mismatch is now
  expressible and the lock is finally **exercisable**.

Genuinely new: a way to mark a capability platform-native, `[[kv_namespaces]]` +
`env.KV` derivation, the two lock diagnostics + effective-platform computation, and the
`cloudflare.Kv` source/binding.

## Decisions

**[DECISION A] How a capability is marked platform-native (lock-inducing).**
- **A1 (recommended)** — first-party metadata. The toolchain registers `cloudflare` as
  a platform and its `Kv` as platform-native, **extending the env-taking-provider
  metadata hook** (0021). No new user-facing syntax; §4 grammar unchanged.
- A2 — a user-facing `platform "<name>"` marker on the adapter declaration.
- *Rationale:* `cloudflare` is first-party and there are no user-authored platform
  adapters yet, so marker syntax is premature. Defer A2 to when third-party platform
  adapters become a goal (it can be added additively then).

**[DECISION B] `Kv` operation error model.**
- **B1 (recommended)** — plain `Effect`: `get(key) -> Effect[Option[String]]`,
  `put/delete -> Effect[()]`. A catastrophic KV failure is an unhandled `Effect`
  rejection (the runtime throws), as elsewhere.
- B2 — a typed `Result[…, KvError]` for operational failures.
- *Rationale:* matches "not extensive" (0023) and Cloudflare KV's common path (null /
  value, no typed error). A `KvError` can land with the v0.22 extension if demand appears.

**[DECISION C] KV namespace identity.**
- **C1 (recommended)** — one fixed binding name (`KV`) from first-party metadata, driving
  one `[[kv_namespaces]]` stanza (placeholder `id`) and the `env.KV` field in the generated
  `Env`.
- C2 — configurable / multiple namespaces (per-consumer binding names).
- *Rationale:* one namespace is enough to prove the path; multiple/named namespaces are
  a v0.22 concern and need a config surface we don't want to design under this increment.

**[DECISION D] Env availability across emit targets — the real wrinkle.**
- KV bindings live **only** on the Worker `env` object — *not* on `globalThis`/
  `process.env` — so 0021's `globalThis` probe (which finds Secrets) **cannot reach KV**.
  `env` must be *threaded* to the `Kv` binding. Under `--target workers` compose already
  threads `env`; under `--target bundle`, 0021 deliberately threads nothing.
- **D1 (recommended)** — extend the **bundle** entry/compose to thread `env` *when a
  platform-native resource is consumed*. A bundle on Cloudflare is a single Worker and
  has `env` at its entry; threading it for platform resources is the honest path and a
  localised compose change.
- D2 — restrict `cloudflare` native capabilities to `--target workers`; bundle
  unsupported for now.
- *Rationale:* D1 keeps bundle-on-Cloudflare a real deployment; D2 is simpler but
  arbitrarily excludes a legitimate topology. Recommend D1 unless it proves invasive,
  in which case fall back to D2 with a clear diagnostic.

> Effective-platform computation itself is **not** a fork — reuse the given-closure walk
> (0017/0019). The rule: a deployment unit's platform set is the union, over its
> in-process `given` closure, of every platform-native capability's platform; a
> service-consume (`consumes` of a context, RPC) edge does **not** contribute.

## Spec delta (sketch — normative prose is written in-place during implementation)

- **§4 Syntactic grammar** — unchanged under [A]=A1; under A2, a `platform` marker
  production on `adapter`.
- **§5 Static semantics** — the lock rules tied to their diagnostics: `vendor_required`
  (a deployment unit uses a platform-native capability but `--platform` selects another
  platform); `vendor_conflict` (one deployment unit's `given` closure spans two
  mutually-exclusive platforms — *within a context* under `workers`, *anywhere in the
  bundle* under `bundle`). Cites 0017 for the deployment-unit definition.
- **§7 Meaning by translation / §8 compilation model** — the `cloudflare` binding reads
  `env.KV` explicitly (extends the 0021 metadata hook); `--platform cloudflare` selects
  the binding; `[[kv_namespaces]]` + `env.KV` typing derived from first-party metadata
  ([C]); the [D] env-threading rule.
- **§9 Diagnostics** — the two codes below.
- **Appendix B** — the v0.19 version-history entry.

## Diagnostics

| Code | Cause |
|---|---|
| `karn.target.vendor_required` | a deployment unit uses a platform-native capability built for another `--platform` |
| `karn.target.vendor_conflict` | one deployment unit's `given` closure spans two mutually-exclusive platforms |

(Both were pre-named in the retired adapters spec as deferred; they land now.)

## Fixtures (shape — not normative emission)

- **Positive** — a context `consumes cloudflare { Kv }` under `--platform cloudflare`,
  exercising get/put/delete; passes the `tsc --strict` gate against the first-party
  binding; one `workers` and one `bundle` variant ([D]).
- **Negative** — the same context built for `--platform node` → `vendor_required`.
- **`vendor_conflict` coverage** — *cannot* be triggered by a real program with only one
  native platform (`node` ships no native capabilities). Options: a **test-only second
  native platform** to exercise the rule now, or implement the rule and **defer its
  fixture** to the `aws` adapter. Recommend the test-only synthetic so the rule isn't
  shipped untested. **(Sub-decision — settle at review.)**
- **Regression guard** — the Secrets fixtures (0021) stay byte-stable; Clock/Random/
  Logger no-arg constructors unchanged.

## Risks

- **Effective-platform across topologies** — the bundle (whole-program union) vs workers
  (per-context) distinction, and correctly treating `given` (propagates) vs
  service-`consumes` (RPC, does not). The walk exists (0019); the *classification* of
  edges is the careful part.
- **[D] bundle env threading** — extending bundle compose without regressing the 0021
  "bundle threads nothing" choice for Secrets.
- **Untestable conflict** — see the fixtures note; don't ship an unexercised rule.
