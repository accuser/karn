# v0.20a — First-class functions + generic functions

- **Status:** Draft (proposal). **Merging this PR is the approval to build.**
- **Builds on (verified):** the structural effect model (`effectful` = result type is
  `Effect[T]`); **effectful free functions already emit `async`/`Promise`**, so
  `(Int -> Effect[Int])` params and v0.20b's `traverse`-shaped combinators rest on shipped
  behaviour; the type-bracket syntax (`List[T]`; `Mock[T](…)` is existing precedent for a
  bracket list after an identifier in *expression* position); `match`'s `=>` and the type
  `->`. *Net-new, confirmed absent:* a function type in `TypeRef` (its generics are the
  closed `Result`/`Option`/`Effect`/`HttpResult` set) and type parameters on `FnDecl`.
- **Spec sections touched:** §4 (syntactic grammar), §5 (static semantics), §6 (type
  system), §7 (emission), §9 (diagnostics), Appendix B.
- **First slice of the functional core (v0.20).** v0.20b — built-in `List`/`Map` + the
  combinator stdlib — follows in its own proposal and depends on this.

## Goal

Add **first-class functions (lambdas)** and **generic functions** — the foundations
collections (v0.20b) need (higher-order combinators) and a **Karn-written** stdlib needs
(generic functions). Useful and testable on their own (a HOF `apply`, a generic
`identity`/`compose`) with no collections. Slicing mirrors v0.10/v0.16.

## Settled (recorded as ADRs ≥0027 this increment)

- **▸ Lambda syntax** — `(params) => expr`, always parenthesised: `(o) => o.paid`,
  `(acc, t) => acc + t`, `() => …`. Param annotations optional where inferable
  (`(o: Order) => …`); block body `(o) => { … }` for multi-statement. `=>` is the **value**
  arrow (shared with `match`); `->` stays the **type** arrow. No `fn(...)` form, no `_`.
- **▸ Generics scope — Open-narrow.** Type parameters on **functions** only
  (`fn name[A, B](…)`); no generic user types, no bounds. `List`/`Map` (v0.20b) stay the
  only generic *types*. A strict, additively-extensible subset; the type-parameter
  representation is kept **bound-capable** so constrained generics remain reachable.
- **▸ Effect-tracking is free.** `A -> B` is effectful iff `B` is `Effect[X]` — the existing
  structural rule decides it; the pure-vs-effectful (`map`-vs-`traverse`) split needs no new
  machinery.
- **▸ Closures over capabilities, and bottom-up lambda effectfulness.** A lambda may close
  over and call a `given` capability; it is then effectful (`A -> Effect[B]`), used with
  `traverse`-shaped combinators. A lambda's effectfulness is inferred **bottom-up**: a body
  that performs an effect operation (a capability call, an effectful function-value call, or
  a `<-` bind) makes the lambda effectful and wraps its result in `Effect`. This resolves the
  apparent circularity (the body is judged effectful by the *presence* of effect operations,
  not by a pre-declared result type), and is what v0.20b's effectful iteration rests on.
- **▸ Function types are confined to non-boundary positions.** A function type is legal only
  in fn/lambda **parameters, returns, and locals**. It is **rejected** anywhere it would
  serialise, persist, or cross a boundary — service/agent handler signatures (Workers wire),
  record fields, agent state, exported types — with `karn.types.function_at_boundary`.
  (Functions cannot serialise; the `serialise_<T>`/`deserialise_<T>` machinery has no case
  for them, so this rule prevents a silent `as any` / emission explosion.)
  **Capability operation signatures also reject function types in v0.20a.** A higher-order
  capability op is technically coherent (capabilities wire in-process via the deps object,
  never serialised — 0008), but it enlarges the surface and interacts with mocks beyond what
  v0.20a needs; allow it later additively if the stdlib wants it.
- **▸ Effect[T] stays non-storable.** Because `Effect` is an eager `Promise`, an effectful
  **function-value** call (`f(x) : Effect[B]`) obeys the *same `<-` confinement* as a
  capability/service call: bound at the call site, never stored or passed as a value. This
  feature does **not** make `Effect[T]` a first-class storable type. (Today effects arise
  only from capability/service calls in `<-` positions; function values must not open a
  back door.)

## Decisions (open — settle at review)

**[DECISION] Type-argument inference + explicit fallback.** Type args **inferred at call
sites** by simple argument-directed unification (*not* full Hindley–Milner — keep errors
local and predictable). A type parameter appearing **only in return position** (a future
`List.empty[T]()`) isn't inferable; accept an explicit **`name[T](…)`** form for those, and
reject otherwise (`karn.generics.uninferable_type_arg`). *Rec:* infer-from-args + explicit
fallback.

**[DECISION] Named functions as values.** Today `karn.resolve.fn_without_call` forbids a
bare function reference ("a function must be called"). With function values, `map(xs, double)`
would hit it, leaving `(x) => double(x)` as the only path. *Rec:* **relax
`fn_without_call`** to permit a bare named-function reference *where a function type is
expected* (contextual and cheap); keep it firing in non-function-typed positions. The
alternative — mandate eta-expansion — is worse ergonomics and must be stated if chosen.

**[DECISION] Lambdas in unconstrained positions.** Inference covers a lambda flowing into a
typed parameter, but not `let f = (x) => x + 1` (no expected type → param type uninferable).
*Rec:* **require param annotations when there is no expected type** (`let f = (x: Int) => …`),
diagnosed by `karn.lambda.unannotated_param`. (`karn.types.lambda_mismatch` doesn't apply —
there's no expected shape to mismatch.)

**[DECISION] Generic methods.** Whether `[A,B]` is allowed on UFCS methods as well as free
functions. *Rec:* **free functions only** in v0.20a; generic methods can follow additively.

## Reconciling existing diagnostics (not green-field)

- `karn.resolve.param_as_function` ("a value/parameter called as a function") **relaxes** —
  a **function-typed** value is now legitimately callable; the code fires only for
  **non-function-typed** values. (This replaces the colliding `karn.fn.not_callable` from the
  earlier draft, which is **dropped**.)
- `karn.resolve.type_as_function` (a *type name* called) — **unchanged**.
- `karn.resolve.fn_without_call` — **relaxed** per the decision above.

## Spec delta (sketch — normative prose written in-place during implementation)

- **§4 Syntactic grammar** — function **type** (`A -> B`; `(A, B) -> C`; `() -> B`), with
  `->` **right-associative** (`A -> B -> C` ≡ `A -> (B -> C)`); **lambda** expression
  (`(params) => expr | block`); **type-parameter list** `[A, B]` on `fn`; **value
  application** (a call on an expression of function type); explicit **type-argument** form
  `name[T](…)`.
- **§5 Static semantics** — lambda typing (params from expected type or annotation; result
  and **effectfulness inferred bottom-up** from the body — an effect operation makes the
  lambda effectful, §closures rule above); the **unconstrained-lambda** annotation rule;
  generic instantiation by
  argument-directed inference + explicit fallback; rejection of generic *type* declarations
  and *bounds*; the **boundary-position** restriction (§5.8) and the **`<-` confinement of
  effectful function-value calls**; relaxations to the three `karn.resolve.*` codes above.
- **§6 Type system** — function types as a type form (§6.5: where they are/aren't admissible
  — non-boundary only); type parameters and their instantiation; effect-structural
  classification of function types.
- **§7 Emission** — lambdas → TS arrow functions (expression/block, lexical capture — the
  same closure shape the emitter already produces for handler bodies); generic functions →
  **erased** TS generics; value application → a TS call; effectful function-value calls
  emitted in `<-` (awaited) position only.
- **§9 Diagnostics** — the codes below.
- **Appendix B** — the v0.20a entry.

## Fixtures (shape — not normative emission)

- **Positive** — HOF `apply(f: Int -> Int, x: Int)`; generic `identity[A]`/`compose`; an
  **effectful** function param `run(f: Int -> Effect[Int], …)` proving the structural split
  (and the `<-` confinement); a **named function passed as a value** (`apply(double, 5)`); an
  explicit type-arg call. Each passes `tsc --strict`.
- **Negative** — function type in a **service signature** and in a **record field** →
  `function_at_boundary`; storing an **effectful function-value call** without `<-` → the
  existing effect-position rule (`karn.effect.*`; pin the exact code at implementation);
  `let f = (x) => x` → `unannotated_param`; `type Box[T]` →
  `no_generic_types`; `fn f[A: Ord]` → `no_bounds`; return-only type param, no explicit arg →
  `uninferable_type_arg`; wrong-arity lambda into a HOF → `lambda_mismatch`.
- **Additive guard** — all prior fixtures byte-stable; the `karn.resolve.*` relaxations must
  not change any *currently-passing* program's diagnostics (only *newly-legal* programs gain
  acceptance).

## Risks

- **Parse ambiguity / lookahead.** `(x) => e` vs a parenthesised expression — and `() => e`
  vs the unit literal `()` — need scan-to-`)`-then-check-`=>` lookahead in `karnc`, and a GLR
  conflict annotation (or equivalent) in tree-sitter. `name[T](…)` parses on settled ground
  (the `Mock[T](…)` precedent); v0.20b's stdlib is combinator-shaped so no `expr[index]`
  collision looms — but reserve that judgement explicitly.
- **Inference scope.** Argument-directed and local only — the explicit `name[T](…)` form is
  the pressure valve; resist creeping toward full HM.
- **Effect eagerness.** The `<-` confinement (settled above) is the load-bearing rule: an
  un-bound effectful function-value call would be a running `Promise`. Enforce it exactly as
  for capability calls, with the negative fixture as the guard.
- **Erased lowering** — no runtime type-arg dispatch (consistent with no reflection/typeclasses).
