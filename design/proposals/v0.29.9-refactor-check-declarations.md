# v0.29.9 — Refactor track, increment 9: decompose `check_v0_5_declarations`

- **Status:** Draft (proposal). **Merging is approval to build.** *(Refactor track item 3, second half
  — the validate-side companion to `29.7`'s `lower_expr`. Gate on `29.2` (which surfaced its home,
  `project/validate.rs`) is **satisfied** (merged); the `29.1` safety net is in place.
  Behaviour-preserving — diagnostics for every fixture are byte-identical.)*

## Context — a 500-line validator over four declaration kinds

- `check_v0_5_declarations` (now `project/validate.rs:438`–`960`, ~522 lines, moved there verbatim by
  the `29.2` split) is, despite its historical name, the semantic-validation pass for **all** context
  declarations. It is a flat sequence of independent passes, each a `for … in table.<kind>` loop
  (verified offsets, relative to `:438`):
  - **capabilities** — `for (name, decl) in &table.capabilities` (`+28`), op params/returns.
  - **cross-context caps** — `for (cap, unit) in &cross_context.flattened_caps` (`+71`).
  - **providers** — `for provider in table.providers.values()` (`+105`), their `given` clauses (`+118`)
    and ops (`+129`).
  - **services** — several passes `for service in table.services.values()` (`+166`, `+197`, `+225`,
    `+249`) over handlers and their `given` clauses.
  - **agents** — `for agent in table.agents.values()` (`+299`), state fields (`+303`, `+341`), handlers
    and their `given` clauses (`+447`, `+449`).
- The passes share inputs (`table`, `cross_context`, `errors`) but are otherwise independent — each is
  a candidate function. The name is a v0.5 artifact that now undersells the scope (a rename candidate,
  tying into queue item 13's marker-convention cleanup).

## Builds on (verified)

- **The seams are `for`-loop boundaries** over distinct `table.<kind>` collections — extraction is
  per-kind: `check_capability_decls`, `check_provider_decls`, `check_service_decls`,
  `check_agent_decls` (+ the cross-context-cap pass), each taking `&UnitTable`/`&CrossContextInfo` and
  `&mut Vec<CompileError>`.
- **`29.2` gave it a home (merged).** After the `project.rs` split this function lives in
  `project/validate.rs` alongside `validate_http_handler`/`validate_cron_handler`/
  `validate_queue_handler` — so the per-kind validators land beside the handler validators they call.
- **Golden fixtures cover the diagnostics** — every declaration-kind error has fixture coverage; that
  is the acceptance gate.

## Settled

- **▸ Behaviour-preserving per-kind extraction.** Pass bodies move into functions verbatim; the parent
  becomes a sequence of calls. Diagnostic text, order, and spans are unchanged.
- **▸ Diagnostic ordering preserved.** The passes emit errors in a fixed sequence (capabilities →
  providers → services → agents); extraction must call them in the same order so multi-error fixtures
  are byte-identical.

## Decisions (open — settle at review)

**[DECISION] Granularity.** *Rec:* one validator **per declaration kind** (capabilities, providers,
services, agents) plus the cross-context-cap pass; the multiple service passes (`+166`/`+197`/`+225`/
`+249`) collapse into `check_service_decls` with internal sub-steps, or stay as separately-named
service sub-validators if each pass is conceptually distinct — decide by reading whether they share
per-service state. Leave any genuinely tiny inline check in place.

**[DECISION] Rename the function?** The `check_v0_5_declarations` name predates providers/services/
agents. *Rec:* rename to `check_context_declarations` (or split entirely into the per-kind validators
with no umbrella function) **as part of this increment**, since it is already being restructured — a
clean application of queue item 13 (lead with the *what*, not the version tag). *Caveat:* it is
`pub(crate)`-or-private; confirm the (few) call sites and update them in the same PR.

**[DECISION] One PR or staged?** *Rec:* **one PR** — the passes are independent and the extraction is
mechanical; unlike `lower_expr`'s `MethodCall`, there is no single dominating arm with deep shared
state. If it proves large, split capabilities+providers from services+agents.

## Spec delta

**None.** No language surface, no `docs/src/spec/` chapter — internal control-flow reshaping.

## Tests

- **No new golden fixtures.** Existing declaration-validation fixtures must pass **unedited**.
- **Optional pins** for any pure sub-helper the carve exposes (e.g. a state-field or op-param
  classifier), per the `29.1` rule, in the same PR.

## Diagnostics / Risks

- **Diagnostic order and dedup.** The single real risk is reordering passes or double-emitting an error
  — caught by multi-error golden fixtures. Preserve the call sequence; if a pass currently `continue`s
  past later passes for a given decl, reproduce that control flow.
- **Shared `errors` borrow.** Each validator takes `&mut Vec<CompileError>`; straightforward, same
  shape as the existing `validate_*_handler` functions.
- **Rename blast radius.** If renamed, update every call site + any doc-comment cross-reference in the
  same PR; a stale reference is a compile error, so the type system enforces completeness.
