# v0.111 — a raw HTTP body: the `Raw` payload shape and a `Raw` (200) variant carrying `(Bytes, String)`, lowered to a body with an author-declared `content-type`

- **Scope:** a single-increment **language** proposal (no track) — one new
  `HttpResult` payload shape and one variant, in the exact 0126/0129 idiom.
  Grammar unchanged (variants are registry rows, not syntax); the change is the
  `HTTP_VARIANTS` registry, three exhaustive arms, and one runtime case. Version
  → **v0.111**.
- **Realises:** the service-tier boundary call — Bynk handlers return *typed
  values* (JSON by default), and non-JSON service bodies (`robots.txt`,
  `sitemap.xml`, `.well-known`, RSS/Atom, CSV downloads, a QR-code PNG) are
  served as an **explicit, named raw body**, not by a templating/view layer. HTML
  *rendering* stays out of scope — presentation is the frontend tier (Cloudflare
  Pages; the client side via the retired in-browser track).
- **Relates:** [ADR 0126](../decisions/0126-httpresult-rfc9110-status-vocabulary.md)
  (the payload-shape registry this extends, exactly as 0126 D2 added `Location`);
  [ADR 0129](../decisions/0129-streaming-http-response.md) (`Streamed` — the
  sibling shape added by one row + three arms, and the 200-only precedent this
  reuses); [ADR 0142](../decisions/0142-bytes-base-type.md) (`Bytes` — the octet
  sequence this body carries, and D8's "typed paths round-trip a `Bytes` fine").

## The shape, in one line

`HttpVariantPayload` gains a `Raw` arm carrying **two** arguments —
`(body: Bytes, contentType: String)` — realised by one dedicated `Raw` variant
at status **200**. A handler returns:

```
Raw(png, "image/png")                          // a QR-code PNG
Raw(Bytes.fromUtf8(xml), "application/xml")     // an XML document
Raw(Bytes.fromUtf8(csv), "text/csv")            // a CSV download
Raw(Bytes.fromUtf8(robots), "text/plain")       // robots.txt
```

`Bytes` (0142) is the body type precisely because it is binary-first: the PNG
flows in directly, and text bodies go through the documented `Bytes.fromUtf8`
constructor, making the charset an explicit author decision rather than a runtime
guess. One shape covers both text and binary — so **XML needs no new variant**;
it is `Raw` today. A *typed* XML codec (an XML analogue of the `Value`→JSON path,
with a real serialiser) is a separate and larger decision, orthogonal to this and
explicitly out of scope.

## Why this is the whole increment (and why it is small)

`HttpResult` is already a closed registry of status-bearing variants over a
closed set of payload **shapes** (`None`/`Value`/`Message`/`Location`/`Streamed`).
0126 D3 made adding a variant a single `HTTP_VARIANTS` row that the resolver, the
LSP completion (`bynk-lsp/src/completion.rs`, sourced from the registry), and the
emitter lowering pick up for free. 0129 proved the pattern for a *new shape*: one
row + three exhaustive arms (construction-check, pattern-binding, the runtime
status map) + one runtime case. This increment is that pattern once more, with the
body type already shipped (0142). Nothing else changes.

## What changes, against the current code

1. **`bynk-syntax/src/ast.rs`** — add `HttpVariantPayload::Raw` (the enum at
   ~line 815) and one `HTTP_VARIANTS` row (`Raw`, `Value`-less, status 200) in
   the 2xx block, beside `Streaming`. This is the single source of truth (0126
   D3); the resolver, completion, and emitter dispatch read it by name.
2. **Construction-check** (`bynk-check/src/checker/expressions.rs`,
   `check_http_variant` at ~line 1341): a `Raw` arm expecting **two** arguments —
   a `Bytes` then a `String` — and returning `HttpResult[()]` (the type parameter
   `T` is unused, as for `Location`/`Streamed`). This is **genuinely new arm
   logic, not reused machinery**: `Raw` is the first two-argument
   `HttpVariantPayload` — every prior shape is 0- or 1-ary — so the arm is the
   first with `args.len() != 2` and two independent `type_of` + compatibility
   checks (arg0 against `Bytes`, arg1 against `String`), with an arity message
   reading "expects 2 arguments". What is reused is the *diagnostic codes*
   (`bynk.types.variant_arity` / `argument_mismatch`), not the logic that raises
   them — this is the arm carrying the most logic of any shape, and the small real
   work of the increment sits here. `Bytes.fromUtf8` already yields a `Bytes`, and
   a refined `String` flows into the `contentType` argument by the existing
   refinement→base widening rule.
3. **`variants_of`** (`bynk-check/src/checker.rs`, the `Ty::HttpResult` arm at
   ~line 2319): add a `Raw` case to the `match v.payload`, returning the
   two-field payload `[("body", Bytes), ("contentType", String)]`. This edit is
   **forced by Rust exhaustiveness**, not by a reachable code path: the `match` is
   exhaustive over `HttpVariantPayload`, so adding the enum variant fails to
   compile until the arm exists. Whether the binding is ever *exercised* is a
   separate question — an `HttpResult` is **constructed** in handler position and
   never **scrutinised** (nothing matches on one; the value goes straight to
   `httpResultToResponse`), so this is the project's **first** two-field variant
   payload but its match-binding is not on any real path. Keep the field names
   `body`/`contentType` **byte-identical** to the runtime union (site 4a below),
   as `stream`/`location`/`message`/`value` already are across both sides.
4. **The runtime `http.ts` — four sites, not one** (`bynk-emit/runtime/src/http.ts`,
   the hand-maintained source of truth marked "Kept in sync with `HTTP_VARIANTS`"
   at ~line 111, regenerated into the shipped `bynk-emit/src/emitter/runtime.ts`
   by the existing bundler/drift guard, 0129 D4). Every variant appears at four
   sites and `Raw` touches all four:
   - **4a — union member** (~lines 5–40):
     `| { readonly tag: "Raw"; readonly body: Uint8Array; readonly contentType: string }`.
   - **4b — constructor factory** (~lines 42–87):
     `Raw: (body: Uint8Array, contentType: string): HttpResult<never> => ({ tag: "Raw", body, contentType })`.
     **This is the site the "registry-driven, no emitter edit" claim actually
     rests on** — a constructed `Raw(png, "image/png")` lowers to a call on this
     factory object, so "no *per-variant emitter* edit" is true only *because* 4b
     exists. The two are the same fact; the claim and the constructor cannot be
     separated.
   - **4c — `HTTP_STATUS`** (~line 112): `Raw: 200`. **Required even though the
     `Raw` case hardcodes 200 and never reads this**, because `HTTP_STATUS` is
     typed `Record<HttpResult<unknown>["tag"], number>` (exhaustive) and omitting
     the key fails `tsc --strict`. `Streaming: 200` sits at ~line 114 for exactly
     this reason.
   - **4d — the switch case** in `httpResultToResponse` (~line 172): **no helper
     needed** (unlike `sseResponse`), because the `Uint8Array` a `Bytes` erases to
     is a valid `BodyInit` on both Workers and Node:

     ```ts
     // Raw — an author-owned body with an explicit content-type. No codec runs.
     case "Raw":
       return new Response(result.body, {
         status: 200,
         headers: { "content-type": result.contentType },
       });
     ```

     `serialiseValue` is bypassed entirely — that is the point of the shape.

## Decisions

**[DECISION A] One shape `(body: Bytes, contentType: String)`, binary-first,
text via `Bytes.fromUtf8` — not a separate `Text(String, …)` variant.** A single
uniform shape matches 0126's one-variant-one-shape model, keeps the runtime a
one-liner, and forces the charset to be an explicit author decision rather than a
runtime assumption. The cost is real — the common text case (XML, `robots.txt`,
RSS) wraps in `Bytes.fromUtf8` — but it is one documented call, and honest.
*Recommend the single shape.* An ergonomic `Text` sugar variant is a named
follow-on if the friction proves out in practice, exactly as 0129 deferred a
structured `SseEvent` behind the minimal `Stream[String]`.

**[DECISION B] 200-only, mirroring `Streaming` (0129 D2) — and it is a feature,
not a limitation.** Keeping `Raw` at a single status preserves the
one-variant-one-status property that gives the registry its single-source-of-truth
and completion-for-free behaviour (0126 D3); letting `Raw` carry a status *value*
would break it. It also aligns with the tier boundary: the genuinely service-tier
raw bodies (`robots.txt`, `sitemap.xml`, `.well-known`, feeds, QR PNG, CSV
download) are overwhelmingly 200, whereas a custom-status raw body — a `404` with
an HTML error page — is the presentation concern this increment deliberately
excludes. *Recommend 200-only.* Non-200 raw bodies are the **re-openable
boundary** (a later `Raw`-family row per status, or a status-carrying variant if
per-status rows prove silly) — noted, not built.

**[DECISION C] `contentType` is an opaque `String`, unvalidated in v1.** Accept
any `String`, exactly as `Message`/`Location` accept any `String`; do not
validate against a media-type registry. This keeps *content-type validation* out
of the checker — the arm still does the genuinely-new two-argument shape work
(change #2), but it adds no media-type logic on top. A refined `MediaType` (a
refined `String`) is a named follow-on.

**[DECISION D] This is *not* the workers-wire `Bytes` boundary — state it so a
reviewer does not conflate them.** `bynk-syntax/src/diagnostics.rs` (~line 1205)
rejects a *bare* `Bytes` in a `workers` wire signature because that erased
cross-context hop does not base64-encode it. The HTTP response body is a
**different** boundary: the runtime writes the `Uint8Array` straight into
`Response`, with no cross-context erasure — so `Raw(Bytes, …)` is one of the
"typed paths that round-trip a `Bytes` fine" 0142 D8 promised. No new diagnostic,
and no interaction with the workers-wire rule.

**[DECISION E] Additive — no version-breaking change, minor bump v0.111.** No
variant or shape is renamed or removed; this is a pure surface widening (0126 D5's
posture). Construction reuses the existing arity/mismatch **diagnostics** (the
codes, not the arm logic — change #2); `Raw` needs no bespoke *diagnostic*.

**[DECISION F] Variant name — `Raw`.** *Recommend `Raw`* over `Content`/`Body`:
it warns the reader that the typed-wire guarantee is deliberately off for this
body — the author owns the encoding, no codec runs. This is the one bikeshed;
flagged for sign-off.

## Fixtures

- **`bynkc/tests/fixtures/positive/242_http_raw_body`** (next free number after
  241; `234_http_streamed_response` is the sibling to mirror): a handler returning
  `Raw(Bytes.fromUtf8(xml), "application/xml")` and a binary `Raw(png,
  "image/png")`, asserting the generated `Response` carries the exact
  `content-type` and the body bytes unmodified, and that it type-checks under
  `tsc --strict` — the same bar 0129 held `Streaming` to.
- A negative fixture: `Raw("not-bytes", "text/plain")` (a `String` where `Bytes`
  is expected) reports the existing `argument_mismatch`, proving no bespoke error
  path.
- A coexistence fixture (the 0129 D2 shape): a handler with a `Raw` branch **and**
  a pre-body `NotFound` branch typing under `HttpResult[()]`, proving raw and
  ordinary variants share one handler.
- **No match-binding fixture** — deliberately. `Raw` is the project's first
  two-field variant payload, but an `HttpResult` is construct-only in handler
  position (change #3): it is never scrutinised, so the two-field binding in
  `variants_of` is not on a reachable path and there is nothing to assert. The
  `variants_of` arm exists to satisfy Rust exhaustiveness, and the construction
  and runtime fixtures above cover the two-field shape end to end. Stated here so
  a reviewer does not expect (or ask for) a match fixture that would test dead
  code.

## Risks & mitigations

- **Charset foot-gun on text bodies.** `Raw(Bytes.fromUtf8(s), "application/xml")`
  is UTF-8; a `content-type` claiming a different charset would mislead. *Mitigation:*
  document the `fromUtf8`↔`charset=utf-8` correspondence in the guide, and leave the
  refined `MediaType` (DECISION C follow-on) as the place to enforce it later. v1
  trusts the author, like `Message`.
- **"Why 200-only?" reports (custom error pages).** Expected, and answered by
  DECISION B's boundary note in the guide — a custom-status raw body is the
  presentation concern held out on purpose; the boundary is re-openable by one
  registry row when a real service-tier case (not a view-layer one) needs it.
- **Drift between the two runtime copies.** The `Raw` case lands in
  `bynk-emit/runtime/src/http.ts` (source) and is regenerated into
  `bynk-emit/src/emitter/runtime.ts`; the existing bundler/drift guard (0129 D4)
  catches a hand-edit to the shipped copy. No new guard needed.
- **Scope creep toward HTML.** The one-line defence: this ships *bytes + a
  content-type*, not a renderer. HTML templating remains explicitly out of scope,
  and the guide says so where a reader would look for it.
