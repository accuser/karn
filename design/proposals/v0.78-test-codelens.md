# v0.78 — Run/Debug a test from the editor: eager discovery + a Test CodeLens

- **Scope:** a single-increment tooling proposal (no track) — a focused `vscode-bynk`
  enhancement. **No language change, no compiler change, no server change**
  (`bynkServerVersion` stays `v0.77.0`; only the extension version → 0.78.0).
- **Realises:** the slice-4 debugging proposal's **DECISION C**, which left a test
  CodeLens *optional*. Real use asked for an in-editor Run/Debug affordance at the test.

## The two parts, and which is load-bearing

There are two separable pieces, and being honest about the split is the crux of this
proposal:

1. **Eager discovery (the load-bearing part).** Today discovery is *lazy* —
   `resolveHandler`/`refreshHandler` only fire when the Testing view opens or you hit
   Refresh (`vscode-bynk/src/testing.ts`), so opening a `.bynk` file and editing creates
   no test items and the editor shows nothing. Make discovery *eager* (also on a `.bynk`
   document opening / becoming active / changing) and the test items exist whenever the
   file is on screen — at which point **VS Code renders its native gutter run/debug
   glyphs at each declaration, for free**. That alone is a per-line Run/Debug affordance.
2. **A Test CodeLens (the discoverability layer).** A bespoke `▷ Run | Debug` text line
   above each `test` **case**, on top of the glyphs.

**Lenses and glyphs land on *case* declarations only.** Discovery gives a location only
for cases — `JsonCase` carries one and `upsertCase` sets the item's `uri`/`range`, whereas
`JsonSuite` has none and `upsertSuite` creates a locationless item (`testing.ts`). So
`testRanges(uri)` returns case ranges, and a `test <suite> { … }` line gets no lens or
glyph. Case-level is the useful granularity anyway; this just preempts a "why not on my
suite line?" report.

**So why the CodeLens, given step 1 already gives a per-line affordance?** One reason,
stated plainly: **discoverability**. The gutter glyph is small, lives in the margin, and
is easy to miss — especially for a new language where users aren't scanning for it. An
explicit `▷ Run | Debug` text label is unmissable and was the literal ask. That is the
*entire* marginal value of part 2 — it adds no capability the glyph lacks (both trigger
the same project run/debug; see Granularity). If that justification doesn't hold for you,
**ship part 1 alone** — the glyphs may well suffice, and part 2 is pure addition that can
follow. This proposal covers both; the open call is whether to include part 2 now.

## The granularity constraint (unchanged, and it bounds *both* affordances)

`bynkc test` has **no single-test filter** — it compiles the project and runs the
aggregated runner, so it always runs the *whole suite*. And `runHandler`
(`testing.ts`) ignores `request.include` — it iterates `doc.suites` regardless. So
**neither the glyph nor the CodeLens can run one test**; both trigger a project run:

- **Run** runs the project's tests via the existing flow (`testing.runAll`); the Test
  Explorer reconciles results, so *this* test's pass/fail shows at its line among the
  rest.
- **Debug** starts the slice-4 session (`bynkc test --inspect` + attach) via
  `debugBynkTests()`; the suite runs under the debugger and the breakpoint you set — in
  this test or the code it exercises — pauses. Placement matters here: you debug from the
  test you're looking at. Interactive debugging doesn't need single-test execution.

True single-test run is a **named follow-on** — it needs a `bynkc test` filter (the
emitted runner running only matching cases), a CLI/runner change out of scope here.

## What this adds (concretely, against the current code)

1. **Eager discovery** (`vscode-bynk/src/testing.ts`): on `onDidOpenTextDocument` /
   `onDidChangeActiveTextEditor` / debounced `onDidChangeTextDocument` for `language ==
   "bynk"`, run the existing `discover(...)` — *coalescing in-flight discoveries* (see
   Cost). Keeps the view-open/refresh triggers.
2. **Expose the discovered locations.** `registerTesting` currently keeps `ctrl` local
   (`testing.ts:58`). Return from it a small read-only **location index** —
   `testRanges(uri): vscode.Range[]` backed by the controller's items (the case ranges
   `upsertCase` already sets) — so the CodeLens provider can place lenses without
   re-parsing the `.bynk`.
3. **A `CodeLensProvider`** registered via `vscode.languages.registerCodeLensProvider({
   language: "bynk" }, …)` (a **new client-side provider — distinct from the existing
   LSP CodeLens middleware** in `codelens.ts`, which only hydrates the server's
   `N references` lenses). For the document, it emits a `▷ Run | Debug` lens at each
   `testRanges(uri)` entry, firing an `onDidChangeCodeLenses` after discovery settles.
4. **Commands the lenses invoke — reusing the existing project run/debug, *not*
   item-scoped:** **Run** → the **existing `bynk.runTests`** command (which already calls
   `testing.runAll`) — no redundant third command; **Debug** → a thin **new
   `bynk.debugTests`** command wrapping `debugBynkTests()`. So exactly one new command is
   registered. No per-item dispatch is built, because the CLI runs the whole suite
   (Granularity) — the per-line *placement* is the convenience; the *command* is
   project-wide. Honest, and avoids decorative plumbing.

**Lens stacking, noted:** above a `test` declaration that also has references, the editor
will show **two** lenses — the server's `N references` (existing middleware) and the new
`▷ Run | Debug`. VS Code composes providers; they coexist. Worth expecting visually.

## Decisions

**[DECISION A] Locate lenses from the discovered items, via an exposed location index —
not by parsing the `.bynk`.** The compiler's own `--no-run` discovery gives exact case
locations; a regex scan would trip on comments / strings containing `test`. The plumbing:
`registerTesting` returns `testRanges(uri)`; the provider reads it. (This is the concrete
dependency that makes DECISION A real.)

**[DECISION B] Suite-granularity run/debug; per-test filtering deferred.** Bounded by the
CLI and by `runHandler` ignoring `include`. Both the glyph and the lens trigger a project
run; the lens is labelled `Run Test` / `Debug Test` for the intent at that line. The
`bynkc test` filter is the follow-on.

**[DECISION C] Reuse the existing run/debug; no new run path, no item dispatch.** Run =
the existing `bynk.runTests` (→ `testing.runAll`); Debug = `debugBynkTests()` via a single
new `bynk.debugTests`. No second compile-and-run, and no `runTestAtItem` machinery that
would only run the suite anyway.

**[DECISION D] A setting to disable the CodeLens.** `bynk.testCodeLens` (default **on**)
for users who want only the gutter glyphs / the Testing view. (Eager discovery — part 1 —
is not gated; it's the prerequisite for the glyphs people *do* want.)

**[DECISION E — open] Part 2 (the CodeLens) now, or part 1 only?** *Recommend shipping
both*, justified by discoverability above — but part 1 alone is a legitimate, smaller cut
if the glyphs are deemed enough. Flagged for sign-off before build.

**[DECISION F] Version — minor v0.78; extension only.** `bynkServerVersion` unchanged.

## Risks & mitigations

- **Discovery is a whole-project compile, not a per-file scan.** `bynkc test . --no-run`
  compiles the project (its tests usually span several files), so a cheap "skip if the
  *active* document has no `test`" pre-scan helps little — editing any file can change
  whether the test files compile. Real mitigations: **debounce generously** (e.g. ~500ms
  after the last keystroke), **coalesce in-flight discoveries** (never stack compiles —
  drop/replace a pending run), and accept that this is *one project compile per
  edit-burst*, the same compile Refresh already runs, not a keystroke-rate cost.
- **CodeLens flicker** on edit. Debounce + fire `onDidChangeCodeLenses` only after
  discovery settles; reuse the prior items on a transient failure (as today).
- **"Run" runs the suite** — honest labelling + docs (DECISION B); the follow-on adds
  real filtering.
- **Two stacked lenses.** Expected and benign; documented above.
