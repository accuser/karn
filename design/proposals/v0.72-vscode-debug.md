# v0.72 — Debug in VS Code: the one-click finale (debugging track, slice 4)

- **Track:** [debugging](../tracks/debugging.md), slice 4 — the finale. Wraps the
  `--inspect` CLIs slices 2–3 built into a one-click VS Code experience: set a
  breakpoint in a `.bynk` file, press Debug.
- **Realises ADR:** [0104](../decisions/0104-debug-launch-model.md) D1/D2/D5 (reuse VS
  Code's JS debugger via a thin `DebugConfigurationProvider`; **no bespoke DAP**;
  `skipFiles` the runtime + glue). Consumes everything slices 1–3 shipped.
- **Posture:** tooling (VS Code extension). TypeScript only — no compiler/language
  change.

## Why this is the finale, and what's actually left

The hard problems are all solved: source maps are per-statement everywhere and compose
through both the Node type-strip path (`bynkc test --inspect`, v0.69) and the
`wrangler`/esbuild bundle (`bynk dev --inspect`, v0.71). A `.bynk` breakpoint already
binds and pauses under both runtimes from the CLI. What's missing is the **editor
surface**: today a user runs an `--inspect` CLI in a terminal and attaches a debugger by
hand. Slice 4 makes it one click.

Per ADR 0104 D1 this is **glue, not a Debug Adapter**: the extension contributes a
`"bynk"` debug type whose `DebugConfigurationProvider` *compiles, starts the inspector
(by shelling the CLI), reads the inspector URL, and hands off to VS Code's built-in
JavaScript debugger* (`pwa-node`). The `vscode-bynk` extension is clean greenfield here
— no `debuggers`/`breakpoints` contribution and no debug code today; its Test Explorer
(`src/testing.ts`) already tracks each case's `.bynk` file + line and shells `bynkc test
--format json`, so a Debug profile slots in beside the existing Run profile.

## The one genuine uncertainty — the attach (spike-first)

Everything else is wiring; the attach is the question, and it differs by runtime:

- **Node (`bynkc test --inspect`).** The CLI launches `node --inspect-brk` and prints a
  `ws://` URL. VS Code's `pwa-node` *attach* to a Node inspector is well-trodden — low
  risk. This is the headline path.
- **workerd (`bynk dev --inspect`).** `wrangler`'s inspector requires an **`Origin`
  header** on the CDP WebSocket (slice 3). Does `vscode-js-debug`'s attach send one, or
  reach the endpoint at all? Unknown. Cloudflare ships its own VS Code attach recipe, so
  it is *possible*, but the exact config (a `pwa-node` attach by port, or a
  `websocketAddress`/`attachSimplePort` form) must be confirmed.

So slice 4 **spikes the attach first** (in the extension / its integration harness):
confirm a `pwa-node` attach binds a `.bynk` breakpoint under `node --inspect-brk`, and
determine whether the same mechanism reaches `wrangler`'s inspector. The spike decides
DECISION B (is workerd-in-VS-Code in this slice, or a follow-on with the CLI as the
interim). No production extension code until the attach is observed.

## What this slice does

1. **Contributions** (`package.json`): a `debuggers` entry (`type: "bynk"`, a
   `configurationAttributes` schema, `initialConfigurations`/`configurationSnippets` for
   `launch.json`) and a `breakpoints` entry (`language: "bynk"`, so the editor lets you
   set breakpoints in `.bynk` files).
2. **A `DebugConfigurationProvider`** (`registerDebugConfigurationProvider("bynk", …)`)
   that resolves a `"bynk"` launch: pick the target (a test, or a `bynk dev` context),
   shell the matching `--inspect` CLI, parse the inspector URL/port from stdout, and
   `startDebugging` a delegated `pwa-node` *attach* session with `skipFiles`/`smartStep`
   over `runtime.ts` and the generated worker glue (ADR 0103 D5).
3. **A Test Explorer Debug profile** (`createRunProfile("Debug",
   TestRunProfileKind.Debug, …)`) — the headline entry. The test gutter/Explorer gains a
   **Debug** action beside Run; it shells `bynkc test --inspect`, attaches, and a
   breakpoint in the test body or the code it exercises pauses. Reuses the existing
   case→`.bynk`-location mapping.
4. **Binary resolution for `bynk`.** The extension resolves `bynkc-lsp` and `bynkc`
   today; `bynk dev --inspect` needs the **`bynk`** front-end too — add a
   `bynk.bynkPath` setting (default: `bynk` on PATH), mirroring `bynk.compilerPath`.
5. **A VS Code integration test** modelled on the existing `@vscode/test-electron`
   suite: start a bynk **test**-debug session in the fixture workspace and assert a
   breakpoint binds / the session starts. Guarded on `bynkc` + `node` being on the
   harness PATH (the harness provisions `bynkc-lsp` today; this extends it). The workerd
   attach is exercised locally / opt-in (CI has no `wrangler`).
6. **Docs:** a "Debugging in VS Code" guide (set a breakpoint, press Debug); the
   extension README/walkthrough entry.

## Decisions

**[DECISION A] Delegate to `pwa-node`; no bespoke DAP.** ADR 0104 D1, restated as the
build constraint: `resolveDebugConfiguration` returns / starts a `pwa-node` attach; we
implement no `DebugAdapter`. The source maps do breakpoint relocation; `names` give
variable panes Bynk identifiers (rich value formatting is Phase 2 / slice 5, not here).

**[DECISION B — sign-off, spike-gated] Node test-debug ships; workerd-in-VS-Code is
gated on the attach spike.** The Node test-debug path (the one-click headline) is low
risk and ships. The `bynk dev --inspect` VS Code launch ships **iff** the spike confirms
`vscode-js-debug` attaches to `wrangler`'s inspector (the `Origin` question); otherwise
it is a documented follow-on — the CLI `bynk dev --inspect` + manual attach already works
(slice 3). *Recommend this scoping* — it guarantees a shippable, high-value finale without
betting it on an unconfirmed third-party attach.

**[DECISION C] The Test Explorer Debug profile is the primary entry; a separate
CodeLens is optional.** VS Code renders Run/Debug affordances from the test profiles, so
a Debug profile gives the gutter action for free; a bespoke "Debug" CodeLens is not
needed for v1 (the extension has no test CodeLens today, only the LSP reference lens).

**[DECISION D — sign-off] The integration test runs the Node path, harness-provisioned.**
The CI harness guarantees `bynkc-lsp`; the debug test also needs `bynkc` + `node` — extend
the harness PATH/setup to provide them (they are built in the same `target/release`). The
workerd attach test stays local/opt-in (no `wrangler` in CI), consistent with slice 3.

**[DECISION E] Version — recommend minor v0.72.** A new extension capability; the
extension version tracks the workspace bump (`bump-version.sh`). No language change → minor.

## Risks & mitigations

- **The workerd attach may need custom work** (`Origin` header). Mitigated by spiking it
  first and gating it (DECISION B); the Node path — the common case — is independent and
  low-risk.
- **`vscode-js-debug` attach-config drift.** The exact `pwa-node` attach shape can vary
  by VS Code version. Mitigated by the pinned `@vscode/test-electron` version and the
  integration test.
- **Harness can't provision `wrangler`.** The workerd debug test is local-only; the Node
  test-debug session test is the CI guard, and slice 3's `bynkc`-side decode goldens keep
  the maps honest regardless.
- **Race between "CLI launched the inspector" and "attach".** The CLI prints the URL when
  ready; the provider waits for that line (and `--inspect-brk` holds the process paused
  until the debugger attaches), so there is no lost-startup window.

- **Spec:** none.
- **Docs:** a "Debugging in VS Code" guide under tooling (breakpoint → Debug → step),

## Done when

- A user can set a breakpoint in a `.bynk` test (or code a test exercises), press **Debug**
  in the Test Explorer, and pause at that line — via `bynkc test --inspect` + a `pwa-node`
  attach, with `runtime.ts`/glue skip-stepped.
- `package.json` contributes `debuggers` (`bynk`) + `breakpoints` (`bynk`); the
  `DebugConfigurationProvider` is registered; `bynk` is resolvable for the `dev` path.
- The workerd `bynk dev --inspect` VS Code launch ships **or** is documented as a
  follow-on, per the spike (DECISION B).
- A VS Code integration test starts a bynk test-debug session (harness-provisioned
  `bynkc` + `node`); the workerd test is local/opt-in.
- "Debugging in VS Code" guide + changelog added; extension version bumped per DECISION E.
