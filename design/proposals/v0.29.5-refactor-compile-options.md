# v0.29.5 — Refactor track, increment 5: collapse the `compile_project*` API into `CompileOptions`

- **Status:** Draft (proposal). **Merging is approval to build.** *(Refactor track item 5. Gate on
  `v0.29.3` (pipeline decomposition) is **satisfied** (merged) — the build/analyse split that retires
  `Mode`/`PipelineResult` needs the phases to be functions, which they now are. Lands after `v0.29.4`
  (merged), with which it doesn't interact. **The one track increment that changes the public Rust
  API** — but not observable compiler behaviour, and the only consumers are in-repo, see below.)*

## Context — a combinatorial entry-point smell (refs post-`v0.29.4`)

- Six public variants over **orthogonal axes** (verified signatures, `karnc/src/project.rs`):

  | fn | target | platform | paths | error shape |
  |---|---|---|---|---|
  | `compile_project:139` | default (Bundle) | default | single `root` | `Vec<CompileError>` |
  | `compile_project_with_target:148` | explicit | default | single | `Vec<CompileError>` |
  | `compile_project_with_platform:157` | explicit | explicit | single | `Vec<CompileError>` |
  | `compile_project_with_split_paths:171` | explicit | default | split (`ProjectPaths`) | `Vec<CompileError>` |
  | `compile_project_with_split_paths_full:181` | explicit | default | split | `ProjectFailure` |
  | `compile_project_full:219` | explicit | explicit | split (`src`/`tests` roots) | `ProjectFailure` |

- Four axes — **target × platform × paths × error-shape** — encoded as a hand-picked subset of their
  product. The set grows *multiplicatively*: the `_full` suffix already doubles entries just to switch
  the error shape (attributed `ProjectFailure` vs flattened `Vec<CompileError>`), and a fifth axis
  would double it again.
- **The error-shape axis is the worst offender** — it is not really a *mode*, it is a *projection* of
  one result. `ProjectFailure` already carries `ProjectFailure::flatten` (called at `compile_project`
  `:177` and `compile_project_inner` `:208`/`:214`); the flattened variants exist only to call it for
  the caller.

## Builds on (verified)

- **Callers are fewer than the queue assumed** — and the LSP is *not* among them:
  - `compile_project*` is called **only from `main.rs`** — `compile_project_with_split_paths_full`
    (`main.rs:50`, `:106`) and `compile_project_full` (`:62`, `:112`, `:336`, `:385`); all six call
    sites want the **attributed** (`ProjectFailure`) shape.
  - **`karn-lsp` does not call `compile_project*` or `analyse_project` directly.** It goes through
    `karnc::diagnose_project` (`karn-lsp/src/main.rs:976`), a `lib.rs` wrapper over
    `project::analyse_project` (`lib.rs:283`). So the LSP is **insulated by the wrapper** — this
    increment does not touch a single LSP call site. *(This corrects
    `design/karn-refactor-proposal-queue.md`, which lists `karn-lsp` as a direct caller — verify
    against the build.)*
  - `lib.rs` re-exports the six names (`lib.rs:41-43`) — the public façade to update.
- **The `Mode`/`PipelineResult` companion cleanup is now enabled (`v0.29.3` landed).** The two
  `unreachable!()` guards exist because one `compile_project_pipeline(.., mode)` returns a
  `PipelineResult` sum (`Build | Analysis`) while each caller statically knows its mode and unwraps the
  other arm:
  - `analyse_project` (`:194`) → `Mode::Analyse` (`:200`), `PipelineResult::Build(_) => unreachable!(…)`
    (`:204`)
  - `compile_project_full` (`:219`) → `Mode::Build` (`:230`), `PipelineResult::Analysis(_) =>
    unreachable!(…)` (`:234`)
  `Mode`, `PipelineResult`, and `finish` now live in **`project/diagnostics.rs`** (`:13`/`:96`/`:106`,
  moved there by the `v0.29.2` split); the pipeline reaches them via the parent's flat
  `use diagnostics::*`, and its mode-dispatch branches are at `:2019`/`:2166` (plus `finish`'s own
  branch in `diagnostics.rs`). Now that the phases are functions, the pipeline splits into **two typed
  entry points** — a build path returning `Result<ProjectOutput, ProjectFailure>` and an analyse path
  returning `ProjectAnalysis` — sharing the phase functions. `Mode`, `PipelineResult`, and **both
  `unreachable!()`s** then disappear (the sum type was a poor man's two return types). The split spans
  `project.rs` (entry points + pipeline) and `project/diagnostics.rs` (the retired types).

## Settled

- **▸ Same compiler behaviour; the change is to the Rust API, not the language.** Compilation results
  for every input are byte-identical; the golden fixtures + `v0.29.1` pins are the gate. The track
  contract (no observable behaviour, no language surface) holds — this increment changes *internal Rust
  signatures*, which is within the track's "internal quality" remit.
- **▸ Safe to break the API now.** The crate is **pre-1.0 and its only `compile_project*` consumer is
  in-repo `main.rs`** (the LSP is wrapper-insulated). No deprecation shims — the variants are removed,
  not aliased.
- **▸ Error shape is a projection, not an entry point.** One attributed result; `.flatten()` for the
  `Vec<CompileError>` view at the call site.

## Decisions (open — settle at review)

**[DECISION] Options shape + the single entry point.** *Rec:* a plain
`CompileOptions { target: BuildTarget, platform: Platform, roots: Roots }` with `Default` (Bundle,
default platform, single-root), where `Roots` models the **single-vs-split** axis explicitly
(`Roots::Single(PathBuf)` | `Roots::Split { project_root, paths }`, the latter resolving to
`src`/`tests` roots as `compile_project_with_split_paths_full` does today, `:181`). One public
`compile(opts: &CompileOptions) -> Result<ProjectOutput, ProjectFailure>`; callers wanting the flat
shape use `.map_err(ProjectFailure::flatten)`. *Alternative:* a builder
(`CompileOptions::new(root).target(..).platform(..)`) — more fluent, but the struct-literal form is
plainer and matches the existing `ProjectPaths`/`EmitProjectCtx` style. Settle which at review.

**[DECISION] Bundle the `Mode`/`PipelineResult` removal here, or split it out?** They are the same
neighbourhood (entry-point rationalisation) and the queue ties them, but one is *input axes* and the
other is *mode dispatch*. *Rec: bundle* — both rewrite the same entry band (`:139`–`:240`, the six
variants + `analyse_project`) plus the pipeline's `mode ==` branches (`:2019`/`:2166`) and `finish`'s
branch in `diagnostics.rs`, so doing them apart means editing that band twice. *Caveat:* if the
combined diff is unwieldy at review, land
`CompileOptions` first and the `Mode` split as an immediate follow-up — they are separable, the
`unreachable!()`s are merely the bonus the API work exposes.

**[DECISION] Keep `analyse_project` / `diagnose_project` signatures stable.** *Rec:* the build/analyse
split changes the pipeline's *internals*, but `analyse_project(root, overlay) -> ProjectAnalysis`
(`:194`) and the `diagnose_project` wrapper (`lib.rs:279`, over `analyse_project` at `lib.rs:283`)
keep their signatures — so **`karn-lsp` stays untouched** (it calls `karnc::diagnose_project`,
`karn-lsp/src/main.rs:224`/`:976`).
The split removes `Mode`/`PipelineResult` behind these stable faces, not through them.

## Spec delta

**None.** No language surface, no `docs/src/spec/` chapter — a Rust-API and internal-dispatch change.

## Tests

- **No new golden fixtures.** Output is identical; existing fixtures + `v0.29.1` pins stay green and
  unedited.
- **Caller-shape tests, if cheap.** A trivial test that `CompileOptions::default()` reproduces
  `compile_project`'s old defaults (Bundle, default platform, single-root) and that `.flatten()` on a
  failing build yields the same `Vec<CompileError>` the old flat variants did — pins the
  defaults/projection contract the collapse must preserve.

## Diagnostics / Risks

- **`main.rs`'s six call sites must map exactly.** Each currently selects target/platform/paths
  explicitly; the rewrite to `CompileOptions` must reproduce the same arguments (esp. the split-paths
  rooting at `:50`/`:106` vs the same-root `:336`). A mis-mapped root silently compiles the wrong
  tree — cover the CLI paths with the existing end-to-end CLI fixtures and read the diff carefully.
- **`Default` must equal the old implicit defaults.** `compile_project` defaulted to `Bundle` +
  `Platform::default()` + single-root; `CompileOptions::default()` must match, or the zero-arg
  ergonomics shift silently. The defaults test above guards this.
- **Scope creep via the `Mode` split.** The build/analyse split is genuinely enabled only **after**
  `v0.29.3`; attempting it before the phases are functions reintroduces duplication. If `v0.29.3` has
  not landed, defer the `Mode` half and ship only `CompileOptions` (still a clean win).
- **Low cross-crate exposure.** Confirmed: `karn-lsp` does not call `compile_project*`; the only
  cross-crate consumer band is `lib.rs` re-exports. Under the freeze there is no concurrent edit to
  contend with.

## Definition of done

The six `compile_project*` variants are replaced by `CompileOptions` + a single
`compile(&CompileOptions) -> Result<ProjectOutput, ProjectFailure>` (callers `.flatten()` for the flat
shape); `lib.rs` re-exports and `main.rs`'s six call sites are updated with identical
target/platform/paths selections; `CompileOptions::default()` reproduces the old implicit defaults;
the build/analyse split removes `Mode`, `PipelineResult`, and both `unreachable!()` guards while
keeping `analyse_project`/`diagnose_project` signatures stable so **`karn-lsp` is untouched**; all
golden fixtures and `v0.29.1` pins pass unedited (plus the defaults/projection test); `cargo
build`/`test`/clippy and CI stay green; changelog notes the API collapse. The implementing PR
**deletes this file**.
