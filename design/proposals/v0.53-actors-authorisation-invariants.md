# v0.53 — Actors, slice 5 (authorisation invariants): refinement actors and the 401/403 split

- **Status:** Draft (proposal). **Merging is approval to build.** *(Feature-track slice — the fifth of
  the actors track, `design/tracks/actors.md`. Admits the reserved refinement form `actor Admin = User
  where <predicate>`: an authorisation invariant carved as a refinement of a base actor, with a
  satisfied-scheme-but-failed-invariant request answered **403**, distinct from the 401 authentication
  channel.)*
- **Sequences after v0.52 (multi-actor sum).** It activates the refinement form the grammar has parsed
  (and rejected) since v0.45, and completes the **401/403 split** (Q6) the sum slice deferred — a sum's
  total failure is 401, but a refinement invariant is the 403 path. Provisional version (`v0.53` assumes
  nothing jumps the queue — renumber if sequencing shifts).
- **Verification-bearing slice** → carries the `/security-review` + `/code-review` gate (ADR 0076) and
  the threat model below. It introduces the **authorisation** decision (may this verified party act?),
  a distinct boundary from authentication; getting the fail-closed 403 right is the thing to defend.

## Context — *who you are* vs *whether you may*

The landed slices answer **authentication**: a party proves identity (`None`/Bearer/Signature) and the
boundary mints a sealed identity, fail-closed → 401. They do not answer **authorisation**: a verified
`User` may still lack the right to a route (an admin-only endpoint). Today that check is hand-written in
the body — the classic place an app forgets the guard, or conflates "not logged in" (401) with "logged
in but not allowed" (403).

This slice makes the compiler carry that invariant. `actor Admin = User where hasClaim("admin")`
declares **Admin is a User who additionally satisfies a predicate**; a handler `by a: Admin` emits, at
the boundary: verify the `User` scheme (fail → **401**), then check the predicate against the verified
party (fail → **403**), then run the body. The two failures are structurally distinct response
channels — the runtime can never answer "who are you" when it means "you may not."

## Builds on (verified)

- **The reserved refinement form** — `actor Admin = User where <predicate>` already parses to
  `ActorRefinement { base, predicate: Expr, span }` (v0.45) and is rejected with
  `karn.actor.refinement_unsupported`. This slice flips it to admitted; the `refinement_in_sum` rule
  (v0.52) already bars a refinement from being a sum peer, so the construct slots in cleanly.
- **The Bearer seam (ADR 0085)** — mints the identity from the JWT `sub`. A refinement predicate reads
  *other* verified claims, so this slice surfaces the verified claims object from the seam to the
  predicate evaluation (the identity itself stays `sub`-minted and sealed).
- **`HttpResult.Forbidden` (403)** — already in the HTTP result sum; the 403 channel is the existing
  `Unauthorized` (401) pattern with a different verdict.
- **Refinement elimination** ("an `Admin` is-a `User`") — the track's Q3 design: a refinement carves a
  subset, so an `Admin` binding is usable anywhere its base `User` is (`a.identity` is the `User`
  identity). This reuses the subtyping intuition the type-refinement system already encodes.

## Settled (from the track / foundational ADRs)

- **▸ Authorisation invariants desugar as refinement** (Q3): `actor Admin = User where p` lowers to
  `{ u: User | p(u) }`, reusing the closed-set + boundary-predicate model — a refinement *carves a
  subset*, it does not add a member, so exhaustiveness and the closed scheme set (0080) stay intact.
  **▸ The 401/403 split** (Q6): authentication-scheme failure → 401; a satisfied scheme but a failed
  refinement invariant → 403; for non-HTTP protocols the verdict maps to that protocol's outcome.
  **▸ Refinements are never sum members** (Q4, enforced since v0.52) — a refinement is a handler's sole
  contract (`by a: Admin`) or a guard inside a resolved arm, never a point on the sum axis.
  **▸ Reject composition/intersection; reserve nominal extension** as the escape hatch only if a
  sub-actor must *add* structure (refinement can restrict, not add).

## Decisions (open — settle at review)

**[DECISION] Predicate vocabulary: a small closed set over verified claims now; general expressions
later.** The refinement predicate is a free-form `Expr` in the AST, but claims are untyped JSON, so an
arbitrary expression has no typed surface to bind against. *Rec:* admit a **closed set of claim
predicates** — `hasClaim("name")` (the claim is present and truthy) and `claimEquals("name", "value")`
(string-equals) — composable with `&&`/`||`/`!`. These plug into the checker as new predicate builtins
(the way `Matches`/`InRange` are closed predicate forms for refined base types). *Alternatives:* (a) a
general typed claims record the predicate reads as `u.claims.admin == true` — needs a claims-typing
story (a `claims:` schema on the actor) that is its own slice; (b) arbitrary boolean expressions over
the identity — but the identity is `sub` only, so there is nothing to test. The closed claim-predicate
set covers the common RBAC shape and is the substrate a typed-claims layer sits on.

**[DECISION] Claims surfacing: the seam binds the verified claims; the predicate reads them, the body
does not.** *Rec:* the Bearer seam already verifies the JWT and has the full claims object in hand;
extend it to evaluate the refinement predicate against those claims **at the boundary**, before the
body. The body still sees only the sealed identity (`a.identity` = the `sub`-minted `User` identity) —
claims are an authorisation-time input, not a body-time value, so they are not threaded into `deps`.
*Alternative:* expose claims to the body — rejected: widens the sealed-identity surface (ADR 0081) and
invites authorisation logic to leak back into the body.

**[DECISION] Scheme applicability: refinements require a claim-bearing scheme (Bearer) for now.** *Rec:*
a refinement predicate tests claims, which only a `Bearer` actor carries; a refinement of a `None` or
`Signature` actor has nothing to test, so restrict refinements to `Bearer` base actors this slice
(`karn.actor.refinement_base_unsupported` otherwise). *Alternative:* allow a refinement over any
actor with a trivially-true/echo predicate — rejected: meaningless and a footgun. Internal-channel
authorisation (a `Caller`-claim equivalent) is a later, separate question.

**[DECISION] Failure response: 403 fail-closed, distinct from 401.** *Rec:* a verified party that fails
the invariant → `HttpResult.Forbidden` (403); an unverified party → `HttpResult.Unauthorized` (401), as
today. The body runs only when both hold. *Alternative:* 401 for both — rejected: it is the exact
authentication/authorisation conflation the slice exists to remove, and Q6 settles the split.

**[DECISION] `is-a` usability: an `Admin` is usable where a `User` is.** *Rec:* refinement elimination —
a handler `by a: Admin` binds `a` with the base `User` identity type; the invariant is discharged at
the boundary, so downstream `a` is just a verified `User`. *Alternative:* a distinct `Admin` identity
type — deferred (would need nominal extension, the reserved escape hatch).

## Spec delta

- **`syntactic-grammar.md`.** No grammar change — the refinement form already parses; document that the
  predicate is now a closed set of claim predicates (`hasClaim`, `claimEquals`).
- **`static-semantics.md`.** Refinement actors are admitted: the base MUST be a declared `Bearer` actor
  (`karn.actor.refinement_base_unsupported`); the predicate MUST be a conjunction/disjunction of the
  admitted claim predicates (`karn.actor.refinement_predicate_unsupported`); a refinement actor is a
  handler's sole `by` contract (never a sum member — already `refinement_in_sum`). The
  `refinement_unsupported` blanket rejection is retired.
- **`emission.md`.** The refinement seam: verify the base scheme (401), evaluate the claim predicate
  against the verified claims (403 fail-closed), then mint the base identity and run the body.
- **`runtime-library.md`.** A claims-predicate helper (or inline emission) over the verified JWT claims;
  no new crypto — it reads the already-verified payload.
- **`diagnostics.md`.** Retire `karn.actor.refinement_unsupported`; add
  `karn.actor.refinement_base_unsupported` and `karn.actor.refinement_predicate_unsupported`.

## Tests

- **Positive golden:** a `from http` route `on … by a: Admin (…)` where `actor Admin = User where
  hasClaim("admin")`; the seam verifies the JWT (401), checks the claim (403), then runs the body;
  `tsc --strict` green.
- **Behavioral (the standing guard, extending the auth tests):** a Node driver signs JWTs and asserts
  the verdict trichotomy — no/invalid token → 401; valid token **without** the claim → 403; valid token
  **with** the claim → body runs; the claim predicate cannot be bypassed.
- **Negative:** a refinement over a non-Bearer base (`None`/`Signature`); an unsupported predicate
  (a non-claim expression); a refinement actor used as a sum member (already `refinement_in_sum`); the
  `is-a` usability (an `Admin` binding readable as its `User` identity).

## Diagnostics / Risks

- **403-vs-401 fail-closed ordering** — the predicate is checked **after** scheme verification and
  **before** the body; a predicate-evaluation error must fail closed to 403, never fall through.
  *Mitigation:* golden + the behavioral trichotomy test.
- **Claims are untrusted-until-verified** — the predicate must read claims from the **verified** payload
  only (post-signature-check), never from an unverified token. *Mitigation:* the seam evaluates the
  predicate only inside the verified branch; the security review covers it.
- **Predicate surface creep** — keeping the predicate a closed claim-set (not arbitrary expressions)
  is what keeps the 403 decision total and reviewable. *Mitigation:* documented scope; the typed-claims
  generalisation is a later slice.

## Security / threat model

*Gates the slice (ADR 0076).*

- **Guarantees.** The body runs only for a party who (a) authenticated (scheme verified, else 401) and
  (b) satisfies the authorisation invariant (claim predicate over the **verified** claims, else 403) —
  fail-closed and total, before the body. Authorisation is a distinct, structurally-separate channel
  from authentication.
- **Claims are read post-verification only** — the predicate never sees an unverified token; the
  identity stays the sealed `sub`-minted value, and claims do not leak into the body.
- **Explicitly out:** typed claims schemas / general predicate expressions, non-Bearer (Internal/
  Signature) authorisation, nominal actor extension (adding structure), and role hierarchies beyond
  conjunction/disjunction of claim predicates.

## New decision records this increment adds

- **ADR — authorisation invariants are refinement actors (`actor Admin = User where <claim predicate>`)
  over a Bearer base; the predicate is a closed claim-predicate set checked against the verified claims
  at the boundary; a failed invariant is 403, distinct from the 401 authentication channel; an `Admin`
  is-a `User`.** Records the closed-predicate-set decision (general typed claims deferred) and the
  401/403 split realisation (closing Q6).

## Out of scope (→ later slices / the track)

- **Typed claims schemas + general predicate expressions**, **non-Bearer authorisation** (Internal/
  Signature), **nominal actor extension**, **role hierarchies / RBAC sugar**, **cross-context
  `Internal` actors** (Q7), and **replay/ordering hints** (Q8).
