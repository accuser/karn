# v0.63 ‚Äî Extract `bynk-emit`: build orchestration + TS emission (crate-decomposition slice 4)

- **Track:** [crate-decomposition](../tracks/crate-decomposition.md), slice 4.
  After this, `bynkc`'s lib is just the CLI surface + the thin compile/diagnose
  glue over the library set.
- **Realises ADRs:** [0099](../decisions/0099-crate-layering-dependency-direction.md)
  (downward layering, published lockstep), [0102](../decisions/0102-foundation-types-boundary.md)
  (the `line_col` source utility belongs in the leaf).
- **Depends on:** slices 1‚Äì3 (`bynk-syntax`, `bynk-fmt`, `bynk-check` ‚Äî shipped).
- **Posture:** a **pure internal refactor** ‚Äî behaviour-preserving, no language
  surface, no emitter/project behaviour change. The whole workspace suite
  (golden emission, `tsc_verify`, e2e, project fixtures) must pass unchanged.

## What this slice does

Create a new published crate **`bynk-emit`** over `bynk-check`, and move the
emission + project-orchestration layer **down** into it. `bynkc` then depends on
`bynk-emit` and re-exports its modules so the public API (`compile_project`,
`ProjectOutput`, ‚Ä¶) and the binary are unchanged.

**Modules that move:** `emitter` (+ `emitter/`: `emit`, `lower`, `serialisation`,
`workers`, `workers_entry`, `wrangler`, and the `runtime.ts` asset) and `project`
(+ `project/`: `consistency`, `diagnostics`, `discovery`, `graph`, `paths`,
`symbols`, `tests_emit`, `validate`). ~5.7k lines.

**`bynk-emit` is the project driver, not just a code generator** (track ¬ßC):
`project.rs` (3.5k lines, the largest file in the tree) conducts discovery, the
dependency graph, consistency, validation, symbols, and paths; `compile_project`
lives here. Read the crate as "build orchestration + TS emission" ‚Äî orchestration
drives emission.

**Dependencies:** `bynk-emit` needs **`bynk-syntax` + `bynk-check` only** ‚Äî
verified the emit set uses no external crates directly (the TS is built as
strings, not serialised). A clean layer with no new third-party surface.

## What stays in `bynkc`

`cli`, **`test_json`**, `main`, and the lib glue (`compile`, `diagnose`,
`diagnose_project`, the render helpers). These become **downward consumers** of
`bynk-emit`.

## Decisions

**[DECISION 1] Move `line_col` down to `bynk-syntax` ‚Äî recommend yes.** The emit
set's *only* edge into `bynkc`'s lib glue is `crate::line_col` (a pure
`(source, byte offset) ‚Üí (line, col)` utility), used by `emitter/lower.rs` **and**
`bynkc`'s rendering. It is a span/source utility, so per ADR 0102 it belongs in
the `bynk-syntax` leaf (`pub fn line_col`); both `bynk-emit` and `bynkc`'s
rendering then call `bynk_syntax::line_col`. This also pre-positions slice 6 ‚Äî
`bynk-render` needs `line_col` for its `short` form and may depend on
`bynk-syntax` **only**. *Alternative:* duplicate it or keep an upward shim ‚Äî
rejected (an upward edge or drift). This is the one prerequisite cleanup.

**[DECISION 2] `test_json` stays in `bynkc` ‚Äî recommend yes.** The track's
slice-4 list is "emitter, project, **tests_emit**"; `tests_emit` is
`project/tests_emit.rs` (moves with `project`). `test_json.rs` is a *different*
module ‚Äî the `bynkc test --format json` document (ADR 0098), CLI-output
structuring with **zero `crate::` edges**, consumed by `main.rs`/lib glue. It is a
CLI concern, not emission, and stays with `cli` in `bynkc`. *(Naming proximity is
the only reason to think otherwise.)*

**[DECISION 3] Version ‚Äî recommend a minor `v0.63`.** New published crate
(`bynk-emit`) ‚Üí a new release surface needing the **bootstrap-seed before trusted
publishing**, added to both publish loops after `bynk-check`, before `bynkc`.
Per-increment minor cadence (v0.62 ‚Üí v0.63).

## Risks & mitigations

- **Largest single file (`project.rs`, 3.5k lines) + many downward consumers.**
  Mitigated by the verified edge map: the move is re-pointing imports
  (`crate::{ast,‚Ä¶}` ‚Üí `bynk_syntax::`, `crate::{checker,resolver,‚Ä¶}` ‚Üí
  `bynk_check::`), plus the one `line_col` relocation. `bynkc`'s re-export keeps
  consumers stable.
- **Boundary surfacings** like slice 3 (private items now crossing a crate edge)
  are expected; they are mechanical `pub(crate)`‚Üí`pub` promotions, surfaced by
  the compiler. (Slice 3 had three; slice 4 may have a few more given
  `project.rs`'s size.)
- **Validation gate:** golden emission, `tsc_verify` (the embedded runtime
  type-checks), `e2e`, and all project-form fixtures must pass byte-identical.
