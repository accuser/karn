# v0.100 — `Stream[T]`, the value-over-time primitive (real-time track, slice 0)

- **Scope:** a single increment — a new built-in parametric type, its minimal
  combinator vocabulary, its classification in the existing non-serialisable
  set, and its lowering. **Language + compiler + emitter + runtime.** No
  protocol, no socket, no Durable Object, no streaming-HTTP terminal (that is
  slice 1).
- **Realises:** [`design/tracks/websocket.md`](../tracks/websocket.md) §3.2 and
  **Slice 0** of its §6 decomposition — "the load-bearing primitive; everything
  downstream cites its ADR". Settles the track's **Q1** (form, and how much to
  commit now), **Q3** (error/termination), and **Q4** (vocabulary breadth).
- **Authorises its own ADR:** the `Stream[T]` model (value-vs-effect,
  classification, the parameter-now / element-semantics-later split). **Number
  assigned when the implementing PR is cut** — the next free number at that
  moment, per the track's no-reservation rule (0126/0127 were taken by unrelated
  work; in-browser's 0126–0130 claim is stale, so nothing forward of 0127 is
  safely ownable until its PR lands).

## 1. What this is, and what it is not

`Stream[T]` is a **lazy, pull-shaped sequence of values produced over time** —
the word the language lacks. `Effect[T]` resolves exactly once; `Query[T]` is a
snapshot read over storage; neither expresses a token feed, a progress stream,
or an incremental response. This slice adds the type and just enough vocabulary
to construct, transform, bound, and observe one — **in isolation**, against a
deterministic in-memory source, before any consumer exists.

It is deliberately *not* useful end-to-end yet. The first real consumer is
slice 1's streaming-HTTP body; the eventual one is a streaming `Ai` capability
(an explicit track non-goal). Slice 0 exists so those land against a settled
primitive rather than co-inventing one. The honest framing: **this slice ships a
type and a conformance test, not a user-facing feature** — see §7's docs delta.

## 2. The model — `Stream[T]` is shaped exactly like `Query[T]`

`Query[T]` is the precedent to copy, almost line for line. Its checker
representation already documents the target classification
(`bynk-check/src/checker.rs:70`):

> `Query[T]` … Non-storable, non-boundary, and not value-comparable — like
> `Effect`/`Fn` (ADRs 0031/0030).

`Stream[T]` joins it in that family. Concretely:

- **Surface type.** A `TypeRef::Stream(Box<TypeRef>, Span)` variant alongside
  `Effect`/`Query`/`List` in `bynk-syntax/src/ast.rs:1416–1454`.
- **Resolved type.** A `Ty::Stream(Box<Ty>)` variant alongside `Ty::Query` in
  `bynk-check/src/checker.rs:49–104`, with the same display (`Stream[{T}]`),
  resolution, and `substitute`/`compatible` handling the other single-parameter
  builtins get.
- **Classification (the load-bearing call).** `Stream[T]` is **non-serialisable,
  non-storable, non-boundary, not value-comparable** — identical to `Query`/
  `Effect`/`Fn`. This is *not* new machinery: it is one new arm in each existing
  rejection site (§3). A live IO source is not a value you can persist or ship.

## 3. Classification reuses existing passes (no new concept)

Every enforcement point is an existing exhaustive `match` that gains one
`Ty::Stream(_)` arm. The track's claim — "extends an existing pass rather than
introducing a concept" — is literally true here:

- **JSON / boundary codec.** `json_codable` (`bynk-check/src/checker/kernels.rs:1400`)
  already returns `false` for `Ty::Effect | Ty::Query | Ty::Fn | …`. Add
  `Ty::Stream(_)` to that arm. This alone makes a `Stream` field, an exported
  `Stream`, a cross-context `Stream` argument/result, and a `Stream` in a
  serialised-closure capture all fail with the established diagnostics
  (the `bynk.types.*_at_boundary` family, cf. fixtures
  `negative/108_query_at_boundary`, `negative/165_list_fn_at_boundary`).
- **Storage fields.** The `store`-field type admission (`checker.rs:322` ff.,
  the `Cell`/`Map`/`Set`/`Cache`/`Log` field-type validation) rejects `Stream`
  exactly as it rejects `Query`/`Effect` today — same path, one more arm.
- **Value comparison / equality.** Wherever `Query`/`Fn` are excluded from
  `==`, `Stream` is too.

**[DECISION A] Classify `Stream[T]` in the existing non-serialisable family, by
adding one arm per site — no new pass, no new diagnostic code.** Recommended.
The alternative (a bespoke `stream_at_boundary` code) buys nothing the
boundary-family messages don't already say and fragments the surface.

## 4. Vocabulary (Q4 — minimal at v1)

Construction, two transformers, and the one terminal needed to *observe* a
stream so the slice is testable standalone. Typed by the same kernel-method
mechanism as `Query` (`is_query_op` / `check_query_kernel_method`,
`bynk-check/src/checker/kernels.rs:611–688`) — a sibling `is_stream_op` /
`check_stream_kernel_method`, or a shared table keyed by receiver.

| Form | Signature | Notes |
|---|---|---|
| `bynk.stream.of(xs: List[T])` | `-> Stream[T]` | The **deterministic in-memory source**. The only constructor this slice ships; live runtime sources arrive with their consumers (slice 1+). Makes the type exercisable with no IO. |
| `.map(f: T -> U)` | `Stream[T] -> Stream[U]` | Builder; lazy, like `Query.map`. |
| `.take(n: Int)` | `Stream[T] -> Stream[T]` | Builder; **bounds** the stream — the language-level backpressure tool the track's §4 names. |
| `.collect()` | `Stream[T] -> Effect[List[T]]` | **Terminal** — drains to a list, the observation point a test asserts on. Mirrors `Query`'s builder→`Effect` terminal split. |

**[DECISION B] Ship `collect` as the one terminal, beyond the track's stated
"construction, `map`, `take`".** The track lists only the builders; but a
builder-only slice is untestable in isolation — you cannot assert on a stream
you cannot drain. `collect` is the minimal observation terminal and the natural
test seam. Recommended; richer consumption (the SSE body, `forEach`, fan-in) is
slice 1+ and earns its own ADR, as the query algebra did. *Stated explicitly
rather than smuggled in.*

**[DECISION C] No `filter`/`scan`/`merge` at v1 (Q4).** A fuller algebra earns
its own slice + ADR. Recommended — same discipline `query-algebra` held.

## 5. Errors and termination (Q3 — in-band `Result`)

**[DECISION D] In-band `Result` elements, not a side channel.** A stream that
can yield failures is `Stream[Result[T, E]]` — a *use* of the type parameter,
needing zero new machinery. An `Err` element is an **outcome** (matched like any
`Result`); a **fault** in the producer aborts the stream as faults abort
handlers. Recommended; it matches the language's existing outcome-vs-fault split
and keeps the type orthogonal to error handling.

## 6. Q1 — keep the parameter, defer the element semantics

The hard-to-reverse surface is the **parametric type**. Retrofitting a type
parameter later is the disruptive move, so commit it now: `Ty::Stream(Box<Ty>)`,
not a monomorphic `StreamChunk`. But **defer the element semantics** that only a
non-chunk, boundary-crossing consumer pins — specifically *whether a stream
element may itself cross a context boundary, and whether such an element must be
serialisable*. Slice 0 never crosses a boundary with a stream element (it
constructs in-memory and drains locally), and `Stream` as a whole is non-boundary
anyway (§3), so that question is **not forced** here.

**[DECISION E] Slice-0 ADR records the type + parameter + classification; the
element-transport/serialisability rule lands with the first consumer that needs
`T ≠ Chunk`.** Recommended. This shrinks the genuinely irreversible commitment to
what is exercised, while avoiding a later parameter retrofit. The value-vs-effect
and non-serialisable calls that *are* made now sit behind the conformance test
(§7).

## 7. Lowering & runtime

- **Production target.** `Stream[T]` lowers to a host **async iterable /
  `ReadableStream`**. `bynk.stream.of(xs)` → an async iterable yielding `xs`;
  `.map(f)` → a lazy mapping wrapper; `.take(n)` → a bounded wrapper; `.collect()`
  → an `async` drain to an array (an `Effect`, i.e. a `Promise<T[]>`, consistent
  with how `Effect`/`Query` terminals already lower — `bynk-emit/src/emitter/runtime.ts`).
- **Test target.** A deterministic in-memory sequence, so a handler's streamed
  output is assertable. Because `of` is in-memory and `collect` drains
  deterministically, slice-0 fixtures are fully reproducible with no IO.
- Runtime helpers (`stream_of` / `stream_map` / `stream_take` / `stream_collect`,
  or a small `Stream` namespace) added to `runtime.ts` beside the existing
  `Result`/`Option` helpers. Strip-only clean (no type-directed constructs), so
  the future `in-browser` JS path is unaffected.

## 9. Docs delta (and why the learning path waits)

- **Now (this slice):** a **reference stub** for `Stream[T]` (the type, the four
  forms, the non-serialisable/non-storable/non-boundary classification) under
  `docs/src/reference/`; the `docs/src/reference/changelog.md` row; currency
  banner + `spec/appendix-version-history.md` advanced.
- **Deferred to slice 1 — stated explicitly per the proposals README, not
  silently omitted:** the **"Understand"/"Do" guide** (the streaming on-ramp).
  *Why:* slice 0 ships no end-to-end story — there is nothing a reader can *do*
  with a stream that only constructs in memory and drains to a list. The guide
  lands with the streaming-HTTP response (slice 1), where "type Bynk, get an
  incremental response" is a real, runnable recipe. Writing a learning page for
  an unobservable-in-production primitive now would be exactly the "aspirational
  prose in the present tense" the README warns against.
- **Roadmap:** mark real-time slice 0 shipped; slice 1 (streaming HTTP) next.

## 10. Tests

- **Positive fixtures** (`bynkc/tests/fixtures/positive/`): construct via `of`,
  `map`, `take`, `collect`; assert the emitted TS and the drained result. A
  `Stream[Result[T, E]]` fixture exercising in-band errors (D-D).
- **Negative fixtures** (`bynkc/tests/fixtures/negative/`, mirroring
  `108_query_at_boundary` / `165_list_fn_at_boundary`): a `Stream` in a `store`
  field; a `Stream` as an exported/handler boundary type; a `Stream` as a
  cross-context call argument; a `Stream` captured by a serialised closure. Each
  asserts the established boundary-family diagnostic — proving §3's "one arm per
  existing site" claim rather than asserting it.
- **The conformance test (E):** a single fixture pinning the load-bearing,
  irreversible calls — value-not-effect, non-serialisable, parameter present — so
  any later drift is a failing build.

## 11. Risks & mitigations

- **Permanent type-system commitment (the track's headline risk).** *Mitigation:*
  D-E splits it by reversibility — parameter now, element semantics with the
  first non-chunk consumer; the small surface that *is* fixed sits behind the
  conformance test; vocabulary stays minimal (D-B/D-C).
- **`Stream` and `Query` look like duplicate machinery.** They are deliberately
  parallel (same classification, same builder/terminal shape) but distinct types
  — `Query` is a snapshot storage read, `Stream` an unbounded temporal source.
  *Mitigation:* the doc-comment on `Ty::Stream` states the distinction at the
  definition site, as `Ty::Query`'s does; a shared kernel-method table (if taken)
  is keyed by receiver so the surfaces can diverge later.
- **A terminal beyond the track's stated vocabulary.** *Mitigation:* D-B states
  it openly and bounds it to the one terminal observability demands.
