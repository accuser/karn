# v0.29 — Surface the A-2 LSP features in the editor (B-1)

- **Status:** Draft (proposal). **Merging is approval to build.** *(A small, **extension-only**
  increment — no `karnc`/server/spec change. Closes the concrete gap v0.28 left: the custom
  semantic token types render with no colour until the client themes them.)*
- **Builds on (verified):**
  - **Most LSP features flow automatically** through the `LanguageClient` — code-action lightbulbs,
    rename, inlay hints, and semantic tokens need *no* client code once the server advertises them.
  - **The one non-automatic piece is theming.** The server's **frozen legend**
    (`semantic_tokens_legend()`, `index_queries.rs:229-241`) declares custom token types
    `capability` / `service` / `agent` / `provider` (alongside standard `type` / `function`) and
    custom modifiers `refined` / `opaque` / `platformNative` (alongside `declaration`). Custom token
    types **do not colour** under default themes unless the extension declares them and maps fallback
    scopes — and `vscode-karn`'s `contributes` has **neither `semanticTokenTypes` nor
    `semanticTokenScopes`** today (only `languages`, `grammars`, `commands`, `configuration`). So
    v0.28's Karn-distinctive tokens are currently invisible.
  - **Inlay hints are server-always-on** (`inlay_hint_provider: Some(OneOf::Left(true))`,
    `main.rs:1089`); the handler doesn't gate on config — so a Karn-specific toggle is a **client**
    concern. The client already syncs `karn.*` config (`configurationSection: "karn"`,
    `extension.ts:90`).
  - Single repo version since #51, so an extension-only change bumps the repo version (v0.29); the
    **language spec version is unchanged**.
- **Spec sections touched:** `design/karn-lsp-spec.md` (a note that the client mirrors the server's
  frozen legend for theming); `vscode-karn/package.json` (`contributes`); `src/extension.ts`
  (an inlay-hint middleware).
- **Scope:** declare + scope-map the semantic-token legend so v0.28's tokens colour out-of-box, plus
  a client-side `karn.inlayHints.enable` toggle. **Not** here: snippets, scaffolding commands,
  getting-started walkthrough (**B-2**), marketplace/Open VSX publishing (**CI Tier 4**).

## Settled (recorded as ADRs ≥0058)

- **▸ The extension mirrors the server's frozen semantic-token legend.** The custom token-type and
  modifier *names* the extension declares and themes are a **cross-component contract** with
  `semantic_tokens_legend()` — they must match exactly, or those tokens silently go unthemed. The
  contract is enforced by a **`karn-lsp` cross-*file* test** that parses `package.json` and checks it
  against the legend (below) — **not** two independently-pinned lists, which enforce nothing.

## Decisions (open — settle at review)

**[DECISION] Declare the custom token types/modifiers and map fallback scopes.** *Rec:* add to
`contributes`:
- **`semanticTokenTypes`**: `capability`, `service`, `agent`, `provider` (with descriptions — for
  theme editors and the *Inspect Editor Tokens* tooling).
- **`semanticTokenModifiers`**: `refined`, `opaque`, `platformNative`.
- **`semanticTokenScopes`**: map each to a standard TextMate scope so **any** theme colours them —
  e.g. `capability → entity.name.type.interface`, `service → entity.name.type`,
  `agent → entity.name.type.class`, `provider → entity.name.function` (standard `type`/`function`
  already theme). Give the Karn-distinctive **modifiers** a visible treatment via **explicit
  per-type keys** — `capability.platformNative`, `type.refined`, etc. *(The `*.modifier` wildcard is
  reliable in users' `semanticTokenColors` but **verify VS Code accepts it in
  `contributes.semanticTokenScopes`** before relying on it; explicit keys are the safe form.)* The
  exact scopes are a best-effort aesthetic call, refined at review — and `vsce package` (CI
  `extension` job) validates the `contributes` schema, so a malformed map fails CI for free.

**[DECISION] `karn.inlayHints.enable` via client middleware (no server change).** *Rec:* a
`provideInlayHints` middleware on the `LanguageClient` returns `[]` when the setting is off — a clean
client-side gate, complementary to VS Code's built-in `editor.inlayHints.enabled`. **Semantic
tokens** rely on the built-in `editor.semanticHighlighting.enabled` (no custom toggle — avoid
over-building). Document both built-ins in the README. *Trade-off (recorded):* the middleware takes
effect on the **next** inlay-hint request (edit/scroll), not the instant the setting flips —
`vscode-languageclient` 9 exposes no public inlay-hints refresh. So the built-in is the **instant**
toggle; `karn.inlayHints.enable` is positioned as the **persistent per-language** preference (one
README sentence makes this explicit).

**[DECISION] Drift guard — a Rust cross-*file* test, not a JS copy.** A JS test asserting
`package.json` equals a JS-side expected list **enforces nothing**: it's a *third* copy of the names,
so a server legend change moves both Rust copies together while the extension silently un-themes —
exactly the drift the guard exists to catch. *Rec:* a **`karn-lsp` test** (it already depends on
`serde_json`) parses **`vscode-karn/package.json`** (a `CARGO_MANIFEST_DIR`-relative
`../vscode-karn/package.json`) and asserts its `contributes.semanticTokenTypes` /
`semanticTokenModifiers` **custom** entries exactly equal `semantic_tokens_legend()`'s custom
entries. One source of truth (the Rust legend), one cross-file assertion, runs in the **existing
`test` CI job** — a legend change without the matching `package.json` edit fails CI.
**Required (now that `karn-lsp` is live on crates.io):** add `exclude = ["tests/legend_drift.rs"]`
to `karn-lsp/Cargo.toml` so this test — which reads `../vscode-karn/package.json`, absent from the
crate tarball — **ships out of the published package**. `exclude` only trims the tarball; in‑repo
`cargo test --workspace` still runs it (drift still caught), but a downstream **standalone
`cargo test`** on the published crate can't fail on the missing sibling. *(Publishing itself was
never at risk — `cargo publish`'s verify is a build, which doesn't compile integration tests; the
concern is purely a downstream `cargo test`. The other `karn-lsp` tests are self‑contained —
`rename_validation.rs` uses `env::temp_dir()` — so only this file needs excluding.)*

## Tests

- **`karn-lsp` (existing `test` CI job):** the **drift-guard** cross-file test — parses
  `vscode-karn/package.json`, asserts its custom token types/modifiers equal `semantic_tokens_legend()`'s.
- **Extension (`extension` CI job):** `vsce package` validates the new `contributes` schema
  (malformed → CI fail) and the smoke/bundle stay green. **No new JS test harness:** `vscode-karn`
  has no test framework or test script today, and standing one up for a three-line `enabled ?
  next(…) : []` middleware isn't worth it — the gate is covered by `tsc --noEmit` + manual
  verification. *(If a JS harness is wanted later, the gate extracts cleanly as a pure function under
  `node --test`.)*
- **Not auto-testable:** the actual colours (theme-dependent) — verified manually.

## Diagnostics / Risks
- **Legend drift** — the dominant risk; enforced by the **`karn-lsp` cross-file test** (parses
  `package.json` against the legend), *not* a JS-side copy, which wouldn't catch it.
- **Scope-mapping aesthetics** — fallback scopes are best-effort and theme-dependent; the goal is
  "coloured and distinguishable," not pixel-perfect across all themes.
- **Toggle redundancy** — `karn.inlayHints.enable` overlaps the built-in; kept because per-language
  Karn control is convenient, and it's a few lines of middleware.
