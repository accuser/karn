# v0.38 — B-2 extension polish

- **Status:** Draft (proposal). **Merging is approval to build.** *(Tooling-queue item 13. Surface the
  mature LSP server to actual users via the VS Code extension: scaffolding **snippets**, **new-project /
  new-context commands**, a **`karnc` problem-matcher + build task** (compile errors → Problems panel),
  and a **getting-started walkthrough**. Mostly extension-only; the one cross-cutting piece is a terse
  `karnc check` output format the problem-matcher keys off.)*

## Context

The server is feature-rich (completion, signature help, codelens, call hierarchy, implementation nav,
member indexing, folding/selection), but the extension still contributes only languages, grammar,
semantic tokens, four server-lifecycle commands, and three settings — nothing that lowers the
first-five-minutes friction. This increment adds the authoring affordances: scaffolds, scaffolding
commands, a whole-project compile task wired to the Problems panel, and an onboarding walkthrough.

## Builds on (verified — scout)

- **`contributes` has the slots free.** `vscode-karn/package.json` (v0.37.0) declares `languages`
  (id `karn`), `grammars`, `semanticTokenTypes/Modifiers/Scopes`, four `commands`
  (`karn.restartServer`/`downloadServer`/`showServerOutput`/`openProjectConfig`), and three settings —
  **no** `snippets`, `taskDefinitions`, `problemMatchers`, or `walkthroughs` entries yet.
- **The command pattern exists.** `extension.ts` `activate()` registers commands via
  `vscode.commands.registerCommand("karn.…", handler)` and pushes to `context.subscriptions` — new
  scaffolding commands slot in beside the four lifecycle ones.
- **Build/package/CI are stable.** esbuild bundle → `out/extension.js`; `npm run package` (`@vscode/vsce
  package --no-dependencies --skip-license`); CI type-checks (`npm run check`), builds, runs the
  bundle-require guard, and packages. Snippets/walkthrough JSON+media are validated by `vsce package`
  against the contributes schema. So the increment is **CI-checkable without a marketplace token**.
- **Canonical scaffolds exist as fixtures.** Idiomatic minimal shapes to base snippets on: a `context`
  with `uses`/`exports` (`fixtures/positive/71_context_uses_commons`), a `capability` + `provides` +
  `service` (`147_cron_with_given`), an `agent` with `key`/`state`/handler
  (`116_auto_lift_in_agent`), and a `commons` + record type.
- **`karnc check` prints rich ariadne diagnostics** (`main.rs` `run_check` → `print_errors` /
  `print_project_failure`) — box-drawing, multi-line (`[karn.cat] Error: msg` header + a separate
  `╭─[ file:line:col ]` location line). There is **no terse format flag** today; `Command::Check`
  (`cli.rs`) takes only `input`.

## Decisions (open — settle at review)

**[DECISION] Problem-matcher input format.** *Rec:* **add a terse `karnc check --format short`** that
emits one line per diagnostic — `path:line:col: error[category]: message` — and match *that*. Rationale:
the ariadne output is pretty-printed across multiple lines with box-drawing glyphs; a problem-matcher
over it is brittle. A one-line format is a small, well-scoped `karnc` lib+CLI addition (the diagnostics
already carry span + category + message; this is a second renderer), and it's independently useful for
CI and scripts. The richer default stays the default. *(Alt: a multi-line ariadne regex — rejected as
fragile; or defer the problem-matcher — but the queue lists it and the terse flag is cheap.)*

**[DECISION] Slicing.** *Rec:* **two slices.** **Slice 1 (extension-only):** snippets + scaffolding
commands + walkthrough — pure `vscode-karn`, F5/CI-validatable, no compiler change. **Slice 2:** the
`karnc check --format short` renderer + the `contributes.problemMatchers` + a `taskDefinitions`/default
build task wiring whole-project compile into the Problems panel. This isolates the one `karnc` change
into its own slice; the final slice deletes the proposal.

**[DECISION] Scaffolding-command UX.** *Rec:* **new context** — prompt for the dotted context name, write
`<name-last-segment>.karn` (or into `src/`) with a `context <name>` skeleton, open it. **new project** —
prompt for a project dir, scaffold `karn.toml` + `src/` + a starter context. Use `vscode.window`
input/quick-pick + `workspace.fs` writes (no external deps). Refuse to overwrite an existing file.

**[DECISION] Snippet set.** *Rec:* `context`, `commons`, `capability` (+ `provides`), `service` handler
(`on call` / `on http` / `on cron`), `agent` (key/state/handler), `type` record, `type` sum — prefixes
that don't collide with keywords, tab-stops through the names. Bodies mirror the fixtures above so they
type-check as written.

**[DECISION] Walkthrough scope.** *Rec:* a 3–4 step getting-started — install/locate the server, create a
project (links the new-project command), write a first context (snippet), run the build task (Problems
panel). Markdown + a couple of small media assets under `walkthroughs/`; keep media light.

**[DECISION] New decision record.** *Rec:* **ADR 0071** — the extension-polish surface; the terse
`karnc check --format short` as the problem-matcher contract; the two-slice split.

## Spec / docs delta (sketch)

- **`docs/src/tooling/vscode-karn.md`** — document the new snippets, commands, build task + problem
  matcher, and walkthrough. **`docs/src/tooling/karn-lsp.md`** or the CLI reference — document
  `karnc check --format short`.
- No `design/karn-lsp-spec.md` change (this is extension/CLI, not the LSP protocol surface).

## Tests

- **Slice 2 `karnc`**: a unit/CLI test that `check --format short` on a fixture with a known error emits
  the `path:line:col: error[category]: message` line (stable, matched by the problem-matcher regex).
- **Extension**: no test-electron harness exists; validation is the **CI gate** (`npm run check` + build +
  bundle-guard + `vsce package` succeed with the new contributes) plus a manual F5 smoke (snippets fire,
  commands scaffold, the build task surfaces an error in Problems). The proposal notes this honestly.

## Diagnostics / Risks

- **Problem-matcher brittleness** — mitigated by the terse format (the whole point of the decision).
- **No automated extension tests** — the contributes schema + bundle guard + package step are the
  enforceable checks; behaviour is F5-smoke. Stated, not hidden.
- **Scaffolding overwrite** — commands must refuse to clobber existing files (a guard + a test of the
  path-exists branch where feasible).

## Out of scope (→ later)

- **Marketplace / Open VSX publishing** — CI Tier 4 (items 15–17), needs tokens.
- A full `@vscode/test-electron` harness (a separate testing-infra increment).
- LSP-protocol additions (this increment adds no server capability).

## Definition of done

The extension contributes scaffolding **snippets**, **`karn.newContext` / `karn.newProject`** commands, a
**`karnc` problem-matcher + build task** (whole-project compile → Problems panel, backed by `karnc check
--format short`), and a **getting-started walkthrough**. `npm run check` + build + bundle-guard + `vsce
package` stay green; the terse format has a `karnc` test; docs updated; ADR 0071 added; changelog;
version → 0.38.0. Lands as two slices (extension-only; then problem-matcher + the `karnc` format); the
final PR **deletes this file**.
