# v0.59 — `bynkc test --format json` and a VS Code test runner

- **Status:** Draft (proposal)
- **Spec:** tooling increment — no language surface change

## Summary

Add a `--format json` surface to the existing `bynkc test` command and wire a VS
Code Test Explorer integration on top of it. `bynkc test` keeps its current
behaviour and home (the `bynkc` binary, `bynkc/src/main.rs::run_test`); the only
new thing it learns is how to report results as structured JSON instead of only
human lines plus an exit code.

This is the minimal, non-breaking increment toward a better test experience. It
deliberately does **not** extract a `bynk-test` crate or add a `bynk test`
driver command — see [Deferred](#deferred-bynk-test-crate-and-bynk-test).

## Decision (settled)

We keep `bynkc test` as it is — same binary, same compile + workers overlay +
`tsc`/`node`/`tsx` chain, same human output and exit code — and add a `--format`
selector. Rationale: the extraction's only forced motivation was a `bynk test`
driver command, and that can be added later by **shelling** `bynkc test --format
json` (no library linkage, so no `bynkc → bynk-test → bynkc` package cycle).
Adding JSON now is small, non-breaking, and unblocks the VS Code work, which is
the actual goal.

## Goals

1. `bynkc test --format json` emits a stable, pinned JSON document of results.
2. The human default is byte-for-byte unchanged.
3. A VS Code Test Explorer view that runs Bynk tests and reports pass/fail with
   click-through from a failing assertion to its `.bynk` source.

## Non-goals

- No change to how tests are written, discovered, validated, or emitted
  (`project/tests_emit.rs`, `process_tests`/`process_integration_tests` stay
  put).
- No `bynk-test` crate and no `bynk test` driver command in this increment.
- No change to the integration-test wire model (ADR 0009).
- No per-test timing in v1 — see [Follow-ups](#follow-ups).

## `[DECISION 1]` `--format` vocabulary

The previous draft claimed this "mirrors `bynkc check --format`". It does not,
and that inconsistency is worth resolving deliberately, because the two live in
the same binary:

- `bynkc check` → `DiagFormat` = `rich` (default) | `short` (`cli.rs:23`); no
  `json`.
- `bynk doctor` (a *different* binary) → `human` | `short` | `json`.

**Fork A — match `check`'s vocabulary.** A shared `bynkc`-wide `--format` set:
`rich` (default; for `test` this is the grouped `✓ / ✗` output) and `json`,
leaving `short` reserved. A user types the same `--format` values regardless of
`bynkc` subcommand.

**Fork B — adopt the driver's `human | json` vocabulary** and accept that
`bynkc check` says `rich` while `bynkc test` says `human` for the same idea.

**Recommendation: Fork A.** In-binary consistency is the higher value; `rich`
reads slightly loosely for the `✓ / ✗` output (it isn't ariadne source-context
rendering) but it consistently means "the human-readable rendering" across
`bynkc`.

Concretely, give `test` its **own** value set `{ Rich, Json }` whose value
*names* match `check`, rather than literally sharing `DiagFormat { Rich, Short }`
+ `Json`. Sharing one enum would make `bynkc test --format short` parse on day
one with no behaviour behind it — a value that parses but errors, which is the
awkward middle. Aligning the vocabulary while keeping a per-command subset avoids
that. `short` for `test` (one terse line per case, sibling to `check --format
short`) is added only when its behaviour is implemented; it is trivial, so the
alternative — implement `short` for `test` now and share the full enum — is also
reasonable. What we do **not** do is expose a `short` value that parses but does
nothing.

## Structured runner output

The JSON surface must not be scraped from the human `console.log`. Upgrade
`emit_test_main` (`bynkc/src/project/tests_emit.rs:2050`) so the generated
`tests/main.ts` emits **NDJSON events** to stdout when
`process.env.BYNK_TEST_FORMAT === "ndjson"`, and keeps today's `✓ / ✗` output
(`:2088`) otherwise. `run_test` sets that env var only in JSON mode.

What the runner already has vs. what is net-new:

- **Have:** per-case `{name, pass, error?}`, grouped per module, for both unit
  and integration tests (both `process_tests` at `:58` and
  `process_integration_tests` at `:294` return `Vec<RunnableTest>` and feed the
  *same* `emit_test_main` → one `tests/main.ts`, so the suite/case model covers
  both uniformly).
- **Have, conditionally:** an assertion failure's source location. The generated
  `AssertionError` (`:1596`) carries `location/start/end` set at each assert
  site, surfaced as `error.location`. The string `"unknown"` is only the
  fallback for a *non-assertion* throw (`:744`, `:2043`).
- **Net-new:** nothing else is required for v1. Timing (`durationMs`) is **not**
  in the runner today and is deferred (see Follow-ups).

NDJSON event stream (one JSON object per line):

```jsonc
{"type":"run-begin","suites":4}
{"type":"suite-begin","name":"commerce.payment","kind":"unit","tests":3}
{"type":"case","suite":"commerce.payment","name":"charges the card","outcome":"pass"}
{"type":"case","suite":"commerce.payment","name":"rejects expired card","outcome":"fail",
 "message":"assertion failed at commerce.bynk:42:5","location":{"path":"src/commerce.bynk","line":42,"col":5}}
{"type":"suite-end","name":"commerce.payment"}
{"type":"run-end","passed":10,"failed":1}
```

### Suite name and kind

Unit suites carry a clean dotted `name` (`commerce.payment`). Integration suites
do **not**: `emit_integration_module` sets `target_name = "integration ·
{suite}"` (`:767`), encoding the kind in a middle-dot prefix. A machine-readable
`name` field should not carry that prefix, so the JSON model splits it out: each
suite gets a `kind: "unit" | "integration"` field and a clean `name`. `run_test`
derives both from `RunnableTest::target_name` (strip the `"integration · "`
prefix → `kind = "integration"`, else `kind = "unit"`). The document golden and
the VS Code `TestItem` labels must expect this shape (label = `name`, optionally
grouped by `kind`), not the raw prefixed string.

### `[DECISION 2]` NDJSON is an internal protocol, not a public surface

The flow is: Node emits NDJSON → `run_test` captures and parses → `run_test`
emits the single pinned JSON **document** (below). VS Code consumes only the
document. So NDJSON is a runner↔`run_test` **internal protocol**, deliberately
chosen over having Node `JSON.stringify` the whole document directly, for three
reasons:

1. **Single document emitter.** The compile-failure path never runs Node, so
   `run_test` must be able to emit the document anyway. Letting Node also emit it
   would mean two emitters of one schema, which drift. With NDJSON, Rust is the
   sole emitter of the public document (one `#[derive(Serialize)]` set).
2. **Crash robustness.** A mid-run `node` crash leaves a *truncated* JSON blob
   (unparseable) but a *complete prefix* of NDJSON — `run_test` keeps every event
   received and folds captured stderr into the document's error object.
3. **A clean path to streaming later** (live progress for long integration
   suites), without changing the public document.

Consequence: NDJSON is **goldened as a regression guard, not framed as a
consumer contract**. VS Code parses the whole document today, so there is no live
progress in v1; streaming is a deferred payoff, not part of this increment.

## CLI change: `bynkc test --format`

`bynkc test` has no `--format` flag today. Add one per Decision 1 (default
`rich`, plus `json`). In `run_test`:

- **`rich` (default):** unchanged. Node's stdio stays inherited; the runner
  prints its own lines; `run_test` returns the same exit code.
- **`json`:** set `BYNK_TEST_FORMAT=ndjson`, **capture** node's stdout (instead
  of inheriting), parse the NDJSON into the result model, and print the pinned
  document. A compilation failure before the run is reported as a JSON error
  object rather than ariadne text, so a JSON consumer always gets JSON; captured
  stderr from a crashed run is folded into a `runtime` error object.

  **Exit code — key it on node's process exit status, not the parsed count.**
  Today `run_test` keys success on node's own status (`main.rs:171–172`:
  `s.success() → SUCCESS`, else `FAILURE`), which correctly catches both test
  failures *and* mid-run crashes. JSON mode now rebuilds from parsed events, so
  deriving the exit from `failed == 0` would regress: a crash whose NDJSON prefix
  shows `failed == 0` with no `run-end` would wrongly exit `0` — exactly the case
  Decision 2 #2 exists to handle. So the rule stays: **exit on node's process
  status** (in the normal case parsed `failed == 0` agrees with it); a missing
  `run-end` event marks the document as a crashed/incomplete run regardless of
  the partial counts.

## Pinned JSON document

Built from `#[derive(Serialize)]` structs in declaration order — the discipline
`bynk/src/report.rs` calls out (field order *is* the contract; avoid `json!` so
workspace `preserve_order` unification cannot reorder it). Golden-tested, like
the `bynk doctor` JSON.

```jsonc
{
  "passed": 10,
  "failed": 1,
  "suites": [
    {
      "name": "commerce.payment",
      "kind": "unit",
      "cases": [
        {"name":"charges the card","outcome":"pass"},
        {"name":"rejects expired card","outcome":"fail",
         "message":"assertion failed at commerce.bynk:42:5",
         "location":{"path":"src/commerce.bynk","line":42,"col":5}}
      ]
    }
  ]
}
```

`error.kind` is a closed enum — `"compile" | "runtime"` — and an `error` is
present only in a terminal failure that is not "ran to completion with some
failing assertions". There are thus three terminal states: a normal run (`suites`
present, no `error`; may have `failed > 0`), a compile failure, and a
crashed/incomplete run.

A **compile failure** (node never ran) yields, instead of `suites`:

```jsonc
{"passed":0,"failed":0,"error":{"kind":"compile","diagnostics":[ /* path:line:col / category / message */ ]}}
```

A **crashed/incomplete run** (node started, then died before `run-end`) keeps
whatever `suites`/counts the NDJSON prefix produced *and* carries a `runtime`
error with the folded stderr:

```jsonc
{
  "passed": 3, "failed": 0,
  "suites": [ /* the prefix observed before the crash */ ],
  "error": {"kind":"runtime","message":"node exited with signal SIGSEGV","stderr":"…captured stderr…"}
}
```

Note `failed == 0` here despite a non-zero exit — which is why the exit code
keys on node's process status, not the count (above).

## Goldening strategy

"Golden the doctor way" only fits the document. The doctor precedent
(`bynk/tests/doctor.rs::golden_report`) hand-builds a model, serializes it, and
compares against a committed file (`tests/golden/`, blessed with `BYNK_BLESS=1`)
— deterministic, no toolchain. The pieces here golden differently and **must not
be conflated**, or a CI gate ends up silently requiring `node`/`tsc` and flaking
on timing:

- **The pinned document** — copy doctor literally. Build a synthetic `TestRun`
  model in Rust, serialize, compare a committed golden. Deterministic, toolchain-
  free. Cover the success and `error.kind == "compile"` shapes.
- **`run_test`'s NDJSON → document parser** — feed *fixture* NDJSON (a
  hand-written event stream, including a truncated/crashed stream) into the
  parser and assert the resulting document. Deterministic, toolchain-free. This
  is where crash-robustness (Decision 2) is actually tested.
- **The runner's NDJSON emission** — snapshot the *emitted runner source* that
  `emit_test_main` produces (the `tests/main.ts` string), exactly as the other
  emitters are unit-tested in `tests_emit.rs`. Deterministic, toolchain-free; it
  verifies the runner *will* emit the right events without running anything.
- **A true end-to-end** (node actually runs, real NDJSON observed) belongs with
  the existing toolchain-gated suite (`bynkc/tests/e2e.rs`, `tsc_verify.rs`),
  **not** the golden CI gate, because it needs `node`/`tsc` and has
  non-deterministic timing. Keep it there and env-gated.

So "golden the event stream" means *snapshot the emitted source + test the parser
on fixtures* — not capture live node output.

## `[PRECONDITION]` Integration-assertion locations

Click-through depends on `location` being a real `path:line:col`. For unit tests
this holds (the `AssertionError` mechanism). For **integration** tests it is
**unverified**: integration assertions run over a *simulated wire* in Node, not
Miniflare (ADR 0009), and the result sites fall back to `location:"unknown"` for
any non-`AssertionError` throw (`:2043`). If integration assertion failures do
not carry real spans, click-through silently fails for that whole test class and
the `location` field becomes load-bearing on something that may not exist.

This must be **verified by a spike before sign-off**, not confirmed after: emit a
deliberately-failing integration assertion, run with `--format json`, and check
`location` is a real span. If it is `"unknown"`, either fix the integration
assert lowering to thread spans or scope click-through to unit tests in v1 and
say so.

## VS Code wiring

The extension links no Rust; it shells binaries and speaks LSP. It consumes
`bynkc test --format json` directly — the same `bynkc` it already resolves for
the `bynkc: check` task (via `bynk.compilerPath`).

### Components (`vscode-bynk/src/testing.ts`, new)

- **`vscode.tests.createTestController("bynk", "Bynk Tests")`** owns the tree.
- **Run profile** (`createRunProfile`, kind `Run`). Spawn `bynkc test --format
  json` at the workspace root, parse the document, and for each `TestItem` call
  `run.passed(...)` / `run.failed(item, new vscode.TestMessage(msg))`. Build a
  `vscode.Location` from each case's `location` so a failure links to the `.bynk`
  line (subject to the precondition above).
- **Discovery — populate lazily from a run (v1).** The tree is built from the
  first run's document. A dedicated `--no-run --format json` *discovery* document
  is **not** in v1: `--no-run` today only emits `main.ts` and prints "tests
  emitted to …" (`main.rs:124`), and case names are not retained Rust-side —
  `RunnableTest` (`:928`) holds only `target_name` and `module_path`; per-case
  names exist only inside the generated TS via `sanitise_case_name` (`:2100`). A
  pre-execution discovery list therefore needs new Rust plumbing to surface case
  names and a separately-defined discovery document (no `outcome`); both are
  deferred (see Follow-ups).
- **Failure handling depends on `error.kind`.** A normal run with `failed > 0`
  reports per-case failures through the Test API (above). When the document
  carries an `error`:
  - `kind == "compile"` → mark the run errored and route the diagnostics through
    the existing `$bynkc`-shaped parsing (`path:line:col: severity[category]:
    message`) so they land in the Problems panel exactly as `bynkc check` does.
    These are not test outcomes.
  - `kind == "runtime"` → the toolchain crashed mid-run; this is **not** a
    `$bynkc`-parseable diagnostic. Surface it as a run-level error (e.g.
    `run.errored` / a `TestMessage` with the folded stderr), not in the Problems
    panel, and leave already-reported cases from the prefix intact.

### Config & commands (`package.json`)

- Reuse the existing `bynk.compilerPath` setting to locate `bynkc`.
- Command `bynk.runTests` ("Bynk: Run Tests") as a palette entry triggering the
  default run profile.
- No new problem matcher; results flow through the Test API, compile errors
  reuse `$bynkc`.

`registerTesting(context)` is called from `extension.ts` beside the existing
`registerTasks`.

## Sequencing

1. **Integration-location spike** (the precondition above) — do this first; it
   can change scope.
2. **Structured runner.** Add NDJSON emission to `emit_test_main` behind
   `BYNK_TEST_FORMAT`; snapshot the emitted runner source (see Goldening
   strategy).
3. **`--format json`.** Add the format value (Decision 1), capture/parse in
   `run_test`, emit the pinned document; golden the document and test the parser
   on fixture NDJSON, including a truncated stream (see Goldening strategy).
4. **VS Code.** Add `testing.ts` and the `bynk.runTests` command; package a new
   VSIX.

## Follow-ups (explicitly out of v1)

- **Per-test timing.** `durationMs` at case/suite/run level is net-new
  instrumentation: each emitted module's `run()` constructs `{name, pass,
  error?}` with no timer (the result sites ~`:740` and ~`:2039`). Adding it means
  wrapping each case runner with `performance.now()` *inside the emitted module*,
  not just in `main.ts`, then threading the field through NDJSON and the
  document. Add when the timing lands; the document gains optional fields
  additively.
- **Pre-execution discovery document.** Surface case names Rust-side and define a
  separate `--no-run --format json` document so the Test Explorer can show the
  tree before running.
- **Live streaming** of NDJSON to VS Code for incremental progress on long
  integration suites (Decision 2 keeps the door open).

## Risks

- **Integration locations** (precondition) — the load-bearing one; spike first.
- **The pinned document can rot.** Golden it (copy the `bynk doctor` JSON
  precedent). The internal NDJSON protocol is goldened too, but as a regression
  guard, not a consumer contract (Decision 2).
- **Capture vs. inherit.** JSON mode captures node's stdout; ensure stderr
  (uncaught exceptions, stack traces) is folded into the `runtime` error object
  rather than dropped, and that the exit code follows node's status so a crash is
  never reported as success (see CLI).

## Deferred: `bynk-test` crate and `bynk test`

Out of scope here, recorded so the door stays open. If a `bynk test` driver
command is wanted later, the clean route given the package-cycle constraint
(`bynkc → bynk-test → bynkc` is illegal) is for the **driver to shell** `bynkc
test --format json` and re-render — exactly how `bynk dev`/`doctor` already shell
`bynkc`. No library extraction, no cycle. The JSON surface this increment adds is
precisely the contract that future `bynk test` would consume, so this increment
is a strict prerequisite either way.
