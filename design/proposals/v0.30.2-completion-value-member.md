# v0.30.2 — Comprehensive completion, slice 3: value-receiver `.method`/`.field`

- **Status:** Draft (proposal). **Merging is approval to build.** *(The daily-driver `.`-member tier:
  `list.map`, `str.split`, `email.field` — where the receiver is a **typed value**. Spans `karnc`
  (enumerable kernel registries + an `expr_types` sink) and `karn-lsp` (a `.`-trigger rewrite + a
  type-at-offset query). Bigger than slices 1–2, and gated on a now-completed feasibility spike.)*

## Context — the spike said yes, with a ceiling

ADR 0062 deferred value-receiver `.`-members behind one question: *can a receiver's type be obtained on
an unparseable mid-edit buffer at all?* A spike answered it **empirically** (probe + code read):

- **The mid-edit parse is defeatable by a rewrite.** A bare trailing `receiver.` (the auto-trigger
  instant) cascades — the parser consumes the closing `}`, the receiver is **lost**, no `FieldAccess`.
  But `receiver.partial` parses **clean** with the receiver intact, and rewriting `receiver.` → `receiver`
  is also clean. So a **completion-specific rewrite** (drop the trailing dot, or insert a placeholder
  member) makes the receiver survive as a typed node.
- **`expr_types` is recoverable cheaply.** The checker already computes `expr_types: HashMap<Span, Ty>`
  (`checker.rs:165`) but **drops it on the LSP `Analyse` path** (`project.rs:2102`). Retaining it is the
  proven `HintSink` pattern (the v0.27 inlay-hints sink: checker → `FileHints` in `ProjectDiagnostics`
  → `Analysis.hints` → consumer — `main.rs:262/821`).
- **The ceiling:** `check_record` **bails per-file on any error** (`checker.rs:176`; `project.rs:2066`
  drops `typed` on `Err`), so the receiver is typed **only when the file otherwise checks clean**. A
  mid-edit file with unrelated errors yields no typed receiver — completion offers nothing there
  (graceful degradation, consistent with "conservative: offer nothing over wrong"). The common case
  (a valid file, you add `email.`) works.

## Builds on (verified — spike + scout)

- **`Ty` maps cleanly to member sets** (`checker.rs:45`): `Base(Int/Float)` → numeric kernel;
  `Base(String)` → string kernel; `Option`/`Result`/`Effect`/`HttpResult`/`List`/`Map` → their kernels;
  `Named { kind }` → a **record**'s fields (AST-enumerable) / refined/opaque (no extra value members) /
  sum (matched via patterns, not `.`); `JsonError`/`ValidationError` → their fixed record fields.
- **The kernel method names exist but are buried** in `check_{list,numeric,string,option,result,map}_kernel_method`
  match arms (`kernels.rs:64/179/254/355/459/773`) as `"name" => …`. They are **not enumerable** — a
  small `karnc` refactor extracts each into a registry the checker dispatches over *and* completion
  lists (one source of truth, a pinning test).
- **Record fields are already AST-enumerable** (`RecordBody.fields`), as slice 2 used for type decls.

## Decisions (open — settle at review)

**[DECISION] Rewrite-on-trigger + re-analyse (the spike's approach).** *Rec:* on a `.`-member trigger
where the receiver is **lowercase** (a value; uppercase is slice 2's name receiver), rewrite the current
buffer — drop the trailing `.`(`partial`) so the receiver is a complete expression — run **analyse** on
the rewritten buffer, and query `expr_types` for the receiver's span. Re-analysing per completion is
acceptable (completion is debounced, and analyse is the existing `didChange` path); the clean-file
ceiling bounds the work. *(Alternative — query the cached analysis — is staler and the receiver span may
not be present; the rewrite is the robust choice.)*

**[DECISION] Retain `expr_types` via a sink.** *Rec:* a `TypeSink` mirroring `HintSink` (or a field on
it) → `FileExprTypes` (a `Vec<(Span, Ty)>` or span-keyed map) in `ProjectDiagnostics`/`Analysis`. A
**type-at-offset** query finds the innermost span covering the receiver. `Ty` stays off the public
*completion* surface — the LSP maps `Ty` → a member list internally.

**[DECISION] Enumerable kernel registries (the `karnc` refactor).** *Rec:* extract each
`check_*_kernel_method`'s arms into a `&[(name, signature)]` table the checker matches over and
completion reads. A **pinning test** asserts the table and the dispatch agree (the v0.28 legend-test
shape) so they can't drift. Item kinds: methods → `METHOD`, record fields → `FIELD`.

**[DECISION] The ceiling, stated.** *Rec:* accept graceful degradation — when the receiver can't be
typed (file has other errors, or the rewrite still doesn't check), offer `[]`. Document it; don't chase
error-tolerant typing in this slice.

**[DECISION] New decision record.** *Rec:* **ADR 0063** — the value-receiver mechanism (rewrite +
`expr_types` sink + enumerable registries) and the clean-file ceiling. Numbering after 0062.

## Tests

- **`karnc`** — the kernel-registry pinning test (table ≡ dispatch); the `expr_types` sink populates for
  a checked file (a binding's span → its `Ty`).
- **`karn-lsp` pure-function** — the rewrite (`x.`/`x.ma` → typeable receiver text); the `Ty` → member
  list mapping per kernel (`List` → `map`/`filter`/`fold`/…; record → its fields; `String` → `split`/…).
- **`karn-lsp` analysis-level** — a clean file with `let xs = [1]` then `xs.` yields the list methods; a
  file with an unrelated error yields `[]` (the ceiling).

## Diagnostics / Risks

- **The clean-file ceiling** — the chief limitation; bounded and graceful, but means value-method
  completion is absent exactly when the file is broken. Stated, not hidden.
- **Re-analyse cost** — analysing on `.`-trigger; mitigate with the existing analyse path + debounce;
  measure against §4.5 targets.
- **Rewrite edge cases** — nested/multiline receivers, dots inside strings/comments, chained
  `a.b().c.`; keep the rewrite minimal (operate on the trigger line, the immediate receiver) and fixture
  the cases; degrade to `[]` when unsure.
- **Registry drift** — the enumerable kernel list must match the checker's dispatch; the pinning test is
  the guard.

## Out of scope (→ slice 4 / later)

- **Locals/params in scope** — a separate, larger effort: no scope-at-offset query exists (the index
  tracks only top-level symbols); needs the checker to record each binding's lexical scope range and the
  index to expose it. Independent of this slice's receiver-typing machinery.
- Signature help; auto-import; postfix expansion; error-tolerant/scoped typing to lift the ceiling.

## Definition of done

On a lowercase `receiver.`(`partial`) trigger, `karn-lsp` rewrites the buffer, analyses it, queries the
retained `expr_types` for the receiver, maps its `Ty` to a member list (kernel methods from the new
enumerable registries; record fields from the AST), and returns `METHOD`/`FIELD` items — or `[]` when
the receiver can't be typed (the ceiling). `karnc` retains `expr_types` via a sink and exposes the
enumerable kernel registries with a pinning test. Pure-function + analysis-level tests cover the rewrite,
the `Ty`→members mapping, and the ceiling; `design/karn-lsp-spec.md` §3.15 gains the row + the slice-4
line; ADR 0063 added; CI green; changelog updated; version → 0.30.2. The implementing PR **deletes this
file**.
