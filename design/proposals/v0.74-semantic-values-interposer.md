# v0.74 — Bynk values through the interposer: semantic values on *both* runtimes (semantic-debugging track, slice 1)

- **Track:** [semantic-debugging](../tracks/semantic-debugging.md), slice 1 — the first
  build on slice 0's settled model. The first real payoff: the **workerd** value
  rendering slice 5 couldn't reach.
- **Realises:** ADR [0105](../decisions/0105-semantic-debug-interposition.md) — its
  editor-side `DebugAdapterTracker` rewrite, applied to value descriptions. D1
  (interpose, don't rebuild), D2 (rung-1 tracker mutation), D4 (rewrite every
  `variables` response), D6 (bind to `__bynkChild`, gate on `bynk.debug.semanticValues`).
- **Posture:** tooling (VS Code extension). No language change, no runtime change.

## The gap slice 5 left

Slice 5 (v0.73) renders `Ok(42)` via js-debug's `customDescriptionGenerator` — a
function evaluated **in the debuggee**. That works on Node but **dies on workerd**,
whose runtime forbids the evaluation (it breaks variable inspection outright), so the
dev-server path still shows the raw `{tag: "Ok", value: 42}`. ADR 0105 settled the fix:
rewrite js-debug's `variables`/`evaluate` **responses** in the extension host instead —
**runtime-agnostic**, because nothing runs in the debuggee. This slice builds that
rewrite and, with it, brings Bynk-vocabulary values to **both** runtimes from one path.

## The one genuine uncertainty — the preview parse (spike-first)

The tracker mutation must be **synchronous**: `onDidSendMessage` rewrites the response
as it is delivered, so the new value must come from data *already in the response* —
js-debug's preview string. A tagged value arrives as its preview, e.g.
`{tag: 'Some', value: 'hi'}` (confirmed in the slice-0/5 logs). The interposer
recognises that shape and re-renders it as `Some("hi")`.

What's **unconfirmed**, and what the slice spikes before wiring anything in:

1. **How legibly can we render from the preview alone — and how deep?** The slice-5
   generator recursed fully in the debuggee (`Ok(Some(42))`). The editor-side parse is
   bounded by js-debug's preview depth: a nested ADT may arrive truncated
   (`{tag: 'Ok', value: {…}}`), which we can only render `Ok(…)` — the inner value one
   expand away (and *its* `variables` response is itself rewritten, so it reads
   `Some(42)` when opened). The spike measures the achievable depth and whether the
   shallow top-level rendering reads well enough.
2. **Parser robustness.** The preview is a display string, not JSON (single quotes,
   bare numbers, `{…}`/`[…]` truncation, embedded quotes). The parser must be **total**
   — never throw, never mangle a non-Bynk value (return it untouched) — and cheap (it
   runs on every variable). Spike it against `Result`/`Option`/sum/nested/strings-with-
   braces before trusting it.
3. **Does it hold on workerd?** The same response-rewrite over `wrangler`'s inspector —
   confirmed the same way slice 5's spike was, flipping that slice's "raw on workerd"
   guard into an assertion that it now renders.

This spike also **decides the slice-5 generator's fate** (DECISION B).

## What this slice does

1. **The value interposer** (`src/debug.ts` or a new `src/semanticValues.ts`): a
   `DebugAdapterTracker` registered for the delegated session, gated on
   `__bynkChild` + `bynk.debug.semanticValues`, that in `onDidSendMessage` rewrites
   `variables` and `evaluate` responses — re-rendering each tagged-value preview into
   Bynk constructor syntax (`Ok(42)`, `Some("hi")`, `None`, `BadRequest("…")`), leaving
   every non-Bynk value byte-for-byte untouched. The renderer reuses slice 5's
   constructor-syntax rules; only its *input* changes (a preview string, not a live
   object).
2. **Both runtimes, one path.** The same tracker serves Node (`bynkc test --inspect`)
   and workerd (`bynk dev --inspect`) — so the dev path gains semantic values.
3. **Resolve the generator (DECISION B).** Retire slice 5's
   `customDescriptionGenerator`, or keep it for Node, per the spike.
4. **Tests:** a Node interposer test (values render through the tracker, not the
   debuggee generator) and a workerd test that **asserts** `Some("hi")` renders —
   replacing slice 5's raw-read guard. Both guarded as before (Node harness-provisioned;
   workerd local/opt-in). The slice-0 mutation guard stays.
5. **Docs:** update the "Values read in Bynk" guide — drop the "workerd shows the raw
   shape" caveat (now both); note any depth limitation. Changelog row; the bump.

## Decisions

**[DECISION A] Editor-side response rewrite, per ADR 0105.** The value layer becomes a
tracker that rewrites `variables`/`evaluate` responses — not an in-debuggee generator.
This is the whole point: it is runtime-agnostic and reaches workerd. No new mechanism;
ADR 0105 already ratified it.

**[DECISION B — recommend, spike-gated] Unify on the interposer; retire the slice-5
generator.** One mechanism for both runtimes is simpler and keeps Node and workerd
identical. *Recommend retiring the generator* — **iff** the spike shows the preview-parse
rendering is acceptable on Node (top-level always; nesting one expand away). If the spike
finds the depth regression material, **keep the generator for Node** (full recursion) and
use the interposer only for workerd — two paths, best-per-runtime. The toggle stays
either way.

**[DECISION C] Descriptions only; reshaping the expanded children is a later slice.**
This slice rewrites the one-line value (the description), matching slice 5's scope.
*Hiding the `tag` child* / showing a `Result` as just its payload is a `scopes`/children
reshape — slice 2's territory (contexts/actors as scopes), not here.

**[DECISION D] Synchronous parse, not async child-fetch.** The rewrite reads only what
the response already carries (the preview); it does **not** issue follow-up `variables`
requests to read fields precisely (they would resolve after the response is delivered).
This bounds depth to the preview — an accepted, ADR-0105-consistent constraint, not a
bug. (A future slice could pre-fetch and cache if depth ever matters enough.)

**[DECISION E] Version — minor v0.74.** A new extension capability (workerd semantic
values); no language change.

## Risks & mitigations

- **Preview-format drift across js-debug versions.** The parser keys on js-debug's
  display format. Mitigated by the pinned `VSCODE_TEST_VERSION` and the new render tests
  (and slice 0's mutation guard) — a format change reddens CI.
- **Parser mangling a non-Bynk value.** A total parser with a tight guard (recognise the
  leading `{tag: '…'` shape; otherwise return the string unchanged), unit-tested in
  isolation against adversarial previews (strings containing `{tag:`, deep objects).
- **Shallow nesting vs slice 5.** Bounded by DECISION D; DECISION B keeps the generator
  for Node if the regression bites. Expanded children are themselves rewritten, so depth
  is *reachable*, just not all inline.
- **workerd reach unproven for this exact path.** Spiked first (the response rewrite over
  `wrangler`'s inspector), gated like every workerd step on this track.
