# v0.102 — held-resource linearity: the `Connection[F]` type and the API-discipline ownership pass (real-time track, slice 2)

- **Scope:** a single increment — the `Connection[F]` held type, its operation
  vocabulary (`send`/`close`, and the held-bearing storage ops), and the
  **linearity-check pass** that enforces ownership/disposal. **Language +
  checker + emitter (interface-level).** **No protocol, no socket, no `from
  WebSocket`, no runtime connection implementation** — those are slice 3. This
  slice is *compile-time*: it is tested against hand-written held sources, in
  isolation, before any transport exists.
- **Realises:** [`design/tracks/websocket.md`](../tracks/websocket.md) §3.5 and
  **Slice 2** of its §6 decomposition — "realise `bynk-type-system.md` §2.9: the
  ownership states, the closed origin, mandatory disposal, the storage rules, the
  step-11 check — against hand-written held sources, **no protocol yet**." Settles
  the track's **Q5** (held-resource fault paths) and **Q6** (linearity
  diagnostics).
- **Realises in the spec:** `bynk-type-system.md` **§2.9** (Held resources,
  "settled in shape" — this slice builds it) and its **§3 step 11**, *"Check
  linearity for held resources"*, currently named but unimplemented. Promotes
  §2.9.7 from *Open* to settled for the within-handler subset.
- **Authorises its own ADR:** the held-resource linearity model (the
  `Connection[F]` type, the operation classifications, the ownership pass, the
  fault-path rules). **Number assigned when the PR is cut** — next free
  (provisionally 0130), per the track's no-reservation rule.

## 1. What this is — the carrier, before the protocol

The streaming leg (slices 0–1) is done. The connection leg needs a value the
language has gone further on than `Stream[T]`: `bynk-type-system.md` §2.9 has
**settled in shape** an "API-discipline-driven linearity" model — `Held[T]` /
`Connection[F]`, four ownership states, a closed origin, mandatory disposal, and
storage rules — but **none of it is built**. This slice builds it, deliberately
**without** the `from WebSocket` protocol that produces connections in
production. The point (the track's words) is that the type machinery is
"testable in isolation against hand-written held sources before the socket
transport exists."

Concretely: a fixture declares a capability that yields a connection —

```bynk
capability Gateway { fn accept() -> Effect[Connection[ServerFrame]] }
```

— and the linearity pass governs every binding of that `Connection` from the
moment it is received to the moment it is disposed. The rules are **compile-time
rejections**, so they need no running socket: a connection that escapes a
handler unstored-and-unclosed is an error; one stored in a `Set` is an error; one
used after `close` is an error. That is the whole slice.

## 2. Scope and non-goals

**In scope.**

- The **`Connection[F]` type** (`F` = the server→client frame type), and the
  `Held` *kind* classification it belongs to (a `is_held(ty)` predicate, closed at
  one instance, extensible later — §2.9.1).
- **Operation classifications** (§2.9.2): `c.send(f)` (non-consuming borrow-and-
  return, frame-typed against `F`), `c.close()` (consuming); and on held-bearing
  storage — `put` (consuming), `take`/`remove` (consuming/transfer-out),
  `forEach`/`update` (borrowing).
- **Storage integration** (§2.9.3): `Cell[Option[Connection[F]]]` and
  `Map[K, Connection[F]]` **admit** held values; `Set`/`Log`/`Cache` **reject**
  them, with diagnostics.
- The **linearity-check pass** — **§3 step 11**: the per-handler-body flow
  analysis tracking each held binding through `fresh → owned → borrowed →
  stored/consumed`, enforcing single-owner, mandatory disposal at scope exit,
  branch unification (§2.9.5), and the loop rules.
- **Classification**: `Connection[F]` is non-serialisable, non-boundary, and not
  value-comparable — joining `Stream`/`Query`/`Effect` in the existing pass (it is
  already named in the §2.5.8 serialisability list).
- **Fault paths** (§2.9.7, Q5): the within-handler rules — implicit runtime
  consumption of owned-at-abort connections, transactional rollback of stored
  ones, and the static rejection of a compensation operating on a still-local
  held value.
- **Diagnostics** (§2.9.9, Q6): the branch-divergence message and the
  consume-after-use / leak / wrong-storage messages.
- Emission at the **interface level** — `send`/`close`/storage ops lower against a
  runtime `Connection` *interface*; the concrete implementation (`TestConnection`,
  hibernatable WS) is slice 3.
- Positive (compiles) and negative (rejected) fixtures, plus Rust unit tests on
  the pass; spec-in-place for §2.9; an ADR.

**Non-goals (and why).**

- **The `from WebSocket` protocol** — slice 3. This slice produces no connection
  from a transport; a capability signature is the hand-written source.
- **`TestConnection` and any runtime connection implementation** — slice 3. Slice
  2 is compile-time; it lowers against an interface and ships **no** runtime
  behaviour test (there is no value to run). The design notes' `TestConnection`
  (capture-and-inspect) is the slice-3 realisation.
- **Record-of-held** (§2.9.9) — whether `record { conn: Connection[F], … }`
  carries the discipline. Explicitly deferred and named in §2.9.9; a held field in
  a user record is rejected for now (it is non-serialisable, so already rejected at
  boundaries; a purely-local held record is the open case).
- **User-defined borrowing functions** (§2.9.9) — only platform-defined held ops
  are borrows; user helpers take and return ownership. The line §2.9.9 draws.
- **Cross-context fault propagation of held values** (§2.9.7/§2.9.9) — a held value
  mid-transfer across a context boundary when a fault interrupts. There is no
  cross-context held transfer until the protocol exists (slice 3); this slice
  settles only the within-handler subset.
- **General linear/affine types** — §2.9 is deliberate: linearity rides a fixed
  vocabulary of runtime operations, not a general system. The slice holds that
  line.

## 3. Internal architecture

### 3.1 The type and the kind

A `Connection[F]` (`TypeRef::Connection`/`Ty::Connection`, wired through
parser/AST/checker/emitter exactly as `Stream` was in slice 0) is the one
concrete instance of the closed `Held` kind. A `fn is_held(&Ty) -> bool` —
`true` for `Ty::Connection(_)`, the single extension point for future held types
— is what the linearity pass and the storage-admission rules test, so adding a
future held type later is one arm, not a new concept. `Connection[F]` joins the
non-serialisable family (one arm per existing site, as `Stream` did), so it is
already rejected at every boundary and in `Set`/`Log`/`Cache` before the
linearity pass runs.

### 3.2 The operation classifications

The fixed vocabulary the discipline rides (§2.9.2):

| Operation | Receiver | Classification | Effect on binding |
|---|---|---|---|
| `c.send(f: F)` | `Connection[F]` | non-consuming (borrow-and-return) | stays **owned** |
| `c.close()` | `Connection[F]` | consuming | → **consumed** |
| `m.put(k, c)` | `Map[K, Connection]` | consuming (transfer-in) | `c` → **consumed** |
| `m.take(k) -> Option[Connection]` | `Map[K, Connection]` | transfer-out | result **owned** by caller |
| `m.remove(k)` | `Map[K, Connection]` | consuming (remove-and-close) | stored value closed by runtime |
| `m.forEach(fn)` / `m.update(k, fn)` | `Map[K, Connection]` | borrowing | `fn` gets a **borrowed** ref |
| cell `:=`/`update`, `take` | `Cell[Option[Connection]]` | as above | — |

The borrow `&Connection[F]` is an **internal flow state, never written by users**
(Q6/§2.9.9): inside a borrowing `fn`, the reference admits the non-consuming ops
(`send`) but rejects the consuming ones (`close`, `put`, `take`) — a `close` on a
borrowed ref is a compile error at the call site.

### 3.3 The linearity pass (the load-bearing complexity)

**§3 step 11** is a new per-handler-body flow analysis. For each held binding it
threads an ownership state through the statement/expression walk:

- a **receive** (a capability op returning `Connection`, a held handler parameter)
  produces an **owned** binding;
- a **consuming** op moves it to **consumed** — any later reference is an error
  (`use-after-consume`);
- a **borrowing** op lends a **borrowed** ref for the callback's lexical scope,
  reverting to **owned** after;
- at **scope exit**, every still-**owned** binding is a **leak** error unless
  disposed (stored/consumed/returned-to-a-consumer);
- at a **branch join** (§2.9.5), all arms must leave each binding in a compatible
  state — divergence (`one branch consumes, the other leaves owned`) is the error
  the Q6 diagnostic targets;
- **loops** (§2.9.5): an owned binding used inside without consumption or borrowing
  is an error (it would cross iteration boundaries with no semantics); the
  per-iteration borrow pattern (`conns.parTraverse(c => c.send(…))`) is the
  canonical valid shape.

The pass is bounded — a fixed operation vocabulary over a four-state machine, no
general dataflow lattice — which is exactly why §2.9 chose API-discipline
linearity over a general affine system.

## 4. Fault paths — Q5 (settles §2.9.7's within-handler subset)

**[DECISION A] Settle the within-handler fault rules; defer the cross-context
one to slice 3.**

- A connection **owned by the handler at abnormal exit** (a propagated `Err`, a
  fault) is **implicitly consumed by the runtime** — the runtime closes it,
  avoiding a leak. The pass therefore does *not* require an explicit `close` on
  every fault path; it requires only that no binding is *used* after consumption.
- A connection **stored in agent state** rides the handler-atomic commit (ADR
  0109): a fault rolls the store back with the rest of the agent's state, so a
  connection `put` earlier in a now-aborting handler is rolled back.
- A `Sagas.compensate` closure may operate on a connection **already in agent
  state** (the storage APIs work as expected) but **not** on one the handler still
  held **locally** before storing it — that value is about to be runtime-consumed,
  so the compiler **rejects the local case statically** (§2.9.7).

*Recommendation:* this is the conservative, leak-free reading the track's Q5
endorsed. The **cross-context** transfer-fault (a held value mid-handoff when a
fault interrupts) has no surface until the protocol exists, so it stays deferred
to slice 3 — which is also where the platform handoff (`acceptWebSocket` into the
DO) gives the single-owner invariant its runtime backing.

## 5. Decisions

**[DECISION B] `c.close()` is a method, not a free `close(c)`.** §2.9 writes
`close(conn)`; but Bynk is method-first (`Stream.of`, `Duration.millis`,
`conn.send`), so `conn.close()` reads consistently and keeps the consuming-op
surface on the receiver. *Recommended;* refines the spec's notation (as
`Stream.of` refined `bynk.stream.of`), recorded in the ADR.

**[DECISION C] The borrow notation stays implicit (Q6).** Users never write
`&Connection[F]`; the checker infers borrows at `forEach`/`update`/`parTraverse`.
Investment goes into the **branch-divergence** message — *"this branch leaves
`conn` owned but the other consumes it"* — with a suggested fix, the most common
real error. *Recommended;* the track's Q6 call, settled here.

**[DECISION D] A held value in a user-defined record is rejected (§2.9.9
deferred).** Record-of-held needs the linearity discipline to thread through a
record, which §2.9.9 leaves open. *Recommendation:* reject a `Connection` field in
a user record with a clear *"held values are not yet allowed in record types"*
diagnostic and a named follow-on, rather than silently mis-handling it. The common
shapes (a connection in a `store Map`/`Cell`, a handler parameter) are fully
supported; the record case is rare and earns its own slice.

**[DECISION E] Emit against a runtime `Connection` interface; no implementation.**
`send`/`close`/storage ops lower to method calls on a runtime `Connection<F>`
interface (`send(frame)`, `close()`), so positive fixtures emit and `tsc`-check
without a concrete connection. The implementation (`TestConnection` for
bundle/test, the hibernatable-WS binding for Workers) lands in slice 3.
*Recommended;* keeps slice 2 honest (compile-time) and gives slice 3 a settled
emission target.

## 8. Tests

- **Positive fixtures** (compile + emit): a handler that receives a `Connection`
  parameter and disposes it three ways — `connections.put(k, conn)` (store),
  `conn.close()` (consume), and a `forEach`/`parTraverse` borrow loop that sends
  without consuming; the §2.9.8 chat-room shapes (`join`/`leave`/`post`) compiling
  against a hand-written `Gateway` capability and a `store Map[UserId,
  Connection[ServerFrame]]`.
- **Negative fixtures** (the discipline's whole point): a connection that escapes a
  handler unstored-and-unclosed (**leak**); a `close` then `send` (**use-after-
  consume**); the branch-divergence case (one arm consumes, one leaves owned);
  `close` on a borrowed ref inside `forEach`; a `Connection` in `Set`/`Log`/
  `Cache`; a `Connection` across a context/boundary; a `Connection` field in a user
  record (D-D); a `Sagas.compensate` on a still-local connection (D-A).
- **Rust unit tests** on the pass: the four-state transitions, branch unification,
  the loop rules — exercised on synthetic `Ty::Connection` bindings, independent of
  fixtures.
- Full `cargo test --workspace` + `fmt` + `clippy -D warnings`; emitted positive
  fixtures `tsc --strict` clean against the runtime `Connection` interface.

## 9. Risks & mitigations

- **The linearity pass is genuinely new checker machinery** — flow-sensitive
  per-binding state, the most complex addition in the track. *Mitigation:* it is
  bounded (fixed vocabulary, four states, no general lattice); built and unit-
  tested in isolation here, with **no protocol** to entangle it; the §2.9.8 chat-
  room is the closure proof at compile time.
- **"Settled in shape" ≠ settled in detail (§2.9.7/§2.9.9).** *Mitigation:* D-A/
  D-C settle the build-blocking subset (Q5/Q6); record-of-held, user borrows, and
  cross-context faults are explicitly deferred and named, not rediscovered
  mid-slice.
- **Emitting against an interface with no implementation could rot.** *Mitigation:*
  the interface is small (`send`/`close` + the storage-op shapes) and is the exact
  contract slice 3's `TestConnection` and WS binding implement; positive fixtures
  `tsc`-check it now.
- **Diagnostic quality is the user-facing surface of a subtle discipline.**
  *Mitigation:* the branch-divergence and use-after-consume messages carry
  suggested fixes (D-C); the negative fixtures pin their wording.
