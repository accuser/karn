# v0.24 — LSP project-wide diagnostics (A-0, slice 1)

- **Status:** Draft (proposal). **Merging is approval to build.** *(The first A-tier LSP
  increment — the foundation. Touches `karnc` (a project-diagnostics entry point) +
  `karn-lsp`.)*
- **Builds on (verified):**
  - The LSP publishes only **single-file** diagnostics: `recompile_and_publish` →
    `karnc::diagnose(text)` per changed doc (`karn-lsp/src/main.rs:90`; `did_open`/
    `did_change`/`did_save`/`did_change_watched_files`). `diagnose` (`lib.rs:84`) checks
    **`Commons` units only** (`lib.rs:109`) — so **context files get lex/parse diagnostics
    only**, no resolve/check. (This is the gap A-0 closes.)
  - `compile_project(root) -> Result<ProjectOutput, Vec<CompileError>>` (`project.rs:228`)
    exists but is a **build** entry point: it **reads files from disk** and **bails**
    (returns `Err`) at the first failing phase (e.g. `project.rs:287/295/334/1750`). It is
    neither per-file nor overlay-aware — unusable as-is for live diagnostics.
  - `CompileError` (`error.rs:14`) carries `category`/`span`/`message`/`labels`/`notes` but
    **no file identity**; only `CompiledFile` has `source_path` (`project.rs:41`). So
    attributing a project diagnostic to its file is new work.
  - LSP project config loading exists (`karn-lsp/src/project.rs` `load_config`;
    `project_src_root`, `state.docs`).
- **Spec sections touched:** `design/karn-lsp-spec.md` (Live diagnostics → project-wide);
  `karnc` (the new `diagnose_project` entry point + file attribution).
- **Scope:** A-0 **slice 1** — lift the LSP from single-file `Commons` diagnostics to
  **project-wide diagnostics for every file, including contexts**. The **reference/symbol
  index** (definitions + references — what `references`/`rename` need) is **slice 2**,
  separate. No new language semantics.

## Settled (recorded as ADRs ≥0052)

- **▸ This is the LSP foundation.** It makes diagnostics — and, through the same project
  analysis, hover/definition — correct on **context files**, which are most real Karn code
  (services, handlers, `given`). It also unblocks the prescriptive **code-action catalogue**
  (the `given`-clause quick-fixes, which fire only for contexts) and everything index-
  dependent. Doing it before code-actions avoids building the code-action plumbing twice.
- **▸ Positions convert against the analysed snapshot, not the live buffer.** Diagnostics
  carry byte spans valid against the text that was analysed; debounce narrows but does not
  close the analyse→publish window, so converting a span against newer buffer text mis-places
  it. `diagnose_project` returns (or the LSP retains alongside the result) the **per-file text
  snapshot it analysed**, and span→line/col conversion uses that snapshot.

## Decisions (open — settle at review)

**[DECISION] A non-bailing project-diagnostics entry point.** *Rec:* add
`karnc::diagnose_project(...) -> Vec<(SourcePath, Vec<Diagnostic>)>` that runs the project's
resolve/check across **all** files and returns **all** diagnostics, never aborting on the
first failure — distinct from `compile_project` (which bails and then emits TS). Implementation
likely **refactors `compile_project` to separate "analyse (collect diagnostics)" from "emit
(requires success)"**, sharing the analysis so the CLI build path is unchanged.

**[DECISION] File attribution — resolved: collection-point tagging.** `CompileError` has no
file (`error.rs:14`) and `Span` is a bare `{start, end}` byte pair with **no source id**
(`span.rs:5`), per-file offsets. So a `span → file` map is **unsound** — every file's spans
start at 0 and collide numerically — and a source-id on `Span` is **not needed**. *Rec:*
**(a′) collection-point attribution** — each per-file phase (lex/parse/resolve/check) already
knows which file it is processing, so `diagnose_project` tags errors with the current file
**as each phase's errors are collected**; the project-level validations in `project.rs`
(name/kind/group consistency, cross-context checks) construct their errors while holding the
offending `ParsedFile`, so they tag **case-by-case at their push sites**. `diagnose_project`
returns `Vec<(SourcePath, Vec<Diagnostic>)>`. *(Purely additive at the project boundary — no
`Span` change, no compiler-wide churn.)*

**[DECISION] Rider — proper rendering for project-level CLI errors.** `render_project_errors`
(`lib.rs:214`) prints `[category] message` + notes only — **no spans, no source context** — so
project-mode CLI errors are bare lines while single-file mode gets full `ariadne` output. The
file attribution this increment builds is exactly what that renderer needs. *Rec:* **scope it
in** — once diagnostics are file-attributed, render project errors with `ariadne` source
context per file, fixing a standing today-bug and making the `karnc` test assertions richer.
*Alternative:* record as the immediate follow-up if it widens the increment too far.

**[DECISION] In-memory overlay (buffer, not disk).** `compile_project` reads disk; an LSP
must diagnose **unsaved** buffers. *Rec:* `diagnose_project` takes an **overlay** (open
`uri → text` from `state.docs`) layered over disk reads, so diagnostics track the buffer.
Saved-only diagnostics are wrong while editing — rejected.

**[DECISION] Publish lifecycle + scope.** *Rec:* on change (**debounced**), re-diagnose the
project and `publish_diagnostics` for **every** file, **clearing** files that newly pass
(track the set of files currently carrying diagnostics, so a fixed file gets an empty
publish). **Single-file mode** (no `karn.toml`) keeps today's `diagnose` path.

**[DECISION] Performance.** *Rec:* full project re-analysis per debounced change is acceptable
at current scale; **incremental/salsa-style recompute is deferred**. Revisit if large projects
feel it.

## Spec delta (sketch — normative prose written in-place during implementation)

- **`design/karn-lsp-spec.md`** — the Live-diagnostics section becomes project-wide: all

## Tests (the LSP analogue of fixtures)

- **`karnc`** *(`cargo test`, no LSP):* a multi-file project fixture where a **context** file
  carries a handler-level semantic error (e.g. `karn.given.unused_capability`, `checker.rs:264`)
  — assert `diagnose_project` returns it **attributed to that file** (today's single-file
  `diagnose` returns nothing for it); a second, clean file asserts grouping. With the rider,
  the rendered output gains source context.
- **LSP publish-set logic** *(unit-tested, no transport):* extract the **diff** — given the
  previous and new diagnostics-by-file, compute the publish set **and the clear-set** (files
  that newly pass need an empty publish) — as a **pure function** and unit-test it directly.
- **End-to-end (transport):** *Decision:* `karn-lsp` has **no test harness today** (no
  `tests/`). Rather than stand up a JSON-RPC-over-stdio harness now, this increment unit-tests
  the pure logic above and **defers the full harness to the first interactive feature**
  (completion/code-actions) that genuinely needs round-trip testing — where it pays rent
  immediately. End-to-end here is manual verification. *(Recorded so the harness is a decision,
  not a silent assumption.)*

## Diagnostics

None new. A-0 changes *where/when* existing diagnostics surface, not the diagnostic set.

## Risks
- **`compile_project` refactor** to a non-bailing analyse path must not regress the CLI build
  (which needs the bail-on-error, all-or-nothing contract). Share analysis; branch at emit.
- **Overlay/path normalisation** between LSP `uri`s and project-relative `source_path`s —
  **compare case-insensitively on macOS/Windows** (a buffer path and a disk path can differ
  only in case, silently dropping the overlay).
- **Debounce is new machinery** (nothing debounces today) — trivial with `tokio`, but a new
  moving part; the analyse→publish race is closed by the snapshot rule (Settled).
- **Performance** on large projects (mitigated by debounce; incremental recompute later).
- *(Attribution is no longer a risk — collection-point tagging, settled above, needs no `Span`
  change.)*

## New decision records this increment adds

  **collection-point file-attributed** `diagnose_project`, distinct from `compile_project`;

  span-id idea is unneeded.)*

## Out of scope (→ later)

- **Code actions** — now unblocked for the `given`-clause catalogue — **return after this**,

- **Incremental recompute**; all other A-tier features (completion, signature help, inlay
  hints, semantic tokens).

## Definition of done

`diagnose_project` lands (non-bailing, overlay-aware, **collection-point file-attributed**)
with the `compile_project` analyse/emit split leaving the CLI build byte-stable; a multi-file
`karnc` test proves a **context-file** diagnostic is produced and attributed; the LSP
publish/clear **diff is a unit-tested pure function** and positions convert against the
analysed snapshot; *(rider)* project-level CLI errors render with `ariadne` source context;
`design/karn-lsp-spec.md` updated; decision record added; changelog + LSP-spec delta land
together; `cargo test`, `clippy -D warnings`, `fmt --check` green. The implementing PR
**deletes this file**.
