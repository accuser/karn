# v0.75 — Contexts & capabilities as Bynk structure: the handler frame reads in Bynk (semantic-debugging track, slice 2)

- **Track:** [semantic-debugging](../tracks/semantic-debugging.md), slice 2. Slice 1
  made *values* read in Bynk; this makes the *handler frame's shape* read in Bynk —
  capabilities and agent state as named Bynk groups, not a flat JS local dump.
- **Realises:** ADR [0105](../decisions/0105-semantic-debug-interposition.md) — the
  same editor-side `DebugAdapterTracker`, now rewriting `variables`/`scopes` *structure*
  (not just value strings). Exercises D4 (own the `variablesReference` graph) and D5
  (inference vs emitter debug-metadata — the line this slice draws).
- **Posture:** tooling (VS Code extension). Inference-first → **no emitter change** this
  slice (the actor, which needs metadata, is the named follow-on).

## The gap

Stopped in a handler, the Variables pane is a flat list of *emitted* locals. From a
real agent handler (`bynkc compile`, `demo.counter`):

```ts
async increment(deps: {}): Promise<Result<number, CounterError>> {
  const currentState = await this.loadState();   // the agent's Bynk state
  const next = currentState.count + 1;
  await this.commitState({ ...currentState, count: next });
  return Ok(next);
}
```

so the pane shows `deps`, `currentState`, `next`, plus `this` and compiler temps — JS
names and machinery, not Bynk structure. Slice 2 regroups the recognisable ones into
Bynk vocabulary: **`deps` → a `Capabilities` group**, **`currentState` → a `State`
group** (agents). A service handler's `deps` (e.g. `{ Logger, Kv }`) likewise reads as
`Capabilities`.

## What's inferable — and what isn't (the D5 line)

Grounded in the emitted output:

| Bynk concept | Emitted shape | Recognition |
|---|---|---|
| Capabilities | the `deps` parameter object | **inferable** — fixed emitter name `deps` |
| Agent state | `currentState` (from `loadState()`) / `this.state` | **inferable-ish** — emitter name; generic |
| The `by` actor | `v: Visitor` — *often elided* when the body doesn't use it | **not reliable** — needs emitter metadata |
| Compiler temps | `__r0`, `__d`, … | inferable (`__`-prefix) — but that's **slice 4** |

So per ADR 0105 D5, slice 2 ships the **inferable** structure (capabilities, state) with
**no emitter change**, and **defers the actor** — it isn't dependably a local, so it is
the case that finally earns the emitter **debug-metadata sidecar** (a future slice).

## The one genuine uncertainty — regroup synchronously (spike-first)

The rewrite is synchronous (ADR 0105 D2/D4), and that bounds *how* we regroup:

- **True synthetic scopes are out of reach cheaply.** Adding a `Capabilities` *scope*
  (a row in the Scopes pane) means injecting a `Scope` into the `scopes` response with a
  `variablesReference` — but `deps`'s real reference only appears later, in the Local
  `variables` response, so we can't synthesise the scope synchronously without managing
  a virtual reference graph and faking `variables` requests for it. That's a separate,
  heavier mechanism.
- **So v1 regroups *within* the Local scope's `variables` response** — relabel the
  recognised locals (`deps` → `Capabilities`, `currentState` → `State`), which are
  already expandable objects, so they read as Bynk groups. This is synchronous, cheap,
  and inference-only.

The spike confirms the relabel rewrite reads well (and orders sensibly via
`presentationHint`), survives child-variable expansion, and holds on both runtimes —
and weighs whether synthetic scopes add enough to justify the virtual-reference machinery
(recommend: not in v1).

## What this slice does

1. **Extend the interposer** (`src/debug.ts` + `src/semanticValues.ts`): in
   `onDidSendMessage`, rewrite the Local-scope `variables` response — relabel `deps` →
   `Capabilities` and the agent state local → `State`, set `presentationHint` so they
   sort to the top, and leave every other local (user bindings, request params)
   untouched. Recognition is by the emitter's fixed names, behind the same
   `bynk.debug.semanticValues` toggle and `__bynkChild` binding.
2. **Both runtimes** — same editor-side path (slice 1's machinery).
3. **Tests:** a Node integration test (stop in an agent handler; assert the pane shows a
   `Capabilities` and a `State` group, user bindings intact) and the parser/relabel unit
   coverage; workerd local/opt-in. The slice-0 mutation guard and slice-1 value tests
   stay green.
4. **Docs:** a short "the handler frame in Bynk" note in the debugging guide; changelog
   row; the bump.

## Decisions

**[DECISION A] Regroup via the interposer, per ADR 0105.** Reuse slice 1's tracker; this
slice rewrites *structure* (labels/grouping) where slice 1 rewrote value strings. No new
mechanism.

**[DECISION B — recommend] Inference-only this slice; the actor + a debug-metadata
sidecar are the named follow-on.** Capabilities and state are recognisable from the
emitter's fixed names (`deps`, `currentState`), so they ship with **no compiler change**.
The `by` actor is not reliably a local (elided when unused), so surfacing it needs an
emitter **debug-metadata** sidecar — the first real D5 "metadata earns its cost" case,
deferred to its own slice. *Recommend this split* — it keeps slice 2 cheap and within
inference, and lets the sidecar be designed once, for the actor and robustness together.

**[DECISION C] Relabel within Local, not synthetic scopes.** The synchronous constraint
makes true Scopes-pane rows expensive (virtual-reference management); relabelling the
already-expandable `deps`/state locals reads as Bynk groups for a fraction of the cost.
Synthetic scopes stay a spike-gated future option, not v1.

**[DECISION D] Temps stay visible (slice 4).** Hiding `__r0`/`__d` is the lowered-temp
slice; this slice only *adds* Bynk grouping, it doesn't yet *remove* noise.

**[DECISION E] Version — minor v0.75.** A new extension capability; no language change.

## Risks & mitigations

- **Name-inference fragility.** Recognising capabilities/state by emitted name (`deps`,
  `currentState`) breaks if a user binding shadows the name or the emitter renames.
  Mitigated by keying on the *exact* emitted names + position (a handler parameter named
  `deps`), unit-tested; the debug-metadata sidecar (DECISION B) is the robust upgrade.
- **`variablesReference` expansion.** Relabelling the top-level local must not disturb its
  children (expanding `Capabilities` still shows `Logger`, `Kv`). The rewrite only
  changes the `name`/`presentationHint`, never the `variablesReference`. Covered by the
  expansion test.
- **Over-reaching the regroup.** Only relabel locals we positively recognise; everything
  else is byte-identical (the slice-1 totality discipline, now for structure).
