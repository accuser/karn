# v0.101 — streaming HTTP response: a `Streamed` `HttpResult` shape over `Stream[T]` (real-time track, slice 1)

- **Scope:** a single increment — a **streamed response body** on the existing
  `from http` protocol, consuming the `Stream[T]` primitive (v0.100) and lowering
  to an SSE (`text/event-stream`) `Response`. **Language + checker + emitter +
  runtime.** No socket, no `Connection`, no Durable Object — those are the
  connection leg (slices 2–4).
- **Realises:** [`design/tracks/websocket.md`](../tracks/websocket.md) §3.3 and
  **Slice 1** of its §6 decomposition — "the early payoff: incremental responses
  with no socket, no DO; stands alone." Settles the track's **Q2** (the streaming
  return shape, reconciled with ADR 0126).
- **Builds on:** [ADR 0128](../decisions/0128-stream-value-over-time-primitive.md)
  (`Stream[T]` — the value-over-time primitive this consumes) and
  [ADR 0126](../decisions/0126-httpresult-rfc9110-status-vocabulary.md) (the
  `HttpResult` **payload-shape registry** this extends with one shape, exactly as
  0126 D2 added `Location`).
- **Authorises its own ADR:** the streaming-HTTP response contract (the `Streamed`
  shape, the 200-only status, the SSE framing). **Number assigned when the PR is
  cut** — the next free number (provisionally 0129), per the track's
  no-reservation rule.

## 1. What this is — the early payoff

`Stream[T]` (slice 0) is settled but not yet useful end-to-end: it constructs in
memory and drains to a list, with no consumer. This slice gives it its first
real one — a `from http` handler that returns a **streamed body** — and with it
the first runnable "type Bynk, get an incremental response" recipe, on the
**existing protocol, existing actors, existing topology**, with no connection
lifecycle and no Durable Object. It is the slice that proves the primitive
carries weight before the heavy connection leg begins, and the seam a future
streaming `Ai` capability plugs into.

A handler streams by returning a new `HttpResult` outcome that carries a
`Stream[String]` instead of a buffered value:

```bynk
service Feed from http {
  on GET("/ticks") -> Effect[HttpResult[()]] {
    Streaming(Stream.of(["tick-1", "tick-2", "tick-3"]).take(3))
  }
}
```

It lowers to a `Response` whose body is the stream, **SSE-framed**
(`data: tick-1\n\n…`), content-type `text/event-stream`.

## 2. The model — a `Streamed` payload shape, mirroring `Location`

ADR 0126 made `HttpResult` a registry of status-bearing variants over a closed set
of payload **shapes** (`Value`/`Message`/`None`/`Location`,
`bynk-syntax/src/ast.rs:798`). A redirect realised its shape (`Location`) as
**dedicated variant rows** (the five 3xx variants), each with a fixed status —
the shape is a property of the *variant*, not a mode the caller selects. Streaming
follows that precedent exactly:

- a new **`HttpVariantPayload::Streamed`** arm (`ast.rs:798`), carrying a
  `Stream[String]`;
- one new registry row in **`HTTP_VARIANTS`** (`ast.rs:823`): **`Streaming`** (status
  **200**, shape `Streamed`).

Per ADR 0126 D3, the registry is the single source of truth; the resolver, LSP
completion, and emitter dispatch pick the row up for free. The only exhaustive
matches that extend — one arm each, mirroring `Location` — are:

1. **construction-checking** — `check_http_variant`
   (`bynk-check/src/checker/expressions.rs:1336`): a `Streamed` arm expects **1
   argument of type `Stream[String]`** and returns `HttpResult[()]` (like
   `Location`/`Message`/`None`), validated against the declared element type;
2. **pattern-binding** — `variants_of` (`bynk-check/src/checker.rs:2312`): bind the
   payload as `stream: Stream[String]`;
3. **the runtime status map** — `httpResultToResponse`
   (`bynk-emit/runtime/src/http.ts:144` **and** its emitter-bundled twin
   `bynk-emit/src/emitter/runtime.ts:353`): a `Streaming` case returning the SSE
   `Response` (§4).

No other site changes — `validate_http_handler` already requires
`Effect[HttpResult[T]]` (`validate.rs:2500`) and a streaming handler satisfies it
with `T = ()`.

## 3. Decisions

**[DECISION A — the load-bearing call] A `Streamed` payload shape realised by one
dedicated `Streaming` (200) variant — not a polymorphic `Ok`, not a status-less
peer.** Recommended, and it follows 0126 D2's `Location` precedent (a shape
realised by dedicated status-bearing rows). The alternative — making `Ok`
carry *either* a `Value` or a `Stream` — breaks 0126's one-variant-one-shape
model and the registry-driven dispatch that rests on it. The slice-1 ADR records
this.

**[DECISION B — the status reality that sharpens the track's Q2] Streamed
responses are **200-only** at v1.** The track's Q2 speculated "a streamed `200`
*and a streamed error* are both real." At the HTTP level they are not symmetric:
**a response commits its status line and headers before the first body chunk**, so
once streaming begins the status is already `200` and cannot change. The two real
failure modes are handled where they actually occur:

- **pre-stream failure** (auth, validation, not-found) — return an ordinary
  non-streamed variant *instead* of `Streaming`. These are `None`/`Message` shape,
  i.e. `HttpResult[()]`, so they sit in the **same handler** alongside `Streaming`
  with no type conflict (e.g. `Unauthorized` / `NotFound(msg)` on the failure
  branch, `Streaming(events)` on success);
- **mid-stream failure** — an **in-band `Result` element** in the stream
  (`Stream[Result[String, E]]`), the outcome-vs-fault split slice 0 settled (ADR
  0128 D4); the HTTP status is already sent.

So one `Streaming` (200) variant is the complete v1 surface. Adding a streamed
`202 Accepted` later is one registry row — named here so a later slice adds it
behind its own ADR rather than rediscovering the boundary. *Recommended; this is
a deliberate narrowing of the track's Q2, justified by how HTTP actually frames a
response.*

**[DECISION C — chunk type & framing] `Stream[String]`, each element framed as one
SSE `data:` event by the runtime.** Each stream element is one event, framed
`data: <element>\n\n`. A structured `SseEvent` record (named `event`/`id`/`retry`
fields, multi-line `data`) is a **named follow-on**, not v1. *Recommended:* a
string element is the ergonomic minimum and is wire-encodable, so it sits squarely
inside slice 0's "only `Stream[Chunk]`" scope (ADR 0128 D5) and does **not** force
the deferred element-semantics call. Multi-line elements are split into multiple
`data:` lines by the framer; the framer is the one place SSE detail lives.

**[DECISION D — framing lives in the runtime, not inline] The SSE framing is in
`httpResultToResponse` (the http runtime helper), not emitted inline.** Unlike the
`Stream` *combinators* (slice 0, emitted inline so non-stream files stay
byte-identical), the `Response`-construction code already lives in the runtime and
is already imported by every http-handling file. So the `Streaming` case — wrap
the `AsyncIterable<string>` in a `ReadableStream<Uint8Array>` that encodes
`data: …\n\n` per chunk — belongs beside the other shape cases. *Recommended;* it
keeps the streaming logic in one tested place and the emitter unchanged
(`workers_entry.rs:546` already calls `httpResultToResponse` uniformly).

**[DECISION E — SSE headers] `content-type: text/event-stream` and
`cache-control: no-cache`.** The minimal correct SSE header set. *Recommended;*
`connection: keep-alive` is HTTP/1.1-specific and platform-managed (Workers/Node
set transport framing themselves), so it is omitted.

## 4. Lowering & runtime

- `Streaming(s)` lowers through `httpResultToResponse` to a streaming `Response`:
  the `Stream[String]` (an `AsyncIterable<string>`, ADR 0128 D6) is wrapped in a
  `ReadableStream<Uint8Array>` that, per element, enqueues
  `TextEncoder().encode("data: " + chunk + "\n\n")`; the `Response` carries status
  `200` and the §3-E headers. A small helper (`sseResponse(stream)`) holds the
  framing so both runtime copies (`runtime/src/http.ts`,
  `emitter/runtime.ts`) share it.
- **Both targets.** A `Response` with a `ReadableStream` body is a Web standard, so
  the same lowering runs on **Workers** and **Bundle/Node** — no DO, no
  platform-lock. On the test target the body is drainable, so a fixture asserts the
  exact SSE frames (`data: tick-1\n\n…`) — the deterministic `Stream.of`→framing
  path makes streamed output assertable, the early-payoff testability the track
  promised.
- Strip-only clean — the helper is plain runtime TypeScript, no type-directed
  constructs; the emitter is unchanged.

## 7. Tests

- **Positive fixture** (`bynkc/tests/fixtures/positive/`, project-form with a
  service): a `from http` handler returning `Streaming(Stream.of([...]).take(n))`;
  assert the emitted `Response` carries `text/event-stream` and the drained body is
  the exact `data: …\n\n` frame sequence. A second handler mixing `Streaming` (200)
  with a pre-stream `NotFound`/`Unauthorized` branch (D-B), proving they share
  `HttpResult[()]` with no type conflict.
- **A `Stream[Result[String, E]]` streaming fixture** exercising the in-band
  mid-stream error path (D-B).
- **Negative fixtures:** `Streaming` with a non-`Stream` argument
  (`bynk.types.argument_mismatch`); `Streaming` constructed with the wrong arity.
- **Runtime behaviour:** the SSE framing helper runs under `node` (mirroring the
  `kv_behaviour`/`json_codec_behaviour` runtime tests) — a streamed handler's body
  reads back as the expected frames; `tsc --strict` clean.

## 8. Risks & mitigations

- **Status commits before the body (the HTTP reality behind D-B).** A handler
  cannot change its mind to an error status once a chunk is sent. *Mitigation:* the
  200-only design makes this structural — failures are either pre-stream variants
  or in-band `Result` elements; the guide states it plainly.
- **Unbounded stream = a long-lived open response (a resource/DoS vector, track
  §4).** *Mitigation:* the language tool is bounded `take` (slice 0); per-response
  duration is bounded by the platform's request limits. Named, not silently
  omitted.
- **Body-type portability across Workers/Node.** `Response(ReadableStream)` is a Web
  standard, but the chunk type must be `Uint8Array`. *Mitigation:* the runtime
  helper encodes with `TextEncoder`; the behaviour test proves both targets.
- **SSE multi-line / special data.** A chunk containing `\n` must become multiple
  `data:` lines. *Mitigation:* the framer splits on `\n`; covered by a fixture
  chunk with an embedded newline.
- **Scope creep toward a structured event type.** *Mitigation:* `SseEvent`
  (event/id/retry) is an explicit follow-on (D-C), not v1.
