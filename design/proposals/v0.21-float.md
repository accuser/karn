# v0.21 — `Float`

- **Status:** Draft (proposal). **Merging this PR is the approval to build.**
- **Builds on (verified):** `BaseType` is `Int`/`String`/`Bool` only (`ast.rs:726`) — `Float`
  is net-new; only `IntLit` is lexed (`lexer.rs:165`) — `FloatLit` is net-new; `Int` division
  **already truncates** (`emitter.rs:3505`, `Math.trunc(l / r)`), and the **`expr_types`
  span-keyed side table already hands the emitter operand types** (`emitter.rs:3767`, the
  v0.20b kernel path), so **operand-typed division is a small change, not new plumbing**; the
  boundary codec's `Number.isInteger` admission (`emitter.rs:1565`) is what the `Float`
  variant drops; the refinement predicates are hard-coded `Int`-only today (`checker.rs:572`)
  and extend to `Float`.
- **Spec sections touched:** §3 (lexical), §4 (grammar), §5 (static semantics), §6 (type
  system), §7 (emission), §9 (diagnostics), Appendix B.
- **A language-core increment of its own** (0023 / the Float-first re-plan). It exists so the
  wider stdlib (v0.22) can model decimal data and the typed JSON codec can represent
  non-integer numbers; the stdlib itself does **not** ship here.

## Goal

Add `Float` as a distinct base type so Karn can express decimal data — prices,
measurements, ratios, coordinates, percentages — which `Int`-as-cents can't. This is the
dependency v0.22's JSON codec needs; building it first keeps that codec from shipping
visibly crippled.

## Settled (recorded as ADRs ≥0040)

- **▸ `Float` is a distinct base type**, not a refinement of `Int`. Both `Int` and `Float`
  lower to TS `number` — the distinction is **Karn-side only**, erased at runtime (like
  generics). Boundary `serialise_`/`deserialise_` handle `Float` → `number`, accepting any
  **finite** JS number including decimals (non-finite rejected on decode — see below; this is
  the v0.22 unblock).
- **▸ No implicit `Int`↔`Float` coercion.** Mixing is a static error; conversion is explicit
  (Karn's strict-by-default posture — silent widening is exactly the surprise to avoid).

## Decisions (open — settle at review)

**[DECISION] Conversion dispatch — fix the numeric-kernel rule now (v0.22 copies it).**
Resolution already has two paths: **value methods** on a receiver (instance dispatch,
`checker.rs:4236`) and **statics** (`List.empty()` / `T.of`, `check_static_call`). *Rec:*
**value methods for operations on a value** — `i.toFloat()`, and `Float -> Int` via named
lossy conversions `f.truncate()` / `f.round()` / `f.floor()` / `f.ceil()` (each `-> Int`); no
ambiguous `toInt`. **Statics reserved for constructors/parsers** — `Float.parse(s) ->
Option[Float]` (no receiver), extending the built-in-statics path that already serves
`List.empty()`. This slots into existing dispatch with no *base-type static op* machinery
beyond what parsers need, reads consistently with the collection kernel (0037), and —
thanks to digit-both-sides literals — `2.5.round()` / `1.toFloat()` lex unambiguously.

**[DECISION] Division is operand-typed.** *Rec:* `Int / Int` → `Math.trunc(a / b)`
(unchanged); `Float / Float` → bare `a / b` (true division). Arithmetic never mixes `Int`
and `Float` operands without an explicit conversion (`karn.types.no_numeric_coercion`).

**[DECISION] Float equality.** *Rec:* allow `==`/`!=` on `Float` for consistency, but
**document the footgun** normatively — both `0.1 + 0.2 != 0.3` **and `NaN != NaN`** (since
`==` lowers to `===`, a `NaN` is unequal to itself — the sharper surprise). No built-in
`approxEq` here — a `Float.approxEq(a, b, eps)` stdlib helper can follow in v0.22.

**[DECISION] Float literals.** *Rec:* `FloatLit` = digits `.` digits (`1.0`, `0.5`) **or** an
exponent (`1e10`, `1.5e-3`); **digit required on both sides** of `.` (reject `1.`/`.5` →
`karn.parse.malformed_float_literal`). `1` stays `Int`; `1.0`/`1e10` are `Float`. **Strictly
additive** — `1e10` currently lexes as `IntLit(1)` + `Ident(e10)` (never valid downstream), so
claiming exponents takes no working syntax; logos priority must keep bare `[0-9]+` as `IntLit`,
and `1e-3` is unambiguous under maximal munch. **Reject overflow** — `1e999` (→ `Infinity` at
parse) is `karn.parse.float_literal_overflow`, mirroring `IntLit`'s out-of-range reject
(`lexer.rs:423`), not a silent non-finite literal. **Store the lexeme** (not a parsed `f64`)
so emission is byte-stable — `f64` `Display` normalises `1e10` → `10000000000`, which is
fixture-visible.

**[DECISION] Refinement over `Float` — a bigger fan-out than it reads.** `type Probability =
Float where InRange(0.0, 1.0)`, and `Float where Positive` (exactly what prices want). It
touches: `PredKind::InRange` is `i64`-typed (`ast.rs:757`) and `ConstLit` needs a `Float`
variant; admission (`checker.rs:401`), bound-inversion (512), predicate equality (5332), range
reasoning (5502 / `emitter.rs:3905`), and the runtime check (`emitter.rs:4682`) — which must
**not** emit the `Number.isInteger` term for a `Float`. (No `Eq`/`Hash` pain: `PredKind` derives
only `Debug`/`Clone`, so `f64` bounds are fine.) Two sub-decisions: **(a)** bound literals must
**match the base type** — `Float where InRange(0, 1)` is an error (reuse `no_numeric_coercion`,
consistent with no-coercion); **(b)** `NonNegative`/`Positive` **extend to `Float`** (today
hard-coded `Int`-only at `checker.rs:572`) — yes, trivially.

**[DECISION] Non-finite — arithmetic and the boundary are different surfaces.** *In-language
arithmetic:* division-by-zero / overflow follow the **host** (`Infinity`/`NaN`), no Karn guard
in v0.21 (as `Int` div-by-zero already does); document it (`isNaN`/`isFinite`/checked helpers
are v0.22). *At the boundary it is **not** safe to leave host-defined:* `JSON.parse("1e999")`
yields `Infinity`, so `deserialise_Float` must require **`Number.isFinite`** (boundary values
are finite) — it swaps the `Int` `isInteger` check (1565) for `isFinite`, not nothing. And
`JSON.stringify(NaN)` produces `null`, breaking round-trip — one normative sentence: serialising
a non-finite `Float` is a contract violation (reject at the boundary).

## Diagnostics (new — `karn.*`)

| Code | Cause |
|---|---|
| `karn.types.no_numeric_coercion` | `Int`/`Float` mixed in an operation — *or* a refinement bound literal not matching the base type (`Float where InRange(0, 1)`) — without explicit conversion |
| `karn.parse.malformed_float_literal` | a float literal missing a digit on either side of `.` (`1.`, `.5`) |
| `karn.parse.float_literal_overflow` | a float literal that parses to a non-finite value (`1e999`) |

## Fixtures (shape — not normative emission)

- **Positive** — `Float` arithmetic incl. **true division** (`5.0 / 2.0 == 2.5`); `Int`
  truncation **unchanged** (`5 / 2 == 2`); conversions (`i.toFloat()`, `f.round()`); a
  refined `Price = Float where Positive` built via `.of`; **decimal boundary
  round-trip** (`serialise_`/`deserialise_` of a `Float` field through `19.99`). Each passes
  `tsc --strict`.
- **Negative** — `1 + 2.0` (mixed) → `no_numeric_coercion`; `Float where InRange(0, 1)`
  (mismatched bound literal) → `no_numeric_coercion`; `1.` / `.5` → `malformed_float_literal`;
  `1e999` → `float_literal_overflow`; **`deserialise_Float` of `1e999` from the wire** →
  rejected (`isFinite`); a refined-`Float` literal outside its range → the existing
  refinement-admission diagnostic.
- **Additive guard** — all prior fixtures byte-stable; `Int`-only programs unchanged
  (division still `Math.trunc`).

## Risks

- **Division regression.** The operand-typed `Div` must keep `Int /` truncating; a careless
  refactor that drops `Math.trunc` is exactly the bless-without-reading trap — pin `5 / 2 ==
  2` as a guard.
- **Erased distinction.** `Int` and `Float` both being `number` means the *checker* is the
  only thing keeping them apart; ensure no path silently treats a `Float` as `Int` (or the
  reverse) without a conversion.
- **`NaN`/`Infinity` semantics** — host-defined *in arithmetic*, documented; but the
  **boundary** is not safe to leave host-defined (`isFinite` on decode, non-finite serialise
  is a contract violation) — the easiest thing to get subtly wrong.
- **Refinement on `Float`** — float-bound admission reuses the `Int`/`String` refinement
  path; confirm float comparisons in predicates behave (and the equality footgun doesn't
  leak into `where ==` bounds surprisingly).

## New decision records this increment adds (≥0040)

- 0040 — `Float` a distinct base type, erased to `number`; boundary codec accepts decimals
  but requires `isFinite` (non-finite from the wire is rejected, not admitted).
- 0041 — no implicit `Int`↔`Float` coercion; **conversions are value methods (`i.toFloat()`,
  `f.round()`), parsers are statics (`Float.parse`)** — the numeric-kernel dispatch rule v0.22 copies.
- 0042 — operand-typed division (`Int` truncates, `Float` true-divides).
- 0043 — float literals (fraction/exponent, digit-both-sides, reject overflow, store lexeme).
- 0044 — refinement over `Float`: `Float` bounds; `NonNegative`/`Positive` extend; bound
  literals must match the base type.

## Out of scope (→ later)

- **The wider stdlib** (String/`Int`/`Option`/`Result`/typed-JSON) → **v0.22**, now
  decimal-complete.
- `Float.parse`, `Float.approxEq`, `isNaN`/`isFinite`, checked division → stdlib helpers, v0.22+
  (v0.21 sets the statics *rule*; v0.22 ships the parser).
- Arbitrary-precision / decimal / bignum money types → not planned; `Int`-cents and `Float`
  cover the near term.
