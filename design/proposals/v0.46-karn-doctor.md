# v0.46 — `karn doctor`: a first-class environment check, and the `karn` driver it stands up

- **Status:** Draft for sign-off. The four design forks below are marked
  `[DECISION]` with recommendations. **Merging is approval to build.** *(Tooling
  increment — no language surface, no grammar/checker/emitter change. The first
  step of a project-lifecycle wrapper arc — `doctor` → `new` → `dev` — that this
  increment **introduces**: today `design/karn-tooling-roadmap.md` only gestures
  at it, in B‑2's "Commands / scaffolding — new project" line (§4 is the VS Code
  extension roadmap, not a driver home). `doctor` is chosen to go first precisely
  because it touches nothing in the compiler and carries no spec delta, so it is
  the safe place to stand up the driver the later commands need.)*
- **Independent of the actors track.** This is a tooling slice and does not
  sequence after v0.44/v0.45; it can land in parallel. The version is
  **provisional** — renumber to the next free version at scheduling.
- **Lands one architectural ADR** (introduce the `karn` driver) plus one
  small contract ADR (the doctor output / exit-code contract).

## Context — make the environment check a command, not a surprise

Karn already depends on a Node-side toolchain it discovers *ad hoc, at the
moment of failure*. `karnc test` walks a runner ladder — `tsc`, then
`npx --yes -p typescript tsc`, then `tsx`, then `npx --yes tsx` — guarded by a
`which`-based `tool_exists`, and only if every rung fails does it print an
actionable "install one of…" message (`karnc/src/main.rs`). The hello-world
README then asks the reader to find `node`, `npx wrangler dev`, and a Cloudflare
login on their own. The dependencies are real and knowable, but a newcomer meets
them one broken command at a time.

`karn doctor` promotes that scattered, reactive detection into a single
**upfront, declarative** check: *given what you want to do with Karn, is your
machine ready, and if not, exactly what do you run to fix it?* It is the
cheapest possible win in the lifecycle arc — pure inspection, no machine
mutation, no language surface — and it is the natural first command of the
`karn` driver that `new` and `dev` will also live in.

## Builds on (verified)

- **The existing detection primitive.** `tool_exists(name)` (`main.rs`) already
  shells `which` and reports presence; `doctor`'s probe is a generalisation of
  it (presence + version + provenance), and the two should share one
  implementation rather than diverge.
- **The known runner ladder.** `karnc test`'s `tsc`/`npx tsc`/`tsx`/`npx tsx`
  sequence (`main.rs`) is the authoritative list of what "can run Karn's emitted
  TypeScript" means today. `doctor`'s `test`-readiness check mirrors that ladder
  exactly, so the two cannot disagree about what counts as satisfied.
- **The Node floor, asserted in two Rust doc comments.** "Node.js (≥ 18)" lives
  as prose at `cli.rs:58` (`CliPlatform::Node`) and `firstparty.rs` (~335, the
  `KARN_NODE_BINDING` doc) — **not** in either README (both say only "node +
  tsc", with no floor). `doctor` reads that floor from **one** constant that both
  doc comments then reference, so the increment *replaces* the duplication rather
  than adding a third reader of free prose.
- **The existing actionable-error voice.** `karnc test`'s failure text already
  prints concrete `npm install -g …` remedies. `doctor` reuses that tone and,
  where possible, the exact strings, so the remedy a user sees from `doctor`
  matches the one they'd see from a failed command.
- **The project-root discovery.** `karn.toml` discovery and the `[paths]`
  manifest already exist (`karnc/src/project/…`, the `cli_rooting` fixture).
  `doctor` reuses it to become **project-aware** when run inside a project (see
  the scope decision).

## Decisions (forks for sign-off)

### [DECISION] Where `doctor` lives — **recommend: introduce the `karn` driver crate, with `doctor` as its first command.**

There is no `karn` binary today; the workspace publishes `karnc`, `karn-fmt`,
`karn-grammar`, `karn-lsp`. Two ways to ship `doctor`:

- **A — `karnc doctor` subcommand.** Cheapest: one new arm on the existing clap
  tree, no new crate, no new release artefact. But it cements environment
  orchestration *inside the compiler*, which the project has deliberately kept
  pure (compile / check / fmt / test are all compiler operations; "is `wrangler`
  installed" is not). And it strands the arc: `new` and `dev` are orchestration,
  not compilation, and want a `karn` home — shipping `doctor` as `karnc doctor`
  now buys a rename/deprecation later.
- **B — a new `karn` driver crate (recommended).** `karn` is to `karnc` what
  `cargo` is to `rustc`: a thin orchestrator that shells the compiler and the
  Node toolchain. `doctor` is the *ideal* first command to stand it up — it has
  no language surface, mutates nothing, and exercises exactly the muscles the
  driver needs (locate `karnc`, locate `karn.toml`, shell out, report). The cost
  is real but one-time: a new published binary, its release-plumbing row, and a
  decision on how the driver finds `karnc` (PATH, then a sibling-of-`karn`
  fallback — mirrors how `vscode-karn` resolves `karnc-lsp`). Fork B also
  *creates* a truth `doctor` must then tell — driver↔compiler version skew (see
  the contract decision below); fork A cannot drift, so it sidesteps that, which
  is the one honest point in A's favour.

Recommendation **B**. The whole point of this thread is the lifecycle wrapper;
`doctor` is the low-risk increment that pays for the driver's foundation before
`dev` (the expensive, watch-bearing command) needs it. If sign-off prefers to
defer the crate, fall back to **A** with an explicit note that `karn` will
re-home it — but the recommendation is to pay the foundation cost here, where
the blast radius is smallest.

### [DECISION] What `doctor` checks, and what "missing" costs — **recommend: capability-grouped checks; hard-fail only on what blocks an action you could take today.**

Group probes by the capability they unlock, not as a flat list:

- **Compile / check / fmt** — needs only `karnc` itself (you have it if `karn`
  found it). Always satisfiable; `doctor` reports the resolved `karnc` path +
  version, and **flags driver↔compiler version skew** — once `karn` is a
  separate binary resolving `karnc` over PATH/sibling, a global `karn 0.46` can
  shell a stale `karnc 0.44`. A mismatch is at least a warning (with the resolved
  paths shown); a major/contract mismatch is an error. This is the one surprise
  *unique to fork B*, and silencing it would defeat the command's whole purpose.
- **`karn test`** — needs Node **and** one of (`tsc` | `tsx`), matching the
  runner ladder. Missing ⇒ this capability is **unavailable**.
- **`dev` / deploy (Cloudflare)** — needs Node and `wrangler` (today via `npx`).
  Missing ⇒ this capability is **unavailable**.
- **Editor (optional)** — `karnc-lsp` on PATH. Missing ⇒ **note**, never a
  failure.
- **Build-from-source (optional, contributors)** — a Rust toolchain ≥ MSRV.
  Reported only when relevant (e.g. inside the Karn repo), never a failure for
  end users.

Checks report **presence + version + provenance** (global PATH vs
project-local `node_modules/.bin` vs npx-fetch-on-demand), and version is
compared against a known floor where one exists (Node ≥ 18).

The exit-code contract turns on *what an invocation asks about* — and the bare
invocation must be pinned, because it is what gets golden-tested:

- **Bare `karn doctor`** is **informational**. It surveys every capability and
  prints the full report, but the only thing it treats as *required* is the
  compile floor (`karnc` itself resolvable and not skewed). It exits **0** even
  with `test`/`dev` unavailable — a compile-only user is healthy, and is not
  failed for a missing `wrangler`. It exits non-zero only if the compile floor is
  broken.
- **A capability filter — `karn doctor --only <capability>`** (e.g. `--only
  test`, `--only deploy`) — is what *promotes* that capability's tools to
  required: `--only deploy` on a machine without `wrangler` exits non-zero. (Spelt
  `--only <cap>` rather than a bare `karn doctor test` positional, to avoid
  colliding with the `test` verb that already means something next to `karnc
  test`.)
- **`--strict`** promotes *all* warnings — including optional gaps (no
  `karnc-lsp`) and version skew — to failures, for an all-green CI gate.

So the two readings are reconciled explicitly: bare is informational (the test
matrix's "compile-only ⇒ exit 0 with test/dev flagged" case), filters and
`--strict` are how you ask for a hard gate. Rejected: making bare `doctor` a flat
pass/fail over every tool — it punishes a compile-only user for a missing
`wrangler` and trains people to ignore the output; and leaving the bare exit code
unspecified — it *is* the scriptable surface, so it cannot be implicit.

### [DECISION] `doctor` reports; it does not install — **recommend: no `--fix` in this slice; print the exact remedy instead.**

`doctor` is diagnostic. For each gap it prints the precise remedy (the same
`npm install -g …` / `npx …` strings the toolchain already emits), but it does
**not** run them. Auto-install (`--fix`) is reserved-and-noted, not built:
mutating a user's global environment is a different trust level and a different
review bar, and it would couple a should-be-portable check to package-manager
specifics. Rejected for now: shipping `--fix` here — it widens the blast radius
of the arc's *first* command for marginal convenience.

Output contract (small contract ADR): human-readable grouped table by default;
`--format short` (one `capability: ok|missing (remedy)` line per row, mirroring
`karnc check --format short`, ADR 0071) and `--format json` for scripting/CI. The
ADR also records the bare-vs-`--only`-vs-`--strict` exit semantics above — they
are the same scriptable contract.

### [DECISION] Cross-platform from the start — **recommend: portable detection via the `which` crate, not Unix-only.**

Today's `tool_exists` shells the POSIX `which`, and its own comment concedes "the
rest of the toolchain has been Unix-only." That is defensible for an *internal*
helper, but `doctor`'s entire job is to tell the truth about the environment — a
truth-teller that is silently blind on Windows is a contradiction the proposal
itself names. So this is a real fork, not a deferral:

- **Unix-only now** — keep shelling `which`; document the limitation. Cheapest,
  but ships the contradiction and buys a migration the moment a Windows user runs
  it.
- **Portable now (recommended)** — back the shared probe with the `which` crate,
  which handles `PATHEXT`/`where` resolution. The cost is one small dependency
  paid *once*, at the driver's foundation, before `new`/`dev` inherit the same
  probe — exactly where a one-time portability cost belongs.

Recommendation: **portable now**. The driver is new code with no Unix-only legacy
to preserve, and `doctor` is the wrong command to ship half-blind.

## Spec delta

**None.** `karn doctor` adds no grammar, no static-semantics rule, no emission,
and no diagnostic in the `karn.*` compiler family — it inspects the host
environment and shells existing tools. This is a deliberate no-op on
`docs/src/spec/`, recorded here per the proposal-README rule that silence on the
spec is treated as an oversight: there is genuinely no normative language
surface to define.

## Tests

- **Driver unit tests** (the shared probe): a tool present on PATH; absent; a
  version below floor (Node < 18) vs at/above; project-local resolution
  preferred over global when a `karn.toml` + `node_modules/.bin` exist;
  npx-fetch-on-demand reported as *provisionable*, not *present* (it must not be
  a false "ok").
- **Capability/exit-code matrix** (the pinned contract): **bare** compile-only
  env ⇒ exit **0** with `test`/`dev` flagged unavailable; everything present ⇒
  exit 0, all green; `--only deploy` on a `wrangler`-less machine ⇒ non-zero;
  `--strict` turns an optional-only gap (and version skew) from 0 to non-zero;
  a broken compile floor (`karnc` unresolvable) ⇒ non-zero even bare.
- **Output golden:** stable `--format short` and `--format json` shapes
  (these are the scriptable contract — pin them); the default human table need
  only be smoke-tested.
- **Driver→compiler seam:** `karn doctor` locates `karnc` via PATH and via the
  sibling fallback; reports its version; and a deliberately skewed pair
  (`karn`≠`karnc`) is flagged (warning by default, failure under `--strict`).
- **Cross-platform probe:** the shared detection resolves a present tool and a
  `PATHEXT`-style executable on the non-Unix path (per the portability decision),
  not only POSIX `which`.
- **No fixture in `karnc/tests/`** — this lives in the new crate. (If sign-off
  picks fork A, the tests move under `karnc` instead.)

## Diagnostics / Risks

- **Cross-platform detection** — *resolved as a `[DECISION]` above* (portable via
  the `which` crate, not Unix-only). Left here only as the residual: the existing
  `tool_exists` shells POSIX `which`, so the shared probe must replace it rather
  than extend it, or the Unix-only assumption leaks back in by reuse.
- **`npx` false-positives.** `npx --yes` will *fetch* a package on demand, so
  "`wrangler` resolves via npx" is not "`wrangler` is installed." `doctor` must
  report provenance and not paint fetch-on-demand as green, or it will certify
  an environment that then pauses to download on first real use.
- **Floor drift.** The Node ≥ 18 floor lives as prose in two Rust doc comments
  today — `cli.rs:58` and `firstparty.rs` (~335) — **not** in the READMEs.
  `doctor` would be a third reader. Mitigation: lift the floor into one constant
  that *both* doc comments reference, so the increment *reduces* duplication
  rather than adding to it.
- **`--only` vs the `test` verb.** A capability filter named `test` would shadow
  the `karnc test` verb in users' mental model; `--only <capability>` sidesteps
  the collision (see the checks decision).
- **Scope creep into `--fix` / installs.** Tempting, but it changes the trust
  and review bar (see the third decision). Mitigation: reserved-and-noted, not
  built.
- **Driver cost if under-used.** Standing up a whole crate for one command looks
  heavy in isolation. Mitigation accepted knowingly: it is the foundation for
  `new`/`dev`, and `doctor` is the cheapest possible thing to validate it with.

  `karnc` compiler.** `karn` shells `karnc` + the Node toolchain; the compiler

## Definition of done

- **Driver.** The `karn` driver crate exists and is published-ready, resolving
  `karnc` via PATH with a sibling fallback (fork B; under fork A, `karnc doctor`
  ships instead), and **flagging driver↔compiler skew** (warn by default, fail
  under `--strict`/on contract mismatch).
- **Detection is portable** — backed by the `which` crate (handles
  `PATHEXT`/`where`), replacing the Unix-only `tool_exists`, with a non-POSIX
  resolution test.
- **Capability probes.** compile/check/fmt always satisfiable; `test` = Node +
  tsc|tsx per the runner ladder; dev/deploy = Node + wrangler; LSP + Rust
  toolchain optional — each reporting presence, version against the
  single-sourced Node floor, and provenance (PATH / project-local / npx), and
  **never painting npx-fetch-on-demand as installed**.
- **Exit contract (pinned).** Bare `doctor` is informational (exit 0 unless the
  compile floor is broken); `--only <capability>` promotes that capability's
  tools to required; `--strict` promotes all warnings (incl. skew); `--format
  short` and `--format json` are pinned by golden tests.
- **Behaviour.** Remedies are printed; nothing is installed.
- **Decisions.** The two ADRs land (driver + contract).
- **Docs.** The book gains its `karn doctor` page; the install page and
  hello-world README lead with it; the roadmap gains a driver/lifecycle section
  (not §4) and the queue records the arc; the changelog + version banner are
  advanced.
- **Gates.** `cargo build`/`test`/clippy and CI stay green;
  `scripts/bump-version.sh` sets the version.
