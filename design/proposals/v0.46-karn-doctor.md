## Context ‚Äî make the environment check a command, not a surprise

Karn already depends on a Node-side toolchain it discovers *ad hoc, at the
moment of failure*. `karnc test` walks a runner ladder ‚Äî `tsc`, then
`npx --yes -p typescript tsc`, then `tsx`, then `npx --yes tsx` ‚Äî guarded by a
`which`-based `tool_exists`, and only if every rung fails does it print an
actionable "install one of‚Ä¶" message (`karnc/src/main.rs`). The hello-world
README then asks the reader to find `node`, `npx wrangler dev`, and a Cloudflare
login on their own. The dependencies are real and knowable, but a newcomer meets
them one broken command at a time.

`karn doctor` promotes that scattered, reactive detection into a single
**upfront, declarative** check: *given what you want to do with Karn, is your
machine ready, and if not, exactly what do you run to fix it?* It is the
cheapest possible win in the lifecycle arc ‚Äî pure inspection, no machine
mutation, no language surface ‚Äî and it is the natural first command of the
`karn` driver that `new` and `dev` will also live in.

## Builds on (verified)

- **The existing detection primitive.** `tool_exists(name)` (`main.rs`) already
  shells `which` and reports presence; `doctor`'s probe is a generalisation of
  it (presence + version + provenance), and the two should share one
  implementation rather than diverge.
- **The known runner ladder.** `karnc test`'s `tsc`/`npx tsc`/`tsx`/`npx tsx`
  sequence (`main.rs`) is the authoritative list of what "can run Karn's emitted
  TypeScript" means today. `doctor`'s `test`-readiness check mirrors that ladder
  exactly, so the two cannot disagree about what counts as satisfied.
- **The Node floor, already asserted in two places.** `CliPlatform::Node` is
  documented as "Node.js (‚â• 18)" (`cli.rs`) and the README repeats it. `doctor`
  reads that floor from **one** constant so the claim stops being duplicated
  prose.
- **The existing actionable-error voice.** `karnc test`'s failure text already
  prints concrete `npm install -g ‚Ä¶` remedies. `doctor` reuses that tone and,
  where possible, the exact strings, so the remedy a user sees from `doctor`
  matches the one they'd see from a failed command.
- **The project-root discovery.** `karn.toml` discovery and the `[paths]`
  manifest already exist (`karnc/src/project/‚Ä¶`, the `cli_rooting` fixture).
  `doctor` reuses it to become **project-aware** when run inside a project (see
  the scope decision).

## Decisions (forks for sign-off)

### [DECISION] Where `doctor` lives ‚Äî **recommend: introduce the `karn` driver crate, with `doctor` as its first command.**

There is no `karn` binary today; the workspace publishes `karnc`, `karn-fmt`,
`karn-grammar`, `karn-lsp`. Two ways to ship `doctor`:

- **A ‚Äî `karnc doctor` subcommand.** Cheapest: one new arm on the existing clap
  tree, no new crate, no new release artefact. But it cements environment
  orchestration *inside the compiler*, which the project has deliberately kept
  pure (compile / check / fmt / test are all compiler operations; "is `wrangler`
  installed" is not). And it strands the arc: `new` and `dev` are orchestration,
  not compilation, and want a `karn` home ‚Äî shipping `doctor` as `karnc doctor`
  now buys a rename/deprecation later.
- **B ‚Äî a new `karn` driver crate (recommended).** `karn` is to `karnc` what
  `cargo` is to `rustc`: a thin orchestrator that shells the compiler and the
  Node toolchain. `doctor` is the *ideal* first command to stand it up ‚Äî it has
  no language surface, mutates nothing, and exercises exactly the muscles the
  driver needs (locate `karnc`, locate `karn.toml`, shell out, report). The cost
  is real but one-time: a new published binary, its release-plumbing row, and a
  decision on how the driver finds `karnc` (PATH, then a sibling-of-`karn`
  fallback ‚Äî mirrors how `vscode-karn` resolves `karnc-lsp`).

Recommendation **B**. The whole point of this thread is the lifecycle wrapper;
`doctor` is the low-risk increment that pays for the driver's foundation before
`dev` (the expensive, watch-bearing command) needs it. If sign-off prefers to
defer the crate, fall back to **A** with an explicit note that `karn` will
re-home it ‚Äî but the recommendation is to pay the foundation cost here, where
the blast radius is smallest.

### [DECISION] What `doctor` checks, and what "missing" costs ‚Äî **recommend: capability-grouped checks; hard-fail only on what blocks an action you could take today.**

Group probes by the capability they unlock, not as a flat list:

- **Compile / check / fmt** ‚Äî needs only `karnc` itself (you have it if `karn`
  found it). Always satisfiable; `doctor` reports the resolved `karnc` path +
  version.
- **`karn test`** ‚Äî needs Node **and** one of (`tsc` | `tsx`), matching the
  runner ladder. Missing ‚áí this capability is **unavailable**.
- **`dev` / deploy (Cloudflare)** ‚Äî needs Node and `wrangler` (today via `npx`).
  Missing ‚áí this capability is **unavailable**.
- **Editor (optional)** ‚Äî `karnc-lsp` on PATH. Missing ‚áí **note**, never a
  failure.
- **Build-from-source (optional, contributors)** ‚Äî a Rust toolchain ‚â• MSRV.
  Reported only when relevant (e.g. inside the Karn repo), never a failure for
  end users.

Checks report **presence + version + provenance** (global PATH vs
project-local `node_modules/.bin` vs npx-fetch-on-demand), and version is
compared against a known floor where one exists (Node ‚â• 18). Exit-code rule:
**`doctor` exits non-zero only when a *required-for-something-you-asked-about*
tool is missing**; optional gaps warn and exit zero. `--strict` escalates any
warning to a failure (for CI), and a capability filter (e.g. `karn doctor test`)
scopes the check to one capability so a user who never touches Cloudflare isn't
told they're "unhealthy" for lacking `wrangler`.

Rejected: a flat pass/fail over every conceivable tool ‚Äî it punishes a
compile-only user for a missing `wrangler` and trains people to ignore the
output.

### [DECISION] `doctor` reports; it does not install ‚Äî **recommend: no `--fix` in this slice; print the exact remedy instead.**

`doctor` is diagnostic. For each gap it prints the precise remedy (the same
`npm install -g ‚Ä¶` / `npx ‚Ä¶` strings the toolchain already emits), but it does
**not** run them. Auto-install (`--fix`) is reserved-and-noted, not built:
mutating a user's global environment is a different trust level and a different
review bar, and it would couple a should-be-portable check to package-manager
specifics. Rejected for now: shipping `--fix` here ‚Äî it widens the blast radius
of the arc's *first* command for marginal convenience.

Output contract (small contract ADR): human-readable grouped table by default;
`--format short` (one `capability: ok|missing (remedy)` line per row, mirroring
`karnc check --format short`, ADR 0071) and `--format json` for scripting/CI.

## Spec delta

**None.** `karn doctor` adds no grammar, no static-semantics rule, no emission,
and no diagnostic in the `karn.*` compiler family ‚Äî it inspects the host
environment and shells existing tools. This is a deliberate no-op on
`docs/src/spec/`, recorded here per the proposal-README rule that silence on the
spec is treated as an oversight: there is genuinely no normative language
surface to define.

## Tests

- **Driver unit tests** (the shared probe): a tool present on PATH; absent; a
  version below floor (Node < 18) vs at/above; project-local resolution
  preferred over global when a `karn.toml` + `node_modules/.bin` exist;
  npx-fetch-on-demand reported as *provisionable*, not *present* (it must not be
  a false "ok").
- **Capability/exit-code matrix:** compile-only env ‚áí exit 0 with `test`/`dev`
  flagged unavailable; everything present ‚áí exit 0, all green; a required tool
  missing under a capability filter ‚áí non-zero; `--strict` turns an
  optional-only gap from 0 to non-zero.
- **Output golden:** stable `--format short` and `--format json` shapes
  (these are the scriptable contract ‚Äî pin them); the default human table need
  only be smoke-tested.
- **Driver‚Üícompiler seam:** `karn doctor` locates `karnc` via PATH and via the
  sibling fallback; reports its version.
- **No fixture in `karnc/tests/`** ‚Äî this lives in the new crate. (If sign-off
  picks fork A, the tests move under `karnc` instead.)

## Diagnostics / Risks

- **Cross-platform detection.** Today's `tool_exists` shells `which` and a code
  comment concedes "the rest of the toolchain has been Unix-only." A *user-facing
  "is my machine ready"* command raises the Windows question more sharply than an
  internal helper does. Mitigation: implement the shared probe portably (the
  `which` crate handles PATHEXT/`where`), or scope `doctor` to Unix in this slice
  and **say so explicitly** in the book ‚Äî but don't quietly inherit the Unix-only
  assumption under a command whose whole job is to tell the truth about the
  environment.
- **`npx` false-positives.** `npx --yes` will *fetch* a package on demand, so
  "`wrangler` resolves via npx" is not "`wrangler` is installed." `doctor` must
  report provenance and not paint fetch-on-demand as green, or it will certify
  an environment that then pauses to download on first real use.
- **Floor drift.** The Node ‚â• 18 floor lives in prose in two places today;
  `doctor` introduces a third reader. Mitigation: source it from one constant
  (and have `cli.rs`'s doc comment reference that constant), so the increment
  *reduces* duplication rather than adding to it.
- **Scope creep into `--fix` / installs.** Tempting, but it changes the trust
  and review bar (see the third decision). Mitigation: reserved-and-noted, not
  built.
- **Driver cost if under-used.** Standing up a whole crate for one command looks
  heavy in isolation. Mitigation accepted knowingly: it is the foundation for
  `new`/`dev`, and `doctor` is the cheapest possible thing to validate it with.

## Definition of done

The `karn` driver crate exists and is published-ready, resolving `karnc` via
PATH with a sibling fallback (fork B; under fork A, `karnc doctor` ships
instead); `karn doctor` probes the capability groups (compile/check/fmt always
satisfiable; `test` = Node + tsc|tsx per the runner ladder; dev/deploy = Node +
wrangler; LSP + Rust toolchain optional), reporting presence, version against
the single-sourced Node floor, and provenance (PATH / project-local / npx),
without ever painting npx-fetch-on-demand as installed; exit code is non-zero
only on a required-for-an-asked-capability gap, with `--strict` escalation and a
capability filter; `--format short` and `--format json` are pinned by golden
tests; remedies are printed, nothing is installed; the two ADRs land; the book
gains its `karn doctor` page, the install page and hello-world README lead with
it, the roadmap/queue record the arc, and the changelog + version banner are
advanced; `cargo build`/`test`/clippy and CI stay green; `scripts/bump-version.sh`
sets the version. The implementing PR **deletes this file**.
