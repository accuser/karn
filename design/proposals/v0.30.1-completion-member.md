# v0.30.1 ‚Äî Comprehensive completion, slice 2: name-receiver `.`-members

- **Status:** Draft (proposal). **Merging is approval to build.** *(Continues v0.30 completion. This is
  the **lexically-resolvable** half of `.`-member completion ‚Äî the receiver before the dot is a
  **name** (type / capability / sum), readable from the line prefix, so it needs neither the typed
  model nor a scope query. `karn-lsp`-only, recovery-parse based, same robustness as slice 1.)*

## Context ‚Äî why this is the right next slice

ADR 0061 deferred "`.`-member + locals" to slice 2 as one lump. A feasibility scout shows that lump is
really **two increments of very different cost**, split by *what sits before the dot*:

- **Name receiver** (`Color.`, `Email.`, `Clock.`) ‚Äî the token is a **type/capability/sum name**, present
  verbatim in the line prefix. Members are enumerable from the **parsed AST** (no checker, no
  `expr_types`, no scope query) ‚Äî the exact recovery-parse approach slice 1 already uses. **This slice.**
- **Value receiver** (`list.map`, `str.split`, `email.field`) ‚Äî needs the receiver's **type**, which is
  the hard, risky part (see *Out of scope*). **A later slice.**

This slice takes the tractable, high-robustness half now and quarantines the typed-value machinery.

## Builds on (verified ‚Äî scout citations)

- **The lexical + recovery-parse engine extends directly.** `completion.rs::for_each_unit`
  (`karn-lsp/src/completion.rs`) already parses the project `.karn` files + the embedded `karn`/
  `cloudflare` surface with recovery and walks each `SourceUnit`. Member enumeration is the same walk,
  one level deeper into a named declaration.
- **All three member kinds live in the AST, fully enumerable, no index needed:**
  - **Sum variants** ‚Äî `SumBody.variants: Vec<Variant>` (`ast.rs:703`); the checker already resolves
    `TypeName.Variant` as a field-access shape (`checker/expressions.rs:1545-1566`), confirming
    `Sum.Variant` is the surface form.
  - **Refined/opaque statics** ‚Äî `TypeBody::{Refined,Opaque}` (`ast.rs:143`); `T.of(value)` /
    `T.unsafe(value)` are the constructor statics (`checker/expressions.rs:1501`). (The emitter renders
    exactly `of`/`unsafe` for these ‚Äî slice 1's `emit_refined_type` neighbourhood.)
  - **Capability ops** ‚Äî `CapabilityDecl.ops: Vec<CapabilityOp>`, each with `name`/`params`/
    `return_type` (`ast.rs:417-434`); capability-name dispatch is `checker/calls.rs:1026-1031`.
- **The receiver name is lexically available.** As with `consumes U {` detection, the identifier before
  the `.` is in the line prefix; we resolve *what kind of declaration it is* by the existing project
  parse, then list that declaration's members. **No `expr_types`, no offset‚Üítype, no scope-at-offset.**
- **The naming convention is a clean discriminator.** Karn types/capabilities/sums are CapitalCase;
  value bindings are camelCase. So an **uppercase-initial** `Name.` is a name receiver (this slice);
  a lowercase-initial `name.` is a value receiver (next slice) ‚Üí this slice yields `[]` for it,
  staying conservative.

## Decisions (open ‚Äî settle at review)

**[DECISION] The contexts + their members.** *Rec:* detect `Ident.` (optionally with a partial member
being typed, `Ident.par`) where `Ident` is uppercase-initial and resolves, via the project/surface
parse, to:
- a **sum type** ‚Üí its variant names (kind `ENUM_MEMBER`);
- a **refined or opaque type** ‚Üí `of`, `unsafe` (kind `METHOD`), with one-line details;
- a **capability** ‚Üí its op names + rendered signatures as `detail` (kind `METHOD`).

If `Ident` resolves to none of these (or to a plain alias/record), yield `[]` (conservative).

**[DECISION] Built-in type statics (`Int.parse`/`Float.parse`, `Json.encode`/`decode`).** *Rec:* include
the well-known first-party statics from a **small static table** (sourced from the same `builtin_names`/
registry spirit as slice 1), since `Int.`/`Float.`/`Json.` are common and not user-declared. *(Honest
note: these aren't in a single registry today; if the table feels like drift-risk, defer them and keep
slice 2 purely structural ‚Äî settle at review.)*

**[DECISION] Disambiguating `.`.** The dot is overloaded ‚Äî qualified unit names (`karn.cloudflare`),
decimal literals (`1.5`), field/method access. *Rec:* fire only when the prefix immediately before the
cursor is `UpperIdent.` with an optional trailing partial identifier, **not** preceded by another
`.`-segment (single-segment receiver only) and not a digit run (excludes `1.`). Cover each with a
fixture; prefer offering nothing over wrong items.

**[DECISION] New decision record?** *Rec:* a **light ADR 0062** recording the re-slice (the `.`-member
split into name-receiver vs typed value-receiver, and that value-`.method` + locals are gated on the
receiver-typing + scope-at-offset machinery 0061 named). Refines 0061's roadmap; numbering after 0061.

## Tests

- **`karn-lsp` pure-function** (`complete()` over crafted prefixes): `Color.` yields the sum's variants;
  `Email.` (a refined type in the doc) yields `of`/`unsafe`; `Clock.` yields the `karn`-surface
  capability's ops; an uppercase name that's a plain alias yields `[]`; a **lowercase** `x.` yields `[]`
  (value receiver, deferred); `1.` (decimal) yields `[]`. The existing slice-1 contexts stay green.

## Diagnostics / Risks

- **`.`-overload misfire** ‚Äî the chief risk; keep detection conservative (single uppercase segment, no
  digit run) and fixture every case.
- **Receiver-kind ambiguity across files** ‚Äî a name may be declared in another unit; reuse
  `for_each_unit` so cross-file types/caps resolve (as slice 1 already does for `consumes`).
- **No typed receiver** ‚Äî by design this slice never types a value; a CapitalCase *value* binding (rare)
  would be mis-offered statics, accepted as minor noise (mirrors slice 1's record-construction note).

## Out of scope (‚Üí slice 3 / later ‚Äî scout-confirmed)

- **Value `.method`/`.field`** ‚Äî needs the receiver's type. `expr_types: HashMap<Span, Ty>` exists in
  the checker (`checker.rs:165`) but is **discarded on the LSP `Analyse` path** (`project.rs:2102`), is
  keyed by span not offset, and ‚Äî fundamentally ‚Äî completion fires mid-edit when the buffer **doesn't
  parse**, so the checker never reaches the cursor expression. Requires: caching `expr_types` to the
  analysis, an offset‚Üítype query, and mid-edit-resilient typing. The scout's **#1 risk**; warrants its
  own design spike before commitment.
- **Locals/params in scope** ‚Äî no scope-at-offset query exists (`index.rs:14` excludes locals; the
  `in_scope` at `index_queries.rs` is span-range *filtering*, not lexical scope). Needs new checker +
  index machinery.
- Signature help; auto-import/add-`consumes` resolve; postfix expansion.

## Definition of done

`completion.rs` recognises the `UpperIdent.`(`partial`) member context and sources members from the
project/surface parse ‚Äî sum variants, refined/opaque `of`/`unsafe`, capability ops (and, if kept,
built-in type statics); `main.rs` maps the new item kinds (`ENUM_MEMBER`/`METHOD`); a pure-function
test suite covers each receiver kind plus the conservative `[]` cases (lowercase receiver, decimal,
plain alias); `design/karn-lsp-spec.md` ¬ß3.15 gains the row and the slice-3 line; decision record
(0062) added; the `extension`/`test` CI jobs stay green; changelog updated. The implementing PR
**deletes this file**.
